# 01 가이드 작성 표준 (AI-Ready Data 매뉴얼)

> 이 문서는 20개 주제 가이드를 **일관된 품질·형식**으로 쓰기 위한 공통 지시사항이다.
> 가이드를 작성·수정하기 전에 반드시 읽는다. (작성 원칙은 `ai-ready-data-guide-en` 스킬의 규격을
> 본 프로젝트의 12-섹션 매뉴얼 형식에 맞게 옮긴 것이다.)

---

## 0. 가이드의 본질 (가장 흔한 실패부터)

### 0-1. 리서치 덤프가 아니라 "한 편의 스토리"
- 가이드의 목적은 처음 보는 사람에게 **(a) 이 주제가 무엇이고 (b) 어떻게 구성·운영하는가**를 가르치는 것.
  찾은 내용을 다 담는 보고서가 아니다.
- 같은 주제도 출처마다 다르게 자른다(어디는 구성요소 5개, 어디는 7개). 다 담으면 **반복·모순·군더더기**가 생긴다.
- **정본(canonical) 모델 하나를 정하고 끝까지 그대로 쓴다.** 쓰기 전에 정의·구성요소·단계·용어를 하나로 확정.
  가장 표준적이고 교과서적인 프레이밍을 택하거나, 깔끔한 일반 모델을 직접 종합한다.
- 출처별 상이한 프레이밍·세부 수치·스펙은 **별첨/백업으로 내리거나** 버린다(폴더에 리서치로 남으니 손실 아님).
- **선택 기준:** "이 내용이 지금 이 위치에서 첫 독자가 What/How를 이해하는 데 도움이 되는가?" 아니면 백업·삭제.

### 0-2. 이 주제가 주연 — 주제 고유의 구체적 방법이 본론
- 20개 주제는 각자 별도 가이드가 있다. **인접 주제(카탈로그·품질·리니지·거버넌스 등)는 조연** — 보통 짧은
  포인터면 충분("이 자산의 품질 기준은 C-2 참고"). 스토리에 꼭 필요하면 간단히만.
- **어느 주제에나 그대로 들어갈 일반론(보일러플레이트)은 잘라낸다.** "데이터는 생성·사용·폐기 주기를 돈다",
  "품질 관리가 중요하다" 류는 어느 가이드에나 통하므로, 정작 독자가 온 이유를 밀어낸다. 많아야 한 줄, 보통 생략.
- 독자가 가져갈 것은 **그 주제의 구체적 메커니즘** — 구조·스키마·필드·포맷·버전 단위·실제 방법.
  예) "Prompt/Harness 자산화"라면 프롬프트를 **어떻게** 자산으로 만드는가(구조·필드·템플릿·버전·저장 위치)가 본론.
- **보일러플레이트 테스트:** 어떤 문장/표가 다른 주제 가이드에 그대로 옮겨도 말이 되면, 본문에 둘 자격이 없다.

### 0-3. "AI 구축"이 아니라 "AI를 위한 데이터 준비"
- 답할 질문은 **"AI를 하려면 데이터를 어떻게 준비해야 하는가"** — "AI를 어떻게 만드는가"가 아니다.
- ✅ 포함: 전처리·정제·구조화·표준화, 메타데이터·태그·용어 정렬·품질·리니지 — *AI가 잘 쓰도록 데이터를 준비하는 법*.
- ❌ 제외: 모델 학습/추론 코드, RAG 앱 구현, Agent 오케스트레이션 코드, 모델 선정 — *AI 만드는 법*.
  (솔루션·도구는 *데이터를 준비하는 도구*로서만 등장.)
- **AI성 주제(Agent·Prompt·평가)도 동일:** 프롬프트·하니스·평가셋은 구조화·관리할 **데이터 자산**이지,
  "AI Agent 만드는 법"을 쓰는 게 아니다.

### 0-4. Key Question = 합격 기준
- `최종 주제.md`는 각 주제를 번호가 매겨진 **Key Question**으로 정의한다(+ Lv.2 세부내용·답변 예시 = 기대 깊이 힌트).
- **메인 흐름이 모든 Key Question에 답해야 한다.** 자연스러운 위치에 녹이되, 별도 Q&A 섹션으로 붙이지 않는다.
- 답하는 슬라이드가 없는 Key Question이 있으면 가이드는 미완성. (보통 3.구성체계·5.대상선정·8/9 구축·운영에 매핑됨.)
- 작성 후 **Key Question → 섹션 매핑표**를 스스로 만들어 누락이 없는지 대조한다.

### 0-5. 일관성 (반복·모순 금지)
1. **개념은 한 곳에서 한 번.** 처음 필요한 위치에서 풀이+예시와 함께 도입, 이후엔 *참조*만(다르게 재설명 금지).
2. **숫자·이름은 어디서나 일치.** 구성요소가 A/B/C/D 4개면 끝까지 그 4개·그 이름. 앞 슬라이드 주장과 뒤가 어긋나면 안 됨.
3. **모든 섹션은 제 몫을 한다.** 앞 내용을 재진술만 하는 슬라이드는 합치거나 삭제. 적고 강한 게 많고 약한 것보다 낫다.
4. **최종 통독(필수).** 작성 후 메인 전체를 슬라이드별이 아니라 한 편의 글로 처음부터 끝까지 읽으며 반복·모순·
   용어 표류·개수 변동·삭제해도 되는 슬라이드를 점검·수정한다.

### 0-6. 형식은 내용을 따른다
- **목록은 실제 개수만큼.** 이유가 2개면 2개. 숫자 채우려고 늘리지 않는다.
- **장치를 다양하게.** 그 포인트를 가장 잘 이해시키는 형식(짧은 서사·전후 대비·다이어그램·완성 예시·인과 사슬)을 쓴다.
  인접 슬라이드가 전부 "제목 + N개 불릿"이 되지 않게.
- 이 표준의 숫자 범위는 **상한이지 목표가 아니다.** "KPI 최대 5개"는 부풀림 방지용 — 3개로 충분하면 3개.

## 1. 12-섹션 권장 구조 (기본 골격 — 주제별 가감)

아래 12개 섹션(H2)은 **권장 기본 골격**이다. 대부분의 주제가 이 흐름으로 쓰면 잘 맞으므로 *기본값*으로 삼되,
**모든 주제에 12개를 똑같이 강제하지 않는다.** H3는 주제별로 다르다.
(섹션별 후보 H3는 `00 전체 목차 (20개 주제).md`에 주제별로 정리돼 있다.)

### 1-0. 주제별로 섹션을 가감하는 규칙
- **더한다:** 그 주제에만 의미 있는 목차가 있으면 신설한다 (예: 온톨로지의 "관계·추론 규칙", 데이터 계약의 "스키마 버저닝·호환성 정책", 프롬프트 자산의 "버전·평가 연결"). 핵심 What/How를 가장 잘 전달하는 자리에 둔다.
- **뺀다·합친다:** 그 주제에 알맹이가 없는 섹션은 한 줄 포인터로 줄이거나 인접 섹션에 합치거나 생략한다 (예: 솔루션 시장이 미성숙한 주제의 6.솔루션 선정).
- **순서를 바꾼다:** 스토리가 더 매끄러우면 순서를 조정한다. 다만 1.개요 → … → 12.로드맵의 큰 흐름(정의→필요성→구성→대상→구축→운영→연계→성과)은 가급적 유지하면 독자가 20개 가이드를 같은 리듬으로 읽을 수 있다.
- **불변 조건(어떤 구성에서나 지킨다):** ① `최종 주제.md` Key Question 전부 커버, ② 정본 모델 일관성(0-5), ③ 이해 장치(2장), ④ 깊이 2층·링크·출처 규칙(3·4장), ⑤ 컬러 스키마(5장). 섹션을 어떻게 짜든 이 다섯은 합격 기준이다.
- 섹션을 크게 바꾸면 상단 **목차(앵커)** 와 **KQ→섹션 매핑표**를 실제 구성에 맞게 갱신한다.

### 권장 기본 12-섹션

| # | 섹션 | 본문에 담을 것 |
|---|------|---------------|
| 1 | 개요 | 정의·목적·적용범위·대상조직·AI-ready 체계 내 역할 (1.5는 전체 조감도에서 이 주제 강조) |
| 2 | 필요성 및 기대 효과 | **2.1 AI 과제 Pain Point**(현업 예시) → 필요성 → 기대효과 → 주요기능. 시장수치·규제 등은 발표자 As-Is로 `[As-Is에서 채움]` 표기 |
| 3 | 구성 체계 | **주제의 정본 모델** — 핵심 구성요소·항목·기준. 가이드 전체에서 이 모델을 일관되게 재사용 |
| 4 | 추진 역할 및 책임 | 담당자 정의·RACI·계열사 상황별 조직 조정 |
| 5 | 현황 조사 및 대상 선정 | Key Question 기반 등록/적용 대상·우선순위·제외 기준 |
| 6 | 솔루션 선정 검토 | 솔루션 유형·후보·기능비교·평가·PoC (상세 비교표는 백업) |
| 7 | 계열사 적용 예시 | 두산 계열사 가상 시나리오 — 실제값이 채워진 완성 예시 |
| 8 | 구축 | To-Be 아키텍처·연동·파이프라인 개발·초기 적재/검증 |
| 9 | 운영 | 변경관리·서비스 연계·권한/보안·운영 역할 |
| 10 | AI-ready 체계 내 연계 범위 | 인접 주제와의 **역할 분담**(겹침이 아니라 경계). 전체 조감도 다이어그램 재사용 |
| 11 | KPI 및 성과 관리 | 최대 5개. 측정 이유 + 무엇을 + 방향(↑·↓) + 기대효과. Why의 목표와 연결 |
| 12 | 고도화 Roadmap | 기본 구축 후 확장 — 수기 → AI 보조 → 자율 단계 서사 |

### 깊이 배분 (가중치, 채워야 할 분량이 아님)
- 가장 무게: **3.구성체계 · 5.대상선정 · 8.구축 · 9.운영** (= What/How의 핵심).
- 1·2·4·10·11·12는 메인 위주로 간결하게. 6.솔루션의 상세 비교표·7.사례의 긴 표는 백업/별첨.

## 2. 이해 장치 (메인 본문 의무)

각 섹션(또는 슬라이드 단위) 본문은 다음을 갖춘다:
1. **👉 한 줄 요약:** 맨 앞에 초심자도 이해하는 한 문장.
2. **용어 즉시 풀이:** 어려운 용어·약어가 처음 나오면 괄호로 그 자리에서 풀이.
   예) `DAG(Directed Acyclic Graph = 되돌아오지 않는 한 방향 흐름)`. 메인에 설명 없는 전문용어 금지.
3. **🏭 제조 현업 예시:** 두산/제조 상황에 근거한 예시 최소 1개
   (예: 진동 센서 → 예지보전 모델, MES 생산 데이터, 품질 검사 판정).
4. **실제값이 채워진 완성 예시(핵심 개념마다 1개 이상):** 빈 양식이 아니라 그럴듯한 두산/제조 값으로 채운다.
   예) 테이블 `PROD_COST_MONTHLY` / 설명 / 출처(SAP) / 주요 컬럼 / 오너(원가팀, 홍길동 책임) / 갱신(월간) /
   보안(대외비) / 태그(#원가 #제조) / 품질점수(94/100) / 타임스탬프까지 채움.

## 3. 깊이 2층 — 본문 vs 백업/별첨

이 가이드는 장표(슬라이드)의 원천이다. 내용은 두 층으로 나눈다.
- **본문(메인) = 주연:** 그 포인트를 *이해*하는 데 필요한 것만. 앞 내용만으로 이해되도록 쉽게·정확하게.
  본문만 순서대로 읽어도 What/Why/How가 남으면 합격.
- **백업/별첨 = 조연:** 표·스펙·스키마·솔루션 비교·표준 약어, 그리고 출처별 세부(벤더 프레이밍·대안 분류·정확 수치)는
  본문 흐름을 끊지 않게 내린다. 깊이는 여기서 충분히 가져간다(겁내서 줄이지 말 것).

표기:
- 본문 핵심 단위는 `### [Slide N] 제목`, 그 백업은 바로 아래 `#### [Backup N-1] 제목 (상세)`.
- 본문 슬라이드 끝에 `> ▸ 백업: [Backup N-1] ◯◯◯` 로 어떤 백업으로 들어가는지 표시.
- 가장 깊은 백업(체크리스트·프레임워크·전체 필드 스펙표)은 문서 끝 `## 별첨(Appendix)` 에 `[Appendix A] 제목`.

> 본문에 표/스키마/솔루션 비교/약어 덤프를 넣지 않는다 — 백업으로.

## 4. 언어·표기·링크·출처 규칙

### 4-1. 언어·표기
- 한국어로 쓴다. **기술용어·솔루션명·표준명은 `한글(English)` 병기** (예: 메타데이터(Metadata), 데이터 계보(Data Lineage)).
- 같은 용어는 매번 같은 한국어로(용어 표류 금지 — 0-5와 연결).

### 4-2. 목차(ToC) — 문서 상단 앵커 링크 (필수)
- 모든 가이드는 한 줄 정의 바로 아래에 **목차**를 둔다. 12개 섹션(+별첨·참고자료)을 **앵커 링크**로 연결해 클릭 시 해당 섹션으로 이동하게 한다.
- 앵커 형식(GitHub 규칙): 헤더 텍스트를 소문자화(영문)·구두점 제거·공백을 `-`로. 예) `## 1. 개요` → `[1. 개요](#1-개요)`, `## 10. AI-ready ...` → `(#10-ai-ready-데이터-체계-내-연계-범위)`.
- H3 단위 세부 목차가 필요하면 What/How 등 핵심 섹션에 한해 추가한다(0-6 섹션 agenda와 병행 가능).

### 4-3. 주제 간 상호 연결 (필수)
- 인접 주제를 언급할 때는 **해당 주제의 가이드 파일로 링크**한다. 파일명 규칙(`A-2 메타데이터.md`)을 그대로 쓰되, 공백은 `%20`으로 인코딩한다 → `[A-2 메타데이터](../가이드%20작성/A-2%20메타데이터/A-2%20메타데이터.md)`.
- 특히 10장(연계 범위)의 역할 분담표·1장 경계표의 주제명은 모두 링크로 연결한다.
- 아직 작성되지 않은 주제도 미리 링크해 둔다(파일 생성 시 자동 연결됨). 깨진 링크가 아니라 "작성 예정" 표시로 본다.

### 4-4. 솔루션·도구·표준 소개 시 출처 링크 (필수)
- **솔루션·제품·도구·표준·프레임워크를 소개·비교할 때는 반드시 출처 URL을 단다.** 독자가 클릭해 직접 확인할 수 있어야 한다.
- 방식: ① 본문/표에서 제품명을 **공식 페이지로 인라인 링크**(`[Collibra](https://www.collibra.com)`), ② 문서 끝 `## 참고자료(References)`에 `[제목](URL) — 접속일/성격(공식 제품 페이지 등)`으로 모아 재정리.
- **가격·버전·지원 범위 같은 변동 수치는 단정하지 않는다.** "공식 문서/벤더 데모/PoC로 확인"이라고 명시하고 출처를 링크한다(0-1 정직성 원칙과 연결).
- 출처가 필요한 일반 사실(표준 정의·통계 등)도 같은 방식으로 References에 링크로 모은다.

### 4-5. 솔루션 2층 구조 — Tech Stack 정본 연결·반영 (필수)
솔루션은 **2층으로 관리**한다. 한 솔루션이 여러 주제에 걸치므로(예: 카탈로그 솔루션 = 메타데이터·계통 솔루션), 주제마다 따로 적으면 중복·불일치가 난다.
- **1층 = 각 주제 가이드의 '솔루션' 섹션** — *그 주제 관점에서 솔루션의 **기능**을 비교*한다(커넥터·자동 수집·온프렘 여부 등). 깊은 PoC·기능 상세는 가이드에 둔다.
- **2층 = Tech Stack 정본** [`전체 목차/01 Tech Stack 비교 (솔루션×주제).md`](../전체%20목차/01%20Tech%20Stack%20비교%20(솔루션×주제).md) — *솔루션 **자체**를 주제 가로질러 묶어 평가·선정*한다(Part B 플랫폼×주제 매트릭스·Part C 묶음 비교).
- **연결:** 가이드의 솔루션 섹션 머리에 정본 링크를 단다 → `> 🔗 **2층 연결:** 솔루션을 묶어서 평가·선정하려면 → [Tech Stack 비교 정본](../../전체%20목차/01%20Tech%20Stack%20비교%20(솔루션×주제).md)`.
- **★ 반영(reflect-back):** 가이드를 작성하며 **새로 조사된 솔루션은 반드시 정본 Part A의 해당 주제 표에 추가**하고, 새 플랫폼이면 Part B 매트릭스 행도 갱신한다. (가이드는 기능을 비교하고, 정본은 그 결과를 누적해 묶음 평가를 가능케 한다.)

## 7. 고객 피드백 워크플로우

작업은 **작성 → 고객 확인 → 피드백 반영**을 주제별로 반복한다.

1. **작성:** 템플릿(`03`)에서 시작, 표준(`01`·`02`) 준수, 자가 체크리스트(6장) 통과 후 초안 완성.
   - frontmatter `status: draft`.
2. **고객 확인:** 고객에게 제시. 피드백을 받는다(구조·내용·표현·범위 등).
3. **반영:** 피드백을 가이드에 반영하고, 가이드 하단 **`## 변경 이력 / 피드백 반영`** 표에 누적 기록.
   - 표 형식: `| 일자 | 버전 | 피드백(누가/무엇) | 반영 내용 | 반영 위치 |`
   - frontmatter `status`를 `review`(확인중) → `confirmed`(확정)으로 갱신, `version` 증가.
4. **커밋:** 피드백 반영은 **별도 커밋**으로 남겨 이력 추적
   (예: `feat(A-1): 고객 1차 피드백 반영 — 솔루션 비교 기준 보강`).

> 표준 자체에 대한 피드백(모든 가이드 공통 규칙 변경)은 이 `01`/`02` 파일을 고치고 별도 커밋한다.
