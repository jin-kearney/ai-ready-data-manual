# 01 가이드 작성 표준 (AI-Ready Data 매뉴얼)

> 이 문서는 20개 주제 가이드를 **일관된 품질·형식**으로 쓰기 위한 공통 지시사항이다.
> 가이드를 작성·수정하기 전에 반드시 읽는다.
> **형식의 정답지는 최신 finalversion인 B-1 데이터 전처리·B-3 온톨로지 두 파일이다** — 아래 규칙과
> 실제 가이드가 어긋나면 B-1·B-3을 따른다. 섹션 흐름은 Why → What → When → How → Tech Stack → Where다(상세 §1).

---

## 0. 가이드의 본질 (가장 흔한 실패부터)

### 0-1. 리서치 덤프가 아니라 "한 편의 스토리"
- 가이드의 목적은 처음 보는 사람에게 **(a) 이 주제가 무엇이고 (b) 어떻게 구성·운영하는가**를 가르치는 것.
  찾은 내용을 다 담는 보고서가 아니다.
- 같은 주제도 출처마다 다르게 자른다(어디는 구성요소 5개, 어디는 7개). 다 담으면 **반복·모순·군더더기**가 생긴다.
- **정본(canonical) 모델 하나를 정하고 끝까지 그대로 쓴다.** 쓰기 전에 정의·구성요소·단계·용어를 하나로 확정.
  가장 표준적이고 교과서적인 프레이밍을 택하거나, 깔끔한 일반 모델을 직접 종합한다.
- 출처별 상이한 프레이밍·세부 수치·스펙은 **별첨/백업으로 내리거나** 버린다(폴더에 리서치로 남으니 손실 아님).
- **선택 기준:** "이 내용이 지금 이 위치에서 첫 독자가 What/How를 이해하는 데 도움이 되는가?" 아니면 백업·삭제.

### 0-2. 이 주제가 주연 — 주제 고유의 구체적 방법이 본론
- 20개 주제는 각자 별도 가이드가 있다. **인접 주제(카탈로그·품질·리니지·거버넌스 등)는 조연** — 보통 짧은
  포인터면 충분("이 자산의 품질 기준은 C-2 참고"). 스토리에 꼭 필요하면 간단히만.
- **어느 주제에나 그대로 들어갈 일반론(보일러플레이트)은 잘라낸다.** "데이터는 생성·사용·폐기 주기를 돈다",
  "품질 관리가 중요하다" 류는 어느 가이드에나 통하므로, 정작 독자가 온 이유를 밀어낸다. 많아야 한 줄, 보통 생략.
- 독자가 가져갈 것은 **그 주제의 구체적 메커니즘** — 구조·스키마·필드·포맷·버전 단위·실제 방법.
  예) "Prompt/Harness 자산화"라면 프롬프트를 **어떻게** 자산으로 만드는가(구조·필드·템플릿·버전·저장 위치)가 본론.
- **보일러플레이트 테스트:** 어떤 문장/표가 다른 주제 가이드에 그대로 옮겨도 말이 되면, 본문에 둘 자격이 없다.

### 0-3. "AI 구축"이 아니라 "AI를 위한 데이터 준비"
- 답할 질문은 **"AI를 하려면 데이터를 어떻게 준비해야 하는가"** — "AI를 어떻게 만드는가"가 아니다.
- ✅ 포함: 전처리·정제·구조화·표준화, 메타데이터·태그·용어 정렬·품질·리니지 — *AI가 잘 쓰도록 데이터를 준비하는 법*.
- ❌ 제외: 모델 학습/추론 코드, RAG 앱 구현, Agent 오케스트레이션 코드, 모델 선정 — *AI 만드는 법*.
  (솔루션·도구는 *데이터를 준비하는 도구*로서만 등장.)
- **AI성 주제(Agent·Prompt·평가)도 동일:** 프롬프트·하니스·평가셋은 구조화·관리할 **데이터 자산**이지,
  "AI Agent 만드는 법"을 쓰는 게 아니다.

### 0-4. Key Question = 합격 기준
- `최종 주제.md`는 각 주제를 번호가 매겨진 **Key Question**으로 정의한다(+ Lv.2 세부내용·답변 예시 = 기대 깊이 힌트).
- **메인 흐름이 모든 Key Question에 답해야 한다.** 본문 답변은 자연스러운 위치에 녹이고 별도 Q&A 섹션으로 빼지 않는다. 독자용 **KQ 가시화 박스**(질문·한 줄 답·본문 링크 표)는 **선택**이다 — 둘 경우 B-1처럼 `## 이 가이드가 답하는 N가지 질문` H2 평문 표로 **도입 문단 다음·문서 맨 앞**에 둔다(B-3는 두지 않는다). 상세 CLAUDE.md 목차 구조 표준.
- 답하는 슬라이드가 없는 Key Question이 있으면 가이드는 미완성. (보통 What·When·How에 매핑됨.)
- 작성 후 **Key Question → 섹션 매핑표**를 스스로 만들어 누락이 없는지 대조한다.

### 0-5. 일관성 (반복·모순 금지)
1. **개념은 한 곳에서 한 번.** 처음 필요한 위치에서 풀이+예시와 함께 도입, 이후엔 *참조*만(다르게 재설명 금지).
2. **숫자·이름은 어디서나 일치.** 구성요소가 A/B/C/D 4개면 끝까지 그 4개·그 이름. 앞 슬라이드 주장과 뒤가 어긋나면 안 됨.
3. **모든 섹션은 제 몫을 한다.** 앞 내용을 재진술만 하는 슬라이드는 합치거나 삭제. 적고 강한 게 많고 약한 것보다 낫다.
4. **최종 통독(필수).** 작성 후 메인 전체를 슬라이드별이 아니라 한 편의 글로 처음부터 끝까지 읽으며 반복·모순·
   용어 표류·개수 변동·삭제해도 되는 슬라이드를 점검·수정한다.

### 0-6. 형식은 내용을 따른다
- **목록은 실제 개수만큼.** 이유가 2개면 2개. 숫자 채우려고 늘리지 않는다.
- **장치를 다양하게.** 그 포인트를 가장 잘 이해시키는 형식(짧은 서사·전후 대비·다이어그램·완성 예시·인과 사슬)을 쓴다.
  인접 슬라이드가 전부 "제목 + N개 불릿"이 되지 않게.
- 이 표준의 숫자 범위는 **상한이지 목표가 아니다.** "KPI 최대 5개"는 부풀림 방지용 — 3개로 충분하면 3개.

## 1. 섹션 구조 (권장 흐름 — 주제별 가감, 강제 아님)

권장 기본 흐름은 **Why → What → When → How → Tech Stack → Where**다. 최신 finalversion인 **B-1 데이터 전처리·B-3 온톨로지**(각 6섹션)가 형식·느낌의 본보기다. **단 이 흐름은 강제 틀이 아니다** — 각 주제에 실제로 들어갈 섹션·내용은 주제마다 다르며, **`00 전체 목차 (20개 주제).md`의 해당 주제 블록을 단일 출처로 따른다.** 주제에 필요한 섹션은 더하고, 의미 없는 섹션은 빼거나 합친다. **섹션 수도 주제 무게로 가감**한다 — 가벼운 주제 6개 내외, 무거운 주제(품질·계통) 8~9개. H3도 주제별로 다르다.

### 1-0. 기본 순서와 작성 규칙

| # | 섹션(H2 제목) | 본문에 담을 것 |
|---|------|---------------|
| 1 | **Why — 왜 (필요한가)** | AI 과제 수행 시 Pain Point(현업 예시) → 필요성(개념·구조로) → 기대효과. 시장수치·규제는 발표 본편 As-Is가 채움. 선택형 주제는 "Why — 왜, 언제 필요한가"로 When 흡수 |
| 2 | **What — 무엇인가 (구성요소)** | 첫 소절 `정의 — … + 체계 내 위치`가 **개요를 흡수**(별도 개요 섹션 없음). 이어서 **정본 모델** — 핵심 구성요소·항목·기준. ㉠ 항목 사전(대표만) |
| 3 | **When — 어디부터 / 언제 하나** | 다 필요한 주제는 "어디부터(우선순위)", 골라 하는 주제는 "언제(적용 판단)". 선택형은 Why로 흡수해 생략 |
| 4 | **How — 어떻게 구축·운영하나** | 방식·도구·**구축 절차**·운영·역할을 묶는다. 예시(효과+흐름)는 worked example로 여기·Why에 흡수. ㉡ Before→After·㉤ 플랫폼 매핑 |
| 5 | **Tech Stack — 솔루션 검토** | 이 주제 관점 기능 비교(1층). 통합형 주제(메타데이터·전처리·해설/주석)는 How·아키텍처·별첨에 흡수 가능 |
| 6 | **Where — 다른 주제와의 관계** | 인접 주제와의 **역할 분담**(경계). 전체 조감도 다이어그램. **마지막 섹션** |

- **영문 축 라벨을 H2 제목 선두에 붙인다**(`## 1. Why — 왜 필요한가`). 한글 제목만 써도 되지만 B-1·B-3과 같은 느낌을 위해 권장.
- **주제 고유 섹션을 더한다:** B-3처럼 `AI — 온톨로지 기반 AI 활용` 등을 How와 Where 사이에 신설할 수 있다. 온톨로지의 "추론 규칙", 데이터 계약의 "버저닝·호환성" 같은 주제 고유 목차는 핵심 What/How를 가장 잘 전달하는 자리에 둔다.
- **성과 지표(KPI)는 신뢰성 주제 C-1·C-2·C-3에만** Where 뒤에 마지막 섹션(`## N. KPI — 성과 지표`, 최대 5개·측정 목적·방향)으로 둔다. 나머지 17개 주제는 Where가 마지막이다.
- **고도화 로드맵 섹션은 (2026-06 결정) 전 주제에서 두지 않는다**(되살릴 때 기준은 §5·02 §5-4).
- **뺀다·합친다:** 알맹이 없는 섹션은 한 줄 포인터로 줄이거나 합친다. 단 **Tech Stack과 Where는 전 주제 독립 섹션으로 유지**(통합형 주제의 Tech Stack 흡수만 예외).
- **불변 조건:** ① Key Question 전부 커버, ② 정본 모델 일관성(0-5), ③ 이해 장치(2장), ④ 깊이 2층·링크·각주식 출처(3·4장), ⑤ 컬러 스키마(5장). 섹션을 어떻게 짜든 이 다섯이 합격 기준이다.
- 섹션을 바꾸면 상단 **목차(앵커)** 와 자가 점검용 KQ→섹션 매핑을 실제 구성에 맞게 갱신한다.

### 깊이 배분 (가중치, 채워야 할 분량이 아님)
- 가장 무게: **What · How** (= 정본 모델 + 구축·운영). When·Tech Stack도 주제에 따라 두껍게.
- Why·Where는 메인 위주로 간결하게. Tech Stack의 상세 비교표는 백업/별첨.

## 2. 이해 장치 (메인 본문 의무)

각 섹션(또는 슬라이드 단위) 본문은 다음을 갖춘다:
1. **👉 한 줄 요약:** 맨 앞에 초심자도 이해하는 한 문장.
2. **용어 즉시 풀이:** 어려운 용어·약어가 처음 나오면 괄호로 그 자리에서 풀이.
   예) `DAG(Directed Acyclic Graph = 되돌아오지 않는 한 방향 흐름)`. 메인에 설명 없는 전문용어 금지.
3. **🏭 제조 현업 예시:** 두산/제조 상황에 근거한 예시 최소 1개
   (예: 진동 센서 → 예지보전 모델, MES 생산 데이터, 품질 검사 판정).
4. **실제값이 채워진 완성 예시(핵심 개념마다 1개 이상):** 빈 양식이 아니라 그럴듯한 두산/제조 값으로 채운다.
   예) 테이블 `PROD_COST_MONTHLY` / 설명 / 출처(SAP) / 주요 컬럼 / 오너(원가팀, 홍길동 책임) / 갱신(월간) /
   보안(대외비) / 태그(#원가 #제조) / 품질점수(94/100) / 타임스탬프까지 채움.

### 2-A. 현업 실행 키트 (★ 와닿게 만드는 5개 장치 — 의무)

가이드는 "가르치는 문서"지만, 현업이 **읽고 바로 따라 만들 수 있어야** 와닿는다. 그래서 각 주제 가이드는 아래 **「현업 실행 키트」**를 싣는다. 출발점은 단 한 질문이다:

> **"이 주제에서 현업이 실제로 *만들어 내는 산출물*은 무엇이고, 그걸 어떻게 채우는가?"**
> (카탈로그=등록 항목, 메타데이터=Meta 값, Glossary=용어 정의, 품질=품질 규칙, API/Tool=Tool 명세, 보안=민감도 분류 …)

키트는 5개 요소로 구성하고, **새 섹션을 만들지 말고 기존 섹션(What·How·별첨)에 끼운다**:

| 키트 요소 | 무엇을 | 넣는 위치(섹션) |
|---|---|---|
| **㉠ 항목 사전** | 산출물을 구성하는 항목을 `항목 \| 쉬운 의미 \| 예시값 \| 필수/선택 \| 작성 주체` 표로 | What 끝 (대표만) + 별첨(전체) |
| **㉡ Before→After 작성 규칙** | "나쁜 예 → 좋은 예" 교정 쌍 2~3개 + 금지 표현 | How (작성·운영) |
| **㉢ 표준값 목록** | 자유입력 대신 *고르는* 허용값(controlled vocabulary) | What 또는 How (*해당 주제만*) |
| **㉣ 빈 템플릿 + 1건 완성 예시** | 그대로 복사해 채우는 빈 표 + 두산/제조 값으로 꽉 채운 1건 | How(worked example) 또는 별첨 |
| **㉤ 플랫폼·도구 구체 매핑** | "실제로 *어디서* 하나"(예: Databricks `information_schema`, Collibra 화면) + 타 플랫폼 매핑 한 줄 | How |

규칙:
- **㉠·㉡·㉣는 사실상 모든 주제에 의무**, ㉢·㉤는 해당될 때만(표준값이 없는 주제, 도구가 특정되지 않는 주제는 생략).
- **㉠ 항목 사전엔 반드시 `필수/선택`과 `작성 주체(누가)` 열을 둔다** — 현업이 "최소 뭘, 누가" 채우는지 알게.
- **분량 규칙(3장과 동일):** 본문엔 **대표 항목 6~10개만**, 30+ 필드 전체 사전은 `## 별첨`에 두거나 외부 마스터시트(엑셀)로 포인터. 본문에 전체 사전 덤프 금지.
- **골라서 하는 주제**(온톨로지·합성데이터·평가데이터 등)는 키트 앞에 `> 이 키트는 ◯◯할 때 쓴다` 적용 조건 한 줄을 단다.

## 3. 깊이 2층 — 본문 vs 백업/별첨

이 가이드는 장표(슬라이드)의 원천이다. 내용은 두 층으로 나눈다.
- **본문(메인) = 주연:** 그 포인트를 *이해*하는 데 필요한 것만. 앞 내용만으로 이해되도록 쉽게·정확하게.
  본문만 순서대로 읽어도 What/Why/How가 남으면 합격.
- **백업/별첨 = 조연:** 표·스펙·스키마·솔루션 비교·표준 약어, 그리고 출처별 세부(벤더 프레이밍·대안 분류·정확 수치)는
  본문 흐름을 끊지 않게 내린다. 깊이는 여기서 충분히 가져간다(겁내서 줄이지 말 것).

표기:
- 본문 핵심 단위는 `### [Slide N] 제목`, 그 백업은 바로 아래 `#### [Backup N-1] 제목 (상세)`.
- 본문 슬라이드 끝에 `> ▸ 백업: [Backup N-1] ◯◯◯` 로 어떤 백업으로 들어가는지 표시.
- 가장 깊은 백업(체크리스트·프레임워크·전체 필드 스펙표)은 문서 끝 `## 별첨(Appendix)` 에 `[Appendix A] 제목`.

> 본문에 표/스키마/솔루션 비교/약어 덤프를 넣지 않는다 — 백업으로.

## 4. 언어·표기·링크·출처 규칙

### 4-1. 언어·표기
- 한국어로 쓴다. **기술용어·솔루션명·표준명은 `한글(English)` 병기** (예: 메타데이터(Metadata), 데이터 계보(Data Lineage)).
- 같은 용어는 매번 같은 한국어로(용어 표류 금지 — 0-5와 연결).

### 4-2. 목차(ToC) — 문서 상단 앵커 링크 (필수)
- 모든 가이드는 제목 아래에 **목차**를 둔다(KQ 박스를 둘 경우 그 링크를 목차 맨 위에). 모든 섹션(+별첨·참고자료·변경 이력)을 **앵커 링크**로 연결한다.
- **각 섹션 바로 위에 명시 HTML 앵커**(`<a id="why"></a>`·`<a id="what"></a>`)를 심고 목차에서 그 id로 링크한다 — B-3는 `#why`·`#what`·`#how`·`#tech-stack`·`#where`처럼 깔끔한 id를, B-1은 `#s11`·`#s21` 식 id를 쓴다. 영문 라벨·`—`·`( )`가 섞인 H2 제목은 GitHub 자동 앵커가 깨지므로 자동 앵커에 의존하지 말고 명시 id로 건다.
- **목차는 H2(섹션)+소절(H3)까지 클릭 링크로 연결한다.**(상세 §8-5)

### 4-3. 주제 간 상호 연결 (필수)
- 인접 주제를 언급할 때는 **해당 주제의 가이드 파일로 링크**한다. 파일명 규칙(`A-2 메타데이터.md`)을 그대로 쓰되, 공백은 `%20`으로 인코딩한다 → `[A-2 메타데이터](../가이드%20작성/A-2%20메타데이터/A-2%20메타데이터.md)`.
- 특히 10장(연계 범위)의 역할 분담표·1장 경계표의 주제명은 모두 링크로 연결한다.
- 아직 작성되지 않은 주제도 미리 링크해 둔다(파일 생성 시 자동 연결됨). 깨진 링크가 아니라 "작성 예정" 표시로 본다.

### 4-4. 솔루션·도구·표준 소개 시 출처 링크 (필수)
- **솔루션·제품·도구·표준·프레임워크를 소개·비교할 때는 반드시 출처 URL을 단다.** 독자가 클릭해 직접 확인할 수 있어야 한다.
- 방식: ① 본문/표에서 제품명을 **공식 페이지로 인라인 링크**(`[Collibra](https://www.collibra.com)`), ② 문서 끝 `## 참고자료(References)`에 `[제목](URL) — 접속일/성격(공식 제품 페이지 등)`으로 모아 재정리.
- **가격·버전·지원 범위 같은 변동 수치는 단정하지 않는다.** "공식 문서/벤더 데모/PoC로 확인"이라고 명시하고 출처를 링크한다(0-1 정직성 원칙과 연결).
- 출처가 필요한 일반 사실(표준 정의·통계 등)도 같은 방식으로 References에 링크로 모은다.

### 4-5. 솔루션 2층 구조 — Tech Stack 정본 반영 (필수)
솔루션은 **2층으로 관리**한다. 한 솔루션이 여러 주제에 걸치므로(예: 카탈로그 솔루션 = 메타데이터·계통 솔루션), 주제마다 따로 적으면 중복·불일치가 난다.
- **1층 = 각 주제 가이드의 '솔루션' 섹션** — *그 주제 관점에서 솔루션의 **기능**을 비교*한다(커넥터·자동 수집·온프렘 여부 등). 깊은 PoC·기능 상세는 가이드에 둔다.
- **2층 = Tech Stack 정본** [`Tech Player/01 Tech Stack 비교 (솔루션×주제).md`](../Tech%20Player/01%20Tech%20Stack%20비교%20(솔루션×주제).md) — *솔루션 **자체**를 주제 가로질러 묶어 평가·선정*한다(Part B 플랫폼×주제 매트릭스·Part C 묶음 비교).
- **본문 노출 금지:** 가이드의 솔루션 섹션은 이 주제 관점의 기능 비교(1층)만 다루고, **2층 정본으로의 링크·포인터 문장을 본문에 두지 않는다**(2026-07-01 고객 결정 — 전 주제에서 제거). 여기엔 섹션 머리의 "2층 연결" 안내 콜아웃뿐 아니라 "솔루션을 묶어 평가·선정하는 일은 2층 정본이 전담한다" 류 도입 문장도 포함한다. 2층 정본과의 관계는 이 표준·CLAUDE.md에만 두고 가이드 본문에는 노출하지 않는다.
- **★ 반영(reflect-back) — 게이트:** 가이드 작성·수정 중에는 **정본을 건드리지 않는다**(매번 갱신은 낭비). **사용자가 그 가이드를 최종 승인하면서 "정본/Tech Stack에 반영" 을 명시 요청할 때만**, 새로 조사된 솔루션을 정본 Part A 해당 주제 표에 추가(새 플랫폼이면 Part B 매트릭스도) 갱신하고 별도 커밋으로 남긴다. (가이드는 기능을 비교하고, 정본은 승인된 결과만 누적해 묶음 평가를 가능케 한다.)

## 7. 고객 피드백 워크플로우

작업은 **작성 → 고객 확인 → 피드백 반영**을 주제별로 반복한다.

1. **작성:** 템플릿(`03`)에서 시작, 표준(`01`·`02`) 준수, 자가 체크리스트(6장) 통과 후 초안 완성.
   - frontmatter `status: draft`.
2. **고객 확인:** 고객에게 제시. 피드백을 받는다(구조·내용·표현·범위 등).
3. **반영:** 피드백을 가이드에 반영하고, 가이드 하단 **`## 변경 이력 / 피드백 반영`** 표에 누적 기록.
   - 표 형식: `| 일자 | 버전 | 피드백(누가/무엇) | 반영 내용 | 반영 위치 |`
   - frontmatter `status`를 `review`(확인중) → `confirmed`(확정)으로 갱신, `version` 증가.
4. **커밋:** 피드백 반영은 **별도 커밋**으로 남겨 이력 추적
   (예: `feat(A-1): 고객 1차 피드백 반영 — 솔루션 비교 기준 보강`).

> 표준 자체에 대한 피드백(모든 가이드 공통 규칙 변경)은 이 `01`/`02` 파일을 고치고 별도 커밋한다.

## 8. 문체·금지·콜아웃·구성 표준 (컨설팅체)

> 근거: 고객 마스터 프롬프트 [`0622_작업지시.md`](../가이드%20작성/B-2%20데이터%20해설·주석/0622_작업지시.md) + **B-3 작성 세션의 양식·문장·구성 피드백을 융합**. **모든 주제 가이드 공통 품질 기준**이다 — "AI스러움"을 빼고 사람이 쓴 컨설팅 보고서 품질로 맞춘다.

### 8-1. 문체 — 선언형 컨설팅체
- **선언형으로 단정한다.** "X는 크게 네 가지다", "X란 …다". **대화형·서술형 금지** — "…인지는 …에서 갈린다", "…해 보면", "벅차다", "여럿이 겹치면 더 분명해진다" 식으로 풀어 말하지 않는다.
- 문장은 **짧고 명확**하되 단문만 반복하지 않는다. 따라서·반면·결과적으로·특히·예를 들어 등으로 논리를 잇는다.
- **정의는 직접형:** "X란 …다"로 먼저 단정한다. "X는 …가 아니다. …다"처럼 부정 먼저 풀지 않는다(대조가 꼭 필요하면 정의 뒤 한 번만).
- **문단은 자기완결적으로.** '자세히는 N절에서 다룬다' 같은 불필요한 전·후방 교차참조는 빼고, 독자가 목차·링크로 찾게 한다(꼭 필요한 연결만).
- **자기참조는 "본 가이드"** — '본 프로젝트' 같은 표현은 쓰지 않는다.
- **독자에게 계산·측정 부담을 지우지 않는다.** "비용 대비 가치를 측정하라" 식으로 지표 작성을 요구하지 말고, "더 단순한 방법으로 안 되는 일인가"처럼 정성 판단으로 묻는다.

### 8-2. 금지 표현 (AI 문체 제거)
- **요약/메타 라벨 금지:** "한 줄 요약", "한 줄 정의", "핵심은", "쉽게 말하면", "정리하면", "요약하면", "한 문장으로". 요약 내용은 일반 문단에 흡수한다.
- **메타 코멘트 금지:** "아래는 작성본", "다음은", "참고로", "설명드리자면". 바로 본문부터 쓴다.
- **이모지·장식 기호 금지:** 👉❓✅❌📌🤖 및 ★※▶■□✓✔. 강조는 문장·표·제목으로. **과도한 `**굵게**` 강조도 지양**(한 문장에 여러 번 금지).
- **짧은 선언문 반복 금지:** "첫 번째 문제는 …이다. 두 번째 문제는 …이다." → `- 항목: 설명` 불릿 구조로.
- **소비자 비유 금지:** 스마트폰 사진·고양이/강아지·넷플릭스·쇼핑몰 추천 등. 제조 사례(외관 검사·설비·클레임·시험 보고서)로 대체한다.
- **판단어 대신 속성어:** 대조 라벨에 '나쁜/좋은' 대신 속성을 쓴다(예: '나쁜 목적/좋은 목적'→'**막연한 목적/측정 가능한 목적**').
- **불필요한 괄호 부가설명 금지:** 동의어·재진술을 괄호로 덧붙이지 않는다(예: '바꾼다(개명)·(병합)·(교정)'→괄호 삭제).

### 8-3. `>` 블록인용(콜아웃) — 용도 한정
- `>`는 **네 가지에만** 쓴다: ① **예시**(예:/제조 예시) ② **용어 풀이**(용어 즉시 정의) ③ **권장·권고·판단 기준** ④ **주의·한계**.
- **주 설명·정의·논리·규칙은 본문 단락**으로 쓴다(`>` 금지). 단순 "자세히는 ○○절/별첨" **포인터는 본문 문장**으로(괄호) 둔다.

### 8-4. 제목·표 머리글·용어
- 소제목은 **개조식 명사구**로 짧게 — **서술형 종결('…한다'·'…하나')·대화형 수식('구축 내내 지키는' 류) 금지**(예: '구축 내내 지키는 설계 원칙'→'설계 원칙', '개념을 3층으로 나눠 배치한다'→'개념의 3층 배치'). AI 영어·과한 외래어(레이어→층, 프레임워크→체계)·"vs"→"와/대"·컨설팅 라벨("— 산출물: …")은 피한다.
- **표 머리글도 개조식 명사구**로 — 의문형('무엇을 담나'·'누가 관리'·'무엇을 하나')은 '상세'·'담당'·'수행 내용'처럼 짧게.
- **용어는 한 개념 한 단어로 끝까지 통일**(예: '도구'와 '솔루션'을 섞지 말고 하나로). 영문 병기는 **최초 정의 시 1회**만. 핵심 질문(CQ)→**핵심 질문**.
- **현업 영문 용어는 유지**: 제조 표준어(Delamination·Wicking·CAF·CCL·동판·균열·프리프레그)는 억지 국문화(들뜸·심지 현상·동박적층판 식 직역)하지 않는다.

### 8-5. 목차·출처·일관성
- **목차는 H2+소절까지 클릭 링크.** 한글 제목 특수문자로 자동 앵커가 깨지면 소절마다 안정 HTML 앵커(`<a id="s11"></a>`)를 심어 그 id로 링크한다(§4-2).
- **출처는 논문식 각주**: 본문 `[N]` → 하단 `## 참고자료`로 이동. **실재하지 않는 문서(가상의 내부 문서 등)는 참고자료에 넣지 않는다.**
- **설명이 부족한 외부 개념은 끌어들이지 않는다.** 본문에서 충분히 풀지 못하는 외부 표준·프로세스를 이름만 인용하면 독자가 헷갈린다 — 자족적으로 쓰거나 인용을 뺀다.
- **개수·단계 수는 문서 전체에서 하나로 일관.** 같은 절차를 어디선 5단계, 어디선 7·9단계로 섞지 않는다.

### 8-6. 구성·예시·근거
- **핵심과 (선택)을 구분한다.** 조건·요소를 나열할 때 '하나만 있어도 도입 근거가 되는 핵심'과 '있으면 가치가 커지는 (선택)'을 표시한다.
- **예시는 '예시'로 명시한다.** 개념·관계·요소를 보일 때 '정해진 N개 표준'이 아니라 도메인에 맞춰 고른 **예시**임을 밝힌다(캐노니컬 숫자·외부 표준 인용처럼 보이지 않게).
- **주장은 근거로 받친다.** 모델·규칙을 단정하기 전 필요하면 외부 표준·리서치로 확인하고, 그 결론에 맞춰 문서 전체를 일관되게 고친다.
