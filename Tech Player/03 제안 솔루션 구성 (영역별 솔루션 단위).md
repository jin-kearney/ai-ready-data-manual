# AI-Ready Data — 제안 솔루션 구성 (영역별 솔루션 단위)

> **목적:** RFP 5개 영역에 20개 주제를 매핑하고, **각 영역 안에서 "실제로 몇 개의 솔루션이 필요한가"**를 정한다. 한 솔루션이 함께 제공하는 주제는 **묶고**, 성격이 다른 주제는 **나누고**, 표준·흡수로 충분한 주제는 별도 솔루션을 **빼는** 단일 관점의 솔루션 구성도다.
> **성격:** 제품은 **예시**다 — 정답이 아니라 계열사가 환경(이미 쓰는 플랫폼 · **온프렘/폐쇄망** · 데이터 민감도)에 맞게 고르는 선택 가이드. 후보 전체 비교는 [01 Tech Stack 비교](01%20Tech%20Stack%20비교%20(솔루션×주제).md), 영역별 상세·아키텍처 다이어그램·온프렘 상세표는 [02 제안 Tech Stack](02%20제안%20Tech%20Stack%20(RFP%205개%20영역).md).

**세 가지 정리 원칙:**
- **묶기(통합):** 한 플랫폼이 여러 주제를 함께 제공하면 솔루션 1개로 묶는다 (예: 카탈로그·메타·글로서리).
- **나누기(성격):** 시장·성격이 다르면 분리한다 (예: 라벨링 ≠ 온톨로지, 프롬프트관리 ≠ 합성데이터).
- **빼기(표준/흡수):** 표준 채택으로 충분하거나 다른 솔루션에 흡수되는 주제는 별도 솔루션을 두지 않는다 (예: D-2 MCP 표준, 계보·접근통제).

`온프렘` 열: ✓ = OSS 자체호스팅 등 폐쇄망 가능 선택지 있음. (상세는 [02 온프렘표](02%20제안%20Tech%20Stack%20(RFP%205개%20영역).md#온프렘폐쇄망-가능-여부-선택의-핵심-축))

## 영역별 솔루션 단위

### ① 데이터 수집·전처리

| 솔루션 단위 | 묶는 주제 | 예시(예시일 뿐) | 온프렘 | 비고 |
|---|---|---|:--:|---|
| **문서 전처리·OCR** | B-1 + F-3(문서) | Docling·Unstructured / Document AI | ✓ OSS | 문서 파싱과 문서 OCR은 한 도구로 묶임 |
| **STT(음성→텍스트)** | F-3(음성) | Whisper / CLOVA Speech | ✓ Whisper | 회의록·현장 음성 데이터 있을 때만 |
| **산업 historian** | D-1 | AVEVA PI·Ignition + OPC UA | ✓ 본래 온프렘 | 설비·센서 데이터 다루는 계열사 |

→ **2~3개.** 문서 OCR은 전처리 도구에 묶이고, 음성·설비는 성격이 달라 분리.

### ② 메타데이터·카탈로그·비즈니스 글로서리

| 솔루션 단위 | 묶는 주제 | 예시 | 온프렘 | 비고 |
|---|---|---|:--:|---|
| **거버넌스·카탈로그 플랫폼 1개** | A-1 + A-2 + A-3 | Collibra·Atlan·Purview / OpenMetadata | ✓ OSS(OpenMetadata) | A-1·A-2·A-3은 보통 한 제품이 함께 제공 — **별도로 사지 않는다.** 이 플랫폼이 ④의 **계보(C-3)·접근통제(F-4a)**까지 함께 커버 |

→ **1개.** 이 영역의 핵심 묶음이자, 다른 영역 일부까지 끌어오는 코어.

### ③ 어노테이션·온톨로지

| 솔루션 단위 | 묶는 주제 | 예시 | 온프렘 | 비고 |
|---|---|---|:--:|---|
| **라벨링·어노테이션** | B-2 | Label Studio·CVAT / Snorkel | ✓ OSS | 학습용 라벨이 필요한 데이터에 |
| **온톨로지·그래프DB** | B-3 | Neo4j·GraphDB / Stardog | ✓ 자체호스팅 | **성격이 달라 분리.** 다중 홉 인과 추론이 필요할 때만 |

→ **2개(온톨로지는 선택).** 라벨링과 그래프는 완전히 다른 시장이라 묶이지 않는다.

### ④ 데이터 품질·계보·감사

| 솔루션 단위 | 묶는 주제 | 예시 | 온프렘 | 비고 |
|---|---|---|:--:|---|
| **품질·관측** | C-2 + C-1 | Great Expectations·Soda / Monte Carlo | ✓ OSS(GX·Soda) | 투입 전 게이트(C-2) + 상시 모니터링(C-1). ②/데이터플랫폼 내장으로 갈음 가능 |
| **비식별** | F-4b | Presidio·파수 / Tonic | ✓ OSS·국내 | 개인정보·민감정보 다룰 때만 (발견+변환 짝) |
| ~~계보(C-3)~~ | — | (②플랫폼이 제공) | — | **별도 솔루션 불필요 — ②에 흡수** |
| ~~접근통제(F-4a)~~ | — | (②/데이터플랫폼 내장) | — | **별도 솔루션 불필요 — 플랫폼 내장** |

→ **신규 0~2개.** 계보·접근통제는 ②플랫폼이 이미 제공하고, 품질은 권장, 비식별은 민감정보 있을 때만.

### ⑤ AI 학습·RAG·Agent 연계

| 솔루션 단위 | 묶는 주제 | 예시 | 온프렘 | 비고 |
|---|---|---|:--:|---|
| **LLMOps** | D-3 + E-3 + E-4 | Langfuse / LangSmith | ✓ OSS(Langfuse) | 프롬프트 버전관리 + 평가셋 + 피드백을 **한 도구로 묶음** |
| **합성데이터** | E-2 | MOSTLY AI·SDV / Omniverse(비전) | ✓ OSS(SDV) | **성격이 달라 분리.** 데이터 부족·희소 케이스 보강할 때만 |
| ~~Tool/API 명세(D-2)~~ | D-2 | 표준 **MCP·OpenAPI** (+레지스트리 선택) | ✓ 표준 | **별도 솔루션 굳이 불필요 — 표준 채택이 핵심.** 레지스트리(API Center·MCP Registry)는 선택 |

→ **LLMOps 1개 + 합성(선택).** D-2는 솔루션이 아니라 표준 채택, 피드백(E-4)은 LLMOps에 흡수.

> **필수 코어 3개** = ① 거버넌스·카탈로그 플랫폼 + ② 문서 전처리·OCR + ③ LLMOps. 제조 설비 계열사는 **+ 산업 historian**. 나머지(품질·관측·라벨링·비식별·온톨로지·합성)는 데이터·과제 성격에 따라 **선택**.

## 별도 솔루션이 필요 없는 주제 (명시)

- **D-2 Tool/API 명세** — **표준(MCP·OpenAPI) 채택**이 본질. 명세를 잘 쓰는 게 핵심이지 별도 제품 구매가 핵심이 아니다(레지스트리는 선택).
- **C-3 계보 · F-4a 접근통제** — ② 거버넌스·카탈로그 플랫폼이 함께 제공한다. 별도로 사지 않는다.
- **E-4 피드백 루프** — ⑤ LLMOps 도구에 포함된다.
- **A-2 메타데이터 · A-3 글로서리** — A-1 카탈로그와 한 플랫폼이다(② 1개로 묶임).
- **범위 외:** E-1 Product화 · F-1 DataOps · F-2 생애주기 — RFP 5개 영역에 직접 매핑되지 않아 이 구성에서 제외(필요 시 [01](01%20Tech%20Stack%20비교%20(솔루션×주제).md) 참조).
