# Lv.1 통합 데이터 플랫폼 (Lakehouse) — 구분 A 개요

> 작성일: 2026-06-10 | 조사 기준: 2025~2026년 최신 릴리스 기준

---

## 1. 구분 정의

**통합 데이터 플랫폼(Lakehouse)**은 데이터 저장·처리·거버넌스를 단일 플랫폼 위에서 제공하는 솔루션 유형이다.  
카탈로그·메타데이터·Business Glossary·데이터 리니지·Data Marketplace·품질 모니터링·접근 제어(RBAC/ABAC)·DataOps·데이터 생애주기 관리가 플랫폼에 내장(또는 네이티브 통합)되어 있어 별도 거버넌스 툴 없이도 AI-Ready Data 체계의 핵심 주제를 커버한다.

### 커버하는 AI-Ready Data 주제

| 주제코드 | 주제명 | 커버 방식 |
|---------|--------|-----------|
| A-1 | 데이터 카탈로그 | 플랫폼 내장 카탈로그(Unity Catalog, Horizon Catalog 등) |
| A-2 | 메타데이터 관리 | 테이블·컬럼·태그·비즈니스 용어 자동 수집 |
| A-3 | 데이터 리니지 | 컬럼 레벨 자동 리니지 캡처 |
| C-3 | Business Glossary | 공인 지표(Certified Metrics), 시맨틱 레이어 내장 |
| E-1 | 데이터 품질 | 내장 DQ 모니터링·프로파일링 |
| C-1 | 데이터 접근 제어 | RBAC/ABAC, 행·열 수준 보안 |
| C-2 | 데이터 분류·민감정보 보호 | 자동 분류·태그 기반 정책(DSPM/DLP) |
| F-1 | DataOps·파이프라인 거버넌스 | 파이프라인 통합·CI/CD 오케스트레이션 |
| F-2 | 데이터 생애주기 관리 | Time Travel, 데이터 보존 정책, 아카이빙 |

## 5. 선택 가이드

### 5-1. 일반 선택 기준

| 상황 | 추천 솔루션 | 이유 |
|------|------------|------|
| Azure 기반 마이크로소프트 생태계 표준화 | Microsoft Fabric + Purview | Power BI 통합, Entra ID 연동, M365 보안 체계 재활용 |
| AWS 중심, 빠른 ML/AI 플랫폼 구축 | Databricks | MLflow·Genie·Unity Catalog 패키지, 업계 표준 Delta Lake |
| 데이터 웨어하우스 중심 분석, SQL 친화적 조직 | Snowflake | 관리 오버헤드 최소, Horizon Catalog 내장, Cortex AI |
| GCP 표준화 또는 멀티모달 AI 워크로드 | Google BigQuery+Dataplex | Vertex AI 통합, Gemini 지원, Knowledge Catalog |
| 하이브리드/온프레미스 필수, 엄격한 데이터 주권 | Cloudera | 하이브리드 배포, SDX 보안, 기존 Hadoop 마이그레이션 경로 |
| 오픈 Iceberg 기반 쿼리 가속·멀티소스 페더레이션 | Dremio | 가상화 아키텍처, SQL 쿼리 레이어, 에이전트 자동화 |

### 5-2. 두산그룹 같은 다계열사 대기업 환경 관점

**핵심 고려사항:**
- **멀티 클라우드·온프레미스 혼재**: 계열사별로 인프라가 다를 경우 Cloudera(하이브리드) 또는 Databricks(3대 클라우드 지원)가 유리
- **계열사 간 데이터 격리와 공유**: Databricks Unity Catalog의 ABAC + Delta Sharing, Snowflake의 Data Sharing이 계열사 간 데이터 제품 유통에 적합
- **기존 ERP/SAP 연동**: Microsoft Fabric은 Azure 생태계 내 SAP 커넥터, Power Platform과의 연동이 강점
- **제조 데이터(IoT·SCADA·MES)**: 대용량 비정형·반정형 데이터 처리에는 Databricks 또는 Cloudera의 스트리밍 처리 능력이 유리
- **거버넌스 중앙화 vs. 계열사 자율**: 중앙 거버넌스 팀이 있다면 Databricks 또는 Fabric+Purview; 계열사 자율 운영이라면 Snowflake의 데이터 공유 모델이 적합
- **Total Cost of Ownership**: 소비 기반 모델(Databricks, Snowflake)은 예측 비용이 어렵고 워크로드 증가 시 급격히 비용 상승. 계열사 규모 예측이 중요
