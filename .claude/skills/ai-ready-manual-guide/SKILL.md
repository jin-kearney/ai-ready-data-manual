# AI-Ready Data 매뉴얼 — 주제 가이드 생산 공정 (멀티 에이전트)

이 스킬은 `최종 주제.md`에 정의된 20개 주제 각각에 대해, 이 레포의 **매뉴얼 가이드 `.md`**를
한 번에 한 주제씩 생산하는 **반복 공정**이다. 형식 정답지는 최신 finalversion인 **B-1 데이터 전처리·B-3 온톨로지**다.

> ★ **형식·원칙은 이 스킬이 다시 정의하지 않는다.** 모든 형식 규칙·작성 원칙·이해 장치·
> 다이어그램 컬러·링크/출처 규칙·합격 기준은 `공통 규칙/`이 단일 권위(Source of Truth)다.
> 이 스킬은 그 규칙을 **런타임에 읽어 따르는** 공정만 담는다. 7섹션 등 옛 규격을 끌어오지 않는다.

## 0. 매 실행 첫 단계 — 규칙부터 읽는다 (생략 금지)

작성/수정에 들어가기 전 아래를 **반드시 읽는다.** 이것이 형식의 단일 출처다.

1. `CLAUDE.md` — 전역 규칙·폴더 구조·목차 구조 표준(고객 피드백 반영 순서)
2. `공통 규칙/01 가이드 작성 표준.md` — 0장(본질)·1장(섹션 구조 Why→What→When→How→Tech Stack→Where·가감 규칙)·2~6장(이해장치·깊이2층·언어/링크/각주식 출처·체크리스트)·7장(피드백 워크플로우)·8장(컨설팅 문체)
3. `공통 규칙/02 다이어그램 표준.md` — 파랑6/초록3 컬러 스키마·테마 헤더·섹션별 다이어그램
4. `공통 규칙/03 주제 가이드 템플릿.md` — 새 가이드의 시작 골격(frontmatter·앵커 목차·KQ 맵)
5. `공통 규칙/최종 주제.md` — **해당 주제의 정의·Key Question(= 합격 기준)**
6. `전체 목차/00 전체 목차 (20개 주제).md` — 주제별 후보 H3·연관관계 조감도
7. 해당 주제 폴더의 `_작업 메모.md`(있으면) — 그 주제의 피드백·결정·주의점

> 이 파일들을 읽지 않고 가이드를 쓰지 않는다. 형식 질문의 답은 항상 이 파일들에 있다.

## 1. 핵심 원칙 (공통 규칙에서 가져온 불변 조건)

- **한 번에 한 주제.** 주제 미지정 시 `최종 주제.md`의 주제 목록을 보여주고 어느 주제인지 확인받는다.
- **데이터 준비 관점 고정.** "AI/에이전트/보안을 만드는 법"이 아니라 "그 AI가 쓸 **데이터를 준비·정비하는 법**". 특히 D-2·D-3·F-4 주의 (CLAUDE.md 목차 표준 마지막 항목).
- **정본 모델 하나로 일관.** 같은 개념 두 번 설명 금지, 용어·개수 끝까지 일치.
- **이 주제가 주연, 인접 주제는 짧은 포인터.** 보일러플레이트 제거.
- **Key Question 전부 답한다(합격 기준).** 작성 후 KQ→섹션 매핑표로 자가 대조.
- **섹션 수는 주제 무게에 따라 가감(고정 금지).** 가벼운 주제 6개 내외(B-1·B-3=6), 무거운 주제(품질·계통) 8~9개. H3는 섹션마다 2~6개로 다르게. Tech Stack·Where는 독립 유지.
- **언어 한국어, 기술용어 `한글(English)` 병기.** 다이어그램은 02 표준 컬러 스키마 그대로.

## 2. 멀티 에이전트 구조

이 세션(Opus)이 **오케스트레이터**다. 직접 쓰기보다 아래를 조율한다(`Agent` 툴, `model: "sonnet"`).

```
Opus 오케스트레이터 (이 세션)
├─ Sonnet 리서치 에이전트 × N   (섹션 클러스터별 웹 리서치 → research/*.md + 출처 URL)
├─ Sonnet 작성 에이전트   × 1   (리서치 종합 → 공통 규칙·B-1·B-3 형식대로 가이드 본문 작성)
└─ Sonnet 출처 검증      × 1   (인용 URL을 web_fetch로 실제 확인 → 보정 + 검증 메모)
```

오케스트레이터는 비용이 큰 종합·일관성·합격 기준(KQ 커버리지) 판단을 맡는다.

## 3. 작업 절차

### 0단계 — 주제 확정
`최종 주제.md`를 읽어 주제코드·주제명·정의·**Key Question**을 추출한다(모든 에이전트 공통 입력). 미지정이면 목록 제시 후 선택받는다. 같은 폴더의 `_작업 메모.md`가 있으면 그 주제의 누적 피드백·결정을 먼저 반영한다.

### 1단계 — 작업 폴더 준비
산출 위치는 **이 레포의 영구 폴더**다(스킬 임시폴더 아님):
```
가이드 작성/<주제코드> <주제명>/
├── <주제코드> <주제명>.md     ← 가이드 본문(피드백 반영하며 버전 갱신)
├── _작업 메모.md              ← 이 주제 전용 규칙·피드백·결정 이력
└── research/                  ← 리서치 원자료 .md + 출처(URL·접속일) (보조 자료)
```
이미 폴더가 있으면(예: 피드백 반영) **기존 본문·_작업 메모.md를 읽고 이어서** 작업한다. 신규 주제면 `03 템플릿`을 복사해 본문 골격을 만든다.

### 2단계 — 리서치 에이전트 스폰 (Sonnet, 병렬)
한 메시지에서 여러 Sonnet 리서치 에이전트를 **동시에** 스폰한다. 클러스터(주제 무게에 맞게 가감):
- **A — What(핵심):** 정의·구성요소·정본 모델 후보·표준/프레임워크·동작 원리. (가장 두껍게)
- **B — How(핵심):** 구축 절차·방법론·수작업 Pain Point→해소 기능·솔루션/플레이어 맵·Tech Spec. (가장 두껍게)
- **C — Why / When:** 필요성을 **개념·구조 중심**으로(외부 통계 의존 X — 정량 현황은 발표 본편 회사 as-is가 채움), 적용 시점·대상 선정 판단.
- **D — KPI(신뢰성 C군만):** 관리 지표 사례 3~5개·측정 목적. **C-1·C-2·C-3 주제일 때만 스폰**한다 — 그 외 주제는 KPI 섹션이 없으므로 생략. (고도화 로드맵은 전 주제 제거 상태라 리서치하지 않는다.)

각 에이전트 지시에 **반드시 포함**: 주제 정의+KQ, "데이터 준비 관점(AI 구축 아님)", 출처 URL 필수, 결과를 `가이드 작성/<주제>/research/research-<클러스터>.md`로 저장. What/How는 시간을 더 배분하고 필요하면 A·B를 각각 2개로 쪼갠다.

### 3단계 — 리서치 검수 (오케스트레이터)
`research/`를 읽어 빈 곳·출처 없는 주장·모순·공백을 찾고 부족하면 보강 스폰한다. **What/How가 충분히 깊은지, KQ를 덮을 재료가 다 모였는지** 특히 확인.

### 4단계 — 작성 에이전트 스폰 (Sonnet, 1개)
입력: `research/` 전체 + **`공통 규칙/01·02·03` + `최종 주제.md`(해당 주제) + `00 전체 목차`(해당 주제 H3)**. 작성 에이전트에 명시할 것:
- `03 템플릿` frontmatter·**상단 H2/H3 앵커 목차**로 시작, 한 줄 정의 포함. **형식 본보기는 B-1·B-3 두 파일** — 새 가이드를 그 둘과 같은 느낌으로.
- **권장 흐름 Why→What→When→How→Tech Stack→Where**(영문 축 라벨 H2 제목, 개요 섹션 없이 What 첫 소절이 정의·체계 내 위치 흡수)는 강제 틀이 아니다 — **실제 섹션·내용은 그 주제의 `00 전체 목차` 블록을 따라 가감**한다(주제마다 다름). 선택형 주제는 Why+When 통합. KPI는 C-1·C-2·C-3만 Where 뒤 마지막에. 로드맵 섹션은 두지 않는다.
- 섹션마다 **한 줄 요약 + 어려운 용어 즉시 풀이 + 제조(두산) 예시 + 실제값 채운 완성 예시**(이모지 라벨 금지 — 01 §8-2).
- **깊이 2층:** 메인은 이해에 필요한 것만, 표·스펙·스키마·비교표·약어는 `#### [Backup]`/`## 별첨`으로.
- **링크:** 인접 주제는 가이드 파일로 링크(`가이드%20작성/...`), 솔루션·도구·표준은 출처 URL 인라인 + **각주식 `## 참고자료(References)`**(본문 `[\[N\]](#refN)` → 맨 아래 `<a id="refN"></a>**[N]** 이름 — <URL>` 분류 정리, B-1·B-3 방식).
- **솔루션 2층(01 표준 §4-5):** 솔루션 섹션은 *이 주제 관점 기능 비교*(1층)로 쓰고, 머리에 Tech Stack 정본(`Tech Player/01 Tech Stack 비교 (솔루션×주제).md`) 링크를 단다.
- 02 표준 컬러 스키마 다이어그램. 본문 끝에 **`## 변경 이력 / 피드백 반영`** 표.
출력: `가이드 작성/<주제>/<주제코드> <주제명>.md`(본문). frontmatter `status: draft`.

### 5단계 — 출처 검증 (Sonnet, 1개)
본문·research의 인용 URL을 web_fetch로 실제 확인 → 죽은/어긋난 출처는 대체하거나 주장 완화·삭제(보수적 보정), 사람 확인 필요 건은 ⚠️ 표시. 가격·버전 등 변동 수치는 단정 금지("PoC/공식문서 확인"). 결과 메모는 `research/`에 남긴다.

### 6단계 — 최종 검수 (오케스트레이터)
`01 표준 6장 체크리스트`로 자가 점검 + **KQ→섹션 매핑표로 누락 대조**. 통독하며 반복·모순·용어/개수 표류 점검. 데이터 준비 관점 이탈 여부 확인. 부족하면 재작업 지시 또는 직접 보정.

### (선택) Tech Stack 정본 reflect-back — ★사용자 명시 요청 시에만
**기본은 하지 않는다.** 가이드 작성·수정 단계에서는 정본을 건드리지 말 것(매번 갱신은 낭비). **사용자가 그 가이드를 최종 승인하면서 "정본/Tech Stack에 반영해줘"라고 명시 요청할 때만** 수행한다 — 이번 주제에서 새로 조사된 솔루션을 `Tech Player/01 Tech Stack 비교 (솔루션×주제).md` **Part A 해당 주제 표에 추가**(중복 합치고 출처 URL 유지), 새 통합 플랫폼이면 **Part B 매트릭스 행/칸**도 갱신. 정본 갱신은 별도 커밋으로 남긴다(01 표준 §4-5).

### 7단계 — _작업 메모.md 갱신 + 커밋·푸시
- `_작업 메모.md`에 이번 작성의 결정·정본 모델·주의점을 누적 기록.
- **신규 파일은 요청을 기다리지 말고 바로 커밋·푸시**(CLAUDE.md 규칙). 주제 단위 의미 커밋. 메시지에 무엇을/왜.
- 사용자에게 본문 링크를 제시하고 한두 문장 요약. 다음 주제/피드백 진행 의사를 묻는다.

## 4. 피드백 반영 모드 (기존 가이드 수정 시)
기존 본문·`_작업 메모.md`를 읽고 → 피드백 항목을 본문에 반영 → 하단 **변경 이력 표**에 `| 일자 | 버전 | 피드백(누가/무엇) | 반영 내용 | 반영 위치 |` 누적 → frontmatter `status`(draft→review→confirmed)·`version` 갱신 → **피드백 반영은 별도 커밋**으로 이력 추적(CLAUDE.md·01 표준 7장).

> 형식·합격 기준이 헷갈리면 추측하지 말고 `공통 규칙/`을 다시 읽는다. 이 스킬은 절차이고, 규칙은 그쪽에 있다.
