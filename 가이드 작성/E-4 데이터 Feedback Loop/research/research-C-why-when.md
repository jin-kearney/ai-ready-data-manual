# E-4 데이터 Feedback Loop — Why·When 클러스터 리서치

> 작성일: 2026-06-26  
> 관점 고정: "AI를 만드는 법"이 아니라 "그 AI가 쓸 데이터를 준비·정비하는 법"  
> 용도: 가이드 본문 초안(Why·When 섹션) 작성을 위한 구조화 소재

---

## 1. Why — 왜 필요한가

### 1-1. 현업 Pain Point: 피드백 환류 체계가 없으면 무엇이 문제인가

#### ① 같은 오류가 반복되지만 원인을 모른다
AI가 틀린 답을 냈을 때 그 결과를 기록·분류하지 않으면, 같은 종류의 오류가 계속 반복된다.
예) 두산 계열사 정비 챗봇이 "베어링 교체 주기" 질문에 엉뚱한 답을 반복 제공하는데, 담당자는
그 오류가 **근거 문서 누락**(A-1/A-2 문제)인지, **라벨 오류**(B-2 문제)인지, **Prompt 미스**(D-3 문제)인지
구분하지 못한 채 방치하게 된다.

- 오류 원인 미파악 → 재발 방지 불가
- "AI가 또 틀렸어"가 반복되며 현업 신뢰도 하락
- 개선에 우선순위를 세우지 못하고 인력이 분산됨

#### ② 사용자 불만이 흩어져 사라진다
챗봇·보고서 생성·검색 화면에서 사용자가 경험한 불만은 대화가 끝나면 흔적 없이 사라진다.
수정한 내용, 재질문, "도움이 안 됐다" 클릭이 데이터로 남지 않으면 어느 부분이 약한지 알 수 없다.

- 사용자 수정 이력이 개선 신호(signal)가 아니라 개인 업무로 소비
- 불만 집계 없이 "느낌"으로만 AI를 평가

#### ③ 일회성 수정에 그쳐 체계적 개선으로 이어지지 않는다
오류를 발견하면 그때그때 Prompt를 수정하거나 담당자가 수동으로 보완한다.
개선 내역이 누적·공유되지 않아 다른 AI 과제나 다음 버전에 학습되지 않는다.

- 같은 패턴의 오류가 다른 AI 과제에서도 반복됨
- 개선 이력이 개인 지식으로 남아 조직에 축적되지 않음

#### ④ "AI가 점점 좋아진다"는 약속이 지켜지지 않는다
AI 도입 시 "운영하면서 계속 개선된다"는 기대를 제시하지만, 피드백 루프 체계 없이는
AI는 출시 당시 수준에서 멈추거나 오히려 데이터 drift(시간 경과에 따른 데이터 패턴 변화)로 퇴보한다.

- 현업: "처음엔 그럭저럭 됐는데 지금은 더 이상해졌다"
- AI의 지속적 가치 창출 주장을 뒷받침할 데이터가 없음

**참고 개념 출처:**
- MLOps 커뮤니티에서 정의한 "Feedback Loop" 필요성: https://ml-ops.org/content/mlops-principles (MLOps Principles, 2024)
- Google의 ML 시스템 기술 채무(Technical Debt) 논문: "Hidden Technical Debt in Machine Learning Systems" (Sculley et al., 2015) — 피드백 없는 ML 시스템이 얼마나 빠르게 퇴행하는지 설명. https://proceedings.neurips.cc/paper_files/paper/2015/file/86df7dcfd896fcaf2674f757a2463eba-Paper.pdf

### 1-2. 기대 효과: 피드백이 체계로 연결되면 무엇이 달라지나

#### ① 반복 실수가 점점 줄어드는 데이터 플라이휠(Data Flywheel)
피드백을 체계적으로 수집 → 분류 → 원인 주제로 연결 → 개선 → 재검증 루프가 돌면,
같은 유형의 오류 빈도가 주기별로 감소한다. "데이터 플라이휠(Data Flywheel)"이란
더 많은 피드백이 더 나은 데이터를 만들고, 그것이 더 나은 AI로 이어지는 선순환이다.

**참고:** https://eugeneyan.com/writing/data-flywheel/ (Eugene Yan, 2021, "Building a Data Flywheel")

#### ② 개선 우선순위가 데이터로 정해진다
피드백 로그를 집계하면 "어떤 질문 유형에서 오류가 가장 많이 발생하고, 그 사업적 영향은 얼마나 큰가"를
수치로 볼 수 있다. 직관·경험이 아니라 빈도×영향도 기준으로 개선 과제를 줄 세울 수 있다.

#### ③ 원인이 추적되어 책임 주제가 분명해진다
오류를 분류하면 원인 주제가 드러난다.
- 근거 문서가 없다 → A-1 데이터 카탈로그 / A-2 메타데이터 정비
- 용어가 엇갈린다 → A-3 비즈니스 Glossary 정비
- 라벨이 빠지거나 틀렸다 → B-2 데이터 해설·주석 보완
- AI가 답의 근거를 못 찾는다 → B-3 온톨로지 또는 D-1 Physical 데이터 정비
- Prompt 지시가 불명확하다 → D-3 Prompt/Harness 자산화
- Tool 호출이 실패한다 → D-2 API/Tool 연계 데이터 정비
- 평가셋이 이 유형을 못 잡았다 → E-3 AI 평가 데이터 확충

이로써 E-4는 "증상을 모아 원인 주제로 되돌리는 허브" 역할을 한다.

### 1-3. Before/After 대조 — 가이드 본문 표 재료

| 구분 | 피드백 루프 없을 때 (Before) | 피드백 루프 있을 때 (After) |
|------|------------------------------|------------------------------|
| 오류 인식 | 사용자가 직접 담당자에게 말하거나 그냥 넘어감 | 오류 발생 시 자동 로그·수집 |
| 원인 파악 | 감으로 추정, 원인 불명 | 오류 유형 분류 → 원인 주제 매핑 |
| 개선 과제 | 발생 건 별로 임시 처리 | 빈도×영향 기준 Backlog 우선순위화 |
| 개선 반영 | 담당자 개인 지식, 다음 버전에 미반영 | 개선 이력 공유·버전 기록 |
| AI 품질 추이 | 점검 기준 없음, 퇴보 감지 불가 | 주기별 오류율 감소 추이 측정 가능 |
| 현업 신뢰 | "AI가 또 틀렸어" 반복 → 불신 누적 | 개선 증거 공유 → 신뢰 회복 |

### 1-4. Worked Example 재료 — 제조 AI 사례 (가상)

**시나리오: 두산 중공업 설비 정비 챗봇**

> 설비 정비 챗봇이 "가스터빈 1호기 베어링 교체 주기"를 묻는 정비팀 질문에
> 매번 "정보를 찾을 수 없습니다"라고 답하거나 다른 기종의 주기를 반환한다.

- **피드백 루프 없을 때:** 정비팀이 챗봇을 포기하고 엑셀 파일을 직접 뒤짐.
  문제는 "챗봇이 나빠서"로 귀결되고 방치됨.

- **피드백 루프 있을 때:**
  1. 챗봇이 "도움이 됐나요? 👍/👎" 버튼을 제공하고, 사용자 수정·재질문 이력을 자동 수집
  2. 집계 결과: "가스터빈 부품 주기" 관련 질문의 실패율 68%, 반복 재질문 비율 최다
  3. 오류 분류: "근거 문서 미등록"(A-1 카탈로그에 설비 매뉴얼 PDF 누락)
  4. 개선 과제: A-1 담당자에게 "정비 매뉴얼 5종 등록" 과제 할당
  5. 재검증: 다음 주기에 같은 질문 실패율 12%로 감소 확인

이처럼 E-4는 "어디를 고쳐야 하는가"를 원인 주제(A군)로 연결하는 체계다.

## 2. When — 어디서 피드백을 모으나 (수집 지점·우선순위)

### E-4의 When 성격: 기본형(다 필요)

데이터 카탈로그나 메타데이터처럼 AI를 쓰는 이상 모든 과제에 적용해야 하는 기본형 주제다.
"어디부터 먼저 할 것인가"(우선순위)가 핵심이지, "할지 말지"를 선택하는 주제가 아니다.

---

### 2-1. 피드백 수집 지점 — 기본 5곳

AI 운영 환경에서 피드백 데이터를 수집할 수 있는 지점은 다음과 같다.

| # | 수집 지점 | 수집하는 것 | 비고 |
|---|-----------|-------------|------|
| 1 | **사용자 상호작용 화면** (챗봇·검색·보고서 생성) | 👍/👎 평가, 재질문, 직접 수정, 세션 중단 | 가장 직접적인 신호 |
| 2 | **사람 승인·거절 단계** (Human-in-the-loop) | 승인/반려 여부, 반려 사유, 수정 전·후 내용 | 고위험 업무 필수 수집 지점 |
| 3 | **AI 실패 로그** | 오류 코드, 타임아웃, 빈 응답, 예외 발생 기록 | 자동 수집 가능 |
| 4 | **Tool 호출 실패 이력** | 호출된 Tool, 실패 이유, 재시도 횟수 | D-2 API/Tool 연계 데이터 연결 |
| 5 | **사용자 우회 행동** | 챗봇 무시 후 직접 조회, 수동 보완 파일 생성 | 수집 어려우나 고신호 |

**참고 개념:** Human-in-the-loop(사람이 AI 결정에 개입하는 구조) — 이 단계는 피드백 데이터의 품질이 가장 높다.
출처: https://humancompatible.ai/papers/human-in-the-loop-machine-learning.html
(또는 일반 참고: Amershi et al., "Software Engineering for Machine Learning", ICSE 2019)

---

### 2-2. 우선순위 — 어디부터 피드백 루프를 거나

모든 지점에서 동시에 루프를 구축하기는 어렵다. 다음 기준으로 우선순위를 정한다.

**기준 ①: 사용량 × 오류 영향도**

| 구분 | 사용량 많음 | 사용량 적음 |
|------|------------|------------|
| 오류 시 손해 큼 | **최우선** — 즉시 루프 | 선별 검토 후 결정 |
| 오류 시 손해 작음 | 2순위 — 자동화 후보 | 후순위 |

- 예) 두산 계열사에서 AI 챗봇이 "납기 일정"을 틀리면 고객사 클레임 → 최우선
- 예) 사내 FAQ 보조 챗봇이 틀려도 담당자가 바로 확인 가능 → 후순위

**기준 ②: 피드백 수집 용이성**
- 수집 자동화가 가능한 지점(실패 로그, Tool 호출 실패)부터 시작
- 사용자 행동 기반 수집(우회·중단)은 UX 설계가 필요해 나중에 추가

**기준 ③: 개선 효과가 확인 가능한 지점부터**
- 수집 → 분류 → 개선 → 재측정의 사이클이 짧은 과제
- 예) 3주 내 "오류율 감소"를 수치로 확인할 수 있는 챗봇 과제부터

**참고:** 오류 영향도 우선순위 기준은 MLOps 커뮤니티의 "Model Risk Tiering" 개념을 데이터 준비 관점으로 재해석한 것.
출처: https://ml-ops.org/content/mlops-principles (MLOps Principles, Continuous Delivery for ML)

### 2-3. 수집 지점별 수집 데이터 항목

각 지점에서 실제로 기록해야 하는 최소 항목이다.

**① 사용자 상호작용 화면**
- 질문/요청 원문, AI 응답 원문
- 사용자 평가(👍/👎 또는 5점)
- 사용자 수정 내용(수정 전·후)
- 재질문 여부 및 재질문 내용
- 세션 종료 방식(정상 완료 / 중단)

**② 사람 승인·거절 단계**
- AI 초안 내용, 승인자 ID
- 승인/반려 여부
- 반려 사유(자유 입력 또는 분류형)
- 반려 후 수작업 수정 내용

**③ AI 실패 로그**
- 요청 ID, 발생 시각
- 오류 유형(코드·메시지)
- 연관 AI 과제 명
- 재시도 여부 및 결과

**④ Tool 호출 실패 이력**
- 호출한 Tool 명
- 입력 파라미터, 반환 오류
- 실패 원인 분류(인증 / 타임아웃 / 스키마 불일치 / 데이터 없음)

### 2-4. "수집하지 말아야 할 것" — 범위 경계

다음은 E-4 범위 밖이다.

| 범위 밖 항목 | 이유 |
|-------------|------|
| AI 모델 성능 지표(정확도·손실 함수 등) | E-3 AI 평가 데이터 영역 |
| 데이터 품질 자체 측정값 | C-2 데이터 품질 관리 영역 |

| 모델 재학습 파이프라인 구현 | AI 구축 영역 (데이터 준비 범위 밖) |

## 4. 가이드 초안 작성을 위한 핵심 구조 메모

### Why 섹션 권장 구성
1. Pain Point 3가지(반복 오류·불만 소멸·일회성 수정) → 제조 현업 예시
2. 기대 효과 3가지(플라이휠·우선순위화·원인 추적) → Before/After 표
3. Worked Example: 설비 정비 챗봇 시나리오(문제→수집→분류→개선→재검증)

### When 섹션 권장 구성
- "E-4는 기본형 주제 — 어디부터 먼저 할 것인가"로 시작
- 기본 5대 수집 지점 표
- 우선순위 결정 기준(사용량×영향도 매트릭스)
- 수집 항목 목록(지점별)
- 범위 경계(무엇은 이 주제가 아닌가)
