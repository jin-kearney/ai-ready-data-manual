# E-4 데이터 Feedback Loop — 작업 메모 (주제 로컬 규칙·결정 이력)

## 정본 모델 (이 주제에서 일관되게 쓰는 것)
- **E-4 = 허브.** 직접 고치지 않는다. 증상을 모아 **원인 유형으로 분류 → 원인 주제로 라우팅**까지만. 실제 수리는 라우팅된 주제(A군·B-2·D-2·D-3·E-3·F-4)에서.
- **닫힌 고리 한 문장:** 수집 → 분류 → 라우팅 → 반영 → 재검증(같은 증상 줄었나) → 다시 수집. 선순환 = 데이터 플라이휠.
- **수집 항목 3갈래:** 사용자 직접(좋아요/싫어요·수정·코멘트) / 행동에서 읽힘(재질문·우회·중단) / 시스템 자동(Tool 실패·근거 누락·승인거절·낮은 점수).
- **오류 유형 6가지:** 데이터 / 라벨 / Prompt / Tool / 권한 / 업무 정책.
- **정본 다이어그램:** 라우팅 지도(2.4)가 "어느 유형→어디로"의 단일 출처. 4.3은 같은 지도에 담당·운영규칙을 더한 것, 6장(Where)은 경계만(받음 E-3·C-1 / 보냄은 2.4 참조).

## 라우팅 표기 결정 (중요)
- `최종 주제.md`의 KQ3 **상세** 줄에 오타가 있다 — "Prompt 문제는 **F3**로, Tool 문제는 **F2**로 연결"은 잘못. 같은 KQ3의 **예시** 줄과 `00 전체 목차` E-4 블록(4.2)이 맞다: **Prompt→D-3, Tool→D-2**. 본 가이드는 D-3/D-2를 따랐다.
- 데이터 품질 경계: "있는데 나쁜"(결측·중복·불일치)=C-2, "없는/안 올라온"=A군. 2.4 라우팅 지도엔 C-2를 main 행으로 넣지 않고(00 목차 준수) 4.3·6장에 경계 주의로만 둠.

## 형식 결정
- 기본형 주제 → When은 "어디부터(수집 지점·우선순위)", 선택형 통합 안 함. 6개 섹션(Why·What·When·How·Tech Stack·Where), KPI 없음(C군 아님), 로드맵 없음.
- "이 가이드가 답하는 4가지 질문" 평문 표 박스 둠(KQ 4개와 1:1, 도입 다음·맨 앞). B-1 방식. 제목·본문에 KQ 코드 노출 안 함.
- 다이어그램: mermaid 3종(2.1 환류 내 위치 / 2.4 라우팅 지도 / 4.1 4단계 / 6 허브) + SVG 1종(`이미지/피드백-여정-모형.svg` — 설비 정비 챗봇 한 건의 여정, 68%→12%). 인라인 SVG 아님(커밋된 .svg 파일 이미지 링크).
- worked example: Why(1.2)에 SVG로 미리보기 + How(4.1)에 "예시로 따라가기" 절차로. 같은 설비 정비 챗봇 사례로 통일(중복 아님 — 표준이 요구하는 2곳 배치).
