## 1. 개념·표준 뒷받침 — DLM / ILM이란

### 1.1 데이터 생애주기 관리(Data Lifecycle Management, DLM)

**DLM**은 데이터가 처음 생성되는 순간부터 최종 폐기될 때까지 시간축 전체를 정책 기반으로 통제하는 체계다.

- **DAMA-DMBOK 2nd Edition**은 "데이터 생애주기 관리"를 데이터 거버넌스의 핵심 실행 영역으로 위치시킨다. 데이터는 생성(Creation) → 저장(Storage) → 사용(Usage) → 아카이빙(Archival) → 폐기(Disposal) 단계를 거치며, 각 단계마다 책임자와 정책이 필요하다고 명시한다.[[1]](#ref1)

- **정보 수명주기 관리(Information Lifecycle Management, ILM)**은 IT 인프라 관점에서 동일 개념을 다룬다. 스토리지 계층(Hot/Warm/Cold)을 데이터의 사용 빈도·중요도에 맞춰 자동 배치함으로써 비용을 절감하면서 필요한 데이터를 적시에 접근할 수 있게 한다.[[2]](#ref2)

- **ISO 15489(Records Management 국제표준)**는 문서·기록물의 생성·등록·보관·폐기 전 주기를 정책으로 통제하도록 요구한다. 제조업 감사·규정 준수 대응의 법적 근거로 자주 인용된다.[[3]](#ref3)

### 1.2 거버넌스 안에서의 위치

DLM은 데이터 거버넌스의 일부다. 데이터를 "수집→활용"에서 끝내지 않고, **"시간이 지난 데이터를 어떻게 할 것인가"** 라는 운영 정책까지 포함해야 거버넌스가 완결된다.

- 카탈로그(A-1)가 "무엇이 어디에 있는가"를 관리하듯, DLM은 "그 데이터가 언제까지 있어야 하는가"를 관리한다.
- 운영관리(F-1)가 파이프라인의 일상 운영을 담당한다면, F-2는 그 데이터 자체의 수명을 결정하는 정책 레이어다.

> **쉬운 정의:** 데이터 생애주기 관리 = "어떤 데이터를 얼마나 오래 어디에 두고, 언제 없앨 것인가"를 조직이 합의한 기준으로 다루는 일.

**→ Why 1.1 개념 설명 + 거버넌스 맥락 연결에 사용**

## 2. 현업 Pain Point — 양면 딜레마

### 2.1 한쪽 극단: 다 쌓아두면

데이터를 "일단 지우지 말자"는 관성으로 운영하면 세 가지 문제가 누적된다.

**① 저장 비용 폭증**
- 클라우드·온프레미스 스토리지 비용은 데이터 용량에 비례한다. 수년간 사용하지 않는 원시 데이터(Raw Data)를 Hot 저장소에 방치하면 운영 예산이 불필요하게 소진된다.
- 제조 현장에서는 센서 데이터·MES 이력·품질 성적서가 하루 수백 GB씩 쌓이는 경우가 있어 방치 시 수년 내 수십 TB 규모로 불어난다.

**② 보안·규정 위반 리스크**
- 개인정보보호법(GDPR 포함)은 "더 이상 필요하지 않은 개인정보는 파기"하도록 의무화한다. 오래된 고객·임직원 정보가 관리 없이 방치되면 규정 위반으로 제재받을 수 있다.[[4]](#ref4)
- 계약서·가격 정보·기술기밀이 불필요하게 장기 보관될 경우 유출 시 피해 범위도 커진다.

**③ 데이터 늪(Data Swamp) — 정작 쓸 데이터를 못 찾음**
- "있는데 못 쓰는 상태"가 Data Swamp다. 오래되고 출처 불명한 데이터가 뒤섞여 있으면 AI·분석에 투입할 신뢰할 수 있는 데이터를 찾아내기 어려워진다.
- 품질(C-2)·카탈로그(A-1)가 정비되어도 폐기 대상 데이터가 시스템에 공존하면 AI가 오래된 노이즈를 정답으로 오인할 수 있다.

### 2.2 다른 쪽 극단: 막 버리면

반대로 보관 기준 없이 용량 부족 때마다 임의로 데이터를 삭제하면 다른 문제가 생긴다.

**① AI 학습 데이터 소실 — 특히 희소·불량·실패 사례**
- AI는 "정상 + 이상"을 함께 학습해야 한다. 그런데 불량·장애·실패 데이터는 발생 빈도 자체가 낮다. 이를 용량 문제로 삭제하면 재현 불가능한 케이스가 영구히 사라진다.
- 연료전지·중공업처럼 내구 시험·장기 운용 데이터가 핵심인 제조업에서는 수개월~수년치 이력 데이터가 AI 성능을 좌우한다. 한 번 잃으면 재수집 자체가 불가능하다.

**② 감사·소송 시 증빙 불가**
- 제조업은 제품 책임(PL)·품질 클레임 대응을 위해 제조 이력·검사 성적서를 일정 기간 보존해야 한다. ISO 9001, IATF 16949 등 품질 규격이 기록 보존을 명시한다.[[5]](#ref5)
- 보존 기간 내 데이터를 임의로 삭제하면 감사나 소송에서 증빙을 제출할 수 없게 된다.

**③ 재현·재분석 불가**
- AI 모델 성능이 저하될 때 "언제부터 나빴는가"를 역추적하려면 과거 입력 데이터가 있어야 한다. 데이터가 삭제되어 있으면 원인 분석 자체가 막힌다.

### 2.3 근본 원인: "기준 없이 개인이 알아서" 상태

두 극단 모두를 만드는 공통 원인은 **조직 차원의 보존·폐기 정책 부재**다.

- 담당자마다 "중요할 것 같으니 보관"/"용량 없으니 삭제" 판단이 다르다.
- PC·공유폴더·개인 NAS에 산재해 있어 어떤 데이터가 어디 있는지 조직이 파악조차 못 한다.
- 담당자 퇴직 시 그 사람만 알던 데이터가 함께 사라지거나, 접근 불가 상태로 방치된다.
- 신규 데이터는 쌓이는데 오래된 데이터의 정리 기준이 없어 저장소는 해마다 커진다.

## 3. 기대 효과

| 기대 효과 | 구체 내용 |
|---|---|
| **저장 비용 절감** | Hot/Warm/Cold 계층화로 자주 쓰지 않는 데이터를 저렴한 저장소로 이동. 중복·불필요 데이터 정리 후 스토리지 용량 확보 |
| **보안·규정 위험 감소** | 만료된 개인정보·계약정보 정기 폐기 → 유출 시 피해 범위 축소, 개인정보보호법·ISO 요건 준수 |
| **AI 학습 데이터 확보** | AI 재사용 가치가 높은 불량·희소·실패 이력을 보존 기준에 명시 → 삭제 방지. 중요 데이터는 아카이빙으로 영구 보존 |
| **감사·규제 대응** | 유형별 보존 연한이 정책으로 명문화 → 감사 시 즉시 증빙 제출 가능 |
| **데이터 품질 향상** | 오래된 노이즈·폐기 대상 데이터를 Active 저장소에서 분리 → 카탈로그·AI 입력이 깨끗해짐 |
| **운영 투명성** | 폐기 승인 로그·이력 → "누가 무엇을 언제 삭제했는가" 추적 가능 |

**→ Why 1.2 기대 효과에 사용**

<a id="worked-example"></a>
## 4. 적용 전/후 대비 — 가상 예시 (두산퓨얼셀)

> **주의: 아래 내용은 두산퓨얼셀의 실제 상황이 아니라, 발전용·건물용 수소연료전지 제조사의 전형적 환경을 바탕으로 구성한 가상 예시다.**

### 배경

두산퓨얼셀은 연료전지 스택(Stack)을 제조하며 다음 데이터를 생성한다.

- **스택 내구 시험 데이터**: 수천 시간 연속 운전 기록 (전압·전류·온도·습도 등 시계열 센서값, 시험당 수 GB~수십 GB)
- **제조 이력**: 스택 조립·검사 공정별 작업자·설비·조건·결과 기록
- **품질 성적서**: 출하 전 검사 결과, 고객 납품 이력
- **C/S(After-Sales) 리포트**: 현장 장애·점검 보고서, 불량 원인 분석 이력

### 전(Before) — 기준 없는 혼재 상태

- 내구 시험 데이터는 담당 연구원 PC·개인 NAS에 분산 보관. 퇴직 시 상당량 소실.
- 서버 용량 부족 시 담당자 판단으로 "오래된 것"을 삭제. 초기 실패 실험 데이터·불량 케이스가 우선 삭제됨.
- 품질 성적서 중 일부는 출력본만 존재. 고객 클레임 시 10년 전 성적서를 찾지 못하는 경우 발생.
- 용량 걱정으로 다음 AI 과제 착수 시 "학습할 데이터가 부족"한 상황 반복.

### 후(After) — 유형별 보존·폐기 정책 적용

| 데이터 유형 | 보존 연한 | 저장 계층 | 폐기 기준 | AI 재사용 태그 |
|---|---|---|---|---|
| 스택 내구 시험 (정상·성공) | 10년 | Warm → Cold 이동 | 연한 만료 후 폐기 승인 | ★ 보존 (AI 학습용) |
| 스택 내구 시험 (실패·이상) | **영구** | Cold 아카이브 | 폐기 금지 | ★★ 핵심 보존 |
| 제조 이력 | 제품 출하 후 15년 | Warm | 연한 만료 후 폐기 | ★ 보존 |
| 품질 성적서 | 제품 출하 후 20년 | Cold | 연한 만료 후 법무 승인 폐기 | 증빙 목적 |
| C/S 리포트 | 10년 | Warm | 연한 만료 후 폐기 | ★ 보존 (원인 분석 AI) |
| 임시 중간 파일·중복 데이터 | 6개월 | Hot | 자동 삭제 | 없음 |

**효과:**
- 저장 비용: 불필요 중복·임시 파일 정리로 Hot 스토리지 용량 30~40% 절감(가상 추산).
- 보안 리스크: 만료 개인정보(임직원·고객 연락처 포함 파일) 정기 폐기 → 유출 피해 범위 축소.
- AI 학습: 실패·불량 케이스 "영구 보존" 태그 → 다음 AI 과제에 즉시 활용 가능.
- 감사 대응: 폐기 승인 로그 + 성적서 Cold 아카이브 → 고객 클레임 시 10년 전 성적서 5분 내 제출 가능.

## 5. AI 시대 관점 보강 — "버리는 것도 전략"

### 5.1 AI 학습 가치 기준의 보존 판단

전통적 DLM은 법적 의무·비용·보안을 중심으로 보존 기간을 결정했다. AI 시대에는 여기에 **"AI 학습·재사용 가치"**가 새 기준으로 추가된다.

| 보존 판단 기준 | 전통 DLM | AI 시대 추가 기준 |
|---|---|---|
| 법적 의무 보존 | 필수 | 그대로 |
| 업무 재사용 가능성 | 고려 | 그대로 |
| 비용(저장소 비용) | 핵심 | 그대로 |
| **AI 학습·희소 케이스 가치** | 고려 안 함 | **새 필수 기준** |
| 보안·개인정보 리스크 | 고려 | 강화 (AI 활용 범위 넓어짐) |

- **중복·정상 패턴 데이터**: 대량으로 있어도 AI 학습 효용이 낮음 → 일정 기간 후 폐기(또는 샘플 보존)로 비용 절감.
- **희소·이상·실패 케이스**: 발생 빈도 자체가 낮고 재현 불가능 → "영구 보존" 또는 장기 아카이빙.
- **노이즈·오류·무결성 불명 데이터**: AI가 잘못 학습할 수 있으므로 적극 폐기 또는 품질 정비 후 보존.

### 5.2 DAMA-DMBOK의 관점

DAMA-DMBOK은 데이터 생애주기를 거버넌스의 "데이터 보존과 폐기(Data Retention and Disposal)" 실천 영역으로 다루며, 다음을 강조한다.

> "조직은 어떤 데이터를 얼마나 오래 보관할지, 언제 어떤 방식으로 폐기할지를 명문화된 정책으로 관리해야 한다. 개인이 임의로 결정하는 구조는 규정 위반과 데이터 손실 양쪽 위험을 동시에 만든다."

— DAMA International, *DAMA-DMBOK: Data Management Body of Knowledge*, 2nd Edition, 2017, Chapter 13 (Data Governance). [[1]](#ref1)

### 5.3 NIST의 관점

NIST SP 800-188(De-Identifying Government Datasets)과 NIST SP 800-53은 데이터 보존·폐기를 정보 보안 통제의 일부로 명시하며, 특히 더 이상 필요하지 않은 민감 데이터의 안전한 폐기(Secure Data Disposal)를 강조한다.[[6]](#ref6)

### 5.4 제조업 규정 요건 (ISO/품질 표준)

- **ISO 9001:2015 §7.5**: 문서화된 정보(기록)의 보존 기간과 처분 방법을 관리하도록 요구.[[5]](#ref5)
- **IATF 16949**: 자동차 공급망 품질 기록 보존(일반적으로 최소 15년).
- **개인정보보호법(한국)**: 목적 달성 후 즉시 또는 일정 기간 내 파기 의무. 미파기 시 과징금 부과.[[4]](#ref4)
