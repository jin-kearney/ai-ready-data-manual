## 1. Why — Pain Point (AI 과제를 막는 현실적 장벽)

### 1-1. 민감정보가 섞인 데이터는 AI에 그대로 넣을 수 없다

제조 현장의 클레임·A/S·영업 데이터는 본질적으로 민감정보 덩어리다. 두산밥캣 딜러 클레임 데이터 한 건에는 딜러명, 담당자 연락처, 최종 계약가, 보증 조건이 섞여 있다. 이런 데이터를 있는 그대로 AI 학습에 쓰거나 외부 LLM(ChatGPT, Claude API 등)에 보내면 고객 개인정보·계약정보가 외부 서버에 저장·학습될 수 있다.

**실제 사고 사례**: 2023년 3월 삼성전자 임직원이 ChatGPT를 업무에 활용하다 기업 내부 코드·회의록 등 기밀이 외부 서버에 노출된 사례가 발생했다. ChatGPT 사용 허용 후 20여 일 만에 일어난 일이다 [[1]](#ref1).

**구조적 원인**: AI 서비스 일부는 사용자 입력을 저장하거나 모델 개선 목적에 활용할 수 있으며, 입력 데이터가 완전히 삭제되지 않고 서비스 기록에 남는 경우도 있다 [[2]](#ref2).

---

### 1-2. 접근 권한이 정리 안 돼 있으면 AI도 누구나 볼 수 있게 된다

기존 시스템에서는 "ERP는 회계만 본다, MES는 생산만 본다"는 묵시적 분리가 있었다. 그런데 RAG(Retrieval-Augmented Generation) 기반 사내 AI 챗봇이 도입되면 이 구분이 사라진다. AI가 사내 문서·DB 전체를 검색해 답변을 생성하기 때문이다.

**위험 구조**: 역할 기반 접근 통제가 없는 벡터 DB(RAG 인덱스)는 "악의적 행위자가 데이터베이스에 액세스해 개인정보 보호를 손상시킬 수 있다." 쿼리 조작으로 민감 정보를 무단 검색하는 것도 가능하다 [[3]](#ref3).

**현업 영향**: 두산밥캣 기준으로 보면, 영업 계약가·딜러 마진 데이터가 RAG 인덱스에 통으로 들어가면 현장 작업자도 AI 질문만으로 계약 조건을 꺼낼 수 있게 된다. 이는 기존에 없었던 정보 유출 경로다.

---

### 1-3. 보고서·로그·Prompt에 민감정보가 그대로 쌓인다

AI 과제를 운영하다 보면 ① 사용자가 Prompt에 고객명·계약번호를 직접 입력하거나, ② AI 출력 로그에 민감정보가 포함되거나, ③ RAG 검색 결과(컨텍스트 창)에 개인정보가 섞인다. 이것들이 로그 파일로 저장되면 "또 다른 개인정보 저장소"가 생기는 것이다.

**로그 실태**: 기업 AI 운영 현장에서 로그 시스템을 점검하면 "필요한 이벤트가 빠져 있거나, 연결이 안 되거나, 보관·열람 절차가 없어 감사 대응에 취약한 경우가 많다." 로그는 쌓이지만 그 안에 개인정보가 있는지, 누가 볼 수 있는지 정리가 안 된 상태다 [[4]](#ref4).

**규제 관점**: 개인정보보호위원회에 따르면, Prompt·AI 출력에는 고객 이름·주민등록번호·연락처뿐 아니라 건강 상태·법률 자문 내용 등 민감정보가 담길 수 있으며, 이러한 정보는 적절한 보호 조치가 필요하다 [[5]](#ref5).

---

### 1-4. 다 가리면 데이터가 쓸모없어지는 딜레마

반대로 보안을 너무 강하게 걸면 문제가 생긴다. 클레임 데이터에서 고객명만 지우는 게 아니라 결함 유형·제품군·공정 정보까지 범주화해 뭉개버리면, AI가 "어떤 공정에서 어떤 결함이 많이 나는가"를 학습할 수 없게 된다.

**핵심 딜레마**: "데이터 공유의 목적은 데이터가 담고 있는 정보를 제공하는 것이기 때문에, 프라이버시 보호와 동시에 비식별화를 통한 정보의 손실을 줄여야 하는 상반된 목표를 달성해야 한다" [[6]](#ref6).

**과도한 수집도 위험**: 보안 과제와 반대 방향의 문제도 있다. "더 많은 데이터가 성능을 높인다"는 생각에 필요 이상의 데이터를 수집하면, 유출 시 피해 범위가 늘어난다. 불필요한 데이터를 AI 파이프라인에 넣을수록 이후 삭제(데이터 권리 행사)도 어려워진다 [[7]](#ref7).

---

### 1-5. AI·RAG·Agent가 만든 새로운 유출 경로

기존 시스템에서는 "사람이 DB에 접속해서 조회한다"는 단일 경로였다. AI 환경에서는 경로가 복잡해진다:

| 경로 | 설명 |
|---|---|
| 외부 LLM 전송 | Prompt에 담긴 내용이 외부 모델 서버로 전송·저장 |
| RAG 인덱스 과다 수집 | 민감 문서를 통째로 벡터화해 인덱스에 저장 → 누구나 검색 가능 |
| AI 출력 노출 | LLM이 학습 데이터 속 개인정보를 출력에 재현 |
| Agent 실행 로그 | Tool 호출 결과·중간 추론 로그에 민감정보 포함 |
| 멀티테넌트 세션 누출 | 세션 분리 미흡 시 다른 사용자 데이터가 컨텍스트로 유입 |

**핵심 인용**: "RAG 시스템이 외부 데이터를 검색·활용하는 과정에서 개인정보 보호 문제가 발생할 수 있으며, LLM은 민감한 정보를 포함하거나 추론하는 응답을 실수로 생성해 데이터 개인정보 보호 위반으로 이어질 수 있다" [[3]](#ref3).

AI 모델은 학습 데이터의 희귀한 데이터 포인트를 기억하고, 겉보기에 무해한 입력에서 민감한 속성을 추론하며, 아무도 명시적으로 프로그래밍하지 않은 방식으로 개인정보를 출력에 드러낼 수 있다 [[7]](#ref7).

## 2. 기대 효과 — 적용 전/후 대조

### Before / After: 두산밥캣 클레임 데이터 예시

**Before (권한·보안 체계 없음)**
- 딜러 클레임 DB에 고객명·딜러 연락처·계약가·보증 조건이 원문 그대로 저장
- AI 학습에 쓰려 했지만 "개인정보가 섞여 있어 못 쓴다"는 법무·보안팀 반대로 중단
- 외부 LLM API 활용도 불허 → AI 과제 자체가 막힘

**After (가명화 + 접근 권한 체계 적용)**
- 고객명 → 가명 ID(예: `CUST-A0024`)로 치환, 딜러 연락처 삭제
- 계약가 → 가격대 범주(`$20K~$30K`)로 범주화
- 결함 유형·공정·제품군·지역 정보는 그대로 보존
- 결과: AI가 "어떤 기후대·공정 조건에서 어떤 부품 결함이 많은가"를 학습 가능해짐
- 외부 분석 파트너에게 가명화 데이터 공유 가능, 개인정보 규제 준수

**핵심 포인트**: 식별정보는 가렸지만 업무 판단에 필요한 정보(제품군·공정·결함 유형·지역 범주)는 살아 있어 AI 활용 가치가 유지됨.

## 3. When — 어디부터/우선순위

### 3-1. F-4는 "기본형" — 모든 AI 과제에 기본 적용, 단 우선순위로 접근

F-4는 "선택형"(예: 합성데이터 E-2)이 아니라 "기본형"이다. AI 과제를 하는 곳이라면 어디든 적용해야 하지만, 모든 데이터를 한 번에 처리하지 않고 위험과 활용 가능성 순으로 접근한다.

기본형 분류 근거: AI 학습·RAG·Agent 어느 맥락에서든 데이터가 쓰이면 접근 통제와 민감정보 처리가 기본적으로 필요하다. 반면 E-2(합성데이터)는 특정 데이터가 부족할 때만 선택한다.

---

### 3-2. 우선 처리 대상 — 위험이 큰 순서

아래 순서로 우선 처리한다:

**1순위: AI·외부 모델에 직접 들어가는 데이터**
- 외부 LLM API에 Prompt로 전송되는 데이터
- RAG 인덱스에 색인되는 내부 문서·DB
- 근거: 외부 전송 = 즉각적 유출 위험. "고객명, 주민번호, 영업 기밀은 처리 전 가명 처리하거나 제거해야 하며, LLM 제공사의 데이터 비학습 정책을 계약서에서 확인하는 것이 중요하다" [[8]](#ref8).

**2순위: 민감정보 밀도가 높은 데이터**
- 고객·딜러 계약 데이터 (계약가·조건·담당자)
- A/S 클레임 데이터 (고객 개인정보 포함)
- 임직원 데이터 (인사·급여)
- 근거: 재식별 가능성이 높고, 유출 시 법적 책임이 크다. 개인정보보호법상 개인정보에 해당하면 가명화 처리 후 활용하거나, 동의 없이 학습에 쓸 수 없다 [[5]](#ref5).

**3순위: AI 운영 로그·Prompt 저장소**
- AI 챗봇 대화 이력
- Agent 실행 로그, Tool 호출 결과 저장
- 근거: 운영이 시작되면 로그가 자동으로 쌓이므로, 초기에 마스킹 기준을 설계해 두지 않으면 나중에 처리할 수 없게 된다.

**4순위: 내부 전용·비정형 문서**
- 사내 보고서·회의록·설계 도면
- 개인정보보다 기술기밀·영업비밀이 주 위험
- 상대적으로 재식별 위험이 낮아 우선순위 후순위.

---

### 3-3. 목적별 차등 — 용도에 따라 처리 강도가 다르다

데이터를 어떤 목적으로 쓰느냐에 따라 비식별화·마스킹 강도가 달라진다. 최소 비식별로 최대 활용성을 유지한다는 원칙이다.

| 활용 목적 | 처리 강도 | 이유 |
|---|---|---|
| 사내 AI 학습 (내부 모델) | 가명화 (Pseudonymization) | 재식별 가능 범위 내, 학습 가능 데이터의 업무 정보 보존 필요 |
| 외부 모델(API) 전송 | 강력한 마스킹 또는 삭제 | 외부 서버 저장·학습 위험, 계약·개인정보 유출 경로 |
| RAG 인덱스 | 접근 권한 기반 분리 | 전체 마스킹보다 누가 어떤 문서에 접근하는지 통제가 더 실용적 |
| AI 평가 데이터 (E-3) | 익명화 또는 합성 대체 | 평가셋은 공유 범위가 넓어 재식별 위험 높음 |
| AI 운영 로그 저장 | 자동 마스킹 후 저장 | 로그 = 감사용, 원문 저장 최소화 원칙 |
| 외부 파트너 공유 (분석·검증) | 익명화 또는 가명화 + 결합금지 | 외부인의 재식별 공격 방어 필요 |

---

### 3-4. 기본형이지만 "한 번에 다 하지 않는다" — 단계적 접근

모든 데이터를 동시에 처리하려 하면 프로젝트가 멈춘다. 실용적 접근은:

1. **AI 과제 착수 전**: 그 과제에 쓸 데이터의 민감정보 유형 파악 → 처리 방침 결정 (3-2 순서)
2. **운영 시작 시**: 로그·Prompt 마스킹 기준 설계 (나중에 소급 처리 불가)
3. **데이터 공유 시**: 외부 전송·파트너 공유 전 반드시 처리 (가장 급한 시점)
4. **정기 점검**: 분기별 접근 권한 재검토, 재식별 위험 점검

## 4. 규제·표준 맥락 (개념 수준)

### 4-1. 개인정보보호법 — 가명정보 활용

한국 개인정보보호법(2020년 개정)은 가명정보를 "추가 정보를 사용하지 않고는 특정 개인을 알아볼 수 없도록 처리한 정보"로 정의하고, 통계작성·과학적 연구·공익 기록 보존 목적으로는 정보주체 동의 없이 활용 가능하다고 규정한다.

개인정보보호위원회는 2024년 2월 가명정보 처리 가이드라인을 개정해 이미지·영상·음성·텍스트 등 비정형 데이터에 대한 가명처리 기준을 추가했다 (기존에는 정형 데이터만 다뤘다). 처리 기준: 목적·환경·민감도를 종합적으로 고려해 합리적 위험도를 판단하고 처리 방법을 설정한다 [[9]](#ref9).

**핵심 구분 — 가명화 vs 익명화**:
- **가명화(Pseudonymization)**: 추가 정보와 결합하면 재식별 가능 → 개인정보보호법 적용 대상, 하지만 활용성 보존에 유리
- **익명화(Anonymization)**: 재식별이 불가능하게 변환 → 법 적용 대상에서 제외, 단 데이터 유용성 손실 큼

→ AI 학습에는 대부분 가명화로 충분하다. 익명화는 외부 공개·파트너 공유처럼 더 넓은 활용 범위에서 사용한다.

---

### 4-2. GDPR — 가명화/익명화 구분 (글로벌 맥락)

GDPR(EU 일반 개인정보 보호 규정)은 가명화와 익명화를 엄격히 구분한다:

- **가명화(Pseudonymization)**: 가역적(reversible). 추가 정보와 결합하면 개인 식별 가능 → 여전히 GDPR 규율 대상이나, GDPR이 적절한 기술적 보호 조치로 인정
- **익명화(Anonymization)**: 비가역적(irreversible). 식별 가능성 완전 제거 → GDPR 규율 제외

단순히 이름·이메일을 지우는 것만으로는 익명화로 인정되지 않는다. 여러 속성이 결합되면 재식별이 가능하기 때문이다 [[10]](#ref10).

두산밥캣처럼 글로벌 딜러 네트워크를 운영하는 기업은 유럽 딜러 데이터를 AI 학습에 쓸 경우 GDPR 준수 여부를 확인해야 한다.

---

### 4-3. 정리 — 규제가 요구하는 최소 수준

| 상황 | 최소 요구 |
|---|---|
| 국내 AI 학습 (개인정보 포함) | 가명화 + 보호조치 (개인정보보호법 §28-3) |
| 외부 AI 모델(API) 전송 | 가명화 또는 마스킹 (개인정보 전송 제한) |
| 유럽 딜러 데이터 활용 | GDPR 가명화 기준 충족 또는 정보주체 동의 |
| 공개 목적 데이터 배포 | 익명화 (재식별 불가 수준) |
