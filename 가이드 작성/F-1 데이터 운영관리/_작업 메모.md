## 주제 정의·합격 기준
- 정의: 데이터 수집·정제·변환·제공 파이프라인을 안정적으로 실행·복구·변경 관리하는 DataOps 체계.
- MECE 경계: **운영 인프라(파이프라인의 실행·복구·변경관리)만**. 감지=C-1, 품질 판정=C-2, 사후 추적=C-3, 수명·보존·폐기=F-2.
- Key Question 4개 → 섹션 매핑:
  - KQ1(운영 대상·중요도) → 3장(운영 대상 고르기·Tier 차등·등록표)
  - KQ2(실행·배포 표준화) → 2.2 + 4.1
  - KQ3(장애 복구) → 2.3(Runbook·재처리/롤백/우회·RTO/RPO·SLA) + 4.1/4.3
  - KQ4(변경 통제) → 2.4(데이터 계약·호환성·확장-수축) + 4.2

## 정본 모델·핵심 결정 (0.1 초안)
- **3기둥 모델로 고정**: ① 실행·배포 표준 ② 장애 대응 절차(Runbook) ③ 변경 관리. What(2.2~2.4)에서 정의하고 How(4장)에서 구축·운영 순서로 전개 — 같은 개념을 두 번 설명하지 않게, What=무엇인가/How=어떻게 순서로 분담.
- **기본형 주제**(선택형 아님): When을 Why로 흡수하지 않고 독립 3장으로 둠("어디부터/얼마나" = 중요도 차등).
- KPI 섹션 없음(신뢰성 C군만) → Where(6장)가 마지막.
- 로드맵·미래전망 섹션 없음(전 주제 제거 결정).
- **현업 실행 키트**: ㉠ 등록표(3.2 대표 + 별첨 A 전체·완성예시), ㉡ Before→After 변경 작성 규칙(4.2), ㉣ 변경 공지문·Runbook 빈 템플릿(별첨 B·C), ㉤ 플랫폼 매핑(4.4).
- **다이어그램 5종**(전부 mermaid, 파랑6/초록3): Why 사일런트 장애 cascade(pain色), 2.1 파이프라인+DataOps 받침, 2.3 Runbook 분기, 2.4 확장-수축, 4.1 구축 4단계, 4.3 변경 sequence, 6장 경계. (인라인 SVG 금지 — mermaid만 사용)

## 솔루션(1층) — 이 주제 관점
- 중심 = 오케스트레이터. 1차 갈림 = 이식성(오픈소스 셀프호스트: Airflow/Dagster/Prefect/NiFi) vs 운영 위임(매니지드). 2차 = 인프라 정합(AWS=MWAA/StepFunctions, Azure=ADF, GCP=Composer, Databricks=Workflows).
- 변경관리 = 단일 제품 X → Git + CI/CD(GitHub Actions/GitLab) + 스키마 호환성(Confluent SR)/데이터 계약(datacontract.com) 조합.
- Great Expectations·Soda는 **C-2 품질** 영역 → "배포 게이트"로만 짧게 인용(경계 명시).
- 2층 정본 reflect-back은 **사용자 명시 요청 시에만** (아직 안 함).
