## 1. 개요

데이터 전처리는 문서·표·이미지처럼 비정형·반정형으로 쌓인 데이터를 AI가 읽고 검색·분석할 수 있는 구조화된 형태로 바꾸는 활동이다. 품질 성적서·설비 점검표·작업표준서·운전 로그는 사람에게는 자연스럽게 읽히지만, AI에게는 어디까지가 한 문단인지, 어떤 숫자가 표의 어느 칸 값인지가 파일에 드러나 있지 않다. 전처리는 이 숨은 구조를 복원해 넘긴다.

이는 파서를 만드는 기술 과제가 아니라 AI-ready Data 관점의 데이터 준비 활동이다. 가이드도 추출 알고리즘이 아니라, 어떤 문서를 어떤 순서로 전처리해 재사용 가능한 데이터 자산으로 만들지를 다룬다.

<a id="s11"></a>

### 1.1 적용 범위와 경계

전처리는 이미 디지털로 존재하는 자료를 AI가 읽을 구조로 바꾸는 자리다. 인접 활동과 경계를 분명히 해 두면 책임이 겹치지 않는다.

- 종이 문서를 대량으로 스캔·OCR해 디지털화하는 일은 [F-3 데이터 디지털화](../F-3%20데이터%20디지털화/F-3%20데이터%20디지털화.md)가 맡는다. 전처리는 그 결과(디지털 자료)를 입력으로 받는다.
- 변환된 데이터에 결함 유형 같은 의미(정답) 라벨을 붙이는 일은 [B-2 데이터 해설·주석](../B-2%20데이터%20해설·주석/B-2%20데이터%20해설·주석.md)이 맡는다.
- 변환 결과를 AI에 써도 되는지 품질을 판정하는 일은 [C-2 데이터 품질 관리](../C-2%20데이터%20품질%20관리/C-2%20데이터%20품질%20관리.md)가 맡는다.

전처리는 형식 변환까지를 책임 범위로 둔다. (인접 주제와의 관계는 [6장](#s6)에서 정리한다.)

<a id="s12"></a>

### 1.2 AI-ready Data 체계 내 위치

전처리는 데이터를 이해할 수 있게(Understandable) 만드는 그룹의 첫 관문이다. 원본 문서를 AI가 읽을 수 있는 형태로 바꾼 뒤, 그 위에서 주석이 의미를 입히고 품질 관리가 사용 가능 여부를 판정한다.

```mermaid

    B2["B-2 주석<br/>의미를 입힘"]:::n
```

학습용 라벨이 필요 없는 검색·RAG(Retrieval-Augmented Generation, AI가 답할 때 우리 문서를 찾아 근거로 함께 쓰는 방식)용 자료는 전처리 결과가 곧바로 활용 환경으로 가기도 한다(위 점선). 20개 주제 전체 조감도는 [전체 목차](../../전체%20목차/00%20전체%20목차%20(20개%20주제).md)에 있다.

## 2. 왜 필요한가 (Why)

데이터가 아무리 많아도 AI가 읽지 못하는 형식이면 활용으로 이어지지 않는다. 계열사 인터뷰에서도 AI 과제의 초기 부담 상당수가 모델이 아니라 문서를 데이터로 바꾸는 단계에 몰려 있었다. 전처리는 이 부담을 표준화해, 쌓여 있지만 못 쓰던 문서를 AI가 바로 쓰는 데이터로 전환한다.

<a id="s21"></a>

### 2.1 현업에서 막히는 지점

제조 현장에서 전처리 없이 AI 과제를 진행할 때 반복적으로 부딪히는 문제는 다음과 같다.

- 같은 신호인데 표기가 제각각이다: 발전소 운전 데이터는 계측기 소스, 제어시스템 입출력, 평균값과 원본값이 섞이고 타임스탬프 기준이 달라, 동일 신호도 분석 전에 매번 다시 맞춰야 한다.
- 분석 전 정제에 수 주가 걸린다: 호기별 성능 테스트의 Raw Data는 제품 한 대에 40~50개 파일, 초당 수백 개 태그가 쌓인다. 내부 엔지니어가 병합·추출 프로그램을 만드는 데만 수 주가 소요된다.
- 성적서·매뉴얼이 PDF·이미지로만 존재한다: 시험성적서, 기술문서, 정비 매뉴얼이 PDF·이미지·스캔본이라 OCR과 파싱을 거치지 않으면 검색에 넣을 수 없다. 오래된 스캔 문서는 글자 인식이 안 돼 담당자가 Word로 다시 만드는 경우도 있다.
- 도면 속 텍스트와 객체가 분리된다: Wiring Diagram, CAD 도면은 부품명·부품번호·도면 객체의 연결이 끊겨 있어, 정비 RAG에 그대로 넣으면 부품과 절차가 이어지지 않는다.
- 핵심 데이터가 수기 Excel에만 있다: 생산계획·품질 데이터의 상당 부분이 수기 Excel로 관리되어, 갱신 주기와 책임자가 불명확하고 AI가 읽을 표 구조로 정리돼 있지 않다.

이 문제들의 공통점은 데이터가 없는 것이 아니라, 있는 데이터가 AI가 읽을 형식으로 정리돼 있지 않다는 데 있다.

<a id="s22"></a>

### 2.2 전처리로 얻는 것

전처리를 표준화하면 세 가지가 달라진다.

- 문서 속 정보가 검색·분석 가능한 데이터로 살아난다. 사람만 해석하던 표·수치·문단이 구조화되어, AI가 바로 검색·집계·분석한다. 폴더에 잠자던 성적서 수천 건이 표를 행·열로 살리고 페이지 위치를 붙이는 순간 불량 원인 질의응답의 검색 자료가 된다.
- 근거를 추적할 수 있어 신뢰가 올라간다. 원본 위치 정보(페이지·표·좌표)를 보존하면 AI가 답과 함께 출처 문서·페이지를 제시한다. 현업이 원본을 열어 확인할 수 있고, 이 위치 정보는 답변 근거 추적([C-3 데이터 계통 Lineage](../C-3%20데이터%20계통%20Lineage/C-3%20데이터%20계통%20Lineage.md))의 출발점이 된다.
- 한 번 만든 변환 자산을 반복 재사용한다. 표준 출력 포맷과 안정장치(파서 버전·골든셋)를 갖추면 같은 양식의 문서는 사람 손 없이 자동으로 변환된다. 전처리가 일회성 수작업이 아니라 반복 가동되는 데이터 자산이 된다.

## 3. 무엇을 갖추나 (What)

이 장은 전처리가 무엇으로 이루어지는지를 설명한다. 구축 절차와 유형별 처리 방법은 5장에서 다루고, 여기서는 전처리를 구성하는 요소와 그 산출물의 형태만 정의한다.

<a id="sec31"></a>

### 3.1 전처리 변환 사슬

전처리는 한 번에 끝나는 작업이 아니라, 원본 문서 한 건이 AI-ready 데이터가 되기까지 거치는 네 단계의 연속이다. 가이드 전체가 이 네 단계를 전처리의 일반론으로 삼아 일관되게 사용한다.

```text
원본 문서 1건
→ ① 추출(Extract)      텍스트·표·이미지·수치를 뽑아냄
→ ② 구조화(Structure)  제목·문단·표로 분류하고 표는 행·열을 복원
→ ③ 위치보존(Provenance) 문서ID·페이지·좌표를 꼬리표로 부착
→ ④ 청킹(Chunk)        검색에 알맞은 단위로 분할
```

구조화까지 살려야 AI가 어느 수치가 어느 칸 값인지 알고, 위치보존이 있어야 답의 근거를 원본으로 역추적할 수 있다. 청킹 크기는 너무 크면 검색이 부정확하고 너무 작으면 문맥이 끊긴다. 단순 검색은 위치보존까지로 충분할 때도 있고, RAG는 청킹까지 간다.

추출·구조화를 실제로 해내는 방법은 규칙 기반 코드 파서와 LLM 위임 두 가지이며, 선택 기준은 [5.1](#s51)에서 다룬다.

<a id="kq3"></a>

### 3.2 구조화 산출물

전처리의 목적은 묻힌 내용을 AI가 읽을 수 있는 구조와 원본 위치 정보로 바꾸는 것이다. JSON·Markdown·표는 그 구조를 담는 그릇일 뿐, 그릇 자체가 목적은 아니다. 그래서 산출 형식은 정해져 있지 않고, 그 데이터를 어디에 쓸지에 따라 고른다.

- RAG·의미 검색에 넣을 때는 사람이 읽듯 자연스러운 Markdown 청크로 내보낸다.
- 집계·BI·대시보드에는 한 행이 한 관측인 레코드(표)로 만든다.
- 벡터 DB 입력이나 프로그램 처리에는 요소별 타입·좌표를 담은 JSON으로 만든다.
- 병합셀이 있는 표는 구조를 정확히 보존하는 HTML·Markdown 표로 둔다.

```mermaid
%%{init: {'theme':'base','themeVariables':{'primaryColor':'#DCE8F5','primaryBorderColor':'#2456A4','primaryTextColor':'#10243f','lineColor':'#2456A4','fontSize':'13px'}}}%%
flowchart LR
    DOC["원본 문서<br/>PDF·성적서·로그"]:::src
    CORE["구조화 핵심<br/>content + provenance<br/>(무엇을 + 어디서)"]:::cat
    DOC -->|추출·구조화·위치보존| CORE
    CORE -->|RAG·의미 검색| MD["Markdown 청크"]:::n
    CORE -->|벡터 DB·프로그램| JS["JSON 레코드"]:::gate
    CORE -->|집계·BI| TB["표(레코드)"]:::loop
    classDef n fill:#B7CDE6,stroke:#6B9AD1,color:#10243f;
    classDef src fill:#79C3E8,stroke:#3F9BD4,color:#0b3a52;
    classDef cat fill:#2456A4,color:#fff,stroke:#163a73,stroke-width:2px;
    classDef gate fill:#3F9BD4,stroke:#2456A4,color:#fff;
    classDef loop fill:#4FA47D,stroke:#2f6b50,color:#fff;
```

형식이 무엇이든 변하지 않는 핵심은 가운데의 두 가지다. 산출물의 한 조각인 청크는 무엇을 담았는지(content)와 어디서 왔는지(provenance)를 함께 갖춰야 한다. 이 두 가지가 있어야 검색되고 근거를 댈 수 있다. (이 가이드의 예시 결과가 JSON이 많은 까닭은 구조와 출처를 한눈에 드러내기 위해서이며, 같은 청크라도 RAG에 넣을 때는 Markdown으로 내보내는 경우가 많다.)

| 항목 | 의미 | 예시값 | 필수/선택 | 작성 주체 |
|---|---|---|---|---|
| chunk_id | 청크 고유 식별자 | QR-2024-0123_p45_tbl2_r07 | 필수 | 파이프라인(자동) |
| type | 요소 유형(문단·표행·슬라이드·메일 등) | table_row | 필수 | 파이프라인(자동) |
| content | 실제 내용(텍스트 또는 표 직렬화) | 용접부/외관/.../합격 | 필수 | 파이프라인(자동) |
| provenance.doc_id | 출처 문서 ID | QR-2024-0123 | 필수 | 파이프라인 + [A-1 카탈로그](../A-1%20데이터%20카탈로그/A-1%20데이터%20카탈로그.md) 연계 |
| provenance.위치 | 페이지·표·행·좌표 | p45, tbl2, r7 | 권장 | 파이프라인(자동) |
| section/title | 상위 섹션·제목 맥락 | 2.1 예열 | 선택 | 파이프라인(자동) |

청킹 단위는 문서 유형에 따라 다르다. PDF·Word 보고서는 제목(섹션) 단위로 나누되 표는 통째로 별도 청크로 두고, PPT는 슬라이드 한 장, Excel·CSV는 표 또는 레코드(행) 단위, 이메일은 정리된 본문 한 통, 회의록은 한 화제(연속 발화) 단위로 묶는다. 표를 행 단위로 쪼개면 머리글과 데이터가 분리되어 어느 열 값인지를 잃으므로, 표는 Markdown이나 HTML로 직렬화해 한 청크로 둔다.

> **권장 — 청크 크기 시작값.** 대략 250토큰에 20~50토큰의 겹침(overlap)으로 시작해 조정한다. 검색이 엉뚱한 조각을 물어오면 청크를 줄이고, 답에 문맥이 잘려 보이면 늘린다. 250토큰은 한글로 짧은 한두 문단 분량이다. (청킹 전략 비교는 [별첨](#s8).)

<a id="s4"></a>

## 4. 어디부터 하나 (Where)

모든 문서를 한 번에 전처리하지 않는다. AI가 바로 읽기 어렵고, 활용 가치가 크며, 반복 사용이 많은 문서부터 골라서 착수한다.

<a id="kq1"></a>

### 4.1 전처리 대상

전처리 대상은 이미 디지털이지만 AI가 바로 읽기 어려운 자료다. 제조 현장에서 가장 자주 만나는 유형은 다음과 같다.

- 품질 성적서·기술문서(PDF): 표·다단 레이아웃·페이지 구조가 묻혀 있다.
- 기술 제안서·교육자료(PPT): 정보가 슬라이드·도형·표로 나뉘어 담긴다.
- 설비 점검표·집계표(Excel): 병합셀·다중 시트·색 의미로 구조가 복잡하다.
- 스캔 문서·도면·이미지: 글자가 픽셀로만 있어 OCR과 객체 인식이 필요하다.
- 운전·설비 로그(CSV): 인코딩·구분자·단위 행이 섞여 그대로는 깨진다.

이메일·작업일지·회의록도 같은 이유로 대상이며, 유형별 처리는 [5.2](#kq2)에서 다룬다.

> **주의 — DB 테이블과 내보낸 CSV의 경계.** 이미 깔끔한 정형 테이블(MES·ERP의 DB 테이블)은 구조가 살아 있어 전처리 대상이 아니다. 등록·탐색은 [A-1 카탈로그](../A-1%20데이터%20카탈로그/A-1%20데이터%20카탈로그.md)가 맡는다. 반면 그 테이블을 파일로 내보낸 CSV·설비 로그는 인코딩·구분자·단위 행이 섞여 깨지므로 전처리 대상이 맞다.

<a id="s42"></a>

### 4.2 우선순위

대상을 정했으면 **AI 활용 가치**와 **변환 난이도** 두 축으로 점수를 매겨 우선순위를 정한다. 두 축은 다음 기준으로 평가한다.

- **AI 활용 가치(높음/중간/낮음):** ① AI 과제(검색·QA·집계)가 이 문서를 자주 참조하는가, ② 여러 팀·계열사가 함께 쓰는가, ③ 같은 양식이 반복·대량이라 자동화 효과가 큰가. '예'가 많을수록 가치가 높다.
- **변환 난이도(낮음/중간/높음):** 텍스트 위주·단순 표는 낮음, 다단 레이아웃·테두리 없는 표는 중간, 스캔·손글씨·병합 다중·색에 담긴 의미는 높음. 사외 반출이 제한되면 로컬 도구로만 처리해야 하므로 난이도가 올라간다.

후보 문서를 두 축으로 평가하면 착수 순위가 나온다.

| 후보 문서 | AI 활용 가치 | 변환 난이도 | 착수 순위 |
|---|---|---|---|
| 작업표준서(텍스트 PDF) | 높음 | 낮음 | 1 — 즉시 |
| 품질 성적서(PDF 표) | 높음 | 중간 | 1 — 즉시 |
| 설비 점검표(Excel 병합·색) | 중간~높음 | 높음 | 2 — 골든셋 갖춘 뒤 |
| 스캔 도면(이미지+치수) | 중간 | 매우 높음 | 3 — 도구 검토 후 |

```mermaid
%%{init: {'theme':'base','themeVariables':{'primaryColor':'#DCE8F5','primaryBorderColor':'#2456A4','primaryTextColor':'#10243f','lineColor':'#2456A4','fontSize':'13px'}}}%%
```

우선순위는 사분면으로 정리된다. **가치가 높고 난이도가 낮은 왼쪽 위(최우선 착수)부터** 손대고, 가치는 높지만 난이도가 높은 오른쪽 위(전략 과제)는 골든셋·도구를 갖춘 뒤 단계적으로 옮겨 간다. 가치가 낮은 아래쪽은 후순위로 미룬다.

그래서 두산에너빌리티라면 변환이 쉬운 작업표준서와 활용 가치가 가장 큰 품질 성적서를 **함께 1순위로** 착수한다. 병합·색이 섞인 설비 점검표는 대표 양식의 골든셋을 만든 뒤 2순위로, 스캔 도면 속 치수처럼 OCR·손글씨가 얽힌 자료는 도구를 검토한 뒤 3순위로 옮긴다. 한꺼번에 다 하지 않고, 효과가 빨리 나는 1순위로 기준선을 만든 다음 어려운 양식으로 넓혀 간다.

## 5. 어떻게 구축·운영하나 (How)

이 장은 전처리를 실제로 어떻게 수행하는지를 다룬다. 먼저 전체 구축 흐름과 두 가지 실행 방식을 정한 뒤, 문서 유형별로 무엇이 어렵고 어떻게 처리하는지를 계열사 사례와 함께 보여준다. 이어서 어려운 양식을 LLM에 위임하는 방법, 안정장치와 도구, 적재, 운영을 설명한다.

<a id="s51"></a>

### 5.1 구축 흐름과 두 가지 실행 방식

전처리 파이프라인은 대상 선정에서 적재까지 여섯 단계로 구축한다. [3.1](#sec31)의 변환 사슬(추출·구조화·위치보존·청킹) 앞뒤에 대상 선정과 적재를 더한 것이다.

```mermaid

    A2["② 추출"]:::n
```

추출과 구조화를 실제로 해내는 길은 두 가지다. 어느 길을 쓸지는 양식이 일정한지 문서마다 제각각인지로 갈린다.

- 규칙·도구 기반 파서(코드): 양식이 일정하고 대량일 때 쓴다. 빠르고 저렴하며 같은 입력에 늘 같은 결과를 낸다.
- LLM 기반 추출: 병합·계층·조건부 기준처럼 양식이 까다롭거나 문서마다 제각각일 때 쓴다. 사람이 읽는 방식을 단계로 쪼개 LLM에 위임한다.

어려운 문서일수록 LLM 위임이 주력이며, 그 방법은 [5.3](#sec62)에서 다룬다. 위치보존과 청킹은 어느 길을 쓰든 동일하게 뒤에 붙인다. 각 단계의 산출물은 구조화된 청크이고, 완료 기준은 골든셋 회귀 테스트와 스키마 검증을 통과하는 것이다([5.4](#kq4)).

### 5.2 문서 유형별 전처리

문서 유형마다 내부 구조가 달라 무엇을 어떻게 뽑을지도 다르다. 아래는 자주 만나는 여덟 유형을 무엇이 어렵고 어떤 도구로 처리하며 결과가 어떻게 달라지는지 가상 예시로 보여준다. 예시는 구조·출처를 드러내려 JSON으로 적었지만, RAG에 넣을 땐 Markdown으로 내보내기도 한다([3.2](#kq3)). 코드·프롬프트는 [별첨 — 유형별 실행 레시피](별첨/B-1%20별첨%20—%20유형별%20실행%20레시피.md)에 있다.

| 유형 | 핵심 난점 | 권장 처리 | 대표 도구 |
|---|---|---|---|
| PDF 성적서·보고서 | 읽기 순서·표 구조가 파일에 없음 | 레이아웃 인식 파싱 + 표 격자 복원 | Docling·Unstructured / Camelot·pdfplumber |
| PPT 기술자료 | 텍스트박스 순서가 뒤섞임, 차트는 내장 데이터 | 슬라이드 1장을 의미 단위로 묶기 | python-pptx |
| Excel 점검표 | 병합셀 빈칸, 색에 담긴 의미, 가로 교차표 | 병합 풀고 교차표를 레코드로 펴기 | openpyxl·pandas |

#### 5.2.1 PDF 성적서·보고서

PDF는 인쇄 레이아웃 기반이라 제목·문단·표라는 논리 구조가 파일에 없다. 그래서 글자만 순서대로 잇는 단순 추출은 표를 뭉개고 다단을 뒤섞는다. 텍스트 PDF와 스캔 PDF를 먼저 구분하고, 레이아웃 인식 파싱으로 페이지를 제목·문단·표·그림 블록으로 분류한 뒤, 표는 테두리 유형에 맞춰 행·열을 복원한다. 반복되는 머리글·쪽번호는 걸러낸다.

**권장 도구.** 레이아웃과 표를 함께 인식하는 [Docling](https://github.com/docling-project/docling)이나 [Unstructured](https://unstructured.io)를 1차 파서로 쓰고, 표가 까다로우면 표 전용인 [Camelot](https://camelot-py.readthedocs.io)·[pdfplumber](https://github.com/jsvine/pdfplumber)로 보완한다. 스캔 PDF는 OCR이 강한 클라우드([Azure Document Intelligence](https://azure.microsoft.com/ko-kr/products/ai-services/ai-document-intelligence)·[Google Document AI](https://cloud.google.com/document-ai))로 보낸다. 출력은 본문은 Markdown, 표는 좌표를 보존한 JSON으로 받는다.

#### 5.2.2 PPT 기술자료

PPT는 한 슬라이드 안에 제목·본문·도형·표·차트가 여러 텍스트박스로 나뉘어 있고, 텍스트박스가 화면 위치가 아니라 만든 순서로 저장돼 있어 그대로 뽑으면 순서가 뒤섞인다. 슬라이드 단위로 순회하며 제목·본문·표·발표자 노트를 차례로 뽑고, 텍스트박스를 좌표 기준으로 다시 정렬한다. 슬라이드 한 장을 청크 단위로 삼고 슬라이드 번호를 위치 정보로 붙인다. 차트는 글자가 아니라 내장 데이터이므로 데이터값을 표로 뽑는다.

**권장 도구.** [python-pptx](https://python-pptx.readthedocs.io)로 슬라이드를 순회하며 제목·본문·표·발표자 노트와 도형 좌표(읽기 순서 재정렬용)를 읽는다. 차트는 슬라이드에 박힌 데이터(내장 XML)에서 값을 뽑고, 그림만 있는 슬라이드는 OCR로 보낸다.

두산에너빌리티 기술 교육자료(가상)의 한 슬라이드를 보자.

```
[전처리 전] 제목·불릿·작은 표·발표자 노트가 한 화면에 따로 놓여 있다

[전처리 후] 슬라이드 1장을 한 청크로 묶고 제목·본문·표·노트를 분류
```

흩어진 정보가 의미 단위로 묶여, 언더컷 검출법을 묻는 질문에 슬라이드 7을 근거로 답한다.

#### 5.2.3 Excel 점검표

Excel은 사람이 보기 좋게 만든 탓에 AI에게는 가장 까다롭다. 병합셀은 빈칸으로 나오고, 색에 담긴 의미는 기계가 읽지 못하며, 가로로 월·항목을 펼친 교차표는 한 행이 한 관측이 아니라 다루기 어렵다. 실제 데이터 표의 머리글 위치를 먼저 잡고, 병합셀은 값을 아래·옆으로 채우며, 색·서식의 의미는 별도 컬럼으로 데이터화한다. 가로 교차표는 세로 레코드로 펴서 한 행이 한 관측이 되게 한다.

**권장 도구.** [openpyxl](https://openpyxl.readthedocs.io)로 병합 범위·셀 색·수식 결과값을 직접 읽고(색은 채움 색을 값으로 변환), [pandas](https://pandas.pydata.org/docs/reference/api/pandas.read_csv.html)로 가로 교차표를 세로로 펴(`melt`) 한 행이 한 관측이 되게 한다. 설비마다 열 구성이 다르면 양식별 파서를 두고 골든셋으로 관리한다.

두산밥캣 설비 점검표(가상)는 윤활 카테고리가 병합되고 주차가 가로로 펼쳐지며 이상이 색으로만 표시된다.

#### 5.2.4 이미지·스캔본·도면

스캔 문서·사진·도면은 글자가 픽셀로만 있어 OCR이 필수다. 기울기·대비를 보정해 인식 정확도를 올리고, OCR로 텍스트화하며 표는 표 구조 인식(TSR)으로 행·열을 복원한다. 셀별 신뢰도를 함께 받아 낮은 셀은 사람 검수로 보낸다. 도면은 텍스트와 함께 부품·기호 같은 객체가 섞여 있어, 객체 사이의 관계까지 구조화하는 일은 [B-3 온톨로지](../B-3%20온톨로지/B-3%20온톨로지.md)와 연계한다.

**권장 도구.** 로컬은 [Tesseract](https://github.com/tesseract-ocr/tesseract)(pytesseract)·PaddleOCR, 정확도나 표 인식이 중요하면 클라우드 OCR([Azure Document Intelligence](https://azure.microsoft.com/ko-kr/products/ai-services/ai-document-intelligence)·[Google Document AI](https://cloud.google.com/document-ai)·[AWS Textract](https://aws.amazon.com/textract/))을 쓴다. 표 구조는 클라우드 OCR이나 표 인식 모델로 행·열을 복원하고, 단어별 신뢰도를 받아 낮은 값에 검수 플래그를 단다.

두산에너빌리티 현장의 손글씨 섞인 검사 일지(가상)를 보자.

#### 5.2.5 이메일 (Outlook)

이메일은 협력사 품질 이슈, 변경 통지, 클레임 회신이 오가는 1차 기록이지만 AI가 읽기에는 손이 가장 많이 가는 유형이다. 핵심 정보가 헤더·본문·첨부에 나뉘어 있고, 본문에는 답장 인용·서명·면책 문구가 섞이며, 검사 결과는 표나 이미지로 본문에 박혀 있기 때문이다. 그래서 "본문 텍스트만" 뽑으면 출처도, 표도, 첨부 성적서도 사라진다. 처리 흐름은 데이터 구조 파악 → 추출 → 메타데이터 분리 → 본문 정리 → 표·이미지·첨부 라우팅 순이다.

**데이터가 어떻게 들어있나.** Outlook 메일은 보통 세 형태로 존재한다.

- `.msg` — 메일 한 통을 Outlook이 저장한 파일. 헤더·본문(평문/HTML)·첨부·내장 이미지가 한 파일에 담긴다.
- `.pst`/`.ost` — 메일함 전체(여러 폴더·수천 통)를 담은 보관 파일.
- `.eml`(MIME) — 표준 형식. 한 메일이 여러 부분(part)으로 나뉜다: 헤더, `text/plain`(평문 본문), `text/html`(표·서식이 든 본문), 그리고 첨부·내장 이미지가 각각 별도 part로 붙는다.

공통점은 한 메일이 헤더 + 평문/HTML 본문 + 첨부 + 내장 이미지의 여러 조각으로 이루어진다는 점이다.

**어떻게 추출하나(도구).** `.msg`는 [extract-msg](https://github.com/mattgwwalker/msg-extractor), `.pst`는 [readpst(libpst)](https://www.five-ten-sg.com/libpst/)로 개별 메일로 풀고, `.eml`은 파이썬 표준 [email](https://docs.python.org/3/library/email.html) 라이브러리로 part를 분해한다. 메일함에 직접 접근할 수 있으면 Microsoft Graph API로 메일·첨부·대화 ID를 구조화된 형태로 받는 편이 안정적이다(파일 변환 단계를 건너뛴다).

**메타데이터를 어떻게 가져오나.** 헤더에서 검색·추적에 쓸 필드를 그대로 뽑아 메타데이터로 둔다.

- 보낸이·받는이·참조(From·To·Cc)·날짜(Date)·제목(Subject) → 검색 필터로 쓴다.
- Message-ID·In-Reply-To·References·Conversation-ID → 같은 사안의 메일을 한 스레드로 묶는다.

이 메타데이터가 있어야 "협력사 OO가 보낸 로트 L-2026-0312 관련 메일"처럼 사람이 묻는 방식으로 검색되고, 주고받은 메일이 한 건으로 이어진다.

**본문을 AI가 이해하게.** HTML 본문은 태그를 걷어 평문화하되 구조는 살리고([html2text](https://github.com/Alir3z4/html2text)), 인용된 과거 메일(`>`·"On … wrote:")과 서명·면책 블록을 잘라 이번 메일의 새 내용만 남긴다([talon](https://github.com/mailgun/talon)·[email-reply-parser](https://github.com/zapier/email-reply-parser)). 규칙으로 깨끗이 안 떨어지면 LLM에 "최신 본문·인용·서명"을 나누게 한다([5.3](#sec62)).

**본문 속 표·이미지는 이렇게 한다.**

- 표: 검사 결과 표는 HTML 본문 안 `<table>`로 들어 있다. 평문화할 때 버리지 말고 Markdown 표로 변환해 본문 청크에 함께 유지한다.
- 이미지: 본문에 박힌 내장 이미지(HTML이 `cid:`로 참조하는 그림)와 진짜 첨부를 구분한다. 내장 이미지는 본문 흐름에 자리표시자로 남기되, 표·스캔·그래프가 찍힌 이미지면 [이미지 경로](#t-img)(OCR)로 보낸다.
- 첨부: 진짜 첨부(PDF·Excel)는 떼어 각 유형 전처리로 보내고, 본문과는 첨부 ID로 연결해 "이 메일이 이 성적서를 가리킨다"가 유지되게 한다.

한 통의 메일이 다섯 갈래로 나뉘어 처리되는 모습을 그림으로 보면 이렇다.

#### 5.2.6 Word·HWP 작업표준서

작업표준서(SOP)와 기술문서는 제목이 큰 글씨가 아니라 스타일(제목1·제목2)로만 구분되는 경우가 많아, 단순 추출하면 제목과 본문이 평평하게 뭉개진다. 문단 스타일을 읽어 제목 계층으로 옮기면 제목 단위 청킹이 쉬워진다. 표는 행·열로, 반복 머리말·쪽번호는 걸러낸다. HWP는 파서가 빈약해 텍스트·XML로 1차 추출하거나 PDF로 변환해 PDF 경로로 처리한다.

**권장 도구.** docx는 [python-docx](https://python-docx.readthedocs.io)로 문단 스타일을 읽거나 [Mammoth](https://github.com/mwilliamson/python-mammoth)로 Markdown으로 변환한다. HWP는 [hwp5(pyhwp)](https://github.com/mete0r/pyhwp)나 HWPX(압축 해제)로 텍스트를 뽑고, 깨지면 LibreOffice로 PDF로 변환해 PDF 경로([5.2.1](#t-pdf))로 보낸다.

두산에너빌리티 용접 작업표준서(가상)를 보자.

```
[전처리 전] 제목이 스타일로만 구분되고, 절차가 번호 목록, 중간에 주의 표가 끼임

[전처리 후] 스타일을 제목 계층으로, 절차는 단계 리스트로, 표는 행·열로
  {"chunk_id":"SOP-용접-03_2.1","heading":["용접 작업표준","2. 본작업","2.1 예열"],
   "steps":["1. 예열 토치로 150℃까지","2. 온도계로 확인","3. 기록지에 기입"],
   "caution_table":"| 항목 | 기준 |\n| 예열온도 | 150±10℃ |","provenance":{"doc_id":"SOP-용접-03"}}
```

평평하게 뭉개졌던 SOP가 어느 작업의 몇 단계인지 계층으로 살아나, 예열 온도 기준을 절·표 근거와 함께 찾는다.

#### 5.2.7 CSV·운전 로그

설비 로그·운전 시계열 CSV는 정형처럼 보이지만 인코딩 깨짐, 구분자 혼재, 단위 표기 행, 가로로 펼친 교차표 때문에 그대로는 쓰지 못한다. 인코딩과 구분자를 먼저 확정하고(한글 깨짐의 주원인), 머리글 위치와 단위 행을 정리한 뒤, 타임스탬프를 표준 형식으로 맞추고 결측·단위를 정규화한다. 센서별로 가로로 펼친 값은 한 행이 한 관측이 되게 세로로 편다. 발전소 운전 데이터처럼 동일 신호의 태그명·타임스탬프가 제각각인 경우, 이 정규화가 분석의 전제가 된다.

**권장 도구.** [charset-normalizer](https://github.com/jawah/charset_normalizer)로 인코딩을 감지하고 [pandas](https://pandas.pydata.org/docs/reference/api/pandas.read_csv.html)로 읽되, 머리말·단위 행을 건너뛰고 전부 문자열로 받은 뒤 숫자를 정제한다. 가로 교차표는 `melt`로 세로로 펴고, 초당 수백 태그 같은 초대용량은 청크 단위(`chunksize`)로 나눠 처리한다.

설비 진동 시계열(가상)을 보자.

```
[전처리 전] cp949 인코딩, 머리글 위 장비정보 2줄, 단위 행, 값에 콤마·단위
  # 설비: PRESS-07   추출: 2026-06-15
  timestamp,vibration,temp
  [-],[mm/s],[℃]
  2026-06-15 09:00:01,"1,024",47.5℃
  2026-06-15 09:00:01,"1,024",47.5℃     ← 중복

[전처리 후] 인코딩 감지 → 단위 행 제거 → 숫자 정제 → 타임스탬프 표준화·중복 제거 → 1분 평균
  [{"ts":"2026-06-15T09:00:00Z","설비":"PRESS-07","vibration":1024.0,"temp":47.5}]
```

깨진 한글·중복·단위 섞인 값이 깔끔한 수치 시계열이 되어, PRESS-07 진동 이상 추세를 바로 분석한다.

#### 5.2.8 채팅·회의록

메신저 로그·회의록은 발화자 표기가 제각각이고, 잡담·중복이 섞이며, 한 줄에 여러 발화가 붙는다. 전처리는 발화자·시각을 분리해 발화 단위로 정렬·정돈하는 형식 변환까지를 책임진다. 패턴에서 시각·발화자·내용을 뽑고, 끊긴 연속 발화를 합치며, 시스템 줄과 잡담을 걸러낸다. 결정사항·요약 같은 의미 추출은 주석([B-2](../B-2%20데이터%20해설·주석/B-2%20데이터%20해설·주석.md))이 맡는다.

**권장 도구.** 메신저 로그는 발화 패턴(`[시각] 발화자: 내용`) 정규식으로 분해하고, 회의 자막은 [webvtt-py](https://github.com/glut23/webvtt-py)로 `.vtt`를 파싱한다. 별칭·사번이 섞이면 정규화 후 표준 용어로 맞춘다.

품질 회의 로그(가상)를 보자.

```
[전처리 전] 발화자 표기 제각각, 연속 발화 줄바꿈, 시스템 줄
  [10:03] 김검사: 로트 0312 표면 결함이요
  재현됩니다
  [10:04] 이엔지(eng.lee): RT 다시 돌려보죠
  홍길동님이 입장했습니다

[전처리 후] 발화자·시각 분리, 별칭 정규화, 연속 발화 병합, 시스템 줄 제거
  [{"time":"10:03","speaker":"김검사","text":"로트 0312 표면 결함이요 재현됩니다"},
   {"time":"10:04","speaker":"이엔지","text":"RT 다시 돌려보죠"}]
```

잡음 없는 발화 단위가 되어 검색·인용이 쉽다. 발화자 표기 불일치(이름·사번·별칭)는 [A-3 비즈니스 Glossary](../A-3%20비즈니스%20Glossary/A-3%20비즈니스%20Glossary.md)의 표준 용어로 맞춘다.

### 5.3 어려운 문서 — 사람의 수작업 단계 분해 후 LLM 위임

먼저 **사람이 이 문서를 손으로 처리한다면 어떤 단계를 거칠지** 분석하고, 그 단계를 그대로 LLM에 시킨다. 병합·계층·제각각인 양식은 규칙 코드로 짜기 어렵지만, 사람의 정리 절차를 단계로 쪼개 위임하면 넘어간다. 핵심은 두 번에 나누는 것이다 — 뽑을 항목(스키마)을 먼저 정해 고정하고, 그다음 그 스키마대로 값만 채우게 한다. 한 번에 "알아서 정리"를 시키면 문서마다 구조가 달라져 쓸 수 없다.

역할은 나뉜다. 사람은 단계를 쪼개고 스키마를 검수·확정하며 애매한 셀·예외를 판단하고, LLM은 스키마 초안을 제안하고 값을 추출해 문서 수천 건에 반복 적용한다. 챗봇을 만드는 게 아니라 LLM을 전처리 도구로 쓰는 데이터 준비 활동이다.

```mermaid
%%{init: {'theme':'base','themeVariables':{'primaryColor':'#DCE8F5','primaryBorderColor':'#2456A4','primaryTextColor':'#10243f','lineColor':'#2456A4','fontSize':'12px'}}}%%
flowchart LR
    DOC["병합 작업표준 엑셀<br/>(코드로는 까다로움)"]:::src
    H1["사람: 단계 분해<br/>한 건 = ?"]:::n
    P1["LLM ①<br/>스키마 초안 제안"]:::gate
    H2["사람: 스키마<br/>검수·확정"]:::n
    P2["LLM ②<br/>스키마대로 값 추출"]:::cat
    V["검증<br/>행수·필수값·표준값"]:::loop
    DOC --> H1 --> P1 --> H2 --> P2 --> V
    V -.어긋나면 스키마·프롬프트 보정.-> H2
    classDef n fill:#B7CDE6,stroke:#6B9AD1,color:#10243f;
    classDef src fill:#79C3E8,stroke:#3F9BD4,color:#0b3a52;
    classDef cat fill:#2456A4,color:#fff,stroke:#163a73,stroke-width:2px;
    classDef gate fill:#3F9BD4,stroke:#2456A4,color:#fff;
    classDef loop fill:#4FA47D,stroke:#2f6b50,color:#fff;
```

첫 단계에서는 문서 일부를 주고 한 레코드를 이루는 항목(이름·의미·예시값·타입·필수여부)을 제안하게 한다. 사람이 항목명·필수·표준값을 검수해 스키마를 확정하면, 이 확정 스키마가 곧 골든 기준이 된다. 다음 단계에서는 확정 스키마를 주고 값만 채우게 하되, 원문에 없으면 null로 두고 지어내지 말 것, 병합된 상위 값은 하위 행에 전파할 것, 숫자·단위·판정은 원문 그대로 둘 것을 명시한다.

<a id="sec623"></a>

다음은 작업표준 검사기준표를 이 방법으로 처리한 예다. 사람은 한 건을 공정·검사항목·기준 상한·하한·단위·주기로 정의하고, 좌측 공정명이 병합되어 여러 항목을 묶으므로 하위 행에 전파하며, ≤0.05나 100±0.1 같은 표기를 상·하한으로 푼다고 단계를 적는다.

```
[입력] 병합 검사기준표 — 사람 눈엔 자연스럽지만 코드로는 까다롭다
 공정    | 검사항목 | 기준      | 단위 | 주기
 용접부  | 외관     | ≤0.05    | mm   | 매 로트   ← '용접부'가 외관·치수 2행에 병합
 (병합)  | 치수     | 100±0.1  | mm   | 매 로트
 도장    | 두께     | 80~120   | ㎛   | 일 1회

[확정 스키마] {공정, 검사항목, 기준_하한, 기준_상한, 단위, 검사주기}

[값 추출 결과]
 [
  {"공정":"용접부","검사항목":"외관","기준_하한":null,"기준_상한":0.05,"단위":"mm","검사주기":"매 로트"},
  {"공정":"용접부","검사항목":"치수","기준_하한":99.9,"기준_상한":100.1,"단위":"mm","검사주기":"매 로트"},
  {"공정":"도장","검사항목":"두께","기준_하한":80,"기준_상한":120,"단위":"㎛","검사주기":"일 1회"}
 ]
```

병합된 용접부가 두 행에 전파됐고, ≤0.05·100±0.1·80~120이 상·하한으로 풀렸다. 규칙 코드로는 양식마다 새로 짜야 하지만, 단계만 바꿔 LLM에 시키면 양식이 달라도 같은 방식으로 넘는다. 프롬프트와 LLM 응답, 검증까지 끝까지 펼친 완성 사례는 [별첨 — 유형별 실행 레시피](별첨/B-1%20별첨%20—%20유형별%20실행%20레시피.md)의 3-F에 있다.

LLM에 맡기더라도 결과는 반드시 검증한다. 추출 행 수가 원문과 같은지, 필수값 누락이 없는지, 표준값·범위를 벗어나지 않았는지 확인하고 어긋나면 사람 검수로 보낸다. 핵심 수치는 원문과 대조한다. 전체를 LLM에 돌리면 느리고 비싸므로 어려운 양식만 LLM에 맡기고 쉽고 정형인 문서는 코드로 처리하며, 사외 반출이 막힌 데이터는 사내·로컬 LLM으로 처리한다. 양식이 고정되면 확정 스키마와 추출 프롬프트를 파이프라인으로 묶어 자동 반복한다.

<a id="kq4"></a>

### 5.4 변환 안정장치와 도구

원천 양식은 반드시 바뀐다. 규칙 기반 파서는 열 순서·머리글·여백이 조금만 바뀌어도 조용히 잘못된 값을 내므로, 깨짐을 사람이 뒤늦게 발견하지 않도록 세 가지 안정장치를 둔다.

- 파서 버전 관리: 어떤 파서·설정으로 언제 변환했는지를 결과에 기록해, 결과가 이상할 때 파서 변경 탓인지 되짚는다.
- 골든 문서셋: 대표 양식·까다로운 사례·과거 양식과 그 정답 출력을 고정 보관한다. 파서를 고치거나 양식이 바뀌면 골든셋을 다시 돌려, 출력이 정답과 어긋나면 파이프라인을 멈춘다.
- 스키마 검증: 파싱 결과가 정해진 구조(필드명·타입·필수값)를 지키는지 매번 확인해 어긋난 입력을 초기에 격리한다.

> **예시 — 골든셋 회귀 감지.** 대표 양식 12종을 골든셋으로 둔 상태에서 검사항목 열이 추가된 새 양식이 들어오자, 회귀 테스트가 표 2번의 판정 열이 한 칸씩 밀린 것을 감지해 파이프라인을 중단했다. 파서를 고쳐 골든셋을 통과한 뒤 재가동했다. 이 장치가 없었다면 밀린 값이 그대로 적재돼 AI 답이 틀어진 뒤에야 문제를 알았을 것이다.

도구는 문서 유형·표 복잡도·데이터 민감도에 따라 고른다. 복잡한 표가 많고 사외 전송이 제한되면 로컬 실행되는 [Docling](https://github.com/docling-project/docling)을 1차 파서로 두고, 표가 까다로운 양식은 [Camelot](https://camelot-py.readthedocs.io)으로 보완한다. 스캔본·다국어가 섞인 문서는 OCR이 강한 클라우드([Azure AI Document Intelligence](https://azure.microsoft.com/ko-kr/products/ai-services/ai-document-intelligence) · [Google Document AI](https://cloud.google.com/document-ai))를 PoC로 검토한다. 가격·버전·한국어 지원 범위는 변동되므로 단정하지 말고 PoC와 공식 문서로 확인한다.

### 5.5 적재 — AI 활용 환경에 제공

같은 전처리 결과라도 쓰임에 따라 적재처가 다르다. 의미 유사도로 문서를 찾는 RAG에는 벡터 DB([pgvector](https://github.com/pgvector/pgvector) · [Milvus](https://milvus.io/docs) 등), 키워드·하이브리드 검색에는 검색 인덱스([Elasticsearch](https://www.elastic.co/docs/solutions/search/vector) · [OpenSearch](https://docs.opensearch.org/latest/vector-search/)), 장기 보관·재처리에는 데이터레이크, 집계·BI에는 분석 테이블로 적재한다.

예를 들어 최근 분기 용접 불량 추세 같은 수치 집계는 분석 테이블로, 유사 불량 원인 사례를 찾는 의미 검색은 벡터 DB로 동시에 적재한다. 적재 후 그 데이터를 AI에 써도 되는지 품질 판정은 [C-2](../C-2%20데이터%20품질%20관리/C-2%20데이터%20품질%20관리.md)가 맡는다.

> **용어 풀이 — 벡터 DB·임베딩(Embedding).** 임베딩은 텍스트를 의미 좌표(숫자 벡터)로 바꾼 것이다. 벡터 DB는 이 좌표를 저장해, 단어가 달라도 의미가 가까우면 찾아 준다. 전처리는 임베딩할 깨끗한 청크를 준비하는 데까지를 책임지며, 임베딩 모델을 만드는 일은 아니다.

### 5.6 운영과 역할

운영은 양식이 바뀌면 골든셋이 감지하고, 어려운 양식만 사람이 검수하며, 결과를 다시 적재하는 흐름으로 돈다. 스키마 검증·골든셋([5.4](#kq4))이 회귀를 잡으면 파서를 고치고 골든셋을 갱신한 뒤 재가동하며, 과거 양식도 하위 호환 테스트용으로 보관한다. 검수는 전수가 아니라 병합 표·저품질 스캔본·도면 속 수치·내부 약어처럼 자동 추출이 약한 곳에 집중한다.

```mermaid
%%{init: {'theme':'base','themeVariables':{'primaryColor':'#DCE8F5','primaryBorderColor':'#2456A4','primaryTextColor':'#10243f','lineColor':'#2456A4','fontSize':'13px'}}}%%
flowchart TB
    CHG["원천 양식 변경<br/>(열 추가·머리글 변경)"]:::src
    CHG --> GS{"골든셋 회귀 테스트<br/>스키마 검증"}:::gate
    GS -->|통과| AUTO["자동 변환·적재"]:::loop
    GS -->|회귀 감지| FIX["파서 수정<br/>(엔지니어)"]:::n
    FIX --> REV["어려운 양식 검수<br/>(병합 표·스캔본)"]:::n
    REV --> GS
    classDef n fill:#B7CDE6,stroke:#6B9AD1,color:#10243f;
    classDef src fill:#79C3E8,stroke:#3F9BD4,color:#0b3a52;
    classDef gate fill:#3F9BD4,stroke:#2456A4,color:#fff;
    classDef loop fill:#4FA47D,stroke:#2f6b50,color:#fff;
```

역할은 세 주체로 나뉜다. 데이터 오너·현업 SME는 대상과 중요 표를 확인하고, 엔지니어는 파서·청킹·적재·골든셋을 만들며, 검수자는 어려운 양식을 보정한다. 사내 약어·설비코드 후처리는 [A-3 비즈니스 Glossary](../A-3%20비즈니스%20Glossary/A-3%20비즈니스%20Glossary.md)의 표준 용어를 활용한다.

## 별첨 (Appendix)

<a id="s81"></a>

### 주요 용어

- 비정형/반정형 데이터: 표 형태의 깔끔한 행·열(정형)이 아닌, 문서·이미지처럼 구조가 묻혀 있는 데이터. PDF·스캔 이미지가 비정형, 표 섞인 Excel·보고서가 반정형이다.
- 파싱(Parsing): 문서 파일을 읽어 그 안의 텍스트·표·이미지를 뽑아내는 것. 그 일을 하는 프로그램이 파서다.
- OCR(Optical Character Recognition): 이미지·스캔본 속 글자를 텍스트로 바꾸는 기술. TSR(Table Structure Recognition)은 이미지 속 표의 선·셀을 인식해 행·열로 복원하는 기술이다.
- 청크(Chunk)·겹침(Overlap): 청크는 검색을 위해 나눈 텍스트 한 조각. 겹침은 앞 청크의 끝 일부를 다음 청크 앞에 겹쳐, 경계에서 문맥이 끊기지 않게 하는 장치다.
- provenance(원본 위치 정보): 각 요소가 어느 문서·페이지·표·좌표에서 왔는지 기록한 꼬리표. 답변 근거 추적의 출발점이다.

<a id="s82"></a>

### 청킹 전략 비교

| 전략 | 방식 | 적합 |
|---|---|---|
| 고정 크기 | 일정 토큰 수로 자르고 겹침을 둠 | 구조가 약한 자유 텍스트 |
| 구조 기반 | 제목·표·문단 경계로 나눔 | 보고서·SOP처럼 구조가 뚜렷한 문서 |
| 표 단위 | 표는 통째로 한 청크 | 표가 핵심인 성적서·점검표 |

구조가 뚜렷한 제조 문서는 구조 기반과 표 단위를 함께 쓰고, 고정 크기는 보조로 둔다.

<a id="backup-6-b-배치-vs-증분-처리"></a>

### 배치 처리와 증분 처리

전체를 주기적으로 한꺼번에 처리하는 배치는 지연이 허용되고 원천 부하를 피해야 할 때 쓰고, 마지막 처리 이후 변경분만 처리하는 증분은 최신성이 중요하고 비용을 줄여야 할 때 쓴다. 변경 감지는 타임스탬프, 변경 로그(CDC), 내용 해시 비교로 한다.

<a id="s84"></a>

### 운영 체크리스트

- 표준 출력 포맷(JSON·Markdown)과 청크 항목(content·provenance)을 정의했는가
- 대표 양식·과거 양식·까다로운 사례를 골든셋으로 보관하는가
- 파서 버전과 스키마 검증을 파이프라인에 붙였는가
- 어려운 양식의 검수 지점과 검수자를 정했는가
- 적재처(벡터 DB·검색 인덱스·분석 테이블)를 용도별로 정했는가

코드·프롬프트 수준의 따라하기 레시피와 완성 사례는 [별첨 — 유형별 실행 레시피](별첨/B-1%20별첨%20—%20유형별%20실행%20레시피.md)에 있다.
