### 1.2 전처리로 얻는 것

현재
- 문서 속 정보가 검색·분석 가능한 데이터로 살아난다.
- 폴더에 잠자던 성적서 수천 건이 표를 행·열로 살리고 페이지 위치를 붙이는 순간 불량 원인 질의응답의 검색 자료가 된다.

수정
- 문서 기반 정보가 검색·분석 가능한 데이터로 전환된다.
- 기존에 활용되지 못하던 성적서 데이터도 표 구조와 출처 정보가 정리되면 불량 원인 분석과 검색에 활용할 수 있다.

### 2.1 데이터 전처리란 + 범위·경계

현재
전처리는 이 숨은 구조를 복원해 넘긴다.

수정
전처리는 문서 내 구조 정보를 추출하여 AI가 활용 가능한 형태로 변환한다.

### 2.3 구조화 산출물

현재
JSON·Markdown·표는 그 구조를 담는 그릇일 뿐, 그릇 자체가 목적은 아니다.

수정
JSON, Markdown, 표는 구조화 결과를 표현하는 형식일 뿐이다.
전처리의 목적은 특정 형식을 만드는 것이 아니라 AI 활용에 필요한 구조와 출처 정보를 확보하는 것이다.

### 2.3 구조화 산출물

현재
AI가 어느 칸 값인지 알고

수정
AI가 값의 의미와 위치를 해석할 수 있고

### 4.2.3 Excel 점검표

현재
Excel은 사람이 보기 좋게 만든 탓에 AI에게는 가장 까다롭다.

수정
Excel은 사람이 활용하기 편하도록 설계되어 있어 AI가 구조를 해석하기에는 상대적으로 복잡하다

### 4.2.4 이미지·스캔본·도면

현재
흐린·손글씨 셀은 자동 신뢰하지 않고 검수로 분리돼 잘못된 값이 들어가지 않는다.

수정
흐린·손글씨 셀 등 신뢰도가 낮은 데이터는 별도 검수 대상으로 분류하여 오인식 데이터가 적재되지 않도록 한다.

### 4.2.5 이메일

현재
이메일은 협력사 품질 이슈, 변경 통지, 클레임 회신이 오가는 1차 기록이지만 AI가 읽기에는 손이 가장 많이 가는 유형이다.

수정
이메일은 협력사 품질 이슈, 변경 통지, 클레임 회신이 오가는 주요 기록이며, 본문·첨부파일·이미지·메타데이터가 혼재되어 있어 전처리 과정이 상대적으로 복잡한 유형이다.

### 4.3 복잡한 문서의 LLM 기반 구조화

현재
챗봇을 만드는 게 아니라 LLM을 전처리 도구로 쓰는 데이터 준비 활동이다.

수정
여기서 LLM은 답변 생성 모델이 아니라 문서를 구조화하는 자동화 엔진으로 활용된다.
