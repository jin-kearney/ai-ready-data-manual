## 1. 구조화 형식: LPG vs RDF

### 한 줄 풀이

| 형식 | 한 줄 풀이 |
|------|-----------|
| **LPG** (Labeled Property Graph, 레이블 속성 그래프) | 노드(개체)·엣지(관계) 각각에 직접 속성(키-값)을 붙이는 그래프 모델. 개발자가 쉽게 다루고 다중 홉(hop) 탐색이 빠르다. |
| **RDF** (Resource Description Framework, 자원 기술 프레임워크) | 모든 사실을 "주어-술어-목적어(S-P-O)" 삼중 쌍(트리플)으로 표현하는 W3C 표준. 어휘 공유·추론·외부 연계에 강하다. |

### 핵심 구조 차이

| 항목 | LPG | RDF |
|------|-----|-----|
| 속성 위치 | 노드/엣지에 직접 부착 | 별도 트리플로 표현 |
| 전역 식별자 | 없음 (로컬 ID) | URI (전역 고유) |
| 추론 지원 | 기본 없음 (플러그인 필요) | OWL + RDFS로 표준 추론 |
| 표준화 | openCypher / ISO GQL | W3C 표준 (RDF, OWL, SPARQL) |
| 외부 연계·상호운용 | 제한적 | 강함 (Linked Data, 표준 어휘) |
| 관계 메타데이터 | 엣지에 직접 속성 부여 가능 | RDF-star 확장 필요 |

### 다중 홉(multi-hop) 탐색 성능 — LPG가 유리한 이유

RDF는 **전역 고유성 요건** 때문에 하나의 사실이 여러 트리플로 분해된다. 깊은 경로를 따라가려면 수십~수백 번의 트리플 조인이 필요하다. 반면 LPG는 관계를 **일급 객체(first-class edge)** 로 저장하므로 포인터를 따라가는 방식으로 탐색하여 조인 오버헤드가 거의 없다.

> "RDF graph traversals are computationally expensive due to the sheer number of triples generated by its globally unique structure, making queries slower and less efficient, particularly in scenarios requiring multi-hop traversals. In contrast, LPGs store relationships as first-class edges, allowing direct, high-speed traversal."  
> — Memgraph Docs, "LPG vs. RDF"

**제조 현업(두산 계열사) 적용:**  
설비 고장 원인을 파악하는 제조 현장의 근본 원인 분석(Root Cause Analysis)은 "설비 A 이상 → 공정 B 편차 → 품질 C 불합격 → 배치 D 출하 지연" 같은 **긴 인과 연쇄(causal chain)**를 탐색해야 한다. 이처럼 6~10홉 이상의 심층 경로 탐색이 빈번한 경우, RDF의 트리플 폭발은 성능 병목이 된다. 최근 제조 시스템 병목 원인 분석 연구도 역방향 깊이 우선 탐색(RDFS) 알고리즘을 그래프 위에서 실행하는 방식을 채택하였다.

출처: Memgraph Docs ✅확인 · TigerGraph Blog ✅확인 · ScienceDirect (causal KG 논문) ⚠️미확인(403)

### LPG vs RDF 선택 기준 표

| 판단 기준 | LPG 선택 | RDF 선택 |
|-----------|---------|---------|
| 주요 쿼리 유형 | 다중 홉 탐색, 실시간 패턴 분석 | 추론 기반 쿼리, 어휘 통합 |
| 외부 데이터 연계 | 사내 시스템만 연동 | 공급망·외부 표준 연계 필요 |
| 추론 필요성 | 없거나 단순 | OWL 클래스 추론·규칙 추론 필요 |
| 팀 기술 수준 | 일반 개발자, SQL 친숙 | 시맨틱 웹·OWL 전문가 필요 |
| 표준 준수 요건 | 내부 운영용 | 규제·W3C 표준 준수 필수 |
| 스키마 변화 속도 | 빠른 진화 | 안정적 도메인 어휘 |
| 성능 우선순위 | 탐색 속도 최우선 | 의미적 정확성 최우선 |

**결론:** 두산 계열사처럼 제조 운영 데이터의 긴 인과 경로 탐색과 실시간 성능이 중요하면 **LPG 우선 선택**. 공급망 표준 연계·외부 기관 데이터 통합이 핵심이면 RDF 고려.

**아키텍처 원칙:** 형식 선택(LPG/RDF)은 DB 선택보다 **먼저** 결정해야 한다. 형식이 결정되면 사용할 쿼리 언어와 DB 옵션이 자동으로 좁혀진다. 형식을 잘못 고르면 이후 전환 비용이 매우 크다.

> "Customers have been known to make the wrong choice with no easy way to reverse direction later."  
> — AWS Database Blog, "Build and deploy knowledge graphs faster with RDF and openCypher"

## 2. 저장소 DB: RDB(관계형) vs Graph DB

### 한 줄 풀이

| DB 유형 | 한 줄 풀이 |
|---------|-----------|
| **RDB** (Relational Database, 관계형 데이터베이스) | 행·열 테이블 구조. 트랜잭션·집계 쿼리에 최적화. 관계가 복잡해지면 JOIN이 기하급수적으로 늘어난다. |
| **Graph DB** (그래프 데이터베이스) | 노드·엣지를 네이티브로 저장. 관계 탐색이 핵심인 데이터에 최적. 연결이 많을수록 RDB 대비 성능 이점이 커진다. |

### RDB로 충분한 경우 vs Graph DB가 필요한 경우

| 기준 | RDB로 충분 | Graph DB 필요 |
|------|-----------|--------------|
| 관계 깊이 | 1~2홉 (단순 외래키 조회) | 3홉 이상의 경로 탐색 |
| 관계 밀도 | 엔티티당 관계 수 적음 | 엔티티당 관계 수 많음 |
| 쿼리 유형 | 집계·보고·CRUD | 패턴 매칭·경로 탐색·유사도 |
| 스키마 안정성 | 고정 스키마 | 자주 변하는 연결 구조 |
| 예시 | 생산 실적 집계, 부품 BOM 테이블 | 설비 고장 원인 추적, 공정 의존성 분석 |

### 두 DB를 함께 쓰는 하이브리드 패턴

제조 현장에서는 **RDB + Graph DB 병행**이 현실적이다:

- **RDB**: 생산 실적, 부품 마스터, 품질 이력 등 **집계·트랜잭션 데이터**
- **Graph DB**: 설비-공정-부품-불량 간 **관계·경로 데이터**
- **메타데이터**: 그래프 전체 정보는 그래프 DB에 분산되므로, **RDB에 저장**이 효율적

> "Information records (nodes) related to one another (edges) should be stored in a graph database… metadata about the entire graph is preferably stored in the relational database."  
> — IEEE Hybrid Database Approach (Gouri Benni, Medium 요약)

**형식 → DB 선택 순서 확인:**  
LPG 형식 선택 → Neo4j / ArangoDB / Memgraph 등 LPG 네이티브 DB 선택  
RDF 형식 선택 → GraphDB / Apache Jena / Amazon Neptune (RDF 모드) 선택  
형식이 결정되어야 DB 옵션이 의미 있게 좁혀진다.

출처: [IEEE Hybrid Database (Medium)](https://medium.com/@gouri.benni/an-overview-on-a-hybrid-database-approach-using-graph-and-relational-database-fb216e0a24f1) ✅확인  
출처: [Linkurious Choosing Graph DB](https://linkurious.com/blog/choosing-the-best-graph-database/) ✅확인

## 3. Query 언어

| 형식 | 쿼리 언어 | 한 줄 설명 | 공식 URL |
|------|----------|-----------|---------|
| **RDF** | **SPARQL** (스파클) | W3C 표준 RDF 쿼리 언어. S-P-O 패턴 매칭·집계·추론 결과 조회 가능. 1.1은 2013년 권고안, 1.2 개발 중. | [w3.org/TR/sparql11-query](https://www.w3.org/TR/sparql11-query/) ✅확인 |
| **LPG** | **openCypher** / **Cypher** | Neo4j가 만들고 오픈소스로 공개한 LPG 쿼리 언어. 12개 이상 DB가 지원. 2024년 ISO GQL의 모태가 됨. | [opencypher.org](https://opencypher.org/) ✅확인 |
| **LPG (표준)** | **ISO GQL** (ISO/IEC 39075) | 2024년 4월 11일 발행된 최초의 국제표준 그래프 쿼리 언어. Cypher/openCypher를 주요 참고로 설계. | [AWS GQL 발표](https://aws.amazon.com/blogs/database/gql-the-iso-standard-for-graphs-has-arrived/) ✅확인 |

**실무 요점:**
- 현재 LPG 프로젝트는 대부분 **Cypher / openCypher** 사용
- 신규 프로젝트는 장기적으로 **ISO GQL** 호환 검토 권장 (표준이 안착 중)
- RDF 프로젝트는 **SPARQL 1.1** (안정) 또는 SPARQL 1.2 (개발 중) 사용

## 4. T-Box vs A-Box + 추론 적재(Materialization) 트레이드오프

### 한 줄 풀이

| 구분 | 한 줄 풀이 | 예시 (두산 계열사) |
|------|-----------|-----------------|
| **T-Box** (Terminological Box, 개념 스키마) | "어떤 개념이 있고 어떤 관계를 맺을 수 있는가"를 정의하는 온톨로지 스키마. 인간이 설계하며 데이터보다 훨씬 작다. | "설비(Equipment)는 공정(Process)에 사용된다", "부품(Part)은 설비의 하위 구성이다" |
| **A-Box** (Assertional Box, 인스턴스 데이터) | 실제 데이터 — 개별 개체와 그 속성·관계. T-Box 스키마에 맞춰 채워지는 운영 데이터. | "D-402 선반은 CNC 밀링 공정에 사용 중", "V-belt #1234는 D-402의 구성품" |

**관계:** T-Box가 규칙을 정의하면, 추론(Inference) 엔진은 A-Box에서 새로운 사실을 도출할 수 있다.  
예) T-Box에 "하위 구성품이 고장나면 상위 설비도 영향을 받는다"라고 정의하면, A-Box의 고장 이벤트로부터 설비 영향 범위를 자동 추론.

### 추론 적재 방식: 사전 적재(Forward Chaining) vs 질의 시 추론(Backward Chaining)

| 방식 | 동작 원리 | 저장소 | 쿼리 속도 | 갱신 시 비용 | 적합한 경우 |
|------|---------|-------|---------|------------|-----------|
| **사전 적재(Forward Chaining / Materialization)** | 데이터 로드 시 모든 추론 결과를 미리 계산해 DB에 저장 | 높음 (추론 결과까지 저장) | 빠름 (미리 계산됨) | 데이터 변경 시마다 재추론 필요 | 쿼리 빈도 높고, 데이터 변경 빈도 낮을 때 |
| **질의 시 추론(Backward Chaining / Query-time)** | 쿼리 수신 시 필요한 추론만 그때그때 수행 | 낮음 (원본 데이터만 저장) | 느림 (실시간 추론) | 없음 (원본만 관리) | 데이터 변경 잦고, 쿼리 빈도 낮을 때 |

> "Forward-chaining starts with the data, infers everything that can be inferred and then stores it in the database. This means that every time data is loaded or updated, you have to do reasoning again."  
> "With backward-chaining approaches, reliable cardinality guesses are not available and query optimization is practically impossible [at scale]."  
> — Ontotext, "What Is Inference?" & "Reasoning with Big Knowledge Graphs"

**실무 권고 (제조 현장):**  
설비 마스터·공정 구조 같은 **안정적 데이터**는 사전 적재(Materialization)로 쿼리 성능 확보.  
품질 이벤트·알람처럼 **실시간으로 생성되는 A-Box 데이터**는 점진적 추론(Incremental Reasoning) 또는 하이브리드 방식 고려.

출처: [Ontotext - What Is Inference?](https://www.ontotext.com/knowledgehub/fundamentals/what-is-inference/) ✅확인  
출처: [Ontotext - Reasoning with Big KGs](https://www.ontotext.com/blog/reasoning-with-big-knowledge-graphs/) ✅확인  
출처: [A-Box T-Box R-Box (The Ontologist)](https://ontologist.substack.com/p/a-box-t-box-r-box-c-box) ✅확인

## 5. IOF (Industrial Ontologies Foundry)

### 개요

| 항목 | 내용 |
|------|------|
| **정식 명칭** | Industrial Ontologies Foundry (산업 온톨로지 파운드리) |
| **거버닝 기관** | OAGi (Open Applications Group Inc.)의 산하 조직 |
| **공식 URL** | https://spec.industrialontologies.org/iof/ (→ GitHub 릴리스로 리다이렉트됨) |
| **GitHub** | https://github.com/iofoundry/ontology |
| **상위 온톨로지** | BFO (Basic Formal Ontology, ISO/IEC 21838-2) |
| **범위** | 제조·유지보수·공급망·제품 수명주기의 개념·관계 표준화 |
| **목표** | 이기종 제조 시스템 간 데이터 통합 비용 절감, 디지털 제조 상호운용성 확보 |

**IOF 계층 구조:**
```
BFO (상위 온톨로지) — ISO/IEC 21838-2
  └─ IOF Core Ontology (제조 공통 개념)
       ├─ IOF 공급망(Supply Chain)
       ├─ IOF 유지보수(Maintenance)
       └─ 도메인별 온톨로지 (BMIC 등)
            └─ 기업 맞춤 애플리케이션 온톨로지
```

### IOF 채택의 트레이드오프

채택이 항상 좋은 것은 아니다. 다음을 반드시 고려해야 한다:

| 장점 | 단점 / 위험 |
|------|-----------|
| 타 IOF 기반 시스템과 즉시 연계 가능 (상호운용성) | BFO 개념 체계(연속자/발생자, 역할/기능 구분 등) 학습 곡선이 가파름 |
| ISO 표준 기반 → 규제·인증 대응 유리 | 현장 개념을 BFO 어휘로 매핑하는 데 전문가 투입 필요 |
| 업계 공통 어휘 → 공급사·고객사와 소통 용이 | IOF Core가 아직 성숙 단계 (v1 beta 수준) |
| 보건의료 OBO Foundry 성공 사례 존재 | 상업적 실제 도입 사례 부족 ("대부분 학술 연구 수준") |
| 도메인별 확장 온톨로지 재사용 가능 | 기업 특수 개념을 표준 틀에 억지로 끼워 맞추면 과잉 설계(over-engineering) |

### 언제 IOF를 채택해야 하는가 vs 커스텀 온톨로지

| 상황 | IOF 채택 권장 | 커스텀 온톨로지 권장 |
|------|-------------|-------------------|
| 외부 연계 필요성 | 공급망 파트너·고객·표준기관과 데이터 교환 필수 | 내부 시스템만 연동 |
| 팀 역량 | 온톨로지 전문가 확보 가능 | 개발팀이 OWL/BFO 비전문가 |
| 도메인 성숙도 | 제조·유지보수 표준 개념이 명확히 정의된 영역 | 회사 고유의 독자적 공정·개념 |
| 프로젝트 단계 | 장기 플랫폼 구축 (2년 이상) | 빠른 파일럿·PoC (6개월 이내) |
| 비용·일정 | 초기 투자를 감수할 수 있는 전략 과제 | 빠른 ROI가 필요한 현장 과제 |

**결론:** 두산 계열사 수준의 제조 기업이 내부 AI 데이터 준비를 목적으로 온톨로지를 처음 구축하는 경우, **IOF 즉시 전면 채택보다는 IOF 어휘를 참조하되 기업 맞춤 온톨로지로 시작**하고, 외부 연계 필요성이 생길 때 IOF 정렬을 점진적으로 확대하는 접근이 현실적이다.
