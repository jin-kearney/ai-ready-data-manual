# B-3 온톨로지 가이드 수정 지시사항 (v0.58 → v0.59)

## 1. 1.2 온톨로지가 필요한 조건 — Explainable 조건 보완

### 수정 배경

현재 문서는 다음과 같이 표현하고 있음.

> "100% 정확하고 설명 가능(Explainable)해야 한다"

그러나 온톨로지 역시 관계 정의 또는 데이터 자체가 잘못되면 잘못된 결과를 반환할 수 있다.

따라서 "온톨로지 = 무조건 100% 정확"으로 읽히지 않도록 전제를 명확히 한다.

### 수정 방향

현재 문장

> 확률이 아니라 선언된 규칙대로

수정안

> 관계와 규칙이 올바르게 정의된 범위 내에서, 확률 기반이 아닌 선언된 규칙에 따라 동일한 결과를 재현해야 한다.

또는

> 관계와 규칙이 정확하게 정의된 경우, 확률적 추론이 아닌 결정적(Deterministic) 방식으로 동일한 결과를 제공해야 한다.

### 수정 원칙

- "100% 정확" 표현은 유지 가능
- 단, "정의된 관계와 규칙 범위 내에서"라는 전제 추가
- 온톨로지가 오류를 만들 수 없다는 의미로 읽히지 않도록 수정

## 2. 2.1 정의 — Semantic Layer 표현 수정

### 수정 배경

현재 문서

> 온톨로지(시맨틱 레이어, Semantic Layer)

그러나 온톨로지는 Semantic Layer 전체와 동일 개념이 아니다.

Semantic Layer는 보다 상위 개념이며,
온톨로지는 Semantic Layer를 구성하는 여러 구현 방식 중 하나다.

Ontology ≠ Semantic Layer

### 수정 방향

현재 문장

> 온톨로지(시맨틱 레이어, Semantic Layer)

수정

> 온톨로지(Ontology)

또는

> 온톨로지(Ontology) — 비즈니스의 개념과 관계를 구조화한 의미 체계

### 수정 원칙

- Semantic Layer 용어 제거
- Ontology와 Semantic Layer를 동의어처럼 표현하지 않음
- Knowledge DB와의 차이 설명은 유지

## 3. 2.4 L1 마스터 객체 — 버전 관리 개념 추가

### 수정 배경

현재 문서

> 한 번 잘 정의하면 거의 안 바뀐다

표현이 다소 모호함.

실제 현장에서는

- SOP 개정
- 설비 폐기
- 공정 변경

등이 발생한다.

따라서 "안 바뀐다"가 아니라
"변경 빈도가 낮고 버전 관리한다"는 의미로 보완 필요.

### 수정안

현재

> 한 번 잘 정의하면 거의 안 바뀐다

수정

> 변경 빈도는 낮지만 SOP 개정, 설비 폐기 등 현실 변화가 발생할 수 있으므로 상태값(활성/비활성) 또는 버전으로 관리한다.

### 수정 원칙

- L1은 안정적이라는 메시지는 유지
- "절대 안 변함"으로 읽히지 않도록 수정

## 4. 2.4 L2/L3 구분 기준 명시

### 수정 배경

현재 예시는 충분하지만
실무자가 L2와 L3를 구분하는 기준이 문서에 명시적으로 적혀 있지 않음.

### 추가 문구

#### 판단 기준

사람의 해석이나 판단이 개입되는가를 기준으로 구분한다.

- 센서 측정값
- 검사 결과
- 설비 이벤트

처럼 자동으로 기록되는 사실은 L2(사건)로 분류한다.

반면

- 원인 판단
- 조치 결정
- 위험도 평가

처럼 사람이 해석하여 내린 결론은 L3(해석)로 분류한다.

#### 예시 표 추가

| 항목 | Layer |
|--------|--------|
| Delamination 검출 | L2 |
| 검사 불합격 | L2 |
| 원인 = 흡습 | L3 |
| 조치 = 건조 | L3 |

### 수정 원칙

- 가장 실무적인 구분 기준 제공
- "시간정보 있음 = L2"보다 직관적인 기준 추가

---

## 5. 2.4 L3 해석 — 버전 관리 원칙 보완

### 수정 배경

현재

> 판단만 버전 갱신한다

로 되어 있으나
구체적으로 어떤 의미인지 불명확함.

### 수정안

L3 판단은 변경될 수 있으므로 기존 판단을 삭제하지 않고 버전 또는 유효기간 기반으로 관리한다.

예를 들어

- 최초 판단: 흡습
- 재분석 후: 온도 미달

로 변경되더라도 기존 판단은 보존하고 새로운 판단을 추가한다.

### 수정 원칙

- 개념 설명 수준 유지
- 특정 제품(Neo4j) 구현 방식까지는 들어가지 않음
- RDF/OWL/Neo4j 모두 적용 가능한 원칙만 기술

## 6. 2.6 코어와 유즈케이스 — 참조 구현 방식 보완

### 수정 배경

현재 문서

> 유즈케이스는 코어를 복사하지 않고 참조한다

라고 설명되어 있으나,
실제 구현 관점의 의미가 부족함.

### 추가 설명

#### 코어 참조 원칙

- 유즈케이스는 코어 개념을 복제하지 않는다.
- 유즈케이스는 코어 개념을 참조하거나 연결 관계만 추가한다.
- 코어 개념이 변경되면 해당 개념을 사용하는 모든 유즈케이스가 동일한 의미를 공유한다.

#### 구현 예시 (개념 수준)

Delamination 개념은 코어에 1회만 정의한다.

- RCA Dashboard
- 품질 Agent
- 원인 분석 Agent

등은 동일한 Delamination 개념을 참조하여 사용한다.

### 수정 원칙

- "복사하지 않는다"를 실제 의미로 풀어서 설명
- 특정 구현 기술(Neo4j, RDF/OWL)에 종속되지 않음

### 추가 권고

3.4 유즈케이스 설계 절에 아래 문단 추가 검토.

#### 코어 참조 규칙

유즈케이스는 코어 온톨로지의 개념을 재정의하거나 복사하지 않는다.

유즈케이스별로 필요한 뷰, 질의, 업무 로직은 추가할 수 있으나,
결함·원인·조치 등 핵심 개념의 정의는 반드시 코어를 참조한다.

이를 통해 여러 AI Agent와 시스템이 동일한 의미 체계를 공유하도록 한다.
