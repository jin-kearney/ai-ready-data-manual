## 아키텍처 방법론 정본 (v0.2, 끝까지 일관)
- **결정 순서**: 형식(LPG/RDF) → 저장소(RDB/GraphDB) → 쿼리 언어 → 추론 적재 방식 → 표준 적용(IOF) → 진단(보안·성능·비용). **형식이 DB·쿼리언어에 선행**(전환비용 큼).
- **형식 = LPG (본 프로젝트=커니 채택, RDF 검토 배제)**. 이유: 제조 원인탐색 추론경로 길어(6홉+) RDF 트리플 조인 폭증→성능 우려. 허훈석 컨설턴트 코멘트 원문 인용(6.2 결정 박스). 일반 원칙: 외부 표준연계·OWL 추론 핵심이면 RDF.
- **저장소 = RDB + Graph DB 혼용(폴리글랏)**. 정형·트랜잭션=RDB, 관계·경로=Graph DB.
- **쿼리 언어**: LPG→openCypher/Cypher(ISO GQL 2024 표준화), RDF→SPARQL(W3C).
- **추론 적재**: T-Box(개념 스키마)/A-Box(인스턴스). 사전 적재(materialization, 용량↑·쿼리 빠름) vs 질의 시 추론(용량↓·쿼리 느림) 트레이드오프 — 데이터 변경/쿼리 빈도로 균형.
- **IOF(+BFO)**: 전면 즉시 채택 X. 어휘 참조하되 커스텀으로 시작, 외부 연계 생기면 점진 정렬. 도입 판단=설계 방법론의 일부.
- **트리플은 개념 단위**(3.2) ↔ 물리 형식(RDF/LPG)은 §6 선택(3.2에 bridge 문장 추가).
