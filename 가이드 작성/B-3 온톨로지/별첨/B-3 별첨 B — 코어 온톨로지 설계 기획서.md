# 별첨 B — 코어 온톨로지 설계 기획서 (빈 템플릿)

> 한 줄: **9단계 중 1~7단계(코어 설계)의 산출물 양식.** 그대로 복사해 채운다. 빈 양식 채우기가 아니라 **구축 착수 가능한 실행안**이어야 한다. 코어는 유즈케이스와 무관하게 이 양식만 보고 설계한다.

> **쓰는 법:** 메인 가이드 [§6.2 코어 설계](../B-3%20온톨로지.md#sec62)를 읽으며 채운다. 설계 각론·**작성 사례(CCL)**는 [별첨 A 각론](B-3%20별첨%20A%20—%20유즈케이스%20온톨로지%20구축%20각론.md), 기술 검토(§9)는 메인 [§7 아키텍처](../B-3%20온톨로지.md#7-아키텍처기술-선택). **★ 표시는 원본 기획서 외 보강 항목**(온톨로지 6요소·표준 방법론 정합용).

**★ 문서 정보**

| 항목 | 내용 |
|---|---|
| 버전 / 작성일 | |
| 작성자 / 검토자 | |
| 도메인 / 적용 계열사 | |
| 참조 문서 (As-Is 분석서 등) | |
| 상태 (초안/검토/확정) | |

**작성 완료 기준 (이게 다 채워져야 제출 완료):**

| 항목 | 완료 기준 |
|---|---|
| 도메인 목적 | 한 문장, 측정 지표 포함 |
| 재사용 검토 ★ | 기존 온톨로지·표준·사내 자산 재사용 여부 판단 기록 |
| 노드 후보 | As-Is 근거 항목과 연결, 데이터 소스 확인 |
| 계층(is-a) ★ | 상위-하위 클래스 트리 정의 |
| 관계 목록 | 방향·업무 의미 설명 포함 |
| 속성·표준값 ★ | 주요 속성의 타입·단위·필수/선택·허용값 |
| 규칙/공리 ★ | 추론 규칙·무결성 제약(SHACL) 명시 |
| 핵심 질문 | 노드·관계 경로로 답변 가능 여부 확인 |
| 기술 검토 | LPG/RDF 선택 근거·추론 시점 판단 포함 |
| 미결 사항 | 담당자·기한 포함 |

**As-Is 분석서 연동 (작성 전 확인):** As-Is 근거가 없는 노드·관계는 작성하지 않는다.

| As-Is 분석서 항목 | 이 기획서 반영 위치 |
|---|---|
| E2E 업무 흐름 | §1 도메인 목적 적용 범위 |
| 현실 물리·조직·기준 객체 | §3 L1 마스터 객체 |
| 시간과 함께 일어나는 사건 | §3 L2 사건 |
| 현업의 판단·귀책·대책 | §3 L3 해석 |
| 데이터 소스 확인 결과 | 각 노드 데이터 소스 / §6 속성 |
| 데이터 미확인·수기 항목 | "미확인" 표시 → §11 미결 |
| 팀 간 용어 불일치·동의어 | §6 alias 속성 |

## 4. ★ 클래스 계층 (is-a) — 6요소의 '계층'

노드를 상위-하위로 묶는다(예: `들뜸 ⊏ 층간결함 ⊏ 결함`). 계층이 있어야 "상위 결함에 정의한 조치"가 하위에 자동 상속·추론된다.

## 6. ★ 속성·표준값 정의 (6요소의 '속성' + 항목 사전·표준값)

노드·관계가 갖는 속성을 정의한다. **허용값(표준값)**이 있으면 자유입력 대신 목록으로 고르게 한다.

| 소속 노드 | 속성명 | 데이터 타입·단위 | 필수/선택 | 허용값(표준값 목록) | 데이터 소스 |
|---|---|---|---|---|---|
| | | | | | |
| | | | | | |

> alias(동의어) 속성으로 팀 간 용어 충돌을 흡수한다([A-3 Glossary](../../A-3%20Glossary/A-3%20Glossary.md) 연계). 명명·ID 규칙은 §9.4·메인 [§7.9](../B-3%20온톨로지.md#sec79).

## 7. ★ 규칙 / 공리 (6요소의 '공리' + 무결성 제약)

AI가 새 사실을 추론하는 규칙과, 데이터가 지켜야 할 제약(SHACL)을 명시한다.

| 유형 | 규칙/제약 (자연어) | 형식 (추론 규칙 / SHACL) |
|---|---|---|
| 추론 규칙 | *(예: 결함이 난 공정의 설비가 열화면 → 그 설비를 원인 후보로)* | |
| 무결성 제약 | *(예: 모든 L2 사건은 시간 속성을 반드시 가진다)* | SHACL |
