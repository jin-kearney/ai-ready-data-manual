# 별첨 A — 유즈케이스 온톨로지 구축 각론 (브릿지)

> 한 줄: **DE센터의 일반 방법론(정론)을 "특정 목적의 유즈케이스가 있을 때 온톨로지를 실제로 어떻게 구축하나"의 각론으로 옮긴 문서.** 정론(메인 가이드)과 전자BG CCL 기획서([별첨 B](B-3%20별첨%20B%20—%20코어%20온톨로지%20설계%20기획서.md)·[별첨 C](B-3%20별첨%20C%20—%20유즈케이스%20레이어%20설계%20기획서.md)) **사이의 브릿지**다.

> **읽는 순서:** ① 정론 = [메인 가이드 `B-3 온톨로지.md`](../B-3%20온톨로지.md)(개념·9단계) → ② **각론 = 이 문서**(목적 유형별 설계 차이·암묵지 처리·확장 원칙) → ③ 사례 = [별첨 B 코어 기획서](B-3%20별첨%20B%20—%20코어%20온톨로지%20설계%20기획서.md)·[별첨 C 유즈케이스 기획서](B-3%20별첨%20C%20—%20유즈케이스%20레이어%20설계%20기획서.md)(전자BG CCL 채운 산출물).

---

## 0. 이 문서의 위치 — 왜 각론이 필요한가

정론(9단계 방법론)은 "어떤 유즈케이스든 통하는 보편 절차"다. 그러나 실제로는 **유즈케이스의 목적이 무엇이냐에 따라 어떤 노드·관계를 어느 레벨로 잡고, 암묵지를 어떻게 처리할지가 달라진다.** 이 차이를 정론은 일부러 비워 둔다(보편성 유지). 각론이 그 빈칸을 채운다.

```
[정론] 일반 방법론(9단계·3계층·철학)   ← 무엇을 해야 하나 (보편)
   │  (이 문서가 다리)
[각론] 목적 유형별 설계 차이·암묵지 처리·확장 원칙   ← 우리 상황에선 어떻게 (구체)
   │
[사례] 전자BG CCL 코어/유즈케이스 기획서   ← 실제로 채운 결과
```

> 메인 가이드 [§6 9단계 How](../B-3%20온톨로지.md#sec-how)의 6.2·6.3에서 이 문서를 참조한다. 정론을 모르면 먼저 메인 가이드를 본다.

## 1. 유즈케이스 목적 유형별 노드·관계 설계 접근 차이

> 👉 같은 9단계를 따라도 **유즈케이스 목적 유형(탐색·예측·모니터링·추천·자동화)에 따라 무엇을 강조해 설계하는지가 다르다.** 코어(현실)는 공유하되, 유형별로 *강조하는 계층·관계·전용 구조*가 달라진다.

| 목적 유형 | 핵심 질문 형태 | 강조 계층 | 강조 관계 | 전용 구조 경향 |
|---|---|---|---|---|
| **탐색(Exploration)** | "무엇이 무엇과 어떻게 연결되나" | L1·L2 폭넓게 | 다양한 관계 + **양방향**(역관계) | 경로 탐색용 인덱스 |
| **예측(Prediction)** | "다음에 무엇이 일어날까" | L2 사건(시계열) | 선후행·`precedes`·`affects` | 피처 추출용 속성·시간 윈도 |
| **모니터링(Monitoring)** | "지금 기준을 벗어났나" | L2 사건 + 상태 | `violates`·`governed-by`(규칙) | 임계·SHACL 규칙 노드 |
| **추천(Recommendation)** | "무엇을 하면 되나" | L3 해석(원인·조치) | `causes`·`remediated-by`·유사도 | 과거 사례 인스턴스 풀 |
| **자동화(Automation)** | "이 산출물을 자동으로 채워라" | L1·L2·L3 + 입출력 경계 | end-to-end 경로 전체 | 입력→출력 매핑·위반탐지 |

🏭 **CCL 사례 = 자동화형(C/S Report 초안 생성).** 그래서:
- **입출력 경계**를 명확히 정의(입력=클레임 3건·Lot / 출력=C/S Report 초안). → [별첨 C 유즈케이스 기획서 §2](B-3%20별첨%20C%20—%20유즈케이스%20레이어%20설계%20기획서.md)
- **end-to-end 경로 전체**가 한 번에 연결돼야 함: `클레임 → Lot commonality → 공정조건 → SOP 위반 → 원인 → 조치`.
- 자동화 특유의 **위반 탐지 규칙**(`공정조건 –violates→ SOP기준`)을 유즈케이스 레이어에 둠(코어는 사실만).
- 만약 같은 CCL 코어 위에 **모니터링** 유즈케이스를 얹는다면? 코어는 그대로, 유즈케이스 레이어에 "실시간 프레스 온도 임계 초과 감시" 규칙 노드만 추가한다 — 이것이 코어/유즈케이스 분리의 실익.

> **원칙:** 목적 유형이 다르다고 **코어를 다시 만들지 않는다.** 코어(현실 객체·사건·해석)는 한 번 만들고, 유형별 강조는 **유즈케이스 레이어**에서 흡수한다(메인 [§4.4 코어/유즈케이스 2층](../B-3%20온톨로지.md#sec34)).

## 2. 암묵지 수집·구조화·적재 방법론 (상황별 처리 분기)

> 👉 온톨로지의 본질은 **현장 전문가의 암묵지를 명시적 구조로 옮기는 것**이다. 수집 → 구조화 → 적재 3단계로 처리하되, **상황에 따라 방식이 분기**한다.

### 2.1 수집

| 원천 | 수집 방법 | 산출물 |
|---|---|---|
| 정형 지식(PFMEA·SOP·8D 보고서) | 문서 파싱·열 구조 매핑 | 후보 개념·관계 목록 |
| 반정형(C/S Report·검사 기록) | 필드 추출 + LLM 후보 제안 | 결함·원인·조치 후보 |
| 순수 암묵지(숙련자 판단) | 현장 인터뷰·디스커버리 워크샵 | "이 증상이면 저 설비 의심" 류 규칙 후보 |

> 워크샵은 **객체화(노드화) 전에** 한다 — 산출물 양식이 아니라 현실에서 개념을 길어 올리기 위함(정론 9단계 2단계, 함정 1 회피).

### 2.2 구조화 — 암묵지를 어디에 담을지 분기 (★ 핵심 결정)

수집한 암묵지를 모두 온톨로지 관계로 만들지 않는다. **온톨로지 관계 vs 쿼리·앱 로직**으로 분류한다(정론 9단계 5단계).

| 암묵지 성격 | 담는 곳 | 왜 |
|---|---|---|
| 현실의 **인과·구조**("흡습이 미경화를 부른다") | **온톨로지 관계**(`causes`) | 여러 유즈케이스가 재사용·추론·설명에 쓰임 |
| 특정 유즈케이스의 **처리 순서·우선순위**("이 화면에선 최근 30일만") | **쿼리·앱 로직** | 코어에 넣으면 유즈케이스에 종속됨(함정 2) |
| **임계·규칙**("온도 185±5°C 벗어나면 위반") | 경계 판단 — 규칙 노드(SHACL) 또는 쿼리 | 재사용되면 규칙 노드, 1회성이면 쿼리 |

🏭 **CCL 분기 예:** "흡습→미경화→들뜸"은 **온톨로지 관계**(코어, 재사용). "여러 클레임을 같은 Lot·기간으로 묶는다"는 **쿼리 로직**(유즈케이스 전용 — [별첨 C §7](B-3%20별첨%20C%20—%20유즈케이스%20레이어%20설계%20기획서.md)). "온도 SOP 위반 판정"은 **규칙 노드**(자동화에서 재사용되므로).

### 2.3 적재 — 상황별 처리 방식 분기

| 상황 | 적용 방식 | 근거 |
|---|---|---|
| 정형 소스(MES·ERP)에 데이터 있음 | ETL/CDC + [개념-데이터-문서 매핑표](../B-3%20온톨로지.md#sec37) → A-Box 자동 적재 | 신규 레코드마다 수작업 불필요 |
| 비정형 문서(보고서)만 있음 | LLM 후보 추출 → **SME 검증 필수** → 적재 | LLM은 계층·용어 충돌 판단 못 함 |
| 수기 관리·시스템 없음 | 워크샵에서 직접 입력 + "데이터 소스 미확인" 표시 → 미결로 | As-Is 근거 없는 노드 금지([별첨 B §7 미결](B-3%20별첨%20B%20—%20코어%20온톨로지%20설계%20기획서.md)) |

## 3. 확장을 고려한 설계 원칙

> 👉 처음부터 한 도메인만 보지만, **나중에 양방향 탐색·자연어 질의·다른 도메인·그룹사로 확대될 것을 전제로** 설계하면 재작업이 줄어든다.

| 원칙 | 무엇을 | 설계 시 행동 |
|---|---|---|
| **양방향 탐색 지원** | 원인→결함뿐 아니라 결함→원인도 따라가야 | 관계에 **역관계(inverse)** 정의(`causes`↔`causedBy`), LPG는 엣지 양방향 탐색 |
| **자연어 질의 지원** | 현업이 평문으로 물어도 매핑되게 | 노드·관계 라벨을 **현업 언어**로([A-3 Glossary](../../A-3%20Glossary/A-3%20Glossary.md) 표준 용어), 동의어 `alias` 속성 |
| **도메인 확장/통합** | 새 도메인(설비진단·안전)을 얹어도 코어 안 깨지게 | 코어/유즈케이스 분리 유지, 새 도메인은 **새 유즈케이스 레이어**로 시작 |
| **그룹사 확대** | 밥캣·에너빌리티 등으로 확장 | **공통/계열사 2계층**(메인 [§4.5](../B-3%20온톨로지.md#sec35)) — 공통 상위클래스 + 계열사 네임스페이스 하위클래스 |

> 이 원칙들은 정론의 "단일 진실 레이어·도메인 먼저·재사용" 철학(메인 [§1.4](../B-3%20온톨로지.md#sec14))의 실천 지침이다.

## 4. 실제 사례 — 전자BG CCL 결함분석 프로젝트 (노드 설계 각론)

> 👉 **이 가이드 전체에서 유일하게 실제 프로젝트 사례가 들어가는 곳.** 정론의 3계층/6요소를 전자BG CCL 들뜸 RCA에 적용하며 **메인 노드·서브 노드를 어떻게 잡았고, 각 노드를 어느 레벨로·어떤 프로퍼티로 정했는지**의 실제 결정 기록이다. *(아래 노드명·값은 프로젝트 예시 — 실데이터 확정 시 갱신.)*

### 4.1 메인 노드 / 서브 노드 구성

메인 노드(허브=핵심 클래스)와 서브 노드(하위 클래스·세부)를 나눈 기준: **여러 사례·유즈케이스가 공통으로 참조하면 메인, 특정 도메인 세부면 서브.**

| 메인 노드(핵심 클래스) | 계층 | 서브 노드(하위 클래스·세부) |
|---|---|---|
| 결함(Defect) | L2 | 층간결함 → 들뜸(Delamination) |
| 원인(Cause) | L3 | 프리프레그흡습 · 열압착온도미달 |
| 조치(Action) | L3 | 프리프레그건조(베이킹) · 프레스온도보정 |
| 설비(Equipment) | L1 | 열압착프레스#2 |
| 공정(Process) | L1 | 열압착공정 |
| 원자재(Material) | L1 | 프리프레그Lot(PPG-2403-17) |
| 기준(SOP) | L1 | SOP-LAM-012 |

```
결함 ── 층간결함 ── 들뜸                 [메인=결함 / 서브=들뜸, L2 검출 사건]
원인 ── 프리프레그흡습 · 열압착온도미달      [L3 해석]
조치 ── 프리프레그건조 · 프레스온도보정       [L3 해석]
설비·공정·원자재·SOP                       [L1 마스터, 들뜸검출이 참조]
```

> **메인/서브를 나눈 실무 판단:** `결함`·`원인`·`조치`·`설비`·`공정`은 다른 결함(스크래치 등)·다른 유즈케이스에서도 재사용되므로 **메인(코어 공통)**. `들뜸`·`프리프레그흡습`처럼 CCL 들뜸 도메인에만 쓰이는 것은 **서브(하위 클래스)**로 메인 아래 매단다. 그룹사 확대 시 메인은 공통 레이어로, 서브는 계열사 네임스페이스로 간다([§3](#3-확장을-고려한-설계-원칙) 확장 원칙).

### 4.2 노드 레벨 결정 — 각 노드를 왜 이 레벨로

| 노드 | 결정 레벨 | 왜 이 레벨인가 (판단) | 프로퍼티 결정 |
|---|---|---|---|
| `열압착프레스#2` | **L1 마스터** | 유즈케이스 없이도 존재·느리게 변함(설비) | `설비ID`·`설치라인` (상태값은 별도 사건으로) |
| `프리프레그Lot PPG-2403-17` | **L1 마스터** | 입고된 원자재 로트는 사실로 고정된 객체 | `Lot ID`·`입고일`·`보관습도`(입고 시점값) |
| `들뜸검출` | **L2 사건** | "언제 검출됐다"는 **시간을 가진 사실**, 불변 | `검출시각`(필수)·`심각도` |
| `열압착집행` | **L2 사건** | Lot별 공정 집행은 시점을 가진 이벤트 | `집행시각`·`온도`·`압력`·`시간` |
| `프리프레그흡습판단` | **L3 해석** | "흡습이 원인"은 사람의 판단 — 재조사로 바뀔 수 있음 | `신뢰도`·`판단자`·`판단시각` |
| `프리프레그건조조치` | **L3 해석** | 시정조치 결정도 판단(다른 조치로 대체 가능) | `결과`·`적용시각` |

### 4.3 결정 노트 (왜 이렇게 나눴나)

- **"들뜸"을 결함 *클래스*가 아니라 `들뜸검출` *사건*으로 둔 이유:** 같은 들뜸도 "검출된 사건"은 시점·Lot마다 다른 사실이다. 결함 유형(`들뜸 ⊏ 층간결함`)은 계층(클래스)으로, *검출된 개별 사건*은 L2 인스턴스로 분리해야 "어느 Lot에서 언제" 추적이 된다.
- **원인을 사건이 아니라 해석(L3)으로 둔 이유:** "흡습이 원인"이라는 판단은 8D 재조사에서 "온도 미달이 주원인"으로 바뀔 수 있다. L3로 분리했기에 **검출 사실(L2)은 그대로 두고 해석만 버전 갱신**한다(참조 역방향 금지 — L2는 L3를 가리키지 않음).
- **보관습도를 `프리프레그Lot`(L1)의 속성으로 둘지, 별도 사건으로 둘지:** 입고 시점 1회값이면 L1 속성, 시간에 따라 측정되는 로그면 L2 사건(`보관습도측정`)으로 승격. CCL에선 입고 시점값 + 보관 이력이 둘 다 있어 **둘 다** 둠(이력은 L2).

> 이 결정 각론을 **빈 양식에 옮겨 채운** 산출물이 [별첨 B 코어 기획서](B-3%20별첨%20B%20—%20코어%20온톨로지%20설계%20기획서.md)다(별첨 B는 빈 템플릿, 채운 사례는 이 문서). 노드 명명·ID·필수 속성 등 **구현 규칙**은 메인 [§7.9 구현 구조](../B-3%20온톨로지.md#sec79).
