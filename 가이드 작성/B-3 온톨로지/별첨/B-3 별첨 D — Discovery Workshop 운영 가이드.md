# 별첨 D — Discovery Workshop 운영 가이드

> 한 줄: **9단계 2단계(밸류체인/E2E 분석 + 현장 수집)에서 여는 워크샵 — 현실의 객체·사건·암묵지를 현장 사람에게서 길어 올려 [코어 기획서(별첨 B)](B-3%20별첨%20B%20—%20코어%20온톨로지%20설계%20기획서.md)의 노드 후보 표를 채울 재료를 만든다.** "어떤 사람을 모아 어떤 식으로 진행하나"의 실행법.

> **워크플로 위치:** [별첨 A 설계 각론](B-3%20별첨%20A%20—%20온톨로지%20설계%20각론.md)(설계 원칙) → **이 워크샵(수집)** → [별첨 B 코어 기획서](B-3%20별첨%20B%20—%20코어%20온톨로지%20설계%20기획서.md)·[별첨 C 유즈케이스 기획서](B-3%20별첨%20C%20—%20유즈케이스%20레이어%20설계%20기획서.md)(산출물 채우기). 9단계 전체는 메인 [§6 How](../B-3%20온톨로지.md#sec-how).

---

## 0. 목적 — 왜 워크샵인가

온톨로지의 가치는 **현장 전문가의 머릿속 암묵지("이 증상이면 보통 저 설비를 의심")를 명시적 구조로 옮기는 데** 있다. 이 암묵지는 문서(PFMEA·SOP)만으로는 다 안 나온다. 그래서 **노드화하기 전에** 현장 사람을 모아 현실을 함께 그린다.

> **핵심 원칙 — 노드화는 끝까지 미룬다.** 워크샵에서는 "현실에 무엇이 있고 무엇이 일어나고 누가 어떻게 판단하나"만 그린다. 클래스·관계로 옮기는 형식화는 워크샵 *후* 코어 기획서에서 한다. 산출물 양식(보고서 표)을 그대로 노드로 옮기지 않는다([함정 1](../B-3%20온톨로지.md#sec-traps)).

## 1. 누구를 모으나 (참석자·역할)

| 역할 | 누구 | 워크샵에서 하는 일 | 필수 |
|---|---|---|---|
| **퍼실리테이터** | 시맨틱 아키텍트·지식 엔지니어 | 진행, "이건 객체/사건/해석"인지 즉석 판단, 노드화 시점 통제 | ★필수 |
| **도메인 SME** | 현장 숙련자·반장·품질 베테랑 | 암묵지 원천 — "이 증상이면 뭘 의심"하는 판단을 말로 꺼냄 | ★필수 |
| **공정/품질 엔지니어** | PFMEA·SOP 작성자 | 고장 모드·원인·기준·공차 지식 | ★필수 |
| **데이터 엔지니어** | MES·ERP·QMS 담당 | 후보 객체·사건마다 "이거 데이터로 있나?" 즉석 확인 | ★필수 |
| **유즈케이스 오너** | 현업 리더 | 목적 한 문장·핵심 질문(CQ) 제시, 우선순위 | ★필수 |
| 현장 작업자 | 라인 오퍼레이터 | 실제 현장 디테일·예외 상황 | 선택 |
| IT/시스템 담당 | 인프라·연동 | 시스템 제약·접근 권한 | 선택 |

> **권장 규모 6~10명.** 그 이상이면 도메인별(예: 용접 / 도금)로 세션을 나눈다. 데이터 엔지니어가 **그 자리에 있어야** "데이터 없는 연결"([함정 4](../B-3%20온톨로지.md#sec-traps))을 즉석에서 거른다.

## 2. 어떻게 진행하나 (세션 흐름)

9단계 1·2단계와 직접 연결된다. 화이트보드·포스트잇으로 함께 그린다.

| 순서 | 세션 | 무엇을 | 산출 | 9단계 |
|---|---|---|---|---|
| 0 | **목적 합의** | 측정 가능한 단일 목적 한 문장 합의 | 목적 문장 | 1단계 |
| 1 | **E2E 흐름 그리기** | 업무를 끝에서 끝까지(예: VOC 접수~C/S 발송) 그림 | 프로세스 맵 | 2단계 |
| 2 | **객체 식별** | "유즈케이스 없이도 독립 존재하나? 사실로 고정되나?" 2질문 통과 → 마스터 객체 후보 | L1 후보 목록 | 3단계 |
| 3 | **사건 식별** | "언제 일어났나(시간 속성)?" → 공정·검사·이탈 이벤트 | L2 후보 목록 | 4단계 |
| 4 | **암묵지 끌어내기** | "이 증상이면 뭘 의심?·왜 그렇게 판단?·그다음 뭘 하나?" → 근본원인·조치 판단 | L3 후보 목록 | 6단계 |
| 5 | **핵심 질문 역검증 + 분류** | CQ 3~5개로 "지금 그린 게 이 질문에 답하나" 점검 + 각 암묵지를 *온톨로지 관계 vs 쿼리/앱 로직*으로 분류 | CQ·분류 표 | 5단계 |

> 세션 2~4는 [3계층(L1/L2/L3)](../B-3%20온톨로지.md#sec33)을 현장 언어로 끌어내는 과정이다. 소유·분류 구분 등 설계 원칙은 [별첨 A 설계 각론](B-3%20별첨%20A%20—%20온톨로지%20설계%20각론.md) §1.2.

## 3. 준비물 (퍼실리테이터가 사전 준비)

- As-Is 분석서 초안 (있으면) · E2E 프로세스 맵 초안 — 빈 양식은 [별첨 E As-Is 분석서](B-3%20별첨%20E%20—%20As-Is%20분석서.md)
- PFMEA·SOP 샘플 (개념 원천)
- 목적 문장 초안 + 핵심 질문(CQ) 초안 3~5개
- 데이터 소스 목록(MES·ERP·QMS 테이블·컬럼) — 즉석 확인용
- 화이트보드/포스트잇(객체=파랑·사건=초록·해석=노랑 색 구분 권장)

## 4. 산출물 → 다음 단계

워크샵이 끝나면 아래가 손에 남는다 → 그대로 [코어 기획서(별첨 B)](B-3%20별첨%20B%20—%20코어%20온톨로지%20설계%20기획서.md)의 노드 후보 표로 옮긴다.

| 워크샵 산출물 | 들어가는 칸(별첨 B) |
|---|---|
| 객체(L1)·사건(L2)·해석(L3) 후보 목록 + 정의 | §2 3계층 노드 후보 |
| 각 후보의 데이터 소스 확인 결과 | 노드별 데이터 소스 / "미확인"은 §7 미결 |
| 후보 간 관계(누가 무엇을 일으키나) | §3 관계 목록 |
| 핵심 질문(CQ) | §4 핵심 질문 |
| 팀 간 용어 충돌·동의어 | 노드 alias([A-3 Glossary](../../A-3%20Glossary/A-3%20Glossary.md) 연계) |

## 5. 운영 팁·주의

- **2~3시간 × 2~3회** 권장(한 번에 다 하려 하지 않는다). 기록 담당을 퍼실리테이터와 별도로 둔다.
- **노드화 미루기**(§0) — 형식화 욕구를 누르고 현실부터.
- **데이터 없는 후보는 "미확인"으로 즉석 표시** — 버리지 말고 확정 계획을 미결로.
- **용어 충돌은 그 자리서 기록**("용접 불량"=MES / "고장 모드"=PFMEA) → A-3 Glossary 표준화로 넘김.
- **사실과 판단을 섞지 않게 유도** — "그건 사실인가요(언제 일어남), 판단인가요(누가 그렇게 봄)?"를 계속 물어 L2/L3를 분리.

> 이 문서는 워크샵을 여는 **일반 운영법**만 담는다(특정 프로젝트의 멤버 구성·진행 기록은 별도로 둔다). 끌어낸 객체·사건·판단을 무슨 기준으로 노드·관계로 옮기는지는 [별첨 A 설계 각론](B-3%20별첨%20A%20—%20온톨로지%20설계%20각론.md)을 따른다.
