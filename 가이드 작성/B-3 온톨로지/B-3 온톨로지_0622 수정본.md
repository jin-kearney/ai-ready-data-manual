## 1. 온톨로지란 무엇인가

온톨로지는 단어 뜻을 모아둔 사전이 아니다.

온톨로지는 개념과 개념 사이의 관계, 계층, 인과관계를 구조화하여 사람과 컴퓨터가 동일한 방식으로 이해할 수 있도록 정의한 지식 구조이다.

설비, 공정, 결함, 원인, 조치와 같은 개념을 연결하여 특정 현상이 왜 발생했고 어떻게 해결할 수 있는지를 추적할 수 있도록 만든다.

따라서 온톨로지는 단순한 분류 체계가 아니라 조직의 암묵지와 업무 지식을 구조화하는 체계라고 볼 수 있다.

---

### 1.1 온톨로지의 정의

조금 더 정확히 말하면, 온톨로지는 특정 도메인의 개념과 개념 사이의 관계를 명시적으로 정의한 공유 지식 체계이다.

예를 들어 품질 영역에서는 다음과 같은 개념이 존재한다.

- 결함
- 원인
- 공정
- 설비
- 조치
- 검사

온톨로지는 이러한 개념을 각각 정의하는 데서 끝나지 않는다.

각 개념 사이의 관계를 함께 정의한다.

예를 들어 다음과 같은 관계를 정의할 수 있다.

```text
프리프레그 흡습
→ 원인이 된다
→ 들뜸

들뜸
→ 발생 공정
→ 열압착 공정

들뜸
→ 조치 방법
→ 프리프레그 건조
```

이러한 관계가 정의되면 사람뿐 아니라 AI도 결함, 원인, 조치 사이의 연결 구조를 이해할 수 있다.

---

### 1.2 온톨로지와 Knowledge DB의 차이

온톨로지와 Knowledge DB는 동일한 개념이 아니다.

온톨로지는 무엇이 무엇과 연결되는지를 정의하는 의미 구조이다.

Knowledge DB는 이러한 의미 구조와 실제 데이터를 저장하는 저장소이다.

| 구분 | 온톨로지 | Knowledge DB |
|--------|--------|--------|
| 역할 | 의미 구조 정의 | 데이터 저장 |
| 목적 | 개념과 관계 정의 | 관계와 사례 저장 |
| 예시 | 결함-원인-조치 구조 | 그래프 DB |
| 변경 주체 | 도메인 전문가 | 시스템 운영 조직 |

즉,

```text
온톨로지 설계
↓
Knowledge DB 저장
↓
AI 활용
```

순서로 활용된다.

온톨로지는 구조이고, Knowledge DB는 그 구조를 담는 그릇이다.

---

### 1.3 왜 필요한가

제조 현장에는 데이터가 부족한 것이 아니다.

데이터가 서로 연결되어 있지 않은 것이 문제인 경우가 많다.

예를 들어 들뜸(Delamination) 결함 하나만 분석하더라도 관련 정보는 여러 시스템에 분산되어 있다.

| 정보 | 위치 |
|--------|--------|
| 검사 결과 | MES |
| 공정 조건 | MES |
| 원자재 정보 | ERP |
| 작업 기준 | SOP |
| 고객 불만 | C/S Report |
| 원인 분석 | 품질 보고서 |

이 정보들이 연결되어 있지 않으면 다음과 같은 질문에 답하기 어렵다.

- 왜 이 결함이 발생했는가
- 어떤 원인이 가장 빈번한가
- 과거에는 어떻게 조치했는가
- 어떤 설비에서 반복적으로 발생하는가

온톨로지는 이러한 개념을 연결하여 하나의 지식 구조로 관리할 수 있도록 한다.

---

### 1.4 알아둘 3가지 기본 원칙

온톨로지를 설계할 때는 다음 세 가지 원칙을 이해해야 한다.

| 원칙 | 설명 |
|--------|--------|
| 정답은 하나가 아니다 | 동일한 현실도 여러 방식으로 모델링할 수 있다 |
| 한 번에 완성되지 않는다 | 반복적인 개선 과정이 필요하다 |
| 현실 중심으로 설계한다 | 문서나 시스템이 아니라 실제 업무를 기준으로 설계한다 |

첫째, 온톨로지에는 정답이 하나만 존재하지 않는다.

중요한 것은 가장 복잡한 구조를 만드는 것이 아니라 실제 업무에 도움이 되는 구조를 만드는 것이다.

둘째, 온톨로지는 처음부터 완성되지 않는다.

초기 버전을 만든 후 실제 활용 과정에서 지속적으로 개선해야 한다.

셋째, 설계 기준은 문서 양식이 아니라 현실 세계여야 한다.

예를 들어 품질 보고서라는 문서 자체를 모델링하는 것이 아니라, 결함, 원인, 조치와 같은 현실 개념을 모델링해야 한다.

---

### 1.5 다루는 범위

온톨로지는 개념 간 관계를 정의하는 역할을 수행한다.

따라서 다른 AI-ready Data 주제와는 명확한 역할 구분이 필요하다.

| 주제 | 역할 |
|--------|--------|
| Glossary | 용어 정의 |
| 메타데이터 | 데이터 속성 정의 |
| 데이터 카탈로그 | 데이터 위치 관리 |
| 데이터 해설·주석 | 학습용 의미 부여 |
| 온톨로지 | 개념 간 관계 정의 |

예를 들어 "들뜸"이라는 용어의 의미를 정의하는 것은 Glossary의 역할이다.

반면 들뜸이 어떤 원인과 연결되고 어떤 조치와 연결되는지를 정의하는 것은 온톨로지의 역할이다.

## 2. 언제 온톨로지를 만드나

온톨로지는 모든 데이터 영역에 구축하는 것이 아니다.

온톨로지를 구축해야 하는지는 기술이 아니라 활용 목적을 기준으로 판단해야 한다.

중요한 질문은 다음과 같다.

```text
어떤 서비스를 만들 것인가?

그 서비스가 어떤 가치를 만드는가?

온톨로지가 아니면 해결할 수 없는가?
```

이 질문에 답할 수 있을 때 온톨로지 구축을 검토한다.

---

### 2.1 먼저 정할 것 — 어떤 서비스에 활용할 것인가

온톨로지의 가치는 구조 자체가 아니라 활용 서비스에서 발생한다.

예를 들어 다음과 같은 서비스가 존재할 수 있다.

| 서비스 | 기대 효과 |
|--------|--------|
| 원인 분석 Agent | 조사 시간 단축 |
| 품질 진단 Agent | 전문가 의존도 감소 |
| 조치 추천 Agent | 대응 표준화 |
| 유사 사례 검색 Agent | 문제 해결 속도 향상 |
| 설비 이상 분석 Agent | 반복 장애 감소 |

따라서 온톨로지 구축 이전에 먼저 다음 사항을 정의해야 한다.

- 어떤 서비스를 만들 것인가
- 누가 사용할 것인가
- 어떤 문제를 해결할 것인가
- 기대 효과는 무엇인가

서비스와 가치가 명확하지 않다면 온톨로지 구축을 서두를 필요가 없다.

---

### 2.2 이럴 때 온톨로지가 필요하다

다음과 같은 경우에는 온톨로지가 효과적이다.

| 판단 기준 | 설명 |
|--------|--------|
| 관계가 복잡하다 | 여러 개념을 연결해야 답이 나온다 |
| 설명 가능한 답이 필요하다 | 근거를 함께 제시해야 한다 |
| 전문가 지식이 중요하다 | 암묵지를 구조화해야 한다 |
| 여러 서비스가 같은 지식을 사용한다 | 재사용 가치가 높다 |
| 데이터가 여러 시스템에 흩어져 있다 | 연결 구조가 필요하다 |

예를 들어 품질 원인 분석은 대표적인 온톨로지 활용 대상이다.

결함, 설비, 공정, 원자재, 검사 결과, 조치 이력을 함께 연결해야 하기 때문이다.

---

### 2.3 이럴 때는 만들 필요가 없다

온톨로지가 항상 최적의 선택은 아니다.

다음과 같은 경우에는 다른 방법이 더 적합할 수 있다.

| 상황 | 권장 방식 |
|--------|--------|
| 용어 통일만 필요 | Glossary |
| 단순 집계 | BI |
| 정형 데이터 분석 | SQL |
| 유사 문서 검색 | 벡터 검색 |
| 단순 챗봇 | RAG |

예를 들어 단순히 고객 불만 문서를 검색하는 수준이라면 벡터 검색과 생성형 AI만으로도 충분할 수 있다.

반면 여러 시스템에 존재하는 정보를 연결하여 원인과 조치를 추적해야 한다면 온톨로지가 필요하다.

---

### 2.4 시작은 작게

처음부터 전사 온톨로지를 구축하는 것은 권장하지 않는다.

온톨로지는 작게 시작하여 점진적으로 확장하는 것이 일반적이다.

예를 들어 다음과 같은 순서로 접근할 수 있다.

```text
들뜸(Delamination) 분석
↓

품질 영역 확장

↓

제품군 확장

↓

계열사 확장

↓

전사 공통 온톨로지
```

초기에는 특정 결함 유형이나 특정 공정을 대상으로 시작하는 것이 바람직하다.

실제 활용 가치를 검증한 이후 범위를 확장하는 것이 성공 가능성이 높다.

---

## 3. 무엇으로 이루어지나 (구성요소)

온톨로지는 단순히 개념 목록을 나열한 구조가 아니다.

개념, 관계, 계층, 규칙을 함께 정의하여 특정 도메인의 지식을 표현하는 체계이다.

또한 제조업 환경에서는 단순한 개념 정의를 넘어 실제 데이터, 문서, AI 서비스와 연결될 수 있어야 한다.

본 장에서는 온톨로지를 구성하는 핵심 요소를 설명한다.

---

### 3.1 기본 구성 요소

온톨로지는 6개의 기본 구성 요소로 이루어진다.

| 구성 요소 | 설명 | 예시 |
|--------|--------|--------|
| 클래스(Class) | 동일한 성격의 개념 그룹 | 결함, 설비, 공정 |
| 인스턴스(Instance) | 실제 개별 사례 | 특정 들뜸 결함 1건 |
| 속성(Property) | 개념이 가지는 정보 | 심각도, 발생 시각 |
| 관계(Relationship) | 개념 간 연결 | 원인이 된다 |
| 계층(Hierarchy) | 상위-하위 구조 | 결함 → 층간 결함 → 들뜸 |
| 공리(Axiom) | 규칙 및 제약 조건 | 특정 관계 규칙 |

온톨로지는 이러한 요소를 통해 현실 세계를 구조화한다.

예를 들어 "들뜸"은 결함 클래스의 하위 개념이며, 특정 원인 및 조치와 관계를 가진다.

---

### 3.2 개념을 3계층으로 구분한다

온톨로지는 모든 개념을 동일하게 취급하지 않는다.

특히 제조업에서는 안 변하는 것, 발생한 사건, 사람의 판단을 구분해야 한다.

이를 위해 3계층 구조를 활용한다.

| 계층 | 설명 | 예시 |
|--------|--------|--------|
| Layer 1 | 변하지 않는 기준 정보 | 설비, 제품, 공정 |
| Layer 2 | 실제 발생한 사건 | 검사 결과, 결함 발생 |
| Layer 3 | 사람의 판단 및 해석 | 원인 분석, 조치 결정 |

예를 들어 들뜸 결함을 분석한다고 가정하면 다음과 같이 구분할 수 있다.

| 항목 | 계층 |
|--------|--------|
| 열압착 프레스 | Layer 1 |
| 들뜸 검출 | Layer 2 |
| 흡습이 원인이라는 판단 | Layer 3 |

이 구분을 통해 사실과 해석을 분리하여 관리할 수 있다.

---

### 3.3 핵심 개념과 관계

온톨로지의 핵심은 개념 자체보다 관계에 있다.

개념 간 관계가 정의되어야 AI가 원인과 결과를 추적할 수 있다.

제조업에서 자주 사용되는 개념은 다음과 같다.

| 개념 | 설명 |
|--------|--------|
| 제품 | 생산 대상 |
| 공정 | 생산 과정 |
| 설비 | 공정을 수행하는 장비 |
| 검사 | 품질 확인 활동 |
| 결함 | 발생한 문제 |
| 원인 | 결함 발생 원인 |
| 조치 | 문제 해결 방법 |

대표적인 관계는 다음과 같다.

| 관계 | 의미 |
|--------|--------|
| 발생한다 | 공정에서 결함이 발생 |
| 검출된다 | 검사에서 결함 발견 |
| 원인이 된다 | 특정 원인이 결함 발생 |
| 조치한다 | 특정 조치가 결함 해결 |
| 사용한다 | 공정이 설비를 사용 |
| 포함한다 | 상위 개념 포함 관계 |

예를 들어 다음과 같은 관계를 구성할 수 있다.

```text
프리프레그 흡습
→ 원인이 된다
→ 들뜸

들뜸
→ 발생 공정
→ 열압착 공정

들뜸
→ 조치
→ 프리프레그 건조
```

### 3.4 코어 온톨로지와 유즈케이스 온톨로지

온톨로지는 재사용을 위해 두 계층으로 구분하는 것이 일반적이다.

| 구분 | 설명 |
|--------|--------|
| 코어 온톨로지 | 현실 세계를 설명하는 공통 구조 |
| 유즈케이스 온톨로지 | 특정 서비스 목적의 확장 구조 |

코어 온톨로지는 가능한 한 안정적으로 유지한다.

예를 들어 다음 개념은 코어 영역에 속한다.

- 제품
- 설비
- 공정
- 결함
- 원인
- 조치

반면 아래 항목은 유즈케이스에 따라 달라질 수 있다.

- 품질 진단 Agent
- 원인 분석 Dashboard
- 조치 추천 서비스

유즈케이스가 변경되더라도 코어 구조는 유지하는 것이 원칙이다.

### 3.5 공통 온톨로지와 계열사 온톨로지

여러 계열사가 온톨로지를 사용할 경우 공통 구조와 계열사 특화 구조를 분리하는 것이 필요하다.

| 구분 | 설명 |
|--------|--------|
| 공통 온톨로지 | 전사 공통 개념 |
| 계열사 온톨로지 | 특정 산업·제품 특화 개념 |

예를 들어 모든 계열사는 "결함"이라는 개념을 사용할 수 있다.

반면 구체적인 결함 유형은 계열사마다 다를 수 있다.

| 공통 개념 | 계열사 특화 예시 |
|--------|--------|
| 결함 | 들뜸 |
| 결함 | 용접 기공 |
| 결함 | 절연 파괴 |

계열사는 공통 구조를 재사용하면서 필요한 개념만 확장한다.

### 3.6 데이터와 연결한다

온톨로지는 실제 데이터와 연결되어야 활용 가능하다.

연결되지 않은 온톨로지는 단순한 설계 문서에 불과하다.

대표적인 연결 대상은 다음과 같다.

| 연결 대상 | 역할 |
|--------|--------|
| Glossary | 용어 정의 |
| 메타데이터 | 데이터 구조 정의 |
| 데이터 카탈로그 | 데이터 위치 정보 |
| SOP | 업무 기준 |
| PFMEA | 품질 분석 기준 |
| C/S Report | 실제 사례 |

예를 들어 들뜸 결함은 다음 데이터와 연결될 수 있다.

| 온톨로지 개념 | 연결 데이터 |
|--------|--------|
| 들뜸 | 검사 결과 |
| 열압착 공정 | 공정 이력 |
| 프리프레그 | 원자재 정보 |
| 흡습 | 원인 분석 보고서 |
| 건조 | 조치 이력 |

온톨로지가 실제 데이터와 연결되어야 AI가 개념과 사례를 함께 활용할 수 있다.
