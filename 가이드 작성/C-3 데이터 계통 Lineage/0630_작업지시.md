# AI-ready Data Guide 공통 작업 지시서 (v3)

> 본 작업 지시서는 AI-ready Data 20개 주제 전체에 공통 적용한다.
> 목적은 기존 문서를 부분 수정하는 것이 아니라, 모든 가이드를 동일한 구조·동일한 문체·동일한 품질 수준으로 통일하는 것이다.
>
> 모든 결과물은 B-1(데이터 전처리), B-2(데이터 해설/주석), B-3(온톨로지)를 기준 문서(Leveling Reference)로 삼는다.

---

# 1. 기본 원칙

## 목적

가이드는 단순 개념 설명서가 아니다.

계열사가 실제 구축하고 운영하며 AI-ready Data를 지속적으로 축적할 수 있도록 하는 실행 가이드이다.

대표이사 피드백을 반영하여 AI-ready Data는 선택적으로 참고하는 Best Practice가 아니라 앞으로 모든 AI 과제에서 공통으로 적용되는 데이터 자산화 체계라는 관점을 유지한다.

따라서 문서는 설명보다 **구축 방법**, **판단 기준**, **운영 방법** 중심으로 작성한다.

---

# 2. 문서 최상위 구조

모든 가이드는 아래 순서를 기본으로 한다.

```
제목

목차

관련 가이드

머릿말 (여기에 주제 정의를 녹이던지)

이 가이드가 답하는 5가지 질문

1. Why
2. What
3. When (필요한 경우)
4. How
5. Tech Stack
6. Where
```

Appendix (선택)
참고문헌 (선택)

Appendix와 참고문헌은 선택 사항으로 한다.

Appendix에는 본문의 이해를 돕는 부가 자료(템플릿, 예시 양식, 상세 표, 추가 사례, 용어집 등)를 포함한다.
참고문헌(References)에는 표준, 논문, 공식 문서, 기술 문서, 공개 자료 등 본문 작성에 참고한 자료를 기재한다.

Appendix와 참고문헌은 본문의 흐름을 방해하지 않는 범위에서만 작성하며, 본문에서 반드시 다루어야 할 내용을 Appendix로 이동하지 않는다.

추가 장은 원칙적으로 만들지 않는다.

---

# 3. 목차 작성 규칙

모든 가이드는 아래 구조를 기본으로 작성한다.

없는 장은 생략 가능하지만

**Why / What / How / Tech Stack은 반드시 포함한다.**

```
1. Why — 왜 필요한가

2. What — 무엇을 준비하는가

3. When — 언제 적용하는가 (필요 시)

4. How — 어떻게 준비·운영하나

5. Tech Stack — 어떻게 구현하는가

6. Where — 어디에 적용되는가
```

# 15. 설명 방식

개념 설명은 최소화한다. 중간 중간 독자의 이해를 돕는 비쥬얼, 다이어그램, 박스 차트, 그래프, 등을 추가하되, RACI 차트 또는 4사분면 그래프이지만 네개의 사분면에 걸쳐 데이터가 고르게 분포되어 있지 않은 경우 등은 사용 금지.

반드시

```
필요성

↓

구성

↓

판단 기준

↓

구축 절차

↓

운영
```

순으로 작성한다.

개념 설명만 길게 이어지는 구조는 지양한다.

# 20. 생성 원칙

기존 문장을 부분 수정하는 것을 목표로 하지 않는다.

아래 기준으로 판단한다.

- 구조가 맞지 않으면 새로 작성한다.
- 문체가 다르면 새로 작성한다.
- 위치가 맞지 않으면 이동한다.
- 중복은 통합한다.
- 설명보다 구축 절차와 운영 기준을 우선한다.

# 21. 분량 원칙

가이드는 길이가 아니라 완성도를 목표로 한다.

억지로 분량을 늘리거나 라인 수를 채우기 위해 동일한 내용을 반복하지 않는다.

설명이 끝났다면 더 이상 문장을 추가하지 않는다.

권장 분량은 **500~700라인**이며, 대부분의 주제는 이 범위 안에서 충분히 설명 가능하다.

주제 특성상 필요한 경우를 제외하고 800라인 이상 작성하지 않는다.

다음과 같은 방식으로 분량을 늘리는 행위를 금지한다.

- 동일한 내용을 문장만 바꾸어 반복
- 이미 설명한 개념을 다른 장에서 다시 설명
- 의미 없는 사례 추가
- 불필요한 표 또는 그림 추가
- 같은 메시지를 여러 표현으로 반복
- 불필요한 서론 및 배경 설명

설명은 최소화하고 판단 기준, 구축 절차, 운영 기준 중심으로 작성한다.

필요한 내용을 모두 설명했다면 더 이상 분량을 늘리지 않는다.

# 22. 최종 Self Review Checklist

생성 완료 후 반드시 아래 항목을 스스로 점검한다.

□ B-1, B-2, B-3 수준과 동일한 문체인가

□ Why → What → When → How → Tech Stack → Where 순서를 유지했는가

□ Second Level 구조가 전체적으로 일관되는가

□ KPI가 How로 통합되었는가

□ Roadmap이 별도 장으로 존재하지 않는가

□ 예시 시나리오가 How에 포함되었는가

□ Canonical Model이 문서 전체에서 일관되게 사용되었는가

□ 다른 주제와의 경계가 Where에서 정리되었는가

□ ChatGPT식 표현이 제거되었는가

□ 보고서 문체로 작성되었는가

□ 동일한 내용을 반복하여 분량을 늘리지 않았는가

□ 권장 분량(500~700라인) 안에서 충분한 내용을 담았는가

□ 설명보다 구축 방법과 운영 방법의 비중이 높은가

□ 기존 내용을 억지로 수정하지 않고 필요한 경우 재작성했는가
