## 2. Why — 제조 AI 데이터 부족의 구조적 유형

### 2-A. 왜 필요한가: 핵심 논리 구조

합성데이터가 필요한 이유는 단일하지 않다. 제조업 AI 과제에서 데이터가 부족한 데에는 **서로 다른 구조적 원인**이 있으며, 각 원인은 합성데이터가 보완하는 방식도 다르다.

```
데이터 부족 원인 유형
  ├─ ① 희소 이벤트 (rare event)
  ├─ ② 데이터 불균형 (class imbalance)
  ├─ ③ 보안·규제 제약
  ├─ ④ 검증 데이터 부족 (극한·고장 시나리오)
  ├─ ⑤ 콜드스타트 (신규 제품·설비)
  └─ ⑥ 확보 리드타임·비용 (부수 효과)
```

---

### 2-B. 유형별 구조 설명

#### ① 희소 이벤트 (Rare Event)

**구조**: 전체 생산 중 불량·장애는 극소수 비율(0.1% 수준). 수십만 건 생산 이력이 있어도 실제 학습에 쓸 수 있는 불량 샘플은 수백 건에 그칠 수 있다.

**왜 문제인가**: AI 모델은 충분한 사례를 반복적으로 학습해야 패턴을 추출한다. 불량 샘플이 절대적으로 적으면 모델은 불량의 특징을 제대로 학습하지 못하고, 결국 "모든 것을 정상으로 판정"하는 방향으로 편향된다.

**CCL Delamination 예시**: 두산전자 CCL 라인에서 Delamination 불량이 전체 생산의 0.1~0.3% 수준이라면, 연간 100만 장 생산 기준으로 불량 샘플은 1,000~3,000건. 그러나 이미지 AI 학습에는 수천~수만 건 이상의 균형 잡힌 데이터가 필요하므로 실데이터만으로는 충분하지 않다.

**합성이 보완하는 방식**: 기존 불량 이미지에서 회전·밝기·노이즈 변형(데이터 증강) 또는 GAN 기반 생성으로 불량 샘플 수를 늘린다.

---

#### ② 데이터 불균형 (Class Imbalance)

**구조**: 희소 이벤트와 연결되나, 여기서는 "학습 데이터셋 내 클래스 비율의 심한 불균형"이 핵심이다. 정상:불량 = 99.8:0.2 같은 비율에서 모델을 훈련하면, 모델은 항상 "정상"이라 예측해도 99.8%의 정확도를 얻으므로 불량 감지 자체를 포기하는 방향으로 수렴한다.

**왜 문제인가**: 정확도(Accuracy) 지표가 높아도 재현율(Recall)이 0에 가까울 수 있다. 제조업 품질 검사에서 불량 미탐(False Negative)은 직접적인 품질·비용 손실로 연결된다.

**합성이 보완하는 방식**: 소수 클래스(불량)를 합성으로 늘려 학습 데이터셋에서 클래스 비율을 균형 있게 맞춘다(오버샘플링). 이는 실제 불량 발생 비율을 바꾸는 것이 아니라 **학습용 데이터셋 구성**의 문제다.

> **출처(통념 수준, 교과서적 사실)**: 클래스 불균형 문제와 합성 오버샘플링(SMOTE 등)은 머신러닝 분류 문제에서 널리 알려진 접근. 특정 통계 수치 인용 없음.

---

#### ③ 보안·규제 제약

**구조**: 실데이터에 개인정보(이름·주민번호·급여), 영업비밀(계약가·거래처), 고객 기밀이 포함된 경우 AI 학습용으로 쓰거나 외부(협력사·솔루션사)에 제공하는 데 법적·정책적 제약이 생긴다.

**F-4 비식별(De-identification)과의 차이**:
- **비식별**: 원본 데이터에서 식별자를 제거·마스킹·익명화해 재활용. 원본 레코드를 기반으로 하므로 완전한 프라이버시 보장이 어려울 수 있음(재식별 위험).
- **합성데이터**: 원본 레코드를 직접 변형하지 않고, 원본의 통계적 패턴만 학습해 새 데이터를 생성. 원본과 1:1 대응 관계가 없으므로 재식별 위험이 구조적으로 낮다.
- **선택 기준**: 원본 레코드 보존이 필요 없고 분포·패턴만 재현하면 충분한 경우 → 합성 적합. 원본 레코드 자체가 필요한 경우(감사·소급 분석 등) → 비식별.

**프라이버시 규제 맥락(가볍게)**:
- GDPR(유럽 개인정보보호규정)은 개인정보 처리 요건을 강화하면서, 합성데이터를 "개인정보가 아닌 대안"으로 활용하는 논의가 학계·산업계에서 진행 중이다. 단, GDPR 규제기관의 공식 입장은 합성 방식·품질에 따라 다르게 해석될 수 있다(확인 필요 — 공식 가이던스는 케이스마다 다를 수 있음).
- 국내 개인정보보호법(2023년 개정)은 가명처리·익명처리 기준을 명시하며, 합성데이터가 익명처리에 해당할 경우 보호법 적용 대상에서 제외될 수 있다는 논의가 있다(확인 필요 — 개인정보보호위원회의 해석 사례 참조 필요).
- **실용적 관점**: 규제 해석 여부와 무관하게, 원본 데이터를 외부에 직접 제공하지 않아도 되는 실무 이점은 명확하다.

> 출처: GDPR 관련 합성데이터 논의는 [European Data Protection Board](https://www.edpb.europa.eu/) 및 [UK ICO](https://ico.org.uk/about-the-ico/research-and-reports/research-and-reports-into-ai-and-data-protection/) 가이드에서 언급되나, 법적 확정 해석은 아님(확인 필요, 접속일: 2026-06-22). 국내: 개인정보보호위원회 ([www.pipc.go.kr](https://www.pipc.go.kr/)), 가명처리·익명처리 기준 가이드라인 참조.

---

#### ④ 검증 데이터 부족 — 극한·고장 시나리오

**구조**: AI 모델은 학습 데이터뿐 아니라 다양한 상황에서 얼마나 잘 동작하는지 검증하는 데이터도 필요하다. 그러나 극한 조건(과열·과압·비상정지)이나 고장 시나리오는 실제 현장에서 의도적으로 재현하기 어렵다 — 재현 자체가 위험하거나, 비용이 크거나, 법적으로 불가하다.

**왜 문제인가**: 검증 데이터가 부족하면 AI 모델이 정상 범위 내에서만 동작하고, 이상 상황에서는 신뢰할 수 없다. 이는 안전 중요 공정(Safety-critical process)에서 특히 치명적이다.

**합성이 보완하는 방식**: 물리 시뮬레이션(디지털 트윈)이나 업무 규칙 기반 시나리오 생성으로 실제 재현이 어려운 상황을 인공으로 만들어 검증셋에 포함한다.

---

#### ⑤ 콜드스타트 (Cold Start) — 신규 제품·설비

**구조**: 신제품을 출시하거나 신규 설비를 도입하면 이상 패턴 데이터가 아예 없다. 기존 유사 제품의 데이터가 있어도 공정 조건·소재·설비 특성이 달라 직접 재사용하기 어렵다.

**왜 문제인가**: 데이터 축적 이전에 AI 품질검사·이상탐지 모델을 바로 가동할 수 없다. 초기 운영 기간에 모델이 없으면 AI 도입의 효과가 지연된다.

**합성이 보완하는 방식**: 유사 공정의 통계적 패턴을 참조하거나, 설비 스펙 기반 시뮬레이션 데이터로 초기 모델을 구성한다. 실데이터가 쌓이면서 합성 비중을 줄이고 실데이터 비중을 높이는 전략으로 전환.

---

#### ⑥ 확보 리드타임·비용 (부수 효과)

실데이터 수집·정제·라벨링은 시간과 비용이 많이 든다. 합성데이터는 이를 부분 대체할 수 있으나, 이것은 위 ①~⑤ 구조적 한계를 해소한 뒤에 오는 부수 효과다. 비용 절감 자체를 합성의 1순위 이유로 내세우면 "실데이터보다 싸게 대체하자"는 오해로 이어지므로 주의.

## 3. When — 언제/어디에 합성하나 (적용 판단)

> 합성데이터는 모든 AI 과제에 필요하지 않다. **골라서 하는 주제**다.

### 3-A. 이럴 때 한다 (적용 신호)

아래 중 하나 이상에 해당하면 합성데이터 적용을 검토한다.

| 신호 | 판단 지표 예시 | 대응 유형 |
|------|---------------|-----------|
| 희소 이벤트 | 불량률 1% 미만, 불량 샘플 < 수백 건 | ① 희소 이벤트 |
| 클래스 불균형 | 정상:불량 비율 100:1 이상 | ② 데이터 불균형 |
| 보안·규제 제약 | 개인정보·영업비밀 포함, 외부 공유 필요 | ③ 보안·규제 |
| 검증 시나리오 부족 | 극한·고장 상황을 실제로 재현할 수 없음 | ④ 검증 부족 |
| 콜드스타트 | 신규 제품·설비 도입 초기, 이력 데이터 없음 | ⑤ 콜드스타트 |

---

### 3-B. 이럴 땐 하지 않는다 (적용 불필요 신호)

아래 조건을 **모두** 만족하면 합성데이터가 불필요하다.

| 조건 | 판단 근거 |
|------|-----------|
| 실데이터 충분 | 학습·검증에 필요한 양과 클래스 균형이 이미 확보됨 |
| 보안 제약 없음 | 원본 데이터를 AI 학습·외부 협업에 직접 활용 가능 |
| 검증 데이터 확보 가능 | 극한·고장 상황을 실제 데이터나 운영 이력으로 커버 가능 |
| 신규 제품·설비 아님 | 이미 충분한 이력이 쌓인 공정·제품 |

→ **실무 판단**: 이 4가지 중 어느 하나도 제약이 없다면 굳이 합성 불필요. 시간과 비용 투자 대비 실익이 없다.

---

### 3-C. ★ 반드시 먼저: 실데이터(검증셋) 확보 원칙

**합성데이터를 시작하기 전에, 비교 기준이 될 실데이터(검증셋)를 반드시 먼저 확보해야 한다.**

이유: 합성데이터가 실제 현상을 제대로 재현하는지(충실도, Fidelity) 확인하려면 실데이터와 비교할 수 있어야 한다. 검증셋이 없으면 합성데이터의 품질을 판단할 기준이 없고, AI 모델이 학습 후 실제 현장에서 제대로 동작하는지 알 수 없다.

**실무 규칙**: 가용한 실데이터를 학습용(train)과 검증용(test/val)으로 나눈다. 합성데이터는 학습용에만 보강하고, 검증은 반드시 실데이터로 한다.

```
[실데이터]
  ├─ 학습셋(train): 실 + 합성으로 보강 가능
  └─ 검증셋(test/val): 반드시 실데이터만 → 모델 성능 기준점
```

이 원칙을 지키지 않으면 합성으로만 학습·검증하여 "현장에서는 동작 안 하는 모델"을 만들 위험이 있다.

---

### 3-D. 적용 판단 흐름도 (개념 구조)

```
현재 AI 과제의 데이터를 점검한다
│
├─[불량 샘플 절대 부족?] ─ 예 → ① 희소 이벤트 → 합성 검토
├─[정상:불량 비율 극단적 불균형?] ─ 예 → ② 데이터 불균형 → 합성 검토
├─[개인정보·기밀 포함 or 외부 공유?] ─ 예 → ③ 보안·규제 → 합성 검토
├─[극한·고장 시나리오 재현 불가?] ─ 예 → ④ 검증 부족 → 합성 검토
├─[신규 제품·설비 도입 초기?] ─ 예 → ⑤ 콜드스타트 → 합성 검토
│
└─[위 5가지 모두 아니오?] → 실데이터로 충분 → 합성 불필요

합성 검토 진입 후
└─ 실데이터 검증셋 확보 가능? ─ 아니오 → 검증셋 확보 먼저
                              └─ 예 → 합성 방식 선택(→ How)
```

## 4. 기대 효과 (Why의 마무리)

합성데이터를 적절히 활용했을 때의 효과를 구조화한다. 이것은 "Why가 해소되면 무엇이 좋아지는가"의 답이다.

| 효과 | 연결되는 Why 유형 | 구체 예시 |
|------|-----------------|-----------|
| 검사·이상탐지 AI 성능 향상 | ①② 희소 이벤트·불균형 해소 | Delamination 탐지 재현율(Recall) 개선 |
| 안전한 외부 데이터 협업 | ③ 보안·규제 | 솔루션사·연구기관에 합성데이터 제공 |
| 검증 범위 확대 | ④ 극한·고장 시나리오 | 비상정지·과열 시나리오 AI 검증 가능 |
| 신규 제품 AI 조기 가동 | ⑤ 콜드스타트 | 출시 전 모델 사전 구성 후 실데이터로 점진 전환 |
| 데이터 확보 리드타임 단축 | ⑥ 부수 효과 | 라벨링 없이 보강 샘플 즉시 생성 가능 |

## 5. 초안 개선 제안 요약

기존 초안의 내용은 대부분 방향이 맞다. 다음 구조 조정이 필요하다.

1. **§2(Why)에 "데이터 불균형(class imbalance)" 유형을 추가**: 지금은 §4에만 있음. Why의 논리 흐름상 불균형이 "왜 문제인가"를 §2에서 먼저 설명해야 §4의 판단 기준이 납득됨.

2. **F-4 비식별과의 차이 한 줄 추가**: §2.2 보안·규제 섹션에 "비식별은 원본 레코드 변형, 합성은 패턴만 학습해 새 데이터 생성 — 재식별 위험 구조가 다름" 한 줄.

3. **"합성 전 실데이터 검증셋 먼저 확보" 원칙**: 현재 초안 어디에도 없음. §4 또는 §2.5 AI-ready Data 관점에 명시 필요.

4. **비용·시간(§2.4)을 부수 효과로 재배치**: 핵심 Why가 아니라 결과/효과 측면. 별도 섹션보다는 "기대 효과" 표의 마지막 항목으로 처리하거나, §2.4를 "기대 효과" 절로 전환.

5. **§4의 판단 기준에 "흐름도(분기)" 추가**: 지금은 나열형. "5가지 신호 중 하나도 없으면 불필요"라는 판단 흐름이 보이도록 구조화.

6. **프라이버시 규제 맥락(가볍게)**: §2.2 마지막에 1~2 문장 추가. GDPR·국내법 문맥에서 합성데이터가 대안으로 주목받는다는 흐름만, 단정 없이.
