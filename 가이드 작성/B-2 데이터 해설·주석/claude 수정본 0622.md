# B-2 데이터 해설·주석 매뉴얼

데이터 해설·주석(Data Annotation)은 사람의 판단 기준을 데이터 자산으로 전환하는 활동이다.

원본 데이터(이미지·문장·로그)에 사람이 의미 정보를 부여함으로써 AI가 학습 가능한 데이터로 만드는 과정이다.

예를 들어 외관 검사 이미지에 결함 유형과 위치를 표시하면, 사람만 해석하던 이미지가 AI 학습 데이터로 전환된다.

이 문서는 데이터 해설·주석을 개념으로 설명하는 데 그치지 않는다.

계열사가 실제로 무엇을, 어떤 순서로, 누가 수행하여 재사용 가능한 데이터 자산을 구축하는지를 설명한다.

# 1. 개요

## 1.1 데이터 해설·주석이란

데이터 해설·주석(Data Annotation)은 원본 데이터에 사람이 의미 정보를 부여하는 활동이다.

AI는 원본 데이터만으로는 데이터의 의미를 이해하지 못한다.

예를 들어 외관 검사 이미지가 존재하더라도 어떤 결함인지, 어느 위치에 존재하는지, 얼마나 심각한 문제인지를 스스로 판단할 수 없다.

고객 클레임 문장이 존재하더라도 어떤 원인에 대한 이야기인지, 어떤 조치가 필요한지를 알 수 없다.

따라서 사람의 판단 기준을 구조화된 데이터 형태로 기록하여 AI가 활용 가능한 학습 데이터로 전환해야 한다.

데이터 해설·주석은 단순한 라벨링 작업이 아니다.

현업 전문가의 판단 기준을 데이터 자산으로 전환하고, 향후 다양한 AI 과제에서 재사용 가능한 형태로 축적하는 활동이다.

---

## 1.2 주요 참여 조직

데이터 해설·주석은 특정 조직만 수행하는 업무가 아니다.

현업 조직, 데이터 조직, AI 조직이 함께 참여해야 한다.

| 조직 | 역할 |
|--------|--------|
| 전사 데이터 조직 | 표준 체계 수립 및 운영 정책 관리 |
| 현업 SME | 분류 체계 정의 및 최종 승인 |
| 데이터 조직 | Taxonomy, Guideline, 품질 체계 구축 |
| 라벨러 | 실제 Annotation 수행 |
| 검수자 | 품질 검증 및 재작업 관리 |
| AI 조직 | 자동화 및 AI 활용 체계 구축 |

특히 현업 SME의 역할이 중요하다.

데이터 해설·주석은 현업의 판단 기준을 데이터로 전환하는 과정이기 때문에 현업 기준 없이 품질을 확보할 수 없다.

---

## 1.3 AI-ready Data 체계 내 위치

데이터 해설·주석은 AI-ready Data 체계에서 전처리 이후 단계에 위치한다.

전처리가 데이터를 AI가 읽을 수 있는 형태로 변환하는 과정이라면, 데이터 해설·주석은 해당 데이터에 의미를 부여하여 AI가 학습 가능한 형태로 완성하는 과정이다.

```mermaid
flowchart LR

A[원본 데이터]
--> B[데이터 전처리]
--> C[데이터 해설·주석]
--> D[AI 학습]
--> E[Registry 등록]
--> F[재사용 및 확산]
```

주석 결과물은 특정 AI 프로젝트에서만 사용되는 데이터가 아니다.

정상적으로 구축된 주석 데이터셋은 Registry에 등록되어 향후 다른 AI 과제에서도 활용 가능한 데이터 자산으로 관리된다.

---

## 1.4 데이터 해설·주석의 최종 목표

데이터 해설·주석의 목적은 AI 모델 하나를 만드는 것이 아니다.

최종 목표는 조직의 판단 기준을 데이터 자산으로 축적하는 것이다.

예를 들어 동판 외관 검사 데이터를 대상으로 구축한 주석 데이터셋은 다음과 같이 활용될 수 있다.

- 외관 검사 자동화
- 결함 원인 분석
- 품질 예측
- 이상 탐지
- 품질 Agent
- 생성형 AI 기반 품질 지원 서비스

따라서 데이터 해설·주석은 프로젝트 산출물이 아니라 지속적으로 관리되는 데이터 자산으로 운영되어야 한다.

---

# 2. 왜 필요한가 (Why)

AI 프로젝트에서 가장 흔하게 발생하는 문제는 모델 성능 부족이 아니라 학습 데이터 품질 부족이다.

많은 조직은 충분한 데이터를 보유하고 있다고 생각하지만 실제로는 AI가 학습할 수 있는 형태의 데이터가 준비되어 있지 않은 경우가 많다.

데이터 해설·주석 체계는 이러한 문제를 해결하기 위해 필요하다.

---

## 2.1 현업 Pain Point

두산전자는 동판 외관 검사 자동화를 위해 결함 이미지 약 5만 장을 확보하였다.

그러나 실제 AI 모델 학습을 시작하는 과정에서 다음과 같은 문제가 확인되었다.

- **정답 데이터의 부재:** 이미지는 존재하지만 결함 유형이 표준화되어 있지 않아 AI 학습에 바로 활용할 수 없다.
- **판단 기준의 불일치:** 동일한 결함에 대해 작업자마다 서로 다른 명칭과 기준을 사용하여 일관된 학습 데이터 구축이 어렵다.
- **운영 이력 및 품질 관리 부재:** 누가 어떤 기준으로 주석을 수행했는지 추적이 어렵고 오류 라벨을 체계적으로 관리하기 어렵다.
- **재사용 체계의 부재:** 프로젝트 종료 이후 주석 데이터가 방치되어 다른 AI 과제에서 재사용되지 못한다.

결과적으로 데이터는 존재하지만 AI가 학습할 수 있는 형태의 정답 데이터는 부족한 상태가 된다.

---

## 2.2 기대 효과

### 데이터 의미의 표준화

사람마다 다르게 해석하던 데이터를 공통 기준으로 관리할 수 있다.

동일한 결함은 누구나 동일한 기준으로 판정하게 된다.

### AI 학습 품질 향상

정확한 정답 데이터를 기반으로 AI를 학습시킬 수 있다.

결과적으로 모델 성능과 신뢰도가 향상된다.

### 운영 효율 향상

AI가 1차 분류를 수행하고 사람은 검수 중심으로 참여할 수 있다.

반복 업무를 줄이고 생산성을 향상시킬 수 있다.

### 데이터 자산화

한 번 구축한 데이터셋을 다양한 AI 과제에서 재사용할 수 있다.

데이터 구축 비용이 일회성 비용이 아니라 장기 자산으로 전환된다.

---

## 2.3 AI-ready Data 관점의 의미

데이터 해설·주석은 AI 모델을 위한 보조 작업이 아니다.

AI-ready Data 체계에서 현업의 판단 기준을 디지털 자산으로 전환하는 핵심 활동이다.

예를 들어 외관 검사 전문가가 10년 동안 축적한 경험은 사람의 머릿속에만 존재할 경우 조직 자산이 아니다.

반면 해당 판단 기준을 Taxonomy, Guideline, 사례집, Gold Dataset 형태로 구조화하면 조직이 지속적으로 활용할 수 있는 데이터 자산이 된다.

따라서 데이터 해설·주석의 핵심 목적은 단순히 AI를 학습시키는 것이 아니라 현업의 판단 기준을 재사용 가능한 데이터 자산으로 만드는 것이다.

---


# 3. 무엇을 갖추나 (What)

데이터 해설·주석은 단순히 라벨을 부여하는 작업이 아니다.

AI가 활용할 수 있는 데이터를 지속적으로 구축하고 재사용하기 위해서는 분류 체계, 품질 기준, 운영 체계, 데이터 자산화 체계가 함께 필요하다.

또한 데이터 해설·주석은 프로젝트 단위 산출물이 아니라 향후 다양한 AI 과제에서 반복적으로 활용될 수 있는 데이터 자산이라는 관점에서 접근해야 한다.

본 장에서는 데이터 해설·주석 체계를 구성하는 핵심 요소를 설명한다.

---

## 3.1 정본 모델

본 가이드는 아래 정본 모델을 기준으로 작성한다.

```text
대상 데이터 선정
→ Taxonomy 설계
→ Guideline 작성
→ Pilot Annotation
→ IAA 측정
→ 본 Annotation
→ QA
→ 자동화 및 자산화
```

데이터 해설·주석은 단순히 라벨을 붙이는 행위가 아니라 위 8개 단계를 통해 품질을 확보하고 재사용 가능한 데이터 자산을 구축하는 과정이다.

대상 데이터를 선정한 이후 분류 체계를 정의하고, 이를 기준으로 Guideline을 구축한다.

이후 시범 주석 작업을 수행하여 기준의 적절성을 검증하고, 작업자 간 일관성을 확보한 뒤 본 주석 작업을 수행한다.

최종적으로는 품질 검증을 거쳐 재사용 가능한 데이터 자산으로 등록하고 운영한다.

본 정본 모델은 5장의 사례 시나리오, 6장의 실행 방법론, 8장의 Roadmap에서도 동일하게 사용한다.

---

## 3.2 Taxonomy 및 라벨 정의 체계

Taxonomy는 데이터를 어떤 기준으로 분류할 것인지 정의하는 체계이다.

주석 품질은 결국 Taxonomy 품질에 의해 결정된다.

좋은 Taxonomy는 다음 조건을 만족해야 한다.

- 상호배타성
- 포괄성
- 일관성

예를 들어 동판 외관 검사 데이터를 대상으로 주석 체계를 설계한다고 가정하면 아래와 같은 구조를 사용할 수 있다.

```text
결함
├─ 균열
├─ 눌림
├─ 오염
└─ 긁힘
```

각 라벨은 정의서를 통해 관리한다.

| 코드 | 라벨 | 정의 |
|--------|--------|--------|
| DEF-01 | 균열 | 표면이 선형으로 갈라진 결함 |
| DEF-02 | 눌림 | 외부 압력에 의해 함몰된 결함 |
| DEF-03 | 오염 | 이물질 부착 |
| DEF-04 | 긁힘 | 표면 긁힘 흔적 |

Taxonomy는 향후 AI 모델의 학습 기준이 되며, 작업자 교육, 품질 검수, 자동화 체계의 기준으로 사용된다.

---

## 3.3 Guideline 및 사례집

Guideline은 작업자들이 동일한 기준으로 데이터를 분류할 수 있도록 만드는 문서이다.

사례집(Casebook)은 Guideline을 실제 사례로 설명하는 자료이다.

Guideline은 다음 요소를 포함한다.

- 정의
- 포함 사례
- 제외 사례
- 경계 사례
- 결정 규칙

예를 들어 균열과 긁힘을 구분하는 경우 아래와 같은 규칙을 정의할 수 있다.

```text
IF 선형 갈라짐 존재
→ 균열

IF 표면 긁힘 흔적 존재
→ 긁힘

IF 판단 불가
→ SME 검토
```

사례집은 대표 사례, 제외 사례, 경계 사례를 포함하며 작업자 교육과 품질 개선에 활용된다.

---

## 3.4 데이터 유형별 주석 방식

주석은 데이터 유형에 따라 적용 방식이 달라진다.

| 데이터 유형 | 대표 주석 방식 | 제조업 활용 예 |
|--------|--------|--------|
| 텍스트 | 문서 분류, 개체 추출, 구간 태깅 | 고객 클레임 분석 |
| 이미지 | 분류, 위치 표시, 영역 표시 | 결함 이미지 분석 |
| 영상 | 객체 추적, 이벤트 구간 표시 | 조립 공정 분석 |
| 음성 | 이벤트 탐지, 음성 전사 | 설비 이상음 분석 |

제조업에서는 이미지 데이터가 가장 대표적인 주석 대상이다.

특히 외관 검사, 품질 검사, 설비 점검 분야에서는 이미지 기반 AI 과제가 지속적으로 증가하고 있다.

과제 목적에 따라 단순 분류, 위치 표시, 영역 표시 중 적절한 방식을 선택해야 한다.

---

## 3.5 품질 기준 체계

주석 데이터 품질은 작업자 개인이 아니라 운영 체계에 의해 관리된다.

대표적인 품질 기준은 다음과 같다.

| 기준 | 목적 |
|--------|--------|
| IAA | 작업자 간 일관성 측정 |
| Gold Dataset | 품질 기준점 제공 |
| QA | 최종 품질 검증 |

IAA는 작업자 간 동일한 결과가 나오는지를 측정한다.

Gold Dataset은 전문가 검증이 완료된 기준 데이터셋으로, 품질 평가와 AI 성능 평가의 기준점 역할을 수행한다.

QA는 최종 데이터셋이 학습에 활용 가능한 수준인지 검증하는 단계이다.

---

## 3.6 데이터셋 버전 관리

주석 데이터는 시간이 지남에 따라 지속적으로 변경된다.

새로운 결함 유형이 추가될 수 있고, 기존 분류 체계가 변경될 수 있으며, 품질 검토 결과 일부 데이터가 재작업될 수도 있다.

따라서 주석 데이터는 일반 문서와 동일하게 버전 관리되어야 한다.

| 항목 | 설명 |
|--------|--------|
| 데이터셋 버전 | 버전 번호 |
| 변경 사유 | 변경 이유 |
| 작업자 | 수정 수행자 |
| 검수자 | 승인자 |
| 변경 일시 | 변경 시점 |
| 활용 이력 | 적용 모델 및 과제 |

버전 관리 체계는 단순한 이력 관리 목적만 갖지 않는다.

향후 데이터 자산화, 재사용, 품질 추적, 문제 분석의 기반이 된다.

따라서 데이터 해설·주석 체계를 구축할 때부터 버전 관리 방안을 함께 설계해야 한다.

---

# 4. 어디부터 하나 (Where)

모든 데이터를 주석 대상으로 선정할 필요는 없다.

주석은 비용과 시간이 많이 소요되는 작업이므로 AI 활용 가치가 높고 사람의 판단이 필요한 데이터부터 우선 적용해야 한다.

## 4.1 주석 대상 선정 기준

주석 대상은 다음 다섯 가지 기준을 활용하여 선정한다.

| 기준 | 설명 |
|--------|--------|
| 사람의 판단 필요성 | 사람의 해석이 있어야 의미를 이해할 수 있는가 |
| AI 활용 가능성 | AI 과제에 활용될 가능성이 높은가 |
| 데이터 확보 수준 | 충분한 데이터가 존재하는가 |
| 품질 영향도 | 업무 결과에 직접적인 영향을 미치는가 |
| 재사용 가능성 | 향후 다른 AI 과제에서도 활용 가능한가 |

예를 들어 동판 외관 검사 이미지는 사람의 판단이 필요하고, AI 활용 가능성이 높으며, 품질에 직접적인 영향을 미치기 때문에 대표적인 주석 대상이다.

---

## 4.2 우선 적용 대상

제조업 환경에서는 다음과 같은 데이터가 대표적인 주석 대상이 된다.

| 데이터 유형 | 주석이 필요한 이유 | 우선순위 |
|--------|--------|--------|
| 결함 이미지 | 결함 유형과 위치를 사람이 판단해야 함 | 높음 |
| 고객 클레임 | 원인과 조치 유형이 자연어에 포함됨 | 높음 |
| 설비 이상 이력 | 이상 유형과 원인 해석이 필요함 | 중간 |
| 시험 결과 보고서 | 판정 기준과 조치 결과가 서술형으로 존재함 | 중간 |
| 일반 생산 실적 | 이미 구조화된 수치·코드 중심 | 낮음 |

결함 이미지와 고객 클레임은 AI 활용 가치가 높고 현업 판단이 중요한 데이터이므로 우선 적용 대상으로 적합하다.

특히 결함 이미지는 외관 검사 자동화, 결함 원인 분석, 품질 예측 등 여러 AI 과제로 확장 가능하기 때문에 데이터 자산화 효과가 크다.

---

## 4.3 적용하지 않아도 되는 경우

데이터가 존재한다고 해서 모두 주석해야 하는 것은 아니다.

다음과 같은 경우에는 별도의 주석 체계 구축을 보류하거나 다른 주제의 관리 대상으로 넘긴다.

| 유형 | 판단 기준 | 처리 방향 |
|--------|--------|--------|
| 이미 코드값으로 관리되는 데이터 | 사람의 해석 없이 의미가 명확함 | 별도 주석 불필요 |
| 단순 수치 데이터 | 라벨보다 품질·이상치 관리가 중요함 | 데이터 품질 관리 대상으로 처리 |
| OCR 전 원본 문서 | 아직 AI가 읽을 수 있는 형태가 아님 | 전처리 이후 주석 검토 |
| 활용 과제가 불명확한 데이터 | 재사용 목적이 없음 | 우선순위 낮음 |
| 기준 합의가 불가능한 데이터 | SME 간 판단 기준 불일치 | Guideline 선행 필요 |

예를 들어 설비 센서의 온도, 압력, 전류 값처럼 이미 수치로 구조화되어 있는 데이터는 주석 대상이라기보다 품질 관리나 이상 탐지 모델링의 대상이다.

반면 고객 클레임 원문처럼 자연어 안에 원인, 현상, 조치가 섞여 있는 데이터는 주석을 통해 의미 구조를 부여해야 한다.

---

## 4.4 주석 착수 전 확인 사항

주석 작업에 착수하기 전에는 최소한 다음 사항을 확인해야 한다.

| 확인 항목 | 확인 질문 |
|--------|--------|
| AI 활용 목적 | 이 데이터를 활용할 AI 과제가 정의되어 있는가 |
| 현업 기준 | SME가 판단 기준을 설명할 수 있는가 |
| 데이터 확보 | 충분한 샘플이 확보되어 있는가 |
| 품질 기준 | 주석 결과를 검증할 기준이 있는가 |
| 재사용 가능성 | 다른 과제에서도 활용 가능한가 |

이 항목이 확인되지 않은 상태에서 주석 작업을 시작하면 대량의 데이터를 만들고도 실제 AI 과제에 활용하지 못할 수 있다.

따라서 주석 작업은 데이터가 많다는 이유만으로 시작하는 것이 아니라, AI 활용 목적과 현업 판단 기준이 함께 정의된 경우에 착수하는 것이 바람직하다.

---

# 5. 어떻게 구축·운영하나 (How)

본 장은 데이터 해설·주석 체계를 실제 제조업 환경에서 구축하고 운영하는 전체 과정을 설명한다.

단순한 방법론 설명이 아니라 두산전자 외관 검사 AI 구축 사례를 함께 활용하여 각 단계에서 무엇을 해야 하는지와 실제로 어떤 문제가 발생하는지를 함께 설명한다.

본 장은 아래 정본 모델을 기준으로 작성한다.

```text
대상 데이터 선정
→ Taxonomy 설계
→ Guideline 작성
→ Pilot Annotation
→ IAA 측정
→ 본 Annotation
→ QA
→ 자동화 및 자산화
```

---

## 5.1 대상 데이터 선정

### 목적

AI 활용 가치가 높고 사람의 판단이 필요한 데이터를 선정한다.

모든 데이터를 주석 대상으로 선정할 필요는 없다.

주석 비용은 높기 때문에 AI 활용 가치와 재사용 가능성이 높은 데이터부터 우선 적용해야 한다.

### 주요 수행 작업

- 후보 데이터 목록 작성
- AI 활용 가능성 평가
- 현업 중요도 평가
- 우선순위 선정

### 두산전자 사례

두산전자 품질혁신팀은 외관 검사 자동화를 추진하기 위해 데이터 현황을 분석하였다.

분석 결과 다음과 같은 데이터가 존재하였다.

| 데이터 | 활용 가능성 |
|--------|--------|
| 외관 검사 이미지 | 높음 |
| 고객 클레임 | 높음 |
| 생산 실적 | 중간 |
| 설비 로그 | 중간 |

검토 결과 외관 검사 이미지가 가장 높은 AI 활용 가치를 가진 것으로 판단하였다.

특히 약 5만 장 이상의 이미지가 이미 확보되어 있었기 때문에 우선 적용 대상으로 선정하였다.

### 산출물

- 대상 데이터 목록
- 우선순위 평가표
- 적용 범위 정의서

### 완료 기준

- 대상 데이터 확정
- SME 승인 완료

---

## 5.2 Taxonomy 설계

### 목적

결함을 어떤 기준으로 분류할 것인지 정의한다.

Taxonomy는 이후 모든 주석 작업의 기준이 된다.

### 주요 수행 작업

- 현업 인터뷰
- 기존 분류 체계 분석
- 라벨 체계 설계
- SME 승인

### 두산전자 사례

초기 분석 결과 동일 결함이 다양한 표현으로 기록되고 있었다.

| 현업 표현 | 통합 라벨 |
|--------|--------|
| 갈라짐 | 균열 |
| 크랙 | 균열 |
| 금감 | 균열 |
| 찍힘 | 눌림 |
| 흠집 | 긁힘 |

품질팀과 데이터 조직은 아래 Taxonomy를 정의하였다.

```text
결함
├─ 균열
├─ 눌림
├─ 오염
└─ 긁힘
```

### 산출물

- Taxonomy 정의서
- 라벨 코드 체계
- 라벨 정의서

### 완료 기준

- Taxonomy 승인 완료
- 표준 라벨 체계 확정

---

## 5.3 Guideline 및 사례집 구축

### 목적

작업자들이 동일한 기준으로 데이터를 주석할 수 있도록 한다.

### 주요 수행 작업

- 포함 사례 정의
- 제외 사례 정의
- 경계 사례 정리
- 결정 규칙 작성

### 두산전자 사례

품질팀 인터뷰 결과 가장 큰 혼동은 균열과 긁힘이었다.

이에 따라 Guideline을 아래와 같이 정의하였다.

```text
IF 선형 갈라짐 존재
→ 균열

IF 표면 긁힘 흔적 존재
→ 긁힘

IF 판단 불가
→ SME 검토
```

또한 대표 사례, 제외 사례, 경계 사례를 별도 사례집으로 구축하였다.

### 산출물

- Guideline
- 사례집(Casebook)
- 결정 규칙

### 완료 기준

- Guideline 승인 완료
- 사례집 구축 완료

---

## 5.4 Pilot Annotation

### 목적

본 운영 이전에 기준의 적절성을 검증한다.

### 주요 수행 작업

- 샘플 데이터 선정
- 독립 주석 수행
- 불일치 사례 분석

### 두산전자 사례

라벨러 3인이 동일한 외관 검사 이미지 500장을 독립적으로 주석하였다.

Pilot 결과 다음 문제가 확인되었다.

- 균열과 긁힘 간 판단 차이
- 일부 경계 사례 해석 불일치
- 작업자별 라벨 편차 존재

### 산출물

- Pilot Dataset
- 불일치 사례 목록
- 개선 과제

### 완료 기준

- Pilot 완료
- 불일치 분석 완료

---

## 5.5 IAA 측정 및 기준 개선

### 목적

작업자 간 일관성을 확보한다.

### 주요 수행 작업

- IAA 측정
- 불일치 원인 분석
- Guideline 개선

### 두산전자 사례

Pilot 결과를 분석한 결과 다음과 같은 결과가 나타났다.

| 결함 유형 | IAA |
|--------|--------|
| 균열 | 0.83 |
| 눌림 | 0.86 |
| 오염 | 0.88 |
| 긁힘 | 0.61 |

분석 결과 긁힘과 균열의 경계가 모호한 것으로 확인되었다.

품질혁신팀은 대표 사례 30건을 추가 확보하고 Guideline을 수정하였다.

재측정 결과는 다음과 같았다.

| 결함 유형 | 개선 후 IAA |
|--------|--------|
| 균열 | 0.87 |
| 눌림 | 0.88 |
| 오염 | 0.90 |
| 긁힘 | 0.79 |

### 산출물

- IAA 결과 보고서
- Guideline 개선안

### 완료 기준

- 목표 IAA 달성
- Guideline 보완 완료

---

## 5.6 본 Annotation

### 목적

전체 데이터를 대상으로 주석 작업을 수행한다.

### 주요 수행 작업

- AI Pre-label 적용
- Annotation 수행
- 작업 이력 관리

### 두산전자 사례

기준 검증이 완료된 이후 전체 5만 장의 이미지를 대상으로 본 Annotation을 수행하였다.

이 단계에서는 AI 기반 Pre-label을 함께 적용하였다.

```text
AI 1차 Annotation
→ 작업자 검토
→ 검수자 확인
→ SME 승인
```

AI는 반복 작업을 줄이고 사람은 최종 품질을 보장하는 역할을 수행하였다.

### 산출물

- Annotation Dataset
- 작업 이력
- 운영 로그

### 완료 기준

- 전체 데이터 처리 완료
- 품질 기준 충족

---

## 5.7 QA

### 목적

최종 데이터셋의 품질을 검증한다.

### 주요 수행 작업

- 샘플 검수
- 오류 분석
- 재작업 수행
- 최종 승인

### 두산전자 사례

QA 단계에서는 다음 항목을 검증하였다.

| 항목 | 목적 |
|--------|--------|
| 라벨 정확도 | 오류 제거 |
| 일관성 | 기준 유지 |
| 누락 여부 | 데이터 완전성 확보 |
| 버전 검증 | 변경 이력 관리 |

QA를 통과한 데이터만 최종 학습 데이터셋으로 승인하였다.

### 산출물

- QA 결과 보고서
- 승인 데이터셋

### 완료 기준

- QA 승인
- 재작업 완료

---

## 5.8 자동화 및 자산화

### 목적

주석 데이터를 재사용 가능한 조직 자산으로 전환한다.

### 주요 수행 작업

- Registry 등록
- 버전 관리
- Gold Dataset 구축
- 재사용 체계 구축

### 두산전자 사례

품질혁신팀은 최종 데이터셋을 Registry에 등록하였다.

| 관리 항목 | 예시 |
|--------|--------|
| Dataset ID | DS-ELEC-001 |
| 버전 | v1.0 |
| 활용 과제 | 외관 검사 AI |
| 책임 조직 | 품질혁신팀 |

또한 다음 자산을 함께 관리하기 시작하였다.

- Annotation Dataset
- Taxonomy
- Guideline
- 사례집
- Gold Dataset
- Annotation Registry

이후 해당 자산은 외관 검사 자동화뿐 아니라 품질 Agent, 결함 원인 분석, 품질 예측 과제에도 재사용되었다.

### 산출물

- Annotation Registry
- Gold Dataset Pool
- 재사용 가이드

### 완료 기준

- Registry 등록 완료
- 재사용 가능 상태 확보
- 운영 체계 수립

---

## 5.9 구축 결과

본 사례를 통해 두산전자는 단순히 AI 모델 하나를 구축한 것이 아니다.

현업 전문가의 판단 기준을 Taxonomy, Guideline, 사례집, Gold Dataset 형태로 구조화하고 이를 지속적으로 활용 가능한 데이터 자산으로 전환하였다.

데이터 해설·주석의 최종 목적은 라벨을 생성하는 것이 아니라 조직의 판단 기준을 재사용 가능한 AI-ready Data 자산으로 축적하는 것이다.

---

# 6. 다른 주제와의 관계

데이터 해설·주석은 AI-ready Data 체계에서 학습용 의미 정보를 생성하는 역할을 담당한다.

그러나 데이터 준비 과정 전체를 담당하는 것은 아니다.

전처리, 품질 관리, 온톨로지, 평가 데이터 등 인접 주제와 명확한 역할 분담이 필요하다.

---

## 6.1 데이터 전처리와의 관계

데이터 전처리는 데이터를 AI가 읽을 수 있는 형태로 변환하는 활동이다.

반면 데이터 해설·주석은 전처리가 완료된 데이터에 의미 정보를 부여하는 활동이다.

예를 들어 PDF 검사 보고서를 텍스트로 변환하는 것은 전처리의 영역이다.

반면 변환된 텍스트에 원인 유형이나 조치 유형을 부여하는 것은 데이터 해설·주석의 영역이다.

| 구분 | 데이터 전처리 | 데이터 해설·주석 |
|--------|--------|--------|
| 목적 | 읽을 수 있는 형태 변환 | 의미 정보 부여 |
| 입력 | 원본 데이터 | 전처리 완료 데이터 |
| 산출물 | 구조화 데이터 | 주석 데이터셋 |

---

## 6.2 비즈니스 Glossary와의 관계

비즈니스 Glossary는 용어의 의미를 정의하는 체계이다.

반면 데이터 해설·주석은 데이터를 특정 용어로 분류하는 체계이다.

| 구분 | 비즈니스 Glossary | 데이터 해설·주석 |
|--------|--------|--------|
| 관리 대상 | 용어 | 데이터 |
| 목적 | 의미 정의 | 의미 적용 |
| 산출물 | 표준 용어집 | 주석 데이터셋 |

Glossary는 데이터 해설·주석의 기준 정보를 제공한다.

---

## 6.3 온톨로지와의 관계

온톨로지는 개념 간 관계를 정의하는 체계이다.

반면 데이터 해설·주석은 개별 데이터를 특정 개념으로 분류하는 체계이다.

| 구분 | 온톨로지 | 데이터 해설·주석 |
|--------|--------|--------|
| 목적 | 개념 관계 정의 | 개별 데이터 분류 |
| 관리 단위 | 개념 | 데이터 |
| 활용 | 추론, 검색 | 학습 데이터 생성 |

온톨로지는 데이터 해설·주석 결과를 활용하는 상위 구조라고 볼 수 있다.

---

## 6.4 데이터 품질 관리와의 관계

| 구분 | 데이터 품질 관리 | 데이터 해설·주석 |
|--------|--------|--------|
| 목적 | 품질 판정 | 주석 데이터 생성 |
| 결과 | 사용 가능 여부 | 학습 데이터 구축 |
| 관점 | 품질 기준 | 의미 기준 |

주석 데이터 역시 품질 관리 대상이 된다.

---

## 6.5 AI 평가 데이터와의 관계

| 구분 | 데이터 해설·주석 | AI 평가 데이터 |
|--------|--------|--------|
| 목적 | 학습 | 평가 |
| 활용 시점 | 모델 학습 전 | 모델 학습 후 |
| 규모 | 대규모 | 소규모 고품질 |
| 역할 | 가르치는 정답 | 채점하는 정답 |

동일한 데이터가 학습 데이터와 평가 데이터로 동시에 사용되어서는 안 된다.

---

## 6.6 데이터 Feedback Loop와의 관계

운영 과정에서 발견되는 오류는 데이터 Feedback Loop를 통해 다시 데이터 해설·주석 체계로 전달된다.

```text
AI 오분류 발생
→ Feedback 수집
→ 원인 분석
→ Guideline 보완
→ 주석 데이터 수정
→ 재학습
```

따라서 데이터 해설·주석은 데이터 Feedback Loop의 핵심 입력 요소 중 하나이다.

---

# 7. KPI 및 Roadmap

데이터 해설·주석은 프로젝트 단위 작업으로 끝나서는 안 된다.

지속적인 운영과 자동화, 자산화를 위해 KPI와 Roadmap을 함께 관리해야 한다.

---

## 7.1 KPI

### 품질 KPI

주석 품질은 데이터 해설·주석 체계의 가장 기본적인 성과 지표이다.

| KPI | 정의 | 측정 방식 |
|--------|--------|--------|
| IAA | 작업자 간 일치도 | Cohen's Kappa 등 |
| 라벨 오류율 | 잘못된 주석 비율 | 오류 건수 / 전체 건수 |
| QA 통과율 | 품질 검수 통과 비율 | 통과 건수 / 전체 건수 |

품질 KPI는 Taxonomy와 Guideline의 적절성을 검증하는 기준이 된다.

---

### 생산성 KPI

생산성 KPI는 주석 작업 효율을 측정한다.

| KPI | 정의 | 측정 방식 |
|--------|--------|--------|
| 처리량 | 시간당 주석 건수 | 총 건수 / 작업 시간 |
| 건당 비용 | 주석 1건당 비용 | 총 비용 / 총 건수 |
| 리드타임 | 작업 완료 기간 | 종료일 - 시작일 |

---

### 운영 KPI

운영 KPI는 주석 체계가 안정적으로 운영되는지를 측정한다.

| KPI | 정의 | 측정 방식 |
|--------|--------|--------|
| 재작업률 | 수정 필요 비율 | 재작업 건수 / 전체 건수 |
| 버전 변경 건수 | 기준 변경 빈도 | 월간 변경 건수 |
| 운영 오류 건수 | 운영 중 발생한 오류 | 월간 집계 |

---

### 자동화 KPI

자동화 KPI는 AI 활용 수준을 측정한다.

| KPI | 정의 | 측정 방식 |
|--------|--------|--------|
| AI Pre-label 채택률 | AI 주석 승인 비율 | 승인 건수 / AI 생성 건수 |
| 자동 처리율 | 사람 개입 없는 처리 비율 | 자동 처리 건수 / 전체 건수 |
| 검수 절감률 | 사람 검수 감소 수준 | 기존 대비 감소율 |

---

### 자산화 KPI

자산화 KPI는 데이터 재사용 수준을 측정한다.

| KPI | 정의 | 측정 방식 |
|--------|--------|--------|
| Registry 등록률 | 자산 등록 비율 | 등록 데이터셋 / 전체 데이터셋 |
| 데이터 재사용률 | 재사용 비율 | 재사용 데이터셋 / 전체 데이터셋 |
| Gold Dataset 활용률 | 표준 데이터 활용 비율 | 활용 과제 수 |
| 계열사 적용률 | 계열사 확산 수준 | 적용 계열사 수 |

# Appendix A. 주요 용어

## 데이터 해설·주석

데이터에 의미 정보를 부여하여 AI가 학습 가능한 형태로 만드는 활동이다.

단순한 라벨링 작업이 아니라 현업의 판단 기준을 데이터 자산으로 전환하는 과정이다.

---

## Taxonomy

데이터를 어떤 기준으로 분류할 것인지 정의하는 체계이다.

---

## Guideline

작업자들이 동일한 기준으로 데이터를 분류할 수 있도록 정의한 규칙 문서이다.

---

## 사례집 (Casebook)

대표 사례, 제외 사례, 경계 사례를 정리한 자료이다.

Guideline을 실제 사례로 설명하고 작업자 교육 자료로 활용한다.

---

## Annotation

데이터에 의미 정보를 부여하는 행위이다.

---

## SME

현업 전문가를 의미한다.

주석 기준 정의와 최종 검증 역할을 수행한다.

---

## IAA

작업자 간 일치도를 의미한다.

---

## Gold Dataset

전문가 검증이 완료된 고품질 데이터셋이다.

---

## Registry

데이터 자산을 등록하고 관리하는 체계이다.

---

## Human-in-the-Loop (HITL)

AI가 생성한 결과를 사람이 검토하고 수정하는 운영 방식이다.

---

## Pre-labeling

AI가 먼저 주석 초안을 생성하고 사람이 검토하는 방식이다.

---

## Active Learning

AI가 가장 어려워하는 데이터를 우선적으로 사람에게 요청하여 학습 효율을 높이는 방식이다.

---

## Weak Supervision

규칙, 키워드, 기존 코드 등을 활용하여 대량의 후보 라벨을 생성하는 방식이다.
