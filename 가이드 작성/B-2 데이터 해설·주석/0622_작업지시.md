# AI-ready Data 가이드 생성 마스터 프롬프트 v1.0

> Part 1 — 목적, 작성 철학, 문체 규칙, 금지 규칙, 용어 규칙

---

# 1. 목적

본 프롬프트의 목적은 AI-ready Data 20개 주제에 대해 계열사가 실제로 구축·운영·확산할 수 있는 수준의 실무 가이드를 일관된 품질로 생성하는 것이다.

가이드는 단순 개념 설명서가 아니다.

가이드는 특정 주제의 필요성, 구성 요소, 구축 방법, 운영 체계, KPI, Roadmap을 포함하는 실행 매뉴얼이다.

문서는 다음 질문에 답해야 한다.

- 이 주제는 AI-ready Data 관점에서 무엇인가
- 왜 필요한가
- 어떤 구성 요소를 갖추어야 하는가
- 어디부터 적용해야 하는가
- 제조업에서는 어떻게 적용되는가
- 데이터 조직과 현업은 무엇을 해야 하는가
- 어떤 산출물과 완료 기준이 필요한가
- 어떻게 운영·확산할 것인가

---

# 2. 최상위 작성 철학

## 2.1 설명보다 실행

가이드는 개념을 설명하기 위해 존재하는 문서가 아니다.

가이드는 독자가 실제로 행동할 수 있도록 만드는 문서이다.

따라서 문서를 작성할 때 항상 아래 질문을 먼저 생각한다.

```text
이 내용을 읽은 사람이
실제로 무엇을 해야 하는가?
```

### 좋지 않은 예

```text
데이터 주석은 데이터에 의미를 부여하는 활동이다.

데이터 주석은 AI 학습에 중요하다.

데이터 주석은 품질이 중요하다.
```

### 좋은 예

```text
데이터 주석 체계를 구축하기 위해서는

1. 대상 데이터를 선정하고
2. Taxonomy를 정의하고
3. Guideline을 작성하고
4. Pilot을 수행한 뒤
5. 본 운영에 착수해야 한다.
```

설명은 행동으로 연결되어야 한다.

독자가 읽은 후 "이해했다" 수준이 아니라 "무엇을 해야 하는지 알겠다" 수준이 되어야 한다.

---

## 2.2 정의는 시작점일 뿐이다

정의는 필요하다.

그러나 정의 자체가 문서의 목적이 되어서는 안 된다.

### 좋지 않은 흐름

```text
정의

정의

정의

정의

예시
```

### 좋은 흐름

```text
문제

왜 필요한가

판단 기준

제조업 사례

수행 방법

산출물

KPI

Roadmap
```

정의는 설명의 시작점이다.

문서는 반드시 실행과 운영 체계까지 연결되어야 한다.

---

## 2.3 AI-ready Data 관점 유지

각 주제는 독립된 기술 주제가 아니다.

모든 주제는 AI-ready Data 체계의 일부로 설명한다.

### 좋지 않은 설명

```text
데이터 전처리는 파서를 만드는 것이다.
```

### 좋은 설명

```text
데이터 전처리는
AI가 데이터를 읽을 수 있도록
구조화하는 활동이다.
```

### 좋지 않은 설명

```text
데이터 해설·주석은
라벨링 작업이다.
```

### 좋은 설명

```text
데이터 해설·주석은
사람의 판단 기준을
데이터 자산으로 전환하는 활동이다.
```

기술 자체보다 AI 활용 관점에서 설명한다.

---

## 2.4 데이터 자산화 관점

모든 주제는 최종적으로 데이터 자산화와 연결되어야 한다.

단발성 프로젝트 결과물이 아니라 재사용 가능한 자산을 만드는 관점으로 작성한다.

문서 작성 시 아래 질문을 항상 고려한다.

```text
이 결과물을
다음 AI 과제에서도
재사용할 수 있는가?
```

가이드는 프로젝트 성공 방법이 아니라 조직 역량 축적 방법을 설명해야 한다.

---

## 2.5 제조업 실무 우선

가이드는 제조업 현업 조직을 대상으로 작성한다.

따라서 일반 소비자 서비스 사례보다 실제 제조업 사례를 우선 사용한다.

### 좋지 않은 사례

```text
스마트폰 사진 분류

고양이 / 강아지 이미지

넷플릭스 추천 시스템

쇼핑몰 상품 추천
```

### 좋은 사례

```text
동판 외관 검사

설비 점검 데이터

고객 클레임

시험 결과 보고서

설비 이상 알람
```

가능하면 두산 계열사에서 실제 활용 가능한 사례를 우선 사용한다.

# 3. 문체 규칙

## 3.1 기본 문체

문체는 컨설팅 보고서 스타일을 따른다.

문장은 짧고 명확하게 작성한다.

그러나 단문만 반복하지 않는다.

### 권장 표현

```text
따라서

반면

이후

결과적으로

특히

예를 들어

이 과정을 통해

다음 단계에서는
```

### 비권장 표현

```text
아무튼

쉽게 말하면

결국은

이런 느낌으로

뚝딱

얼른
```

---

## 3.2 독자 관점

독자는 다음 4개 그룹이다.

| 독자 | 관심사 |
|--------|--------|
| 경영진 | 왜 필요한가, 기대 효과 |
| 현업 SME | 판단 기준, 역할 |
| 데이터 조직 | 구축 방법, 운영 |
| AI 조직 | 활용 방법, 자동화 |

문서는 특정 그룹만 이해할 수 있는 형태로 작성하지 않는다.

---

## 3.3 문장 길이

### 좋지 않은 예

```text
데이터 품질이 중요하다.

데이터 품질은 필요하다.

품질 관리가 중요하다.

품질은 필수다.
```

### 좋은 예

```text
데이터 품질은 AI 성능에 직접적인 영향을 준다.

따라서 데이터 품질 기준을 정의하고,
운영 과정에서 지속적으로 관리해야 한다.
```

문장은 짧게 쓰되, 논리적 연결을 유지한다.

# 4. 금지 규칙

## 4.1 요약 박스 금지

다음 표현은 사용하지 않는다.

```text
한 줄 요약

한 줄 정의

쉽게 말하면

정리하면

핵심은

요약하면

한 문장으로 표현하면

실무 한 줄 정리
```

### 좋지 않은 예

```text
한 줄 요약

데이터 주석은
AI를 위한 정답 데이터이다.
```

### 좋은 예

```text
데이터 주석은
AI 학습에 필요한 의미 정보를
데이터에 부여하는 활동이다.

주석 품질은
AI 성능에 직접적인 영향을 준다.
```

요약 내용은 일반 문단 안으로 흡수한다.

## 4.2 이모티콘 금지

---

## 4.3 메타 코멘트 금지

다음 표현은 사용하지 않는다.


## 4.4 비유 사용 금지

# 5. 용어 규칙

## 5.1 국문 우선

본문은 국문 중심으로 작성한다.

영문 병기는 최초 정의 시에만 사용한다.

### 좋은 예

```text
데이터 해설·주석(Data Annotation)

현업 전문가(SME)

Human-in-the-Loop(HITL)
```

이후에는

```text
주석

SME

HITL
```

로 통일한다.

## 5.2 현업 용어 우선

기술적으로 문제가 없는 경우 현업 용어를 사용한다.

### 지양

```text

```

### 권장

```text

균열

결함 유형
```

---

## 5.3 용어 통일

동일 개념은 문서 전체에서 동일 용어를 사용한다.

### 좋지 않은 예

```text

```

### 좋은 예

```text

```

## 5.4 제조업 용어 우선 규칙

가능하면 현업에서 실제 사용하는 표현을 우선 사용한다.

### 권장

```text
동판
```

# 추가 작성 규칙 (B-2 작성 과정 Lessons Learned 반영)

## 1. 장 제목은 Why / What / Where / Example / How를 일관되게 유지한다

가이드는 다음 구조를 기본으로 사용한다.

```text
1. 개요

2. 왜 필요한가 (Why)

3. 무엇을 갖추나 (What)

4. 어디부터 하나 (Where)

5. 예시 시나리오 (Example)

6. 어떻게 준비·운영하나 (How)

7. 다른 주제와의 관계

8. KPI 및 Roadmap
```

Why, What, How만 표기하고 Where를 생략하지 않는다.

독자가 각 장의 역할을 직관적으로 이해할 수 있도록 Why / What / Where / Example / How를 일관되게 표기한다.

---

## 2. 5장 사례는 단순 사례 소개가 아니라 End-to-End 수행 시나리오여야 한다

5장은 배경 설명이나 적용 효과를 소개하는 장이 아니다.

5장은 실제 제조업 환경에서 해당 주제를 처음부터 끝까지 수행하는 과정을 보여주는 End-to-End 시나리오이다.

### 좋지 않은 예

```text
적용 배경

기대 효과

결과
```

또는

```text
간단한 사례 예시

결함 이미지 예시

고객 클레임 예시
```

수준

### 좋은 예

```text
대상 데이터 선정
→ Taxonomy 설계
→ Guideline 작성
→ Pilot 수행
→ IAA 측정
→ 본 운영
→ QA
→ Registry 등록
```

실제 수행 흐름

---

5장은 3장과 4장에서 언급된 개별 예시를 반복하는 장이 아니다.

5장은 다음 질문에 답해야 한다.

```text
우리 회사가 실제로 이 체계를 구축한다면

어떤 데이터를 고르고

누가 참여하고

무엇을 만들고

어떤 문제를 만나고

어떻게 해결하며

최종적으로 어떤 자산이 남는가
```

---

## 3. 5장과 6장의 차이를 명확히 유지한다

### 5장

실제 사례

예)

```text
두산전자 외관 검사

두산밥캣 정비 이력

두산에너빌리티 설계 문서
```

등 특정 사례

---

### 6장

일반화된 방법론

예)

```text
대상 선정

Taxonomy

Pilot

QA

Registry
```

모든 계열사에 적용 가능한 표준 방법론

---

5장이 6장의 축약본이 되어서는 안 된다.

6장이 5장의 반복 설명이 되어도 안 된다.

---

## 4. 정본 모델은 텍스트 프로세스 형태로 먼저 정의한다

3장의 정본 모델은 가능하면 아래와 같은 형태를 우선 사용한다.

```text
대상 데이터 선정
→ Taxonomy 설계
→ Guideline 작성
→ Pilot Annotation
→ IAA 측정
→ 본 Annotation
→ QA
→ 자동화 및 자산화
```

독자가 전체 구조를 한눈에 이해할 수 있도록 텍스트 프로세스 형태를 우선 사용한다.

B-2에서 이 방식이 가장 이해도가 높았다.

---

## 5. 약지도 대신 Weak Supervision 사용

다음 표현은 사용하지 않는다.

```text
약지도
```

다음 표현을 사용한다.

```text
Weak Supervision
```

최초 1회만 영문 병기 후 동일 용어 사용.

---

## 6. 5장 사례는 실제 기업 사례처럼 작성한다

사례는 단순 예시가 아니다.

실제 제조업 프로젝트 수행 문서처럼 작성한다.

### 좋지 않은 예

```text
결함 이미지를 선택하였다.

Taxonomy를 만들었다.

주석을 수행하였다.
```

### 좋은 예

```text
두산전자 품질혁신팀은
동판 외관 검사 이미지 5만 장을 확보하였다.

초기 분석 결과

동일 결함이

균열
갈라짐
크랙

등으로 혼재되어 있었다.

품질팀 SME와 데이터 조직은
결함 4종 체계를 정의하였고

Pilot 결과
스크래치와 균열 간 IAA가 0.61로 나타났다.

이에 따라 Guideline을 보완하고
대표 사례집을 추가 구축하였다.
```

실제 수행 과정과 문제 해결 과정을 보여준다.

---

## 7. 사례와 방법론의 역할을 구분한다

3장

```text
개념
```

4장

```text
적용 대상
```

5장

```text
실제 수행 사례
```

6장

```text
실행 방법론
```

역할이 겹치지 않도록 한다.

특히 5장에서 3장·4장의 단순 예시를 반복하지 않는다.

---

## 8. 로드맵은 Phase 체계를 사용한다

Level 기반 로드맵보다 Phase 기반 표현을 우선 사용한다.

권장 형식

```text
Phase 1
Preparation

Phase 2
AI Ready

Phase 3
Automation

Phase 4
Assetization
```

Level 1, Level 2 표현은 가급적 사용하지 않는다.

---

## 9. 짧은 선언문 반복 금지

다음과 같은 문장 구조는 사용하지 않는다.

```text
첫 번째 문제는 정답 데이터의 부재이다.

두 번째 문제는 판단 기준의 불일치이다.

세 번째 문제는 운영 이력의 부재이다.
```

대신

```markdown
- 정답 데이터의 부재:
  ...

- 판단 기준의 불일치:
  ...

- 운영 이력의 부재:
  ...
```

형태를 사용한다.

짧은 선언문 반복보다 불렛 기반 구조를 우선 사용한다.
