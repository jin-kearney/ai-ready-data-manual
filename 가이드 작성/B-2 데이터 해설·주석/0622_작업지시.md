# AI-ready Data 가이드 생성 마스터 프롬프트 v1.0

> Part 1 — 목적, 작성 철학, 문체 규칙, 금지 규칙, 용어 규칙

---

# 1. 목적

본 프롬프트의 목적은 AI-ready Data 20개 주제에 대해 계열사가 실제로 구축·운영·확산할 수 있는 수준의 실무 가이드를 일관된 품질로 생성하는 것이다.

가이드는 단순 개념 설명서가 아니다.

가이드는 특정 주제의 필요성, 구성 요소, 구축 방법, 운영 체계, KPI, Roadmap을 포함하는 실행 매뉴얼이다.

문서는 다음 질문에 답해야 한다.

- 이 주제는 AI-ready Data 관점에서 무엇인가
- 왜 필요한가
- 어떤 구성 요소를 갖추어야 하는가
- 어디부터 적용해야 하는가
- 제조업에서는 어떻게 적용되는가
- 데이터 조직과 현업은 무엇을 해야 하는가
- 어떤 산출물과 완료 기준이 필요한가
- 어떻게 운영·확산할 것인가

---

# 2. 최상위 작성 철학

## 2.1 설명보다 실행

가이드는 개념을 설명하기 위해 존재하는 문서가 아니다.

가이드는 독자가 실제로 행동할 수 있도록 만드는 문서이다.

따라서 문서를 작성할 때 항상 아래 질문을 먼저 생각한다.

```text
이 내용을 읽은 사람이
실제로 무엇을 해야 하는가?
```

### 좋지 않은 예

```text
데이터 주석은 데이터에 의미를 부여하는 활동이다.

데이터 주석은 AI 학습에 중요하다.

데이터 주석은 품질이 중요하다.
```

### 좋은 예

```text
데이터 주석 체계를 구축하기 위해서는

1. 대상 데이터를 선정하고
2. Taxonomy를 정의하고
3. Guideline을 작성하고
4. Pilot을 수행한 뒤
5. 본 운영에 착수해야 한다.
```

설명은 행동으로 연결되어야 한다.

독자가 읽은 후 "이해했다" 수준이 아니라 "무엇을 해야 하는지 알겠다" 수준이 되어야 한다.

---

## 2.2 정의는 시작점일 뿐이다

정의는 필요하다.

그러나 정의 자체가 문서의 목적이 되어서는 안 된다.

### 좋지 않은 흐름

```text
정의

정의

정의

정의

예시
```

### 좋은 흐름

```text
문제

왜 필요한가

판단 기준

제조업 사례

수행 방법

산출물

KPI

Roadmap
```

정의는 설명의 시작점이다.

문서는 반드시 실행과 운영 체계까지 연결되어야 한다.

---

## 2.3 AI-ready Data 관점 유지

각 주제는 독립된 기술 주제가 아니다.

모든 주제는 AI-ready Data 체계의 일부로 설명한다.

### 좋지 않은 설명

```text
데이터 전처리는 파서를 만드는 것이다.
```

### 좋은 설명

```text
데이터 전처리는
AI가 데이터를 읽을 수 있도록
구조화하는 활동이다.
```

### 좋지 않은 설명

```text
데이터 해설·주석은
라벨링 작업이다.
```

### 좋은 설명

```text
데이터 해설·주석은
사람의 판단 기준을
데이터 자산으로 전환하는 활동이다.
```

기술 자체보다 AI 활용 관점에서 설명한다.

---

## 2.4 데이터 자산화 관점

모든 주제는 최종적으로 데이터 자산화와 연결되어야 한다.

단발성 프로젝트 결과물이 아니라 재사용 가능한 자산을 만드는 관점으로 작성한다.

문서 작성 시 아래 질문을 항상 고려한다.

```text
이 결과물을
다음 AI 과제에서도
재사용할 수 있는가?
```

가이드는 프로젝트 성공 방법이 아니라 조직 역량 축적 방법을 설명해야 한다.

---

## 2.5 제조업 실무 우선

가이드는 제조업 현업 조직을 대상으로 작성한다.

따라서 일반 소비자 서비스 사례보다 실제 제조업 사례를 우선 사용한다.

### 좋지 않은 사례

```text
스마트폰 사진 분류

고양이 / 강아지 이미지

넷플릭스 추천 시스템

쇼핑몰 상품 추천
```

### 좋은 사례

```text
동판 외관 검사

설비 점검 데이터

고객 클레임

시험 결과 보고서

설비 이상 알람
```

가능하면 두산 계열사에서 실제 활용 가능한 사례를 우선 사용한다.

# 3. 문체 규칙

## 3.1 기본 문체

문체는 컨설팅 보고서 스타일을 따른다.

문장은 짧고 명확하게 작성한다.

그러나 단문만 반복하지 않는다.

### 권장 표현

```text
따라서

반면

이후

결과적으로

특히

예를 들어

이 과정을 통해

다음 단계에서는
```

### 비권장 표현

```text
아무튼

쉽게 말하면

결국은

이런 느낌으로

뚝딱

얼른
```

---

## 3.2 독자 관점

독자는 다음 4개 그룹이다.

| 독자 | 관심사 |
|--------|--------|
| 경영진 | 왜 필요한가, 기대 효과 |
| 현업 SME | 판단 기준, 역할 |
| 데이터 조직 | 구축 방법, 운영 |
| AI 조직 | 활용 방법, 자동화 |

문서는 특정 그룹만 이해할 수 있는 형태로 작성하지 않는다.

---

## 3.3 문장 길이

### 좋지 않은 예

```text
데이터 품질이 중요하다.

데이터 품질은 필요하다.

품질 관리가 중요하다.

품질은 필수다.
```

### 좋은 예

```text
데이터 품질은 AI 성능에 직접적인 영향을 준다.

따라서 데이터 품질 기준을 정의하고,
운영 과정에서 지속적으로 관리해야 한다.
```

문장은 짧게 쓰되, 논리적 연결을 유지한다.

---

# 4. 금지 규칙

## 4.1 요약 박스 금지

다음 표현은 사용하지 않는다.

```text
한 줄 요약

한 줄 정의

쉽게 말하면

정리하면

핵심은

요약하면

한 문장으로 표현하면

실무 한 줄 정리
```

### 좋지 않은 예

```text
한 줄 요약

데이터 주석은
AI를 위한 정답 데이터이다.
```

### 좋은 예

```text
데이터 주석은
AI 학습에 필요한 의미 정보를
데이터에 부여하는 활동이다.

주석 품질은
AI 성능에 직접적인 영향을 준다.
```

요약 내용은 일반 문단 안으로 흡수한다.

---

## 4.2 이모티콘 금지

다음 표현은 사용하지 않는다.

```text
👉
❓
🏭
🏢
✅
❌
📌
🤖
```

이모티콘 대신 일반 문장을 사용한다.

---

## 4.3 메타 코멘트 금지

다음 표현은 사용하지 않는다.

```text
아래는 작성본입니다

다음은

참고로

제가 작성한 내용은

설명드리자면
```

문서는 바로 본문부터 시작한다.

---

## 4.4 비유 사용 금지

AI는 아래 비유를 매우 자주 생성한다.

```text
스마트폰 사진

고양이

강아지

넷플릭스 추천

쇼핑몰 상품 분류
```

원칙적으로 사용하지 않는다.

실제 제조업 사례를 사용한다.

---

# 5. 용어 규칙

## 5.1 국문 우선

본문은 국문 중심으로 작성한다.

영문 병기는 최초 정의 시에만 사용한다.

### 좋은 예

```text
데이터 해설·주석(Data Annotation)

현업 전문가(SME)

Human-in-the-Loop(HITL)
```

이후에는

```text
주석

SME

HITL
```

로 통일한다.

---

## 5.2 현업 용어 우선

기술적으로 문제가 없는 경우 현업 용어를 사용한다.

### 지양

```text
CCL

Crack

Defect Class
```

### 권장

```text
동판

균열

결함 유형
```

---

## 5.3 용어 통일

동일 개념은 문서 전체에서 동일 용어를 사용한다.

### 좋지 않은 예

```text
주석

Annotation

라벨링

태깅
```

혼용

### 좋은 예

```text
주석
```

일관 사용

---

## 5.4 제조업 용어 우선 규칙

가능하면 현업에서 실제 사용하는 표현을 우선 사용한다.

### 지양

```text
Copper Clad Laminate

Crack

Delamination
```

### 권장

```text
동판

균열

박리
```

최초 정의 이후에는 대표 용어 하나로 통일한다.
