# AI-ready Data 가이드 생성 마스터 프롬프트 v1.0

> Part 1 — 목적, 작성 철학, 문체 규칙, 금지 규칙, 용어 규칙

---

# 1. 목적

본 프롬프트의 목적은 AI-ready Data 20개 주제에 대해 계열사가 실제로 구축·운영·확산할 수 있는 수준의 실무 가이드를 일관된 품질로 생성하는 것이다.

가이드는 단순 개념 설명서가 아니다.

가이드는 특정 주제의 필요성, 구성 요소, 구축 방법, 운영 체계, KPI, Roadmap을 포함하는 실행 매뉴얼이다.

문서는 다음 질문에 답해야 한다.

- 이 주제는 AI-ready Data 관점에서 무엇인가
- 왜 필요한가
- 어떤 구성 요소를 갖추어야 하는가
- 어디부터 적용해야 하는가
- 제조업에서는 어떻게 적용되는가
- 데이터 조직과 현업은 무엇을 해야 하는가
- 어떤 산출물과 완료 기준이 필요한가
- 어떻게 운영·확산할 것인가

---

# 2. 최상위 작성 철학

## 2.1 설명보다 실행

가이드는 개념을 설명하기 위해 존재하는 문서가 아니다.

가이드는 독자가 실제로 행동할 수 있도록 만드는 문서이다.

따라서 문서를 작성할 때 항상 아래 질문을 먼저 생각한다.

```text
이 내용을 읽은 사람이
실제로 무엇을 해야 하는가?
```

### 좋지 않은 예

```text
데이터 주석은 데이터에 의미를 부여하는 활동이다.

데이터 주석은 AI 학습에 중요하다.

데이터 주석은 품질이 중요하다.
```

### 좋은 예

```text
데이터 주석 체계를 구축하기 위해서는

1. 대상 데이터를 선정하고
2. Taxonomy를 정의하고
3. Guideline을 작성하고
4. Pilot을 수행한 뒤
5. 본 운영에 착수해야 한다.
```

설명은 행동으로 연결되어야 한다.

독자가 읽은 후 "이해했다" 수준이 아니라 "무엇을 해야 하는지 알겠다" 수준이 되어야 한다.

---

## 2.2 정의는 시작점일 뿐이다

정의는 필요하다.

그러나 정의 자체가 문서의 목적이 되어서는 안 된다.

### 좋지 않은 흐름

```text
정의

정의

정의

정의

예시
```

### 좋은 흐름

```text
문제

왜 필요한가

판단 기준

제조업 사례

수행 방법

산출물

KPI

Roadmap
```

정의는 설명의 시작점이다.

문서는 반드시 실행과 운영 체계까지 연결되어야 한다.

---

## 2.3 AI-ready Data 관점 유지

각 주제는 독립된 기술 주제가 아니다.

모든 주제는 AI-ready Data 체계의 일부로 설명한다.

### 좋지 않은 설명

```text
데이터 전처리는 파서를 만드는 것이다.
```

### 좋은 설명

```text
데이터 전처리는
AI가 데이터를 읽을 수 있도록
구조화하는 활동이다.
```

### 좋지 않은 설명

```text
데이터 해설·주석은
라벨링 작업이다.
```

### 좋은 설명

```text
데이터 해설·주석은
사람의 판단 기준을
데이터 자산으로 전환하는 활동이다.
```

기술 자체보다 AI 활용 관점에서 설명한다.

---

## 2.4 데이터 자산화 관점

모든 주제는 최종적으로 데이터 자산화와 연결되어야 한다.

단발성 프로젝트 결과물이 아니라 재사용 가능한 자산을 만드는 관점으로 작성한다.

문서 작성 시 아래 질문을 항상 고려한다.

```text
이 결과물을
다음 AI 과제에서도
재사용할 수 있는가?
```

가이드는 프로젝트 성공 방법이 아니라 조직 역량 축적 방법을 설명해야 한다.

---

## 2.5 제조업 실무 우선

가이드는 제조업 현업 조직을 대상으로 작성한다.

따라서 일반 소비자 서비스 사례보다 실제 제조업 사례를 우선 사용한다.

### 좋지 않은 사례

```text
스마트폰 사진 분류

고양이 / 강아지 이미지

넷플릭스 추천 시스템

쇼핑몰 상품 추천
```

### 좋은 사례

```text
동판 외관 검사

설비 점검 데이터

고객 클레임

시험 결과 보고서

설비 이상 알람
```

가능하면 두산 계열사에서 실제 활용 가능한 사례를 우선 사용한다.

---

# 3. 문체 규칙

## 3.1 기본 문체

문체는 컨설팅 보고서 스타일을 따른다.

문장은 짧고 명확하게 작성한다.

그러나 단문만 반복하지 않는다.

### 권장 표현

```text
따라서

반면

이후

결과적으로

특히

예를 들어

이 과정을 통해

다음 단계에서는
```

### 비권장 표현

```text
아무튼

쉽게 말하면

결국은

이런 느낌으로

뚝딱

얼른

```

---

## 3.2 독자 관점

독자는 다음 4개 그룹이다.

| 독자 | 관심사 |
|--------|--------|
| 경영진 | 왜 필요한가, 기대 효과 |
| 현업 SME | 판단 기준, 역할 |
| 데이터 조직 | 구축 방법, 운영 |
| AI 조직 | 활용 방법, 자동화 |

문서는 특정 그룹만 이해할 수 있는 형태로 작성하지 않는다.

---

## 3.3 문장 길이

### 좋지 않은 예

```text
데이터 품질이 중요하다.

데이터 품질은 필요하다.

품질 관리가 중요하다.

품질은 필수다.
```

### 좋은 예

```text
데이터 품질은 AI 성능에 직접적인 영향을 준다.

따라서 데이터 품질 기준을 정의하고,
운영 과정에서 지속적으로 관리해야 한다.
```

문장은 짧게 쓰되, 논리적 연결을 유지한다.

---

# 4. 금지 규칙

## 4.1 요약 박스 금지

다음 표현은 사용하지 않는다.

```text
한 줄 요약

한 줄 정의

쉽게 말하면

정리하면

핵심은

요약하면

한 문장으로 표현하면

실무 한 줄 정리
```

### 좋지 않은 예

```text
한 줄 요약

데이터 주석은
AI를 위한 정답 데이터이다.
```

### 좋은 예

```text
데이터 주석은
AI 학습에 필요한 의미 정보를
데이터에 부여하는 활동이다.

주석 품질은
AI 성능에 직접적인 영향을 준다.
```

요약 내용은 일반 문단 안으로 흡수한다.

---

## 4.2 이모티콘 금지

다음 표현은 사용하지 않는다.

```text
👉
❓
🏭
🏢
✅
❌
📌
🤖
```

이모티콘 대신 일반 문장을 사용한다.

---

## 4.3 메타 코멘트 금지

다음 표현은 사용하지 않는다.

```text
아래는 작성본입니다

다음은

참고로

제가 작성한 내용은

설명드리자면
```

문서는 바로 본문부터 시작한다.

---

## 4.4 비유 사용 금지

AI는 아래 비유를 매우 자주 생성한다.

```text
스마트폰 사진

고양이

강아지

넷플릭스 추천

쇼핑몰 상품 분류
```

원칙적으로 사용하지 않는다.

실제 제조업 사례를 사용한다.

---

# 5. 용어 규칙

## 5.1 국문 우선

본문은 국문 중심으로 작성한다.

영문 병기는 최초 정의 시에만 사용한다.
핵심 질문(CQ) --> 핵심 질문으로 통일

### 좋은 예

```text
데이터 해설·주석(Data Annotation)

현업 전문가(SME)

Human-in-the-Loop(HITL)

하지만 Delamination은 괜찮음
```

이후에는

```text
주석

SME

HITL
```

로 통일한다.

---

## 5.2 현업 용어 우선

기술적으로 문제가 없는 경우, 문서에는 현업에서 실제 사용하는 용어를 우선 사용한다.

국문 중심 작성 원칙은 유지하되, 현업 문서·도면·품질 리포트·클라이언트 사이트에서 영문 용어를 그대로 사용하는 경우에는 무리하게 국문화하지 않는다.

### 지양

```text
현업에서 거의 쓰지 않는 직역 표현

심지 현상
도전성 양극 필라멘트
층간 들뜸
동박적층판
```

### 권장

```text
현업에서 실제 사용하는 표현

Wicking
CAF
Delamination
CCL
동판
균열
결함 유형
```

예를 들어 두산전자 품질 문서에서 Delamination, Wicking, CAF, CCL처럼 영문 용어가 표준적으로 사용되는 경우에는 해당 용어를 그대로 사용한다.

단, 모든 용어를 영문화하지 않는다.

현업에서 국문으로 자연스럽게 쓰는 표현은 국문을 유지한다.

---

## 5.4 제조업 용어 우선 규칙

제조업 사례에서는 번역의 정확성보다 현업 사용성을 우선한다.

문서 작성 시 아래 순서로 대표 용어를 결정한다.

```text
1. 클라이언트 사이트·현업 문서에서 실제 사용하는 표현
2. 품질 리포트·SOP·도면·검사 기준서에 등장하는 표현
3. 현업 SME가 인터뷰에서 사용하는 표현
4. 일반적인 국문 번역어
```

### 지양

```text
현업에서 쓰지 않는 번역어를 억지로 적용

들뜸
심지 현상
도전성 양극 필라멘트
동박적층판
```

### 권장

```text
현업 문서 기준 대표 용어 사용

Delamination
Wicking
CAF
CCL
동판
균열
결함 유형
```

최초 등장 시에는 필요한 경우에만 간단히 설명한다.

```text
Delamination은 층간 박리성 결함을 의미하며,
이후 본문에서는 Delamination으로 표기한다.
```

이후에는 대표 용어 하나로 통일한다.

# 추가 작성 규칙 (B-2 작성 과정 Lessons Learned 반영)

## 1. 장 구조는 8장을 기본으로 사용한다

가이드는 기본적으로 다음 구조를 사용한다.

```text
1. 개요

2. 왜 필요한가 (Why)

3. 무엇을 갖추나 (What)

4. 어디부터 하나 (Where)

5. 어떻게 구축·운영하나 (How)

6. 다른 주제와의 관계

7. KPI 및 Roadmap

8. Appendix
```

Why, What, Where, How를 일관되게 사용한다.

Example 장은 별도로 분리하지 않는다.

실제 사례는 How 장 내부에 포함한다.

독자가 각 장의 역할을 직관적으로 이해할 수 있도록 Why / What / Where / How를 일관되게 표기한다.

---

## 2. How 장은 방법론과 사례를 함께 설명한다

How 장은 단순한 방법론 설명이 아니다.

실제 제조업 사례를 활용하여 각 단계에서 무엇을 해야 하는지와 실제로 어떤 문제가 발생하는지를 함께 설명한다.

각 단계는 다음 구조를 사용한다.

```text
목적

주요 수행 작업

제조업 사례
(또는 두산 사례)

산출물

완료 기준
```

사례는 독립 예시가 아니라 하나의 프로젝트가 끝까지 이어지는 End-to-End 시나리오여야 한다.

---

## 3. How 장의 사례는 End-to-End 수행 시나리오여야 한다

How 장은 실제 제조업 환경에서 해당 주제를 처음부터 끝까지 수행하는 과정을 보여주는 End-to-End 시나리오를 포함해야 한다.

방법론 설명과 사례 설명은 분리하지 않고 함께 작성한다.

### 좋지 않은 예

```text
적용 배경

기대 효과

결과
```

또는

```text
간단한 사례 예시

결함 이미지 예시

고객 클레임 예시
```

### 좋은 예

```text
대상 데이터 선정
→ Taxonomy 설계
→ Guideline 작성
→ Pilot 수행
→ IAA 측정
→ 본 운영
→ QA
→ Registry 등록
```

How 장은 다음 질문에 답해야 한다.

```text
우리 회사가 실제로 이 체계를 구축한다면

어떤 데이터를 고르고

누가 참여하고

무엇을 만들고

어떤 문제를 만나고

어떻게 해결하며

최종적으로 어떤 자산이 남는가
```

---

## 4. 정본 모델은 텍스트 프로세스 형태로 먼저 정의한다

3장의 정본 모델은 가능하면 아래와 같은 형태를 우선 사용한다.

```text
대상 데이터 선정
→ Taxonomy 설계
→ Guideline 작성
→ Pilot Annotation
→ IAA 측정
→ 본 Annotation
→ QA
→ 자동화 및 자산화
```

독자가 전체 구조를 한눈에 이해할 수 있도록 텍스트 프로세스 형태를 우선 사용한다.

---

## 5. What 장은 구성요소 설명만 수행한다

What 장은 "무엇이 있는가"를 설명하는 장이다.

How 장처럼 절차를 상세하게 설명하지 않는다.

### What 장에서 설명하는 내용

- 구성 요소 정의
- 구성 요소 역할
- 구성 요소 간 관계

### What 장에서 설명하지 않는 내용

- 입력값
- 수행 절차
- 완료 기준
- 운영 방법

예를 들어 Taxonomy는

- 무엇인지
- 왜 필요한지
- 어떤 요소로 구성되는지

까지만 설명한다.

설계 절차는 How 장에서 설명한다.

---

## 6. 단계별 입력값·출력값 표는 기본적으로 사용하지 않는다

AI는 실행 방법론을 작성할 때 아래 구조를 과도하게 반복하는 경향이 있다.

```text
목적

입력값

수행 작업

예시

출력값

완료 기준
```

이 구조는 문서를 불필요하게 길게 만든다.

권장 구조

```text
목적

주요 수행 작업

사례

산출물

완료 기준
```

입력값과 출력값은 반드시 필요한 경우에만 사용한다.

---

## 7. 사례는 반드시 동일한 데이터로 끝까지 이어진다

How 장의 사례는 단계마다 다른 예시를 사용하지 않는다.

하나의 데이터와 하나의 프로젝트를 기준으로 끝까지 연결한다.

### 좋지 않은 예

```text
대상 선정
→ 고객 클레임

Taxonomy
→ 설비 로그

Pilot
→ 이미지 데이터

QA
→ 시험 결과 보고서
```

### 좋은 예

```text
동판 외관 검사 이미지

↓

대상 선정

↓

Taxonomy

↓

Guideline

↓

Pilot

↓

IAA

↓

QA

↓

Registry
```

독자는 하나의 프로젝트가 실제로 진행되는 과정을 따라갈 수 있어야 한다.

---

## 8. 약지도 대신 Weak Supervision 사용

다음 표현은 사용하지 않는다.

```text
약지도
```

다음 표현을 사용한다.

```text
Weak Supervision
```

최초 1회만 영문 병기 후 동일 용어 사용.

---

## 9. 로드맵은 Phase 체계를 사용한다

Level 기반 로드맵보다 Phase 기반 표현을 우선 사용한다.

권장 형식

```text
Phase 1
Preparation

Phase 2
AI Ready

Phase 3
Automation

Phase 4
Assetization
```

Level 1, Level 2 표현은 사용하지 않는다.

---

## 10. 짧은 선언문 반복 금지

다음과 같은 문장 구조는 사용하지 않는다.

```text
첫 번째 문제는 정답 데이터의 부재이다.

두 번째 문제는 판단 기준의 불일치이다.

세 번째 문제는 운영 이력의 부재이다.
```

대신

```markdown
- 정답 데이터의 부재:
  ...

- 판단 기준의 불일치:
  ...

- 운영 이력의 부재:
  ...
```

형태를 사용한다.

짧은 선언문 반복보다 불렛 기반 구조를 우선 사용한다.

---

## 11. 실제 기업 사례처럼 작성한다

사례는 단순 예시가 아니다.

실제 제조업 프로젝트 수행 문서처럼 작성한다.

### 좋지 않은 예

```text
결함 이미지를 선택하였다.

Taxonomy를 만들었다.

주석을 수행하였다.
```

### 좋은 예

```text
두산전자 품질혁신팀은
동판 외관 검사 이미지 5만 장을 확보하였다.

초기 분석 결과

동일 결함이

균열
갈라짐
크랙

등으로 혼재되어 있었다.

품질팀 SME와 데이터 조직은
결함 4종 체계를 정의하였고

Pilot 결과
긁힘과 균열 간 IAA가 0.61로 나타났다.

이에 따라 Guideline을 보완하고
대표 사례집을 추가 구축하였다.
```

실제 수행 과정과 문제 해결 과정을 보여준다.

## 12. 이모티콘 및 장식 기호 사용 금지

다음 표현은 사용하지 않는다.

```text
👉
❓
🏭
🏢
✅
❌
📌
🤖

★
※
▶
■
□
✓
✔
```

이모티콘 및 장식 기호를 사용하지 않는다.

강조는 문장 구조, 표, 제목을 통해 표현한다.

별표(**)를 활용한 과도한 강조도 지양한다.

예외적으로 표 항목명이나 짧은 불렛 제목에 한해 제한적으로 사용할 수 있다.

---

## 13. Appendix는 본문 보조 수준으로 유지한다

Appendix는 본문보다 길어져서는 안 된다.

권장 항목

- 주요 용어
- Gold Dataset 구축 기준
- 버전 관리 기준
- 운영 체크리스트

도구 목록, KPI 예시, 본문에서 이미 설명한 내용은 Appendix에 반복하지 않는다.

---
## 14. 현업 영문 용어는 무리하게 국문화하지 않는다

가이드는 국문 중심으로 작성하되, 제조 현장에서 영문 용어가 사실상 표준으로 쓰이는 경우에는 해당 영문 용어를 유지한다.

특히 품질 결함, 공정명, 자재명, 설비명, 검사 항목처럼 현업 문서에서 반복적으로 등장하는 용어는 국문 번역보다 실제 사용 표현을 우선한다.

### 적용 원칙

```text
현업에서 영문으로 쓰는 용어는 영문 유지

현업에서 국문으로 쓰는 용어는 국문 유지

처음 등장할 때만 필요한 경우 국문 설명 병기

이후에는 대표 용어 하나로 통일
```

### 예시

```text
Delamination → Delamination 유지
Wicking → Wicking 유지
CAF → CAF 유지
CCL → CCL 유지

균열 → 균열 유지
동판 → 동판 유지
결함 유형 → 결함 유형 유지
프리프레그 → 프리프레그 유지
```

### 좋지 않은 예

```text
Delamination을 문서 전체에서 들뜸으로 번역

Wicking을 심지 현상으로 번역

CAF를 도전성 양극 필라멘트로 번역

CCL을 동박적층판으로 반복 번역
```

### 좋은 예

```text
두산전자 품질혁신팀은 CCL 외관 검사 이미지 5만 장을 확보하였다.

초기 분석 결과,
동일 결함이 Delamination, Wicking, 균열 등으로 기록되어 있었고,
일부 작업자는 Delamination을 들뜸으로 표현하고 있었다.

품질팀 SME와 데이터 조직은 현업 문서에서 가장 일관되게 사용되는 Delamination을 대표 용어로 정하고,
들뜸, 층간 박리 등은 동의어로 관리하였다.
```

용어 선택의 목적은 번역의 일관성이 아니라 현업이 실제로 이해하고 사용할 수 있는 기준을 만드는 것이다.
