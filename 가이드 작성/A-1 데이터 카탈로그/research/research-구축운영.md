## 1. 구축 단계 Best Practice

### 1.1 구축 전 전제조건

데이터 카탈로그 구현이 실패하는 가장 흔한 원인은 기술적 문제가 아니라 조직 준비도 부족이다. "실패 지점의 거의 전부가 기술적이지 않으며, 가장 흔한 실수는 문제 정의보다 도구를 먼저 선택하는 것이다." [src-002]

플랫폼 도입 전 반드시 갖춰야 할 것 [src-002]:
- 특정 비즈니스 Pain에서 나온 명확히 정의된 사용 사례
- 결과에 책임지는 경영진 스폰서 (단순 지원이 아닌)
- 데이터 도메인 매핑 (개략적이라도)
- 초기 배정 포함 데이터 오너십 모델
- 경량 거버넌스 정책·프레임워크
- IT 인프라 준비 상태 평가 (클라우드 vs 온프레미스, SSO, 통합 환경)
- 비즈니스·IT 연결하는 내부 프로젝트 리드

### 1.2 구축 단계 Best Practice (10단계)

OvalEdge의 엔터프라이즈 데이터 카탈로그 구현 10단계 프레임워크 [src-001]:

**단계 1: 목표·성과 지표 정의**  
비즈니스 운영 우선순위와 연결되는 명확한 목표 수립. 예: 데이터 발견 효율성 향상, 오너십 가시성 개선, 중복 자산 감소.

**단계 2: 이해관계자·오너·스튜어드 지정**  
경영진 스폰서, 데이터 스튜어드, 엔지니어링팀, 컴플라이언스 담당 4그룹에 거버넌스 책임 배분. 오너십을 IT에만 맡기지 않는다 — 비즈니스 참여 필수. [src-001]

**단계 3: 데이터 도메인 우선순위 설정**  
재무 보고, 컴플라이언스 민감 데이터 등 고임팩트 영역부터 단계적 접근. "운영 복잡도 감소, 구현 통제 강화." [src-001]

**단계 4: 메타데이터·거버넌스 표준 수립**  
배포 전 명명 규칙, 오너십 구조, 분류 체계, 문서화 표준, 보존 정책, 용어집 정렬 절차 정의. [src-001]

**단계 5: 커넥터·통합 구성**  
클라우드 웨어하우스, DB, BI 플랫폼, ETL 도구, SaaS를 단계적으로 연결하여 메타데이터 품질 검증. [src-001]

**단계 6: 원천 시스템에서 메타데이터 수집·프로파일링**  
기술·비즈니스·운영 메타데이터를 자동화 수집. 프로파일링으로 거버넌스 갭 식별. [src-001]

**단계 7: 비즈니스 용어집·오너십 구조 구축**  
정의 승인, 스튜어드십 검토, 오너십 배정을 위한 운영 프로세스 수립. "비즈니스 컨텍스트가 카탈로그 채택에 핵심적 역할을 한다." [src-001]

**단계 8: 데이터 리니지·분류·품질 규칙 활성화**  
리니지 시각화, 거버넌스 통제, 품질 모니터링 구현. [src-001]

**단계 9: 온보딩·채택 프로그램 실행**  
채택을 지속적 프로세스로 인식. 1회성 교육이 아니라 메타데이터 품질·검색 연관성에 집중. [src-001]

**단계 10: 채택 측정·확장**  
사용자 활동, 메타데이터 완성도, 리니지 커버리지, 스튜어드십 참여도 모니터링. 지표 기반 도메인 확장. [src-001]

### 1.3 파일럿 → 전사 확장 패턴

성공적인 구현은 "하나의 도메인, 하나의 사용 사례, 높은 가시성"에서 시작한다. 범위 확장 압박을 모두 거부하고, 8~12주 파일럿 후 성과가 입증된 다음 도메인을 확장한다 [src-002].

파일럿 성공 기준 [src-002]:
- 대상 도메인 자산 커버리지 >70%
- 오너십 커버리지 >80%
- 스튜어드 작업 완료율 >75%

대기업의 전체 구현 기간: 9~18개월 [src-002].

## 2. 자동 메타데이터 수집 (Automated Metadata Harvesting)

### 2.1 수동 입력의 한계

"데이터 카탈로그를 메타데이터로 수동으로 채우는 것을 피하라. 대신 Collibra의 자동화 커넥터, AI 기반 메타데이터 추출, API 등 자동화 워크플로우를 활용하여 메타데이터가 최신 상태로 흐르게 한다." [src-005]

전통적 데이터 카탈로그는 팀이 데이터가 진화함에 따라 수동으로 데이터 항목과 업데이트를 입력하는 방식에 의존 — 시간 집약적이며 상당한 수작업이 필요하다. [src-010]

### 2.2 자동화 수집 방식

**커넥터(Connector) 기반 자동 수집:**  
카탈로그 도구는 원천 시스템에 직접 연결하는 커넥터를 통해 메타데이터를 자동 수집한다. 수집 대상:
- **기술 메타데이터**: 스키마, 테이블·컬럼명, 데이터 타입, 행 수 [src-005]
- **운영 메타데이터**: 사용 빈도, 최종 업데이트 타임스탬프, 데이터 리니지 [src-010]
- **비즈니스 메타데이터**: 설명, 비즈니스 용어 매핑, 오너 정보 (이 부분은 자동화 한계, 수동 보강 필요) [src-001]

**연속 수집(Continuous Harvesting):**  
"연속 메타데이터 수집은 스키마가 변화하고, 파이프라인이 변경되고, 새로운 도구가 도입되더라도 카탈로그가 현실을 반영하도록 보장한다. 이 자동화는 정확도를 높일 뿐 아니라, 카탈로그가 항상 최신 데이터 환경 상황을 보여줌으로써 채택을 가속한다." [src-005]

**Collibra Edge (네트워크 격리 환경):**  
방화벽 뒤 또는 제한된 네트워크 환경에서는 Collibra Edge를 통해 내부 인프라를 외부에 노출하지 않고 안전하게 메타데이터를 수집한다 [src-005].

**DataHub Push/Pull 이중 방식:**  
직접 API 호출 또는 Kafka 스트림을 통한 두 가지 수집 방식 제공. 80+ 프로덕션급 커넥터, 컬럼 수준 리니지(SQL 쿼리 파싱), 사용 통계, 프로파일링 포함 [src-009].

### 2.3 자동 vs 수동의 경계

| 영역 | 자동화 가능 | 수동 보강 필요 |
|---|---|---|
| 기술 메타데이터 | ✅ 스키마, 데이터 타입, 행 수, 타임스탬프 | - |
| 비즈니스 메타데이터 | △ AI 초안 생성 (사람 검토 필요) | ✅ 정의, 비즈니스 용어, 목적 |
| 오너십 | △ 규칙 기반 초기 배정 | ✅ 도메인 오너 확인·승인 |
| 분류(PII 등) | ✅ 패턴 기반 자동 감지 | ✅ 예외 케이스 검토 |
| 리니지 | ✅ SQL 파싱, ETL 메타데이터 | △ 수기 코드 변환 부분 |

자동화가 정확성·연관성을 위해 강력한 오류 검사·검증이 필요하다 [src-001]. 자동 수집만 믿고 큐레이션을 안 하면 카탈로그가 기술 메타데이터 덤프로 전락한다 [src-003].

## 3. 연동 (Integration)

### 3.1 연동 유형

**커넥터 자동 연동 (메인 방식):**  
Collibra는 100+ 네이티브 커넥터 제공 [src-005]:
- 클라우드 데이터 플랫폼: Snowflake, Databricks, Redshift, BigQuery
- 온프레미스 DB: Oracle, SQL Server, PostgreSQL, MySQL
- 데이터 레이크: S3, ADLS, HDFS
- BI 도구: Tableau, Power BI, Looker, Qlik
- ETL 도구: Informatica, Talend, SSIS, DataStage

DataHub는 80+ 커넥터 및 JDBC 기반 커넥터로 온프레미스 DB 연결 지원 [src-009].

**수동 업로드 파이프라인 (레거시 시스템):**  
"레거시 기술용 커넥터 부재로 원천 메타데이터 가용성 저하." [src-007]  
커넥터가 없는 레거시 시스템(구형 ERP, MES, 전용 DB 등)의 경우:
- API/SDK를 통한 커스텀 수집 파이프라인 구현
- CSV/파일 기반 수동 메타데이터 업로드
- 커스텀 스크립트로 주기적 배치 수집

**리니지(데이터 계보) 연계:**  
Collibra Lineage Harvester는 데이터 원천 가까이에서 실행되어 SQL 스크립트·ETL 스크립트 등 변환 로직을 수집하고 Collibra Data Lineage 서버에 전송 [src-005].  
DataHub는 SQL 쿼리 파싱으로 컬럼 수준 업스트림·다운스트림 의존관계를 자동 표시 [src-009].

### 3.2 숨겨진 변환 문제

ETL 프로세스와 프로그래밍 언어(Python, Scala, Java)에 내재된 변환 로직은 카탈로그가 자동으로 파악하지 못하는 경우가 많다 [src-007]. 이 부분은 파이프라인 코드에 메타데이터 에밋(emit) 로직을 삽입하거나 수동 문서화가 필요하다.

## 4. 거버넌스·운영 (Governance & Operations)

### 4.1 역할 구조

**데이터 오너 (Data Owner):**  
특정 데이터 자산·도메인의 전반적 품질, 접근, 활용에 책임지는 시니어 담당자(경영진·부서장). 데이터 정책 정의 및 데이터 스튜어드에게 책임 위임 권한 보유 [src-004].

**데이터 스튜어드 (Data Steward):**  
거버넌스 정책 시행의 일상적 운영 책임자. 메타데이터 관리, 데이터 정의, 품질 통제, 접근 관리, 데이터 자산의 신뢰성 보장 [src-004].  
유형:
- **비즈니스 스튜어드**: 비즈니스 단위·프로세스 관점 운영
- **운영 스튜어드**: 거버넌스 정책·절차 일상 실행, 품질 점검, 접근 요청 처리
- **메타데이터 스튜어드**: 데이터 사전, 리니지, 카탈로그 관리

**데이터 관리자 (Data Custodian):**  
IT 담당자로 데이터 저장, 유지보수, 보안의 기술적 측면 담당. 인프라 관리 및 보안 조치 구현 [src-004].

### 4.2 변경 관리 워크플로우

**Microsoft Purview의 승인 워크플로우 예시** [src-006]:

*데이터 접근 요청 흐름:*
1. 사용자가 데이터 제품 접근 요청 제출
2. Pending on all: 모든 승인자(데이터 오너) 동의 필요 / Pending on any: 1명 동의로 충분
3. 조건 분기: If yes → 접근 승인, If no → 요청 거절
4. 이메일 알림 자동 발송

*데이터 제품·용어집 게시 워크플로우:*
1. 데이터 제품 오너·스튜어드가 드래프트 단계에서 협업
2. 최종 검토 후 게시 승인 워크플로우 실행
3. If yes → 카탈로그에 게시, If no → 드래프트 유지

**Collibra의 기본 제공 워크플로우 11종** [src-005]:  
- Assign Owner To Data Set: 신규 데이터셋 오너 자동 배정
- Post Data Ingestion: 새로 등록된 스키마에 오너·기술 스튜어드 배정
- Simple Approval: 단일 단계 자산 승인
- Escalation Process: 작업 에스컬레이션

**자동화 큐레이션의 인간 감독 원칙:**  
AI가 생성한 메타데이터 값은 "실제 저장 전 미리 볼 수 있어, 인간 감독을 유지하면서 지루한 수작업을 제거한다." 실제로 SAP 테이블 문서화 작업에서 6주 수작업을 약 15분으로 단축한 사례가 있다 [src-004].

### 4.3 스튜어드십 운영화의 현실

스튜어드십 책임은 별도 전담 직책 신설보다 기존 역할에 분산 배정이 현실적:
- 비즈니스 분석가 → 메타데이터 관리
- 데이터 엔지니어 → 기술 품질 유지
- IT 관리자 → 거버넌스 정책 구현
- 부서 책임자 → 도메인 내 품질 기준 시행 [src-004]

## 5. 최신성 유지 (Freshness)

### 5.1 왜 최신성이 중요한가

"오래된 오너 필드·비즈니스 설명이 있는 자산은 사람들을 적극적으로 오도한다." [src-010]  
"오래된 또는 불완전한 메타데이터 = 데이터 불신의 주요 원인. 분석가들이 데이터를 수동으로 검증하거나 작업을 중복 수행하는 데 시간을 낭비하게 된다." [src-010]

### 5.2 갱신 방식

**① 주기 갱신 (Scheduled Refresh):**  
커넥터를 통해 일간·주간 등 정기 스캔으로 기술 메타데이터를 자동 동기화.

**② 변경 시점 자동 갱신 (Event-Driven Refresh):**  
"스키마 변경, 품질 이슈 발생, 신규 원천 등장 시 카탈로그가 리니지·품질 점수·거버넌스 메타데이터를 실시간 자동 업데이트." [src-010]

**③ 신선도 자동화 (Freshness Review Automation):**  
data.world의 사례:
- **Refreshed On 필드**: 리소스가 마지막으로 갱신된 시점 추적
- **Shelf life(유효 기간)** 설정: 갱신 유효 기간(일 단위)
- **Refresh Due 필드**: 유효 기간 초과 시 주의 필요 표시
- 임계값 초과 시 담당 스튜어드에게 자동 알림 발송 [src-010]

**④ 데이터 파이프라인 이벤트 연동:**  
파이프라인 이벤트가 워크플로우 트리거로 연결되어 새로운 배치 실행 후 자동으로 카탈로그를 업데이트. [src-002]

### 5.3 운영 메타데이터 (신선도 판단의 근거)

사용자가 데이터의 신선도·신뢰성을 판단하기 위해 카탈로그에 있어야 할 항목 [src-010]:
- 갱신 빈도
- 최종 업데이트 타임스탬프
- 사용 패턴·쿼리 통계
- 성능 지표

## 6. 흔한 실패 요인 및 교훈

### 6.1 구조적 실패 패턴 (요약)

| 실패 요인 | 교훈 |
|---|---|
| 문제 정의 전 도구 먼저 도입 | 사용 사례부터 명확히 정의 후 플랫폼 선정 [src-002] |
| 오너십 없이 출시 | 출시 전 도메인 오너 배정 완료 필수 [src-002] |
| 전체 동시 카탈로깅 시도 | 한 도메인, 한 사용 사례부터 파일럿 [src-002] |
| 자동 수집만 믿고 큐레이션 안 함 | 자동화 + 스튜어드 검토의 하이브리드 필수 [src-001] |
| 비즈니스 설명 없는 기술 메타데이터 덤프 | 비즈니스 용어집·도메인 연결 작업 병행 [src-003] |
| 카탈로그를 완성 프로젝트로 취급 | "지속적 거버넌스 운영 모델"로 인식 전환 [src-001] |
| 변경 관리 과소평가 | 도구 교육보다 업무 습관 변화에 더 많은 투자 [src-002] |
| 레거시 커넥터 부재로 핵심 자산 누락 | 커스텀 파이프라인 또는 수동 업로드 보완책 마련 [src-007] |

### 6.2 자동 수집의 함정

"카탈로그는 방대한 메타데이터를 수집할 수 있지만, 적절한 의미론·설명·분류·분류 체계 없이는 관련 데이터를 찾고 활용하기 매우 어렵다." [src-007]

기업 카탈로그에서 자산의 40%가 잘못 분류되거나 노후화된 디렉토리에 있는 경우가 많다 — 원래 분류 체계를 구축한 팀이 이동하고 아무도 업데이트 부담을 지지 않으려 하기 때문. [src-007]

### 6.3 메타데이터가 업무 책임에 포함되지 않는 문제

"도메인 데이터 오너들이 명명 표준·포괄적 정의를 위한 가이드라인을 제공받지 못하며, 메타데이터 관리가 업무 책임·OKR에 포함되지 않는다." [src-007]

교훈: 메타데이터 관리 참여를 성과 지표(OKR/KPI)에 반영하고, 스튜어드에게 명확한 업무 시간·도구·권한을 부여해야 한다.
