## D-1 Physical 데이터  *(기본 필수 — 설비 데이터 범위가 핵심)*
> 한 줄: 센서·설비가 쏟아내는 현장 신호를 AI가 바로 쓸 수 있게 시간·단위·설비ID를 맞춰 연결하는 것.
> 핵심 계열사: 두산에너빌리티

```
1. 개요
   1.1 Physical 데이터란 (센서·설비·현장 시스템이 만드는 신호를 AI 입력으로 연결)
   1.2 적용 범위와 체계 내 위치 (사용해도 되는지 최종 판정은 사용성(C-2), 문서 전처리는 비정형(B-1)과 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (시간·단위·설비 이름이 제각각이라 데이터를 합치거나 비교할 수 없음)
   2.2 기대 효과 (이상 감지·예지보전·품질 분석에 현장 데이터를 바로 활용)
3. 무엇을 갖추나
   3.1 수집 대상 목록 (센서값·설비 상태·검사 장비값·알람 로그·작업 조건·환경 데이터)   ← KQ1
   3.2 표준 3종 맞추기 (시간 동기화·단위·설비ID/공정ID·샘플링 주기)   ← KQ3, 쉽게: "언제·무슨 단위·어느 설비"를 통일
   3.3 기초 정제 기준 (센서 잡음·결측·이상값·신호 끊김을 AI에 넣기 전 1차 정리)   ← KQ4
4. 수집 대상 우선순위 (핵심 설비부터)   ← 기본 필수이므로 "전 태그 일괄 X, 순서"
   4.1 우선 대상 (고장·품질 영향 큰 핵심 설비·핵심 태그부터)
   4.2 처리 방식 구분 (즉시 대응용 실시간 vs 분석·리포팅용 배치)   ← KQ5, 쉽게: 급한 신호와 모아서 보는 신호를 나눔
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (설비마다 시간·단위가 달라 합산 불가 → 표준을 맞추면 여러 설비를 한 화면에서 비교·이상 감지)
   5.2 흐름 미리보기 (핵심 설비 태그 선정 → 수집 연결 → 시간·단위·설비ID 표준화 → 잡음·결측 1차 정제 → AI 입력)
6. 어떻게 준비·운영하나 (상세)
   6.1 수집·연결 방식 (설비·센서·Edge·IoT 플랫폼·MES/QMS 연결, 실시간/파일/수동 입력 구분)   ← KQ2
   6.2 구축 절차 (대상 태그 정의 → 수집 연결 → 시간·단위·설비ID 표준화 → 기초 정제 → 검증)
   6.3 운영 (태그 추가·설비 교체 시 표준 갱신, 정제 기준 점검) / 역할·책임
7. 다른 주제와의 관계
   7.1 사용 가능 최종 판정(C-2)·문서 전처리(B-1)·복구/백업(F-1)과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (표준화율·결측/이상값 비율·실시간 수집 지연)
   8.2 고도화 (이상값·결측 자동 보정, 신규 태그 자동 표준 매핑)
```

## D-2 API/Tool 연계 데이터  *(선택 적용 — Agent를 쓸 때만)*
> 한 줄: AI 에이전트가 외부 시스템을 안전하게 부르도록 "Tool 사용 설명서(기능·입력·출력·제약)"를 정리하는 것.
> 핵심 계열사: 두산로보틱스

```
1. 개요
   1.1 API/Tool 연계 데이터란 (AI 에이전트가 호출할 기능의 설명서·규격)
   1.2 적용 범위와 체계 내 위치 (호출 절차/실행 흐름은 Prompt·Harness(D-3), 로그 활용·개선은 운영개선(E-4)과 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (설명이 부실하면 AI가 엉뚱한 Tool을 부르거나 위험한 동작을 실행)
   2.2 기대 효과 (에이전트가 맞는 도구를 정확·안전하게 호출, 오작동·사고 예방)
3. 무엇을 갖추나
   3.1 Tool 후보 목록 (조회·계산·보고서 생성·시스템 검색·업무 실행)   ← KQ1
   3.2 Tool 설명서 (이름·기능·목적·입력값·출력값·제약·호출 예시)   ← KQ2
   3.3 입출력 규격 (필수값·데이터 타입·허용값·응답 구조·오류 응답)   ← KQ3, 쉽게: 무엇을 넣고 무엇이 나오는지 약속
4. 연계 대상 선정 (조회형부터, 위험도 분류)   ← 선택 적용이므로 "무엇을 연결하나"
   4.1 적용 판단 (에이전트가 실제로 외부 시스템을 부를 때만, 단순 답변형이면 불필요)
   4.2 위험도 분류·우선순위 (조회형 → 추천형 → 실행형 → 외부 전송형, 위험할수록 사람 승인)   ← KQ4
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (에이전트가 어떤 시스템을 어떻게 부를지 몰라 실패 → 설명서를 주면 정확히 조회하고 위험 동작은 승인 거침)
   5.2 흐름 미리보기 (조회형 Tool 골라 설명서 작성 → 입출력 규격 정의 → 위험도 분류·승인 규칙 → 등록 후 에이전트가 호출)
6. 어떻게 준비·운영하나 (상세)
   6.1 설명서·규격 작성 도구·표준 검토 (Tool 등록소 Registry 형태)
   6.2 구축 절차 (Tool 식별 → 설명서·입출력 규격 작성 → 위험도/승인 규칙 → 호출 테스트 → 등록)
   6.3 운영 — 버전·오류 관리 (기능·규격·응답이 바뀌면 버전·변경 이력 관리해 오호출 방지)   ← KQ5
   6.4 호출 기록 (호출 시점·입력·출력·주체·성공/실패·오류를 로그로 남김 — 활용·개선은 E-4)   ← KQ6
7. 다른 주제와의 관계
   7.1 호출 절차·실행 흐름(D-3)·로그 활용 개선(E-4)·거버넌스 승인과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (호출 성공률·오호출률·위험 Tool 승인 준수율)
   8.2 고도화 (시스템 명세에서 설명서 초안 자동 생성 → 사람 검수)
```

## D-3 Prompt/Harness 자산화  *(선택 적용 — 반복·고영향 업무만)*
> 한 줄: 사람이 하던 업무 절차·판단 기준을 AI가 그대로 실행할 Prompt로 정리하고 버전으로 관리하는 것.
> 핵심 계열사: 두산밥캣

```
1. 개요
   1.1 Prompt/Harness 자산화란 (업무 지시·판단을 AI 실행용 Prompt로 구조화, Harness = 묶음 패키지)
   1.2 적용 범위와 체계 내 위치 (잘했는지 평가 기준은 평가데이터(E-3), Tool 명세 자체는 API/Tool(D-2)과 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (사람마다 Prompt를 따로 만들어 결과가 들쭉날쭉하고 노하우가 흩어짐)
   2.2 기대 효과 (검증된 Prompt를 공용 자산으로 재사용 → 결과 일관·재작업 감소·노하우 축적)
3. 무엇을 갖추나
   3.1 자산 목록 (업무별 Prompt·시스템 지시문·역할 정의·판단 기준·출력 형식·예외 처리)   ← KQ1
   3.2 업무→Prompt 변환 틀 (입력·판단 단계·참조 기준·출력 형식·금지 행동으로 나눠 적기)   ← KQ2
   3.3 Harness (Prompt + 입력 데이터 + Tool 호출 순서 + 출력 형식을 묶은 반복 실행 단위)   ← KQ6
4. ★ 무엇을 자산화하나 (반복·고영향 업무만)   ← 선택 적용의 핵심 판단
   4.1 적용 판단 (자주 쓰거나 업무 판단에 영향이 큰 것만, 일회성·실험용은 제외)
   4.2 우선순위 (현업 노하우가 담긴 반복 업무부터 작게 시작)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (담당자마다 Prompt가 달라 결과가 제각각 → 검증된 공용 Prompt로 누가 써도 같은 품질)
   5.2 흐름 미리보기 (반복·고영향 업무 선정 → 업무 절차를 Prompt로 변환 → 의존성 연결 → 버전 등록 → Harness로 재사용)
6. 어떻게 준비·운영하나 (상세)
   6.1 관리 도구·표준 검토 (Prompt 등록소 Registry·버전 관리 도구)
   6.2 구축 절차 (대상 업무 선정 → 절차→Prompt 변환 → 의존성 연결 → 테스트 → Harness 패키징)
   6.3 버전·의존성 관리 (변경 이력·사유·승인자 기록, 어떤 데이터·용어(Glossary)·Tool에 의존하는지 연결)   ← KQ3, KQ4
   6.4 운영 — 성능 저하 감지·개선 (평가 결과·사용자 수정·실패 사례·정책 변경으로 개선 판단, 영향 큰 것은 승인 후 배포)   ← KQ5
7. 다른 주제와의 관계
   7.1 평가 기준(E-3)·Tool 명세(D-2)·용어(Glossary, A-3)와의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (재사용률·사용자 수정률·업무 결과 일관성)
   8.2 고도화 (성능 저하 자동 감지 → 개선안 초안 추천, 사람 검수 후 배포)
```
