# AI-Ready Data 매뉴얼 — 상세 목차 (논리 순서·현업 눈높이) v3

> **작성 원칙**
> 1. **12개를 다 채우지 않는다.** A-1 12-섹션은 참고용 풀일 뿐. 주제에 필요한 것만, 보통 7~9개.
> 2. **순서는 논리 흐름으로**: 왜(Why) → 뭐고 뭘로 구성(What) → 언제/어디에(When) → 어떻게 준비·운영(How) → 다른 주제와 관계 → 지표·고도화 → **계열사 예시(맨 끝)**.
> 3. **계열사 예시는 맨 끝.** 구축 절차는 사례와 별개로 "어떻게(How)"에 둔다.
> 4. **세부 H3 목차는 둔다.** 각 섹션 아래 번호 매긴 소목차(3.1·3.2…)는 있어야 한다 — 단 **내용은 쉽고 얕게**. 어렵고 딥한 전문 내용·약어는 넣지 않는다.

## 기본 흐름 (필수 아님 · 주제별 가감)

1. 개요 (이게 뭐고 체계 내 어디) · 2. 왜 필요한가 · 3. 무엇을 갖추나 · 4. 언제/어디에 하나 · 5. 어떻게 준비하나(구축 포함) · 6. 다른 주제와의 관계 · 7. 성과 지표·고도화 · 8. 계열사 예시(맨 끝)

> 가벼운 주제는 일부 섹션을 합쳐 6~7개로, 무거운 주제만 9개 가까이.

## A-2 메타데이터  *(기본적으로 다 필요한 주제)*
> 한 줄: 데이터의 "설명서" — 이 항목이 무슨 뜻이고, 단위·갱신주기·담당자가 누구인지 적어두는 것.
> 핵심 계열사: 두산에너빌리티

```
1. 개요
   1.1 메타데이터란 (데이터의 설명서)
   1.2 적용 범위와 체계 내 위치 (소재는 카탈로그, 용어 뜻은 Glossary와 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (항목 이름·단위만으론 뜻을 몰라 AI가 잘못 해석)
   2.2 기대 효과 (재사용이 쉬워지고 오해석이 줄어듦)
3. 무엇을 갖추나
   3.1 표준 항목 (타입·단위·생성 시스템·갱신 주기·담당자)
   3.2 항목 설명서 (약어·코드 풀이 — 예: DEF_CD = 결함 유형 코드)
   3.3 단위·기준일·집계 기준 (숫자를 잘못 비교하지 않도록)
4. 어디부터 하나 (우선순위)   ← 메타데이터는 다 필요하므로 "순서"를 다룸
   4.1 정비 우선 대상 (설명이 비었거나 단위가 뒤죽박죽인 데이터부터)
   4.2 자동으로 모으고 사람은 검수만 (★ 일일이 입력하지 않는다)
5. 어떻게 준비하나
   5.1 수집·관리 도구 검토
   5.2 구축 절차 (자동 수집 연결 → 설명 보완 → 검증)
   5.3 운영 (항목이 바뀌면 갱신·승인) / 역할·책임
6. 다른 주제와의 관계
   6.1 카탈로그(소재)·Glossary(용어)·품질과의 경계
7. 성과 지표·고도화
   7.1 성과 지표 (작성 완성률·자동 수집률·설명 충실도)
   7.2 고도화 (AI가 설명 초안 작성 → 사람 검수)
8. 계열사 예시 — 두산에너빌리티
   8.1 상황 (현장 약어·설비 단위가 제각각)
   8.2 적용 (표준 항목·설명서·단위 정리)
   8.3 효과 (AI가 설비 데이터를 바로 해석)
```

## B-3 온톨로지  *(골라서 하는 주제 — "언제 하나"가 중요)*
> 한 줄: 개념들 사이의 "관계 지도" — 결함 ↔ 원인 ↔ 조치처럼 무엇이 무엇과 이어지는지 정리.
> 핵심 계열사: 두산에너빌리티

```
1. 개요
   1.1 온톨로지란 (개념 사이의 관계 지도)
   1.2 적용 범위와 체계 내 위치 (단어 뜻이면 Glossary, 학습용 분류면 주석과 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (단순 검색으론 "왜 생겼고 어떻게 조치하나"를 못 찾음)
   2.2 기대 효과 (원인 분석·유사 사례·조치 추천이 가능)
3. 무엇을 갖추나
   3.1 개념 목록 (제품·공정·결함·원인·조치)
   3.2 개념 사이 관계 (발생한다·원인이 된다·조치한다)
   3.3 개념과 실제 데이터·문서 연결
4. ★ 언제 온톨로지를 하나   ← 전부 다 하는 게 아님
   4.1 적용 판단 (원인 분석·추천처럼 "관계"가 중요할 때만, 단어 통일이면 Glossary로 충분)
   4.2 작게 시작 (한 업무부터, 전사 거대 모델 금지)
   4.3 우선 만들 영역 고르기 (지식이 많고 AI 활용 가치 큰 곳)
5. 어떻게 준비하나
   5.1 관계 저장·검색 도구(그래프 DB) 검토
   5.2 구축 절차 (개념·관계 정리 → 데이터·문서 연결 → 검증)
   5.3 운영 (개념·관계가 바뀌면 전문가 검토 후 갱신) / 현업 전문가 참여
6. 다른 주제와의 관계
   6.1 Glossary(단어 뜻)·주석(분류 라벨)·AI 검색/에이전트와의 경계
7. 성과 지표·고도화
   7.1 성과 지표 (관계 채움 정도·추천 적중률)
   7.2 고도화 (문서에서 관계를 AI가 초안 추출 → 전문가 검수)
8. 계열사 예시 — 두산에너빌리티
   8.1 상황 (품질 지식이 문서·머릿속에 흩어져 있음)
   8.2 적용 (결함→원인→조치 관계도 구성)
   8.3 효과 (유사 사례·조치 추천)
```

## E-2 합성데이터  *(골라서 하는 주제 + 만드는 방식이 핵심)*
> 한 줄: 실제 데이터가 부족할 때 인공으로 만들어 채우는 데이터.
> 핵심 계열사: 두산전자

```
1. 개요
   1.1 합성데이터란 (인공으로 만든 데이터 — 원본 대체가 아님)
   1.2 적용 범위와 체계 내 위치
2. 왜 필요한가
   2.1 현업 Pain Point (불량 사례가 너무 적거나, 보안 때문에 원본을 못 씀)
   2.2 기대 효과 (부족한 사례를 채워 학습·검증 가능)
3. 무엇을 갖추나
   3.1 무엇을 합성할지 (불량·장애·예외 사례)
   3.2 만드는 방식 (통계·시뮬레이션·생성모델)
   3.3 검증 항목 + "합성" 표시
4. ★ 언제 합성하나   ← 실데이터로 충분하면 안 함
   4.1 적용 판단 (데이터가 부족·희소하거나 보안으로 원본을 못 쓸 때만)
   4.2 비교 기준이 될 실데이터 먼저 확보
5. 어떻게 만드나
   5.1 만드는 방식·도구 비교·선정 (시범 적용)
   5.2 만드는 절차 (생성 → 검증 → "합성" 표시·등록)
   5.3 운영 (재생성, 위험 점검: 편향·현실 왜곡·재식별)
6. 다른 주제와의 관계
   6.1 평가셋(E-3)·비식별(F-4)·사용 판정(C-2)과의 경계
7. 성과 지표·고도화
   7.1 성과 지표 (실데이터와 닮은 정도·모델 성능 개선)
   7.2 고도화 (검증 자동화)
8. 계열사 예시 — 두산전자
   8.1 상황 (특정 불량 사례가 극히 적음)
   8.2 적용 (희소 불량 합성 → 검증 → 학습)
   8.3 효과 (검사 AI 정확도 향상)
```
