# AI-Ready Data 매뉴얼 — 상세 목차 (논리 순서·간결판) v3

> **작성 원칙**
> 1. **12개를 다 채우지 않는다.** A-1 12-섹션은 내용 풀(참고)일 뿐. 주제에 필요한 것만, 보통 7~9개.
> 2. **순서는 논리 흐름으로**: 왜(Why) → 뭐고 뭘로 구성(What) → 언제/어디에(When) → 어떻게 준비·운영(How) → 다른 주제와 관계 → 지표·고도화 → **계열사 예시(맨 끝)**.
> 3. **계열사 예시는 맨 끝.** 사례는 위 전체를 한 시나리오로 꿰는 것이라 마지막. **구축 절차는 사례와 별개로 "How"에** 둔다. (← A-1이 사례를 7번 중간에 둔 건 어색해서 바로잡음)
> 4. **현업 눈높이·간결**: 쉬운 표현, 약어 최소, 섹션당 핵심 1~3개.

## 기본 흐름 (필수 아님 · 주제별 가감)

| 순서 | 섹션 | 축 |
|------|------|----|
| 1 | 개요 (이게 뭐고 체계 내 어디) | What |
| 2 | 왜 필요한가 (현업 Pain Point·효과) | Why |
| 3 | 무엇을 갖추나 (구성 요소) | What |
| 4 | 언제/어디에 하나 (대상·적용 판단) | When |
| 5 | 어떻게 준비하나 (방식·도구·구축 절차·역할) | How |
| 6 | 어떻게 운영하나 | How |
| 7 | 다른 주제와의 관계 | (경계) |
| 8 | 성과 지표·고도화 | How |
| 9 | 계열사 적용 예시 (전체를 한 사례로) | 종합 |

> 가벼운 주제는 5+6을 합치거나 일부를 빼 6~7개로. 무거운 주제만 9개 가까이.

## A-2 메타데이터  *(기본적으로 다 필요한 주제)*
> 한 줄: 데이터의 "설명서" — 이 항목이 무슨 뜻이고, 단위·갱신주기·담당자가 누구인지 적어두는 것.
> 핵심 계열사: 두산에너빌리티

```
1. 개요 — 메타데이터란, 체계 내 위치 (소재는 카탈로그, 용어 뜻은 Glossary와 구분)
2. 왜 필요한가
   - 항목 이름·단위만으론 뜻을 몰라 AI가 잘못 해석함
   - 정비하면: 재사용 쉬워지고 오해석 줄어듦
3. 무엇을 갖추나
   - 표준 항목 (타입·단위·생성 시스템·갱신 주기·담당자)
   - 항목 설명서 (약어·코드 풀이 — 예: DEF_CD = 결함 유형 코드)
   - 단위·기준일·집계 기준 (숫자를 잘못 비교하지 않도록)
4. 어디부터 하나  ← 메타데이터는 다 필요하므로 "우선순위"
   - 설명이 비었거나 단위가 뒤죽박죽인 데이터부터
5. 어떻게 준비하나
   - 자동으로 모으고 사람은 검수만 (★ 일일이 입력하지 않는다)
   - 수집·관리 도구 검토 / 역할·책임
6. 어떻게 운영하나 (항목이 바뀌면 갱신·승인)
7. 다른 주제와의 관계 (카탈로그=소재 / Glossary=용어 / 품질·거버넌스)
8. 성과 지표·고도화 (완성률·자동 수집률 / AI가 설명 초안 작성 → 사람 검수)
9. 계열사 예시 — 두산에너빌리티: 현장 약어·설비 단위를 정리해 AI가 바로 해석
```

## B-3 온톨로지  *(골라서 하는 주제 — "언제 하나"가 중요)*
> 한 줄: 개념들 사이의 "관계 지도" — 결함 ↔ 원인 ↔ 조치처럼 무엇이 무엇과 이어지는지 정리.
> 핵심 계열사: 두산에너빌리티

```
1. 개요 — 온톨로지란, 체계 내 위치 (단어 뜻이면 Glossary, 학습용 분류면 주석과 구분)
2. 왜 필요한가
   - 단순 검색으론 "왜 생겼고 어떻게 조치하나"를 못 찾음
   - 만들면: 원인 분석·유사 사례·조치 추천이 가능
3. 무엇을 갖추나
   - 개념 목록 (제품·공정·결함·원인·조치)
   - 개념 사이 관계 (발생한다·원인이 된다·조치한다)
   - 개념과 실제 데이터·문서 연결
4. ★ 언제 온톨로지를 하나  ← 전부 다 하는 게 아님
   - 원인 분석·추천처럼 "관계"가 중요한 업무일 때만 (단어 통일이면 Glossary로 충분)
   - 한 업무부터 작게 시작 (전사 거대 모델 금지)
5. 어떻게 준비하나
   - 개념·관계 정리 → 데이터·문서 연결 → 검증
   - 관계 저장·검색 도구(그래프 DB) 검토 / 현업 전문가 참여
6. 어떻게 운영하나 (개념·관계가 바뀌면 전문가 검토 후 갱신)
7. 다른 주제와의 관계 (Glossary=단어 뜻 / 주석=분류 라벨 / AI 검색·에이전트)
8. 성과 지표·고도화 (관계 채움 정도·추천 적중률 / 문서에서 관계 자동 추출 → 검수)
9. 계열사 예시 — 두산에너빌리티: 품질 관계도(결함→원인→조치)로 조치 추천
```

## E-2 합성데이터  *(골라서 하는 주제 + 만드는 방식이 핵심)*
> 한 줄: 실제 데이터가 부족할 때 인공으로 만들어 채우는 데이터.
> 핵심 계열사: 두산전자

```
1. 개요 — 합성데이터란 (인공으로 만든 데이터, 원본 대체가 아님), 체계 내 위치
2. 왜 필요한가
   - 불량 사례가 너무 적거나, 보안 때문에 원본을 못 씀
   - 만들면: 부족한 사례를 채워 학습·검증 가능
3. 무엇을 갖추나
   - 무엇을 합성할지 (불량·장애·예외 사례)
   - 만드는 방식 (통계·시뮬레이션·생성모델)
   - 잘 만들었는지 확인 항목 + "합성" 표시
4. ★ 언제 합성하나  ← 실데이터로 충분하면 안 함
   - 데이터가 부족·희소하거나 보안으로 원본을 못 쓸 때만
5. 어떻게 만드나
   - 만드는 방식·도구 비교 → 시범 적용 → 생성 → 검증 → "합성" 표시·등록
6. 어떻게 운영하나 (재생성, 위험 점검: 편향·현실 왜곡·재식별)
7. 다른 주제와의 관계 (평가셋=E-3 / 비식별=F-4 / 사용 판정=C-2)
8. 성과 지표·고도화 (실데이터와 닮은 정도·성능 개선 / 검증 자동화)
9. 계열사 예시 — 두산전자: 희소 불량 사례를 합성해 검사 AI 학습
```
