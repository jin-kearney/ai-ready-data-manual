# AI-Ready Data 매뉴얼 — 상세 목차 (논리 순서·현업 눈높이) v3

> **작성 원칙**
> 1. **12개를 다 채우지 않는다.** A-1 12-섹션은 참고용 풀일 뿐. 주제에 필요한 것만, 보통 7~8개.
> 2. **순서는 논리 흐름으로**: 왜(Why) → 뭐고 뭘로 구성(What) → 언제/어디에(When) → **예시 시나리오** → 어떻게 준비·운영(How) → 다른 주제와 관계 → 지표·고도화.
> 3. **★ How 바로 앞에 "예시 시나리오"**: "이걸 적용하면 이렇게 좋아진다(효과) + 흐름 미리보기"를 계열사 맥락으로 먼저 그려준다. 그다음 How에서 상세화. (이게 worked example 역할 → 맨 끝 별도 계열사 예시는 생략, 중복 방지)
> 4. **세부 H3 목차는 둔다.** 번호 매긴 소목차(3.1·3.2…)는 있어야 한다 — 단 **내용은 쉽고 얕게**, 어려운 전문 내용·약어는 빼고.

## 기본 흐름 (필수 아님 · 주제별 가감)

1. 개요 · 2. 왜 필요한가 · 3. 무엇을 갖추나 · 4. 언제/어디에 하나 · **5. 예시 시나리오(효과+흐름 미리보기)** · 6. 어떻게 준비·운영하나(구축 포함, 상세) · 7. 다른 주제와의 관계 · 8. 성과 지표·고도화

## 진행 상태

| 단계 | 상태 |
|------|------|
| 데모 4개 (A-2·B-2·B-3·E-2) — 예시 시나리오(How 앞) 추가 | ⏳ 검토 중 |
| 나머지 16개 | ⬜ 확정 후 진행 |

## A-2 메타데이터  *(기본적으로 다 필요한 주제)*
> 한 줄: 데이터의 "설명서" — 이 항목이 무슨 뜻이고, 단위·갱신주기·담당자가 누구인지 적어두는 것.
> 핵심 계열사: 두산에너빌리티

```
1. 개요
   1.1 메타데이터란 (데이터의 설명서)
   1.2 적용 범위와 체계 내 위치 (소재는 카탈로그, 용어 뜻은 Glossary와 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (항목 이름·단위만으론 뜻을 몰라 AI가 잘못 해석)
   2.2 기대 효과 (재사용이 쉬워지고 오해석이 줄어듦)
3. 무엇을 갖추나
   3.1 표준 항목 (타입·단위·생성 시스템·갱신 주기·담당자)
   3.2 항목 설명서 (약어·코드 풀이 — 예: DEF_CD = 결함 유형 코드)
   3.3 단위·기준일·집계 기준 (숫자를 잘못 비교하지 않도록)
4. 어디부터 하나 (우선순위)   ← 메타데이터는 다 필요하므로 "순서"
   4.1 정비 우선 대상 (설명이 비었거나 단위가 뒤죽박죽인 데이터부터)
   4.2 자동으로 모으고 사람은 검수만 (★)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (AI가 'DEF_CD'를 못 알아봄 → 설명서를 달면 '결함 유형 코드'로 바로 해석)
   5.2 흐름 미리보기 (자동 수집 → 빈 설명·단위 채우기 → 담당자 검수 → AI가 바로 활용)
6. 어떻게 준비·운영하나 (상세)
   6.1 수집·관리 도구 검토
   6.2 구축 절차 (자동 수집 연결 → 설명·단위 보완 → 검증)
   6.3 운영 (항목이 바뀌면 갱신·승인) / 역할·책임
7. 다른 주제와의 관계
   7.1 카탈로그(소재)·Glossary(용어)·품질과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (작성 완성률·자동 수집률·설명 충실도)
   8.2 고도화 (AI가 설명 초안 작성 → 사람 검수)
```

## B-2 데이터 해설/주석  *(★ 라벨/주석 — 사용자가 든 예)*
> 한 줄: AI가 학습할 수 있게 데이터에 라벨·해석을 붙이는 것 (예: 결함 이미지에 결함 유형 표시).
> 핵심 계열사: 두산전자

```
1. 개요
   1.1 데이터 해설/주석이란 (학습용으로 라벨·해석을 붙이는 것)
   1.2 적용 범위와 체계 내 위치 (평가용 정답은 평가데이터(E-3), 개념 관계는 온톨로지와 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (라벨이 없거나 기준이 제각각이면 AI 학습 품질이 떨어짐)
   2.2 기대 효과 (분류 자동화·판단 기준 일관·재작업 감소)
3. 무엇을 갖추나
   3.1 분류 체계(taxonomy)·라벨 정의 (결함·원인·조치 유형)
   3.2 주석 기준·사례집 (헷갈리는 경우의 판단 기준)
   3.3 라벨 이력(버전) 관리 (누가·언제·왜 바꿨나)
4. 어디부터 하나 (대상 선정)
   4.1 주석 대상 (결함 이미지·고객 클레임·원인 분석 등)
   4.2 우선순위·규모 (자주 쓰고 영향 큰 것부터)
5. 예시 시나리오 (한눈에)   ← How 앞 ★
   5.1 적용 전/후 (결함 이미지를 사람이 일일이 분류 → 라벨을 달면 AI가 자동 분류, 검사 시간↓·기준 일관↑)
   5.2 흐름 미리보기 (분류 체계 정하기 → AI가 1차 라벨 → 사람은 헷갈리는 것만 검수 → 학습 데이터 완성)
6. 어떻게 준비·운영하나 (상세)
   6.1 분류 체계(taxonomy) 만드는 기준 (유형 간 경계·중복 없이)
   6.2 라벨링 시키는 법 (AI가 1차 라벨 + 가이드라인 + 사람 검수 워크플로)
   6.3 운영 (라벨 변경·버전 관리) / 역할·책임
7. 다른 주제와의 관계
   7.1 평가 정답(E-3)·개념 관계(B-3)·용어 뜻(A-3)과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (라벨 정확도·검수 합격률·작업 생산성)
   8.2 고도화 (AI 1차 라벨 비중↑ → 사람은 어려운 것만 검수)
```

## B-3 온톨로지  *(골라서 하는 주제 — "언제 하나"가 중요)*
> 한 줄: 개념들 사이의 "관계 지도" — 결함 ↔ 원인 ↔ 조치처럼 무엇이 무엇과 이어지는지 정리.
> 핵심 계열사: 두산에너빌리티

```
1. 개요
   1.1 온톨로지란 (개념 사이의 관계 지도)
   1.2 적용 범위와 체계 내 위치 (단어 뜻이면 Glossary, 학습용 분류면 주석과 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (단순 검색으론 "왜 생겼고 어떻게 조치하나"를 못 찾음)
   2.2 기대 효과 (원인 분석·유사 사례·조치 추천이 가능)
3. 무엇을 갖추나
   3.1 개념 목록 (제품·공정·결함·원인·조치)
   3.2 개념 사이 관계 (발생한다·원인이 된다·조치한다)
   3.3 개념과 실제 데이터·문서 연결
4. ★ 언제 온톨로지를 하나   ← 전부 다 하는 게 아님
   4.1 적용 판단 (원인 분석·추천처럼 "관계"가 중요할 때만, 단어 통일이면 Glossary로 충분)
   4.2 작게 시작 (한 업무부터, 전사 거대 모델 금지) / 우선 영역 고르기
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 ("균열" 검색하면 문서만 나옴 → 관계도가 있으면 균열의 원인·조치까지 추천)
   5.2 흐름 미리보기 (개념·관계 정리 → 문서·데이터 연결 → AI가 원인→조치 추천)
6. 어떻게 준비·운영하나 (상세)
   6.1 관계 저장·검색 도구(그래프 DB) 검토
   6.2 구축 절차 (개념·관계 정리 → 데이터·문서 연결 → 검증)
   6.3 운영 (개념·관계가 바뀌면 전문가 검토 후 갱신) / 현업 전문가 참여
7. 다른 주제와의 관계
   7.1 Glossary(단어 뜻)·주석(분류 라벨)·AI 검색/에이전트와의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (관계 채움 정도·추천 적중률)
   8.2 고도화 (문서에서 관계를 AI가 초안 추출 → 전문가 검수)
```

## E-2 합성데이터  *(골라서 하는 주제 + 만드는 방식이 핵심)*
> 한 줄: 실제 데이터가 부족할 때 인공으로 만들어 채우는 데이터.
> 핵심 계열사: 두산전자

```
1. 개요
   1.1 합성데이터란 (인공으로 만든 데이터 — 원본 대체가 아님)
   1.2 적용 범위와 체계 내 위치
2. 왜 필요한가
   2.1 현업 Pain Point (불량 사례가 너무 적거나, 보안 때문에 원본을 못 씀)
   2.2 기대 효과 (부족한 사례를 채워 학습·검증 가능)
3. 무엇을 갖추나
   3.1 무엇을 합성할지 (불량·장애·예외 사례)
   3.2 만드는 방식 (통계·시뮬레이션·생성모델)
   3.3 검증 항목 + "합성" 표시
4. ★ 언제 합성하나   ← 실데이터로 충분하면 안 함
   4.1 적용 판단 (데이터가 부족·희소하거나 보안으로 원본을 못 쓸 때만)
   4.2 비교 기준이 될 실데이터 먼저 확보
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (특정 불량 샘플이 10건뿐이라 학습 불가 → 합성으로 보강하면 검사 AI 정확도↑)
   5.2 흐름 미리보기 (만들 대상 정하기 → 방식 골라 생성 → 검증 → "합성" 표시 후 학습)
6. 어떻게 만들고 운영하나 (상세)
   6.1 만드는 방식·도구 비교·선정 (시범 적용)
   6.2 만드는 절차 (생성 → 검증 → "합성" 표시·등록)
   6.3 운영 (재생성, 위험 점검: 편향·현실 왜곡·재식별)
7. 다른 주제와의 관계
   7.1 평가셋(E-3)·비식별(F-4)·사용 판정(C-2)과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (실데이터와 닮은 정도·모델 성능 개선)
   8.2 고도화 (검증 자동화)
```
