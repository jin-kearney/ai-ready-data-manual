# AI-Ready Data 매뉴얼 — 상세 목차 (현업 눈높이·간결판) v3

> **작성 원칙**
> 1. **A-1 12-섹션 템플릿을 base로** 따르되, 주제별로 안 맞는 건 빼고 필요한 건 더한다.
> 2. **현업이 읽고 바로 이해할 수준**으로 쉽게. 전문용어·약어는 빼거나 한 줄로 풀이.
> 3. **분량 최소** — 섹션당 핵심 1~3개만. "많이 적기"가 아니라 "필요한 것만".
> 4. **Why·What·When·How 4축**은 반드시 담는다.

## 4축 = 어떤 섹션인가 (전 주제 공통)

| 축 | 질문 | 섹션 |
|----|------|------|
| **Why** 왜 필요 | 안 하면 뭐가 문제인가 | 2장 |
| **What** 무엇 | 이게 뭐고 뭘로 구성되나 | 1·3장 |
| **When** 언제 | 어디에·언제 해야 하나 | 5장 |
| **How** 어떻게 | 어떻게 준비·운영하나 | 6·7·8·9·11·12장 |

> **When(5장) 규칙:** 온톨로지·합성데이터처럼 **골라서 하는 주제**는 5장에 "언제 해야 하나(적용 판단)"를 먼저 둔다. 데이터 카탈로그·메타데이터처럼 **기본적으로 다 해야 하는 주제**는 5장을 "어디부터(우선순위)"로 쓴다.

## A-2 메타데이터  *(기본적으로 다 필요한 주제)*
> 한 줄: 데이터의 "설명서" — 이 항목이 무슨 뜻이고, 단위·갱신주기·담당자가 누구인지 적어두는 것.
> 핵심 계열사: 두산에너빌리티  ·  A-1에서 뺀 것: 카탈로그 조회 방식 / 넣은 것: 항목 설명서·단위 기준

```
1. 개요
   1.1 메타데이터란 (데이터의 설명서)
   1.2 목적과 적용 범위 (항목·테이블 단위 / 소재는 카탈로그(A-1), 용어 뜻은 Glossary(A-3))
   1.3 대상 조직과 체계 내 역할
2. 왜 필요한가
   2.1 현업 Pain Point: 항목 이름·단위만으로는 뜻을 몰라 AI가 잘못 해석
   2.2 기대 효과: 재사용 쉬워지고 오해석 줄어듦
3. 무엇을 담나 (구성)
   3.1 표준 항목 (타입·단위·생성 시스템·갱신 주기·담당자)
   3.2 항목 설명서 (약어·코드 풀이 — 예: DEF_CD = 결함 유형 코드)
   3.3 단위·기준일·집계 기준 (숫자를 잘못 비교하지 않도록)
4. 역할·책임 (누가 담당하고 누가 승인하나)
5. 어디부터 정비하나  ← 메타데이터는 다 필요하므로 "우선순위"
   5.1 설명이 비었거나 단위가 뒤죽박죽인 데이터부터
   5.2 자동으로 모으고 사람은 검수만 (★ 사람이 일일이 입력하지 않는다)
6. 도구(솔루션) 검토 (항목 자동 수집·관리 도구)
7. 계열사 예시 — 두산에너빌리티: 현장 약어·설비 단위 정리
8. 구축 절차 (자동 수집 연결 → 설명 보완 → 검증)
9. 운영 (항목이 바뀌면 갱신·승인)
10. 다른 주제와의 관계 (A-1 소재 / A-3 용어 / 품질·거버넌스)
11. 성과 지표 (작성 완성률·자동 수집률·설명 충실도)
12. 고도화 (AI가 설명 초안 자동 작성 → 사람 검수)
```

## B-3 온톨로지  *(골라서 하는 주제 — "언제 하나"가 중요)*
> 한 줄: 개념들 사이의 "관계 지도" — 결함 ↔ 원인 ↔ 조치처럼 무엇이 무엇과 이어지는지 정리.
> 핵심 계열사: 두산에너빌리티  ·  A-1에서 뺀 것: 정형 데이터 중요도·취합 방식 / 넣은 것: 적용 판단(언제 하나)

```
1. 개요
   1.1 온톨로지란 (개념 사이의 관계 지도)
   1.2 목적과 범위 (단어 뜻만이면 Glossary(A-3), 학습용 분류면 주석(B-2) — 그건 온톨로지 아님)
   1.3 대상 조직과 체계 내 역할
2. 왜 필요한가
   2.1 현업 Pain Point: 단순 검색으론 "왜 생겼고 어떻게 조치하나"를 못 찾음
   2.2 기대 효과: 원인 분석·유사 사례·조치 추천이 가능
3. 무엇을 담나 (구성)
   3.1 개념 목록 (제품·공정·결함·원인·조치)
   3.2 개념 사이 관계 (발생한다·검출된다·원인이 된다·조치한다)
   3.3 개념과 실제 데이터·문서 연결
4. 역할·책임 (현업 전문가 참여가 핵심)
5. ★ 언제 온톨로지를 하나  ← 전부 다 하는 게 아님
   5.1 적용 판단: 원인 분석·추천처럼 "관계"가 중요한 업무일 때만
   5.2 작게 시작 (한 업무부터, 전사 거대 모델은 금지)
   5.3 우선 만들 영역 고르기 (지식이 많고 AI 활용 가치 큰 곳)
6. 도구(솔루션) 검토 (관계를 저장·검색하는 그래프 DB 등)
7. 계열사 예시 — 두산에너빌리티: 품질 관계도(결함→원인→조치)
8. 구축 절차 (개념·관계 정리 → 데이터·문서 연결 → 검증)
9. 운영 (개념·관계가 바뀌면 전문가 검토 후 갱신)
10. 다른 주제와의 관계 (A-3 단어 뜻 / B-2 분류 라벨 / AI 검색·에이전트)
11. 성과 지표 (관계 채움 정도·추천 적중률)
12. 고도화 (문서에서 관계를 AI가 초안 추출 → 전문가 검수)
```

## E-2 합성데이터  *(골라서 하는 주제 + 만드는 방식이 핵심)*
> 한 줄: 실제 데이터가 부족할 때 인공으로 만들어 채우는 데이터.
> 핵심 계열사: 두산전자  ·  A-1에서 바꾼 것: 6장 "솔루션 선정"→"만드는 방식 선정", 8장 "구축"→"만드는 절차"

```
1. 개요
   1.1 합성데이터란 (실데이터 부족 시 인공으로 만든 데이터 — 원본 대체가 아님)
   1.2 목적과 범위
   1.3 대상 조직과 체계 내 역할
2. 왜 필요한가
   2.1 현업 Pain Point: 불량 사례가 너무 적거나, 보안 때문에 원본을 못 씀
   2.2 기대 효과: 부족한 사례를 채워 학습·검증 가능
3. 무엇을 담나 (구성)
   3.1 무엇을 합성할지 (불량·장애·예외 사례 등)
   3.2 만드는 방식 (통계·시뮬레이션·생성모델 등)
   3.3 잘 만들었는지 확인 항목 + "합성" 표시
4. 역할·책임
5. ★ 언제 합성하나  ← 실데이터로 충분하면 안 함
   5.1 적용 판단: 데이터가 부족·희소하거나 보안으로 원본을 못 쓸 때만
   5.2 비교 기준이 될 실데이터 먼저 확보
6. 만드는 방식·도구 선정 (방식별 장단점 비교 → 시범 적용)
7. 계열사 예시 — 두산전자: 희소 불량 사례 합성
8. 만드는 절차 (생성 → 검증 → "합성" 표시·등록)
9. 운영 (재생성, 위험 점검: 편향·현실 왜곡·재식별)
10. 다른 주제와의 관계 (평가셋 E-3 / 비식별 F-4 / 사용 판정 C-2)
11. 성과 지표 (실데이터와 닮은 정도·모델 성능 개선)
12. 고도화 (검증 자동화)
```
