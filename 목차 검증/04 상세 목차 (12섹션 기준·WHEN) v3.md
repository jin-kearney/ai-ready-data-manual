# AI-Ready Data 매뉴얼 — 상세 목차 (논리 순서·현업 눈높이) v3

> **작성 원칙**
> 1. **12개를 다 채우지 않는다.** A-1 12-섹션은 참고용 풀일 뿐. 주제에 필요한 것만, 보통 7~8개.
> 2. **순서는 논리 흐름으로**: 왜(Why) → 뭐고 뭘로 구성(What) → 언제/어디에(When) → **예시 시나리오** → 어떻게 준비·운영(How) → 다른 주제와 관계 → 지표·고도화.
> 3. **★ How 바로 앞에 "예시 시나리오"**: "이걸 적용하면 이렇게 좋아진다(효과) + 흐름 미리보기"를 계열사 맥락으로 먼저 그려준다. 그다음 How에서 상세화. (이게 worked example 역할 → 맨 끝 별도 계열사 예시는 생략, 중복 방지)
> 4. **세부 H3 목차는 둔다.** 번호 매긴 소목차(3.1·3.2…)는 있어야 한다 — 단 **내용은 쉽고 얕게**, 어려운 전문 내용·약어는 빼고.

## 기본 흐름 (필수 아님 · 주제별 가감)

1. 개요 · 2. 왜 필요한가 · 3. 무엇을 갖추나 · 4. 언제/어디에 하나 · **5. 예시 시나리오(효과+흐름 미리보기)** · 6. 어떻게 준비·운영하나(구축 포함, 상세) · 7. 다른 주제와의 관계 · 8. 성과 지표·고도화

## 진행 상태

| 단계 | 상태 |
|------|------|
| 데모 4개 (A-2·B-2·B-3·E-2) — 예시 시나리오(How 앞) 추가 | ⏳ 검토 중 |
| 나머지 16개 | ⬜ 확정 후 진행 |

## A-2 메타데이터  *(기본적으로 다 필요한 주제)*
> 한 줄: 데이터의 "설명서" — 이 항목이 무슨 뜻이고, 단위·갱신주기·담당자가 누구인지 적어두는 것.
> 핵심 계열사: 두산에너빌리티

```
1. 개요
   1.1 메타데이터란 (데이터의 설명서)
   1.2 적용 범위와 체계 내 위치 (소재는 카탈로그, 용어 뜻은 Glossary와 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (항목 이름·단위만으론 뜻을 몰라 AI가 잘못 해석)
   2.2 기대 효과 (재사용이 쉬워지고 오해석이 줄어듦)
3. 무엇을 갖추나
   3.1 표준 항목 (타입·단위·생성 시스템·갱신 주기·담당자)
   3.2 항목 설명서 (약어·코드 풀이 — 예: DEF_CD = 결함 유형 코드)
   3.3 단위·기준일·집계 기준 (숫자를 잘못 비교하지 않도록)
4. 어디부터 하나 (우선순위)   ← 메타데이터는 다 필요하므로 "순서"
   4.1 정비 우선 대상 (설명이 비었거나 단위가 뒤죽박죽인 데이터부터)
   4.2 자동으로 모으고 사람은 검수만 (★)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (AI가 'DEF_CD'를 못 알아봄 → 설명서를 달면 '결함 유형 코드'로 바로 해석)
   5.2 흐름 미리보기 (자동 수집 → 빈 설명·단위 채우기 → 담당자 검수 → AI가 바로 활용)
6. 어떻게 준비·운영하나 (상세)
   6.1 수집·관리 도구 검토
   6.2 구축 절차 (자동 수집 연결 → 설명·단위 보완 → 검증)
   6.3 운영 (항목이 바뀌면 갱신·승인) / 역할·책임
7. 다른 주제와의 관계
   7.1 카탈로그(소재)·Glossary(용어)·품질과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (작성 완성률·자동 수집률·설명 충실도)
   8.2 고도화 (AI가 설명 초안 작성 → 사람 검수)
```

## B-2 데이터 해설/주석  *(★ 라벨/주석 — 사용자가 든 예)*
> 한 줄: AI가 학습할 수 있게 데이터에 라벨·해석을 붙이는 것 (예: 결함 이미지에 결함 유형 표시).
> 핵심 계열사: 두산전자

```
1. 개요
   1.1 데이터 해설/주석이란 (학습용으로 라벨·해석을 붙이는 것)
   1.2 적용 범위와 체계 내 위치 (평가용 정답은 평가데이터(E-3), 개념 관계는 온톨로지와 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (라벨이 없거나 기준이 제각각이면 AI 학습 품질이 떨어짐)
   2.2 기대 효과 (분류 자동화·판단 기준 일관·재작업 감소)
3. 무엇을 갖추나
   3.1 분류 체계(taxonomy)·라벨 정의 (결함·원인·조치 유형)
   3.2 주석 기준·사례집 (헷갈리는 경우의 판단 기준)
   3.3 라벨 이력(버전) 관리 (누가·언제·왜 바꿨나)
4. 어디부터 하나 (대상 선정)
   4.1 주석 대상 (결함 이미지·고객 클레임·원인 분석 등)
   4.2 우선순위·규모 (자주 쓰고 영향 큰 것부터)
5. 예시 시나리오 (한눈에)   ← How 앞 ★
   5.1 적용 전/후 (결함 이미지를 사람이 일일이 분류 → 라벨을 달면 AI가 자동 분류, 검사 시간↓·기준 일관↑)
   5.2 흐름 미리보기 (분류 체계 정하기 → AI가 1차 라벨 → 사람은 헷갈리는 것만 검수 → 학습 데이터 완성)
6. 어떻게 준비·운영하나 (상세)
   6.1 분류 체계(taxonomy) 만드는 기준 (유형 간 경계·중복 없이)
   6.2 라벨링 시키는 법 (AI가 1차 라벨 + 가이드라인 + 사람 검수 워크플로)
   6.3 운영 (라벨 변경·버전 관리) / 역할·책임
7. 다른 주제와의 관계
   7.1 평가 정답(E-3)·개념 관계(B-3)·용어 뜻(A-3)과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (라벨 정확도·검수 합격률·작업 생산성)
   8.2 고도화 (AI 1차 라벨 비중↑ → 사람은 어려운 것만 검수)
```

## B-3 온톨로지  *(골라서 하는 주제 — "언제 하나"가 중요)*
> 한 줄: 개념들 사이의 "관계 지도" — 결함 ↔ 원인 ↔ 조치처럼 무엇이 무엇과 이어지는지 정리.
> 핵심 계열사: 두산에너빌리티

```
1. 개요
   1.1 온톨로지란 (개념 사이의 관계 지도)
   1.2 적용 범위와 체계 내 위치 (단어 뜻이면 Glossary, 학습용 분류면 주석과 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (단순 검색으론 "왜 생겼고 어떻게 조치하나"를 못 찾음)
   2.2 기대 효과 (원인 분석·유사 사례·조치 추천이 가능)
3. 무엇을 갖추나
   3.1 개념 목록 (제품·공정·결함·원인·조치)
   3.2 개념 사이 관계 (발생한다·원인이 된다·조치한다)
   3.3 개념과 실제 데이터·문서 연결
4. ★ 언제 온톨로지를 하나   ← 전부 다 하는 게 아님
   4.1 적용 판단 (원인 분석·추천처럼 "관계"가 중요할 때만, 단어 통일이면 Glossary로 충분)
   4.2 작게 시작 (한 업무부터, 전사 거대 모델 금지) / 우선 영역 고르기
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 ("균열" 검색하면 문서만 나옴 → 관계도가 있으면 균열의 원인·조치까지 추천)
   5.2 흐름 미리보기 (개념·관계 정리 → 문서·데이터 연결 → AI가 원인→조치 추천)
6. 어떻게 준비·운영하나 (상세)
   6.1 관계 저장·검색 도구(그래프 DB) 검토
   6.2 구축 절차 (개념·관계 정리 → 데이터·문서 연결 → 검증)
   6.3 운영 (개념·관계가 바뀌면 전문가 검토 후 갱신) / 현업 전문가 참여
7. 다른 주제와의 관계
   7.1 Glossary(단어 뜻)·주석(분류 라벨)·AI 검색/에이전트와의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (관계 채움 정도·추천 적중률)
   8.2 고도화 (문서에서 관계를 AI가 초안 추출 → 전문가 검수)
```

## E-2 합성데이터  *(골라서 하는 주제 + 만드는 방식이 핵심)*
> 한 줄: 실제 데이터가 부족할 때 인공으로 만들어 채우는 데이터.
> 핵심 계열사: 두산전자

```
1. 개요
   1.1 합성데이터란 (인공으로 만든 데이터 — 원본 대체가 아님)
   1.2 적용 범위와 체계 내 위치
2. 왜 필요한가
   2.1 현업 Pain Point (불량 사례가 너무 적거나, 보안 때문에 원본을 못 씀)
   2.2 기대 효과 (부족한 사례를 채워 학습·검증 가능)
3. 무엇을 갖추나
   3.1 무엇을 합성할지 (불량·장애·예외 사례)
   3.2 만드는 방식 (통계·시뮬레이션·생성모델)
   3.3 검증 항목 + "합성" 표시
4. ★ 언제 합성하나   ← 실데이터로 충분하면 안 함
   4.1 적용 판단 (데이터가 부족·희소하거나 보안으로 원본을 못 쓸 때만)
   4.2 비교 기준이 될 실데이터 먼저 확보
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (특정 불량 샘플이 10건뿐이라 학습 불가 → 합성으로 보강하면 검사 AI 정확도↑)
   5.2 흐름 미리보기 (만들 대상 정하기 → 방식 골라 생성 → 검증 → "합성" 표시 후 학습)
6. 어떻게 만들고 운영하나 (상세)
   6.1 만드는 방식·도구 비교·선정 (시범 적용)
   6.2 만드는 절차 (생성 → 검증 → "합성" 표시·등록)
   6.3 운영 (재생성, 위험 점검: 편향·현실 왜곡·재식별)
7. 다른 주제와의 관계
   7.1 평가셋(E-3)·비식별(F-4)·사용 판정(C-2)과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (실데이터와 닮은 정도·모델 성능 개선)
   8.2 고도화 (검증 자동화)
```

## F-1 데이터 운영관리 (DataOps)  *(기본적으로 다 필요한 주제 — "어느 파이프라인부터/얼마나" 차등)*
> 한 줄: 데이터가 흐르는 길(수집→정제→변환→제공)이 멈추지 않게 실행·복구·변경을 관리하는 것 — 장애 나면 되살리고, 바꿀 때 사고 안 나게.
> 핵심 계열사: 두산에너빌리티

```
1. 개요
   1.1 데이터 운영관리(DataOps)란 (데이터 흐르는 길을 안 멈추게 운영)
   1.2 적용 범위와 체계 내 위치 (이상 감지는 Observability(C-1), 추적은 Lineage(C-3), 품질 판정은 품질관리(C-2)와 구분 — 여기는 '실행·복구·변경')
2. 왜 필요한가
   2.1 현업 Pain Point (야간 수집이 조용히 실패 → 다음 날 AI가 옛 데이터로 잘못 답함)
   2.2 기대 효과 (멈춰도 빨리 되살리고, 바꿔도 사고가 안 남 → AI 결과가 끊기지 않음)
3. 무엇을 갖추나
   3.1 운영 표준 (수집·정제·변환·적재·제공의 실행 주기와 배포 절차)
   3.2 장애 대응 절차(Runbook) (실패하면 다시 돌리기·되돌리기(롤백)·담당자 알림·복구 확인)
   3.3 변경 관리 (양식·연결방식·제공방식이 바뀔 때 미리 알리고 영향 점검)
4. 어디부터/얼마나 하나 (운영 대상 파이프라인 우선순위)   ← 다 필요하므로 "차등"
   4.1 운영 대상 고르기 (AI 서비스·RAG·리포팅에 쓰는 핵심 흐름)
   4.2 중요도 차등 (멈추면 타격 큰 흐름은 촘촘히, 그 외는 가볍게)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (야간 수집 실패를 아침에야 발견 → 운영 체계가 있으면 즉시 재처리·알림으로 그날 안에 정상화)
   5.2 흐름 미리보기 (핵심 흐름 정하기 → 실행·배포 표준화 → 실패 시 자동 재처리·알림 → 복구 확인 → 변경은 사전 공지)
6. 어떻게 준비·운영하나 (상세)
   6.1 운영·자동화 도구 검토 (스케줄·배포·복구 자동화)
   6.2 구축 절차 (핵심 파이프라인 등록 → 실행/배포 표준 → 장애 대응 절차 마련 → 변경 절차 연결)
   6.3 운영 (정기 점검·재처리·변경 승인) / 역할·책임
7. 다른 주제와의 관계
   7.1 이상 감지(C-1)·추적(C-3)·품질 판정(C-2)과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (파이프라인 성공률·평균 복구 시간·약속 시간(SLA) 준수율·재처리 건수·제공 지연)
   8.2 고도화 (이상 감지(C-1)와 연동해 자동 복구 범위 확대)
```

## F-2 데이터 생애주기 관리  *(기본적으로 다 필요한 주제 — "무엇을 언제까지/언제 버릴지")*
> 한 줄: 데이터를 언제까지 보관하고 언제 버릴지 정하는 것 — 단, 재사용 가치 있는 과정 데이터(실패한 실험 등)는 일부러 남긴다.
> 핵심 계열사: 두산퓨얼셀

```
1. 개요
   1.1 데이터 생애주기 관리란 (생성→사용→보관→정리(아카이빙)→폐기의 시간축 관리)
   1.2 적용 범위와 체계 내 위치 (무엇이 어떻게 바뀌었나=변환 이력(C-3), 민감정보 보호=보안(F-4)과 구분 — 여기는 '얼마나 두고 언제 버리나')
2. 왜 필요한가
   2.1 현업 Pain Point (다 쌓아두면 비용·위험↑, 막 버리면 나중에 AI가 학습할 데이터가 사라짐)
   2.2 기대 효과 (보관 비용·보안 위험은 줄이고, 쓸모 있는 데이터는 남겨 AI 재사용)
3. 무엇을 갖추나
   3.1 생애주기 단계 정의 (생성·사용·보관·아카이빙·폐기·재사용 + 단계별 책임자)
   3.2 보존 기간 기준 (법적 의무·업무 가치·AI 재사용 가치·비용으로 유형별 기간 결정)
   3.3 ★ 과정 데이터 보존 기준 (실패한 실험·중간 결과·검토 의견처럼 재사용 가치 있는 것은 일부러 보존)
4. 어디부터 하나 (보존/폐기 대상 우선순위)   ← 다 필요하므로 "대상 선정"
   4.1 우선 정리 대상 (보존 기간 끝난 것·안 쓰는 것부터 폐기/아카이빙)
   4.2 우선 보존 대상 (AI 학습 가치·감사 필요성 큰 것은 먼저 챙겨 보존)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (실패 실험 기록을 그때그때 삭제 → 보존 기준이 있으면 그 데이터로 불량 원인 AI를 학습 가능)
   5.2 흐름 미리보기 (단계·기간 정하기 → 폐기/아카이빙 자동화 → 과정 데이터는 따로 보존 → 저장소 계층으로 비용 관리)
6. 어떻게 준비·운영하나 (상세)
   6.1 보존·폐기 자동화 도구 검토 (저장소 계층화(Hot/Warm/Cold) 포함)
   6.2 구축 절차 (단계 정의 → 유형별 보존 기간 → 폐기/아카이빙 절차 → 과정 데이터 보존 규칙)
   6.3 운영 (폐기 승인·복원 절차·보존 예외 관리) / 역할·책임
7. 다른 주제와의 관계
   7.1 변환 이력(C-3)·보안(F-4)·품질(C-2)과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (보존 정책 적용률·미사용 데이터 비중·저장 비용 절감·과정 데이터 재사용 건수)
   8.2 고도화 (사용 빈도를 보고 보관 등급을 자동 추천 → 사람 승인)
```

## F-3 데이터 디지털화  *(골라서 하는 주제 — "무엇부터 디지털로 바꾸나"가 핵심)*
> 한 줄: 종이·수기·사진·머릿속 노하우(암묵지)를 디지털로 바꾸고, 앞으로는 처음부터 디지털로 입력(Digital-first)하게 만드는 것.
> 핵심 계열사: 두산에너빌리티

```
1. 개요
   1.1 데이터 디지털화란 (아날로그 자산을 AI가 쓸 수 있는 디지털로 전환)
   1.2 적용 범위와 체계 내 위치 (이미 디지털인 문서를 AI용으로 다듬는 건 전처리(B-1), 쓸 수 있나 판정은 품질(C-2)과 구분 — 여기는 '아날로그→디지털 전환')
2. 왜 필요한가
   2.1 현업 Pain Point (수기 검사표·현장 노하우가 종이·머릿속에만 있어 AI가 아예 못 봄)
   2.2 기대 효과 (사라질 뻔한 자산을 데이터로 살려 AI 학습·검색에 활용)
3. 무엇을 갖추나
   3.1 아날로그 자산 목록(Inventory) (종이·수기 검사표·사진·현장 메모·작업자 노하우)
   3.2 전환 방식 (스캔·글자 인식(OCR/ICR)·음성→텍스트·모바일/태블릿 입력·인터뷰 구조화)
   3.3 전환 품질 검증 + Digital-first 기준 (원본 대비 검수 / 앞으로는 처음부터 디지털 입력)
4. 무엇부터 하나 (디지털화 우선순위)   ← 다 못 하므로 "고르기"
   4.1 적용 판단 (소실 위험 크거나 활용 가치 큰 것부터, 안 쓸 자료는 굳이 안 함)
   4.2 작게 시작 (한 업무·한 양식부터 → 효과 확인 후 확대)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (수기 검사표가 캐비닛에만 있어 분석 불가 → 디지털화하면 AI가 불량 패턴을 분석)
   5.2 흐름 미리보기 (자산 조사 → 우선순위 정하기 → 방식 골라 전환·검수 → 자산 등록 → 신규는 Digital-first로 입력)
6. 어떻게 준비·운영하나 (상세)
   6.1 전환 방식·도구 비교·선정 (대상 특성에 맞춰 시범 적용)
   6.2 전환 절차 (조사 → 우선순위 → 전환 → 검수 → 자산 등록)
   6.3 운영 (개인 PC·엑셀 산재 데이터를 공식 자산으로 승급·지속 수집 / Digital-first 정착) / 역할·책임
7. 다른 주제와의 관계
   7.1 전처리(B-1)·품질 판정(C-2)·카탈로그 등록(A-1)과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (우선 대상 전환율·글자 인식 정확도·검수 합격률·Digital-first 적용률)
   8.2 고도화 (AI가 글자·표를 1차 추출 → 사람은 헷갈리는 것만 검수)
```

## F-4 AI 데이터 보안  *(기본적으로 다 필요한 주제 — "막는" 게 아니라 "쓸 수 있게 변환")*
> 한 줄: 개인·민감정보를 찾아 가리고(마스킹)·가명으로 바꿔(가명화) AI에 안전하게 쓰게 하는 것 — 차단이 목적이 아니라 활용이 목적.
> 핵심 계열사: 두산밥캣

```
1. 개요
   1.1 AI 데이터 보안이란 (민감정보를 탐지·마스킹·가명화해 안전하게 활용)
   1.2 적용 범위와 체계 내 위치 (누가 볼 수 있나 접근 차단은 품질/권한(C-2), 부족분 인공 생성은 합성(E-2)과 구분 — 여기는 '데이터를 안전하게 변환')
2. 왜 필요한가
   2.1 현업 Pain Point (보고서·로그에 고객명·계약가가 섞여 있어 AI에 그냥 못 씀 → 그렇다고 다 막으면 활용도 안 됨)
   2.2 기대 효과 (식별정보는 가리되 업무 정보는 살려 AI를 안전하게 활용)
3. 무엇을 갖추나
   3.1 보호 대상 분류 (개인정보뿐 아니라 고객사명·가격·설계정보·기술기밀·영업비밀)
   3.2 탐지·처리 방식 (정형·비정형 모두에서 탐지 / 삭제·마스킹·가명화·익명화)
   3.3 ★ 활용성 보존 기준(Utility-Preservation) (식별자는 가리되 제품군·공정·결함유형은 남겨 AI 성능 유지)
4. 어디부터 하나 (보안 처리 대상 우선순위)   ← 다 필요하므로 "대상 선정"
   4.1 우선 처리 대상 (AI 학습·RAG·외부 모델에 들어가는, 민감정보 많은 데이터부터)
   4.2 활용 목적별 차등 (학습·RAG·로그·외부 모델 여부에 따라 처리 강도를 다르게)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (고객명이 들어 있어 클레임 데이터를 AI에 못 씀 → 고객명만 가명화하면 원인 분석 AI에 안전하게 활용)
   5.2 흐름 미리보기 (보호 대상 정하기 → 탐지 → 목적별 처리(마스킹·가명화) → 학습 가능 등급 부여 → 재식별 위험 점검)
6. 어떻게 준비·운영하나 (상세)
   6.1 탐지·마스킹 도구 검토 (정규식·사전·이름찾기(NER)·LLM 문맥 탐지 조합)
   6.2 구축 절차 (보호 대상 정의 → 탐지 → 목적별 처리 정책 → 학습 가능 등급 부여)
   6.3 운영 (AI 입력(Prompt)·실행 로그도 저장 전 마스킹 / 재식별 위험 정기 점검) / 역할·책임
7. 다른 주제와의 관계
   7.1 접근 차단·권한(C-2)·합성(E-2)·평가데이터(E-3)와의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (민감정보 탐지율·마스킹 적용률·학습 가능 등급 분류율·재식별 위험 점검 건수)
   8.2 고도화 (LLM 문맥 탐지로 탐지 자동화↑ → 사람은 고위험만 검수)
```
