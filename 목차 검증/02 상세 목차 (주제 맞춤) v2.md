# AI-Ready Data 매뉴얼 — 상세 목차 (주제 맞춤) v2

> **이 파일은 "강제 골격을 푼" 주제 맞춤 목차입니다.**
> v1(`01 상세 목차 (20개 주제) v1.md`)은 12-섹션 H2 골격을 전 주제에 동일 적용한 버전 — **비교용으로 보존**합니다.
> v2 원칙: 12-섹션은 **참고만**. 각 주제는 "그 주제의 가이드를 쓰는 데 실제로 필요한 목차"로 섹션을 **가감·병합·재배열·신설**합니다. 무거운 주제(A-1·B-3·C-2·D-1·F-4)는 섹션이 많이 남고, 가볍거나 메타·방식 중심 주제(A-3·E-2·E-4)는 과감히 줄입니다.
> **필수 3축 (Why·What·How):** 골격은 자유롭되 모든 주제는 최소한 ① **Why** 왜 필요한가(Pain Point·필요성) ② **What** 그래서 이게 뭔가(정의·구성 체계) ③ **How** 어떻게 준비하는가(대상선정·구축·운영) 를 반드시 담습니다. 단 Why는 장황하게 늘이지 않고 간결히 — 무게는 What·How에. 섹션 제목을 literally "WHY/WHAT/HOW"로 달지 않고 자연스러운 제목 안에 세 축을 분포시킵니다(각 주제 헤더에 매핑 표기).
> **깊이 균일:** 길이(섹션 수)는 주제마다 달라도, 콘텐츠 깊이는 비슷한 밴드(섹션 7~10개, leaf 약 24~32개)로 맞춥니다. 얇은 stub·60항목 비대화 모두 금지.
> 합격 기준은 v1과 동일 — `최종 주제.md`의 Key Question에 전부 답할 것(섹션 위치·이름은 자유). 본문 leaf의 `[KQ#]`/`(KQ#)` 표기는 Key Question 매핑 추적용.

# A. 발견·의미 (Discovery & Meaning)

## A-1 데이터 카탈로그  *(가장 무거운 주제 — 솔루션·구축이 실제로 두꺼움. 과밀 항목은 합쳐 상단 밴드로 정돈)*
> 파일명: `A-1 데이터 카탈로그.md`
> 정의: AI와 사용자가 데이터 자산의 존재·위치·오너·접근 경로를 찾을 수 있도록 등록하는 자산 목록(Catalog) 체계. **"어디에 무엇이 있는가"만** 다루고, 의미 해석은 A-2/A-3.
> 핵심 계열사 예시: 두산전자 (ERP·MES·QMS·LIMS·SharePoint 산재 자산)
> 골격 변경: 추진역할·계열사 13단계를 독립 섹션에서 제거(운영·예시에 흡수), v1 §5(현황조사)+§8(구축)을 압축 통합, 구성체계+필수항목을 What 1섹션으로 병합, 연계범위 11항목→경계 포인터로 축소. 솔루션은 카탈로그의 실제 무게라 살림. → 9개 섹션.
> Why·What·How: Why=1.2 / What=1.1·2장(구성·필수항목) / How=3·4·5·6·7·8장(대상선정·통합수집·솔루션·구축·운영)
> ★ 핵심 원칙: 사람이 모든 데이터를 다 찾아 정리하지 않는다 — 자동 수집으로 수작업을 최소화한다.

```
1. 카탈로그란 / 왜 필요한가
   1.1 정의와 범위 (자산의 소재·오너·접근경로 목록 — 의미는 A-2/A-3, 흐름은 C-3)
   1.2 Pain Point: 데이터가 어디 있는지 몰라 AI 과제마다 자산 찾기부터 반복 → 탐색 시간 소진·중복 수집
   1.3 기대 효과와 체계 내 위치 (AI-ready 진입점 — 모든 주제가 카탈로그에서 자산을 찾는다)
2. 카탈로그 구성 체계 — 무엇을 기록하나  [KQ2]
   2.1 카탈로그 필수 등록 항목 (데이터명·보유 시스템·저장 위치·오너·보유 부서·접근 경로·갱신 주기)
   2.2 항목 구성 기준 (자산을 찾기 위한 최소 정보에 집중 — 의미·품질 속성은 A-2가 채움)
   2.3 데이터 Type·조회 방식 작성 기준 (정형·반정형·비정형, 시스템·문서·DB 구분)
3. 무엇을 등록할 것인가 — 대상 선정  [KQ1]
   3.1 등록 대상 범위 (시스템 데이터·문서·보고서·품질·실험·설비 데이터)
   3.2 우선순위 기준 ★ (AI 활용 가능성·업무 중요도·재사용성 — 한 번에 다 넣지 않는다)
   3.3 등록 제외 / 보안 검토 기준 (저활용·임시·폐기 예정 / 민감 자산 노출 점검 — 마스킹은 F-4)
4. 흩어진 자산을 하나로 — 통합 수집  [KQ3]
   4.1 원천별 수집 방식 (ERP·MES·QMS·LIMS·SharePoint·파일서버·데이터레이크)
   4.2 자동 수집 vs 수동 등록 구분 ★ (자동 연동 가능 자산 vs 수기 등록 자산)
   4.3 미연동·수동 자산 수집 Pipeline 기준
5. 검색·탐색 설계 — 어떻게 찾게 하나  [KQ4]
   5.1 검색 분류 체계 (업무 도메인·데이터 유형·보유 조직·시스템·활용 목적)
   5.2 탐색 화면 구성 기준 (자산을 찾기 위한 분류·탐색만 — 의미 해석 아님)
   5.3 AI·사용자 탐색 연계 (자연어 탐색·서비스 연계 — 표준 용어 변환은 A-3 공급)
6. 솔루션 선정 (카탈로그의 실제 무게 — 살림)
   6.1 솔루션 유형 (전용 카탈로그·메타 통합형·오픈소스) 및 적용 범위
   6.2 후보군 검토 + 기능 비교 기준 (자동 수집 커버리지·연동·검색·권한·연계)
   6.3 평가·선정·PoC (실제 연동 환경에서 자동 수집·검색 정확도 검증)
7. 구축 (압축)
   7.1 취합 데이터 검토 → 정합성 보완 요청
   7.2 To-Be 아키텍처 설계 (솔루션 기반 + Legacy DB 연동)
   7.3 연동·수동 업로드 Pipeline 개발, 솔루션 초기 환경 구성
   7.4 초기 적재 및 검증
8. 운영 — 최신 상태 유지  [KQ5]
   8.1 변경 트리거 (신규 생성·폐기·위치 변경·오너 변경 발생 시 갱신)
   8.2 갱신 프로세스 (자동 감지 → 오너 확인 → 정기 점검) ★
   8.3 변경 요청·승인·반영, 접근 권한·보안 관리
   8.4 미등록 데이터 점검 / 전처리·분석·질의 서비스 연계
9. 경계·KPI·고도화
   9.1 인접 주제 경계 (A-1=소재 / A-2=속성 / A-3=용어 / C-3=흐름·이력 / F-2=생애주기 / E-1=소비)
   9.2 KPI (카탈로그 활용도·신규 AI 과제 활용 수·탐색 Lead-time 단축·재사용/전처리 비용 절감)
   9.3 고도화 (수기 등록 → 자동 수집 → AI 메타 초안 생성 → Semantic Layer 자연어 탐색 → Agent 연계)
```

## A-2 메타데이터  *(자산 단위 속성 — 자동 수집이 핵심 원칙)*
> 파일명: `A-2 메타데이터.md`
> 정의: 데이터 자산의 구조·형식·단위·생성 기준·갱신 주기 등 기술·운영 속성(Metadata)을 설명해 AI가 자산을 정확히 해석하게 하는 관리 정보 체계. **테이블·필드 단위 속성만** 다룬다.
> 핵심 계열사 예시: 두산에너빌리티 (다수 레거시 DB·MES, 설비/공정 수치 데이터)
> 골격 변경: 추진역할 제거, 계열사 13단계→핵심 흐름 1섹션, 구성체계를 What 1섹션으로 통합하되 필드 설명서·단위/기준 정의는 차별 포인트라 leaf로 살림, 구축 압축, 연계범위→경계 포인터. → 8개 섹션.
> Why·What·How: Why=1.2 / What=1.1·2장(표준 항목·스키마·필드설명·단위) / How=3·4·5·6장(정비대상·솔루션·구축·운영)
> ★ 핵심 원칙: 메타데이터는 사람이 일일이 입력하지 않는다 — 자동 수집 후 오너 검토가 원칙.

```
1. 메타데이터란 / 왜 필요한가
   1.1 정의와 범위 (테이블·필드·파일 단위 속성 — A-1=소재 / A-2=속성 / A-3=표준 용어)
   1.2 Pain Point: 구조 미상 데이터, 단위·기준일·집계 불일치로 AI 오해석, 약어·코드 필드 해석 불가
   1.3 기대 효과와 체계 내 위치 (A-2 속성이 A-1 등록 항목을 채우고, A-3·B-3·전처리에 공급)
2. 메타데이터 구성 체계 — 무엇을 기술하나  [KQ1·KQ3·KQ4]
   2.1 AI-ready 표준 항목 (타입·단위·포맷·허용값·생성 시스템·갱신 주기·오너) [KQ1]
   2.2 메타데이터 유형 (기술 Technical·운영 Operational·비즈니스 Business)
   2.3 필드 설명서 / 컬럼 Dictionary ★ (약어·코드·현장식 명칭에 자연어 설명 — 예: DEF_CD="품질 결함 유형 식별 코드") [KQ3] (★ 필드 설명까지가 A-2, 표준 용어 정의는 A-3)
   2.4 단위·기준일·집계 방식 정의서 ★ (단위·기준 빠진 수치는 AI가 잘못 비교한다) [KQ4]
3. 메타데이터 스키마 — 메타-메타데이터  [KQ2]
   3.1 그룹 표준 스키마 정의 (항목·구조·필수/선택 — 어떤 속성을 어떻게 기술할지의 표준)
   3.2 스키마 진화·버전 관리 ★ (표준 자체가 바뀔 때의 변경·버전 통제) [KQ2]
4. 무엇을 정비할 것인가 — 대상 선정
   4.1 정비 대상 기준 (메타데이터 부재·불완전·노후 자산)
   4.2 우선순위 ★ (의미 모호 필드: 약어·코드·현장 명칭 / 단위·기준 불일치 잦은 수치 테이블 집중)
   4.3 정비 제외·보안 검토 (저활용·폐기 예정 / 민감 필드 메타 노출 점검 — 마스킹은 F-4)
5. 솔루션 선정
   5.1 솔루션 유형 (메타 관리·프로파일링·자동 수집 Scanner·카탈로그 내장형)
   5.2 후보군·기능 비교 (자동 수집 커버리지·스키마 변경 감지·필드 설명·버전·연계)
   5.3 평가·PoC (실제 DB·ETL 환경에서 자동 수집·변경 감지 정확도 검증)
6. 구축 (압축)
   6.1 수집 대상 자산·스키마 검토 → 메타-메타데이터 스키마 정의
   6.2 To-Be 아키텍처 (자동 수집 → 스키마 레지스트리 → 카탈로그 연계)
   6.3 자동 수집·프로파일링 기반 초기 생성 + 필드 설명·단위/기준 정의서 작성(현업 협업)
   6.4 자동 수집·변경 감지 테스트 → 초기 적재·검증(오너 검토)
7. 운영 — 자동 수집·갱신  [KQ5]
   7.1 변경 감지 (스키마·신규 컬럼·타입·코드값 변경 시 갱신 후보 자동 생성) ★ [KQ5]
   7.2 검토·승인 (자동 수집 후 오너 검토 → 반영·버전 관리)
   7.3 신규 자산 메타 생성·등록, 수집 Pipeline 운영
   7.4 카탈로그·전처리·AI 해석 서비스 연계, 민감 필드 메타 노출 통제
8. 경계·예시·KPI·고도화
   8.1 인접 주제 경계 (A-1=소재 / A-2=속성 / A-3=표준 용어 정의 / B-3=개념 노드 매핑 / C-3=변환 이력)
   8.2 계열사 예시 — 두산에너빌리티 (현장 약어·설비 코드 필드 설명 + 단위·기준 표준화 → 스키마 변경 시 갱신 후보 자동 생성 → 오너 승인 → 효과)
   8.3 KPI (메타 완성률·자동 수집률·필드 설명 커버리지·단위/기준 커버리지·갱신 리드타임·AI 오해석 감소)
   8.4 고도화 (수기 정의 → 자동 수집·변경 감지 → AI Description·단위 추론 제안 → 저신뢰·고영향 오너 검수 → 스키마 변경 시 자동 갱신·연계)
```

## A-3 비즈니스 Glossary  *(가벼운 주제 — 도입부·솔루션·계열사 슬림)*
> 파일명: `A-3 비즈니스 Glossary.md`
> 정의: 업무 용어·약어·동의어를 표준 정의로 통일해 데이터 해석·AI 검색의 일관성을 확보하는 용어 사전.
> 핵심 계열사 예시: 두산밥캣 (글로벌 현장 용어 상이)
> 골격 변경: 추진역할·솔루션선정·구축을 독립 섹션에서 제거(운영·연계에 흡수) → 7개 섹션.
> Why·What·How: Why=1.2 / What=1.1·2장 / How=3·4·5장

```
1. Glossary란 무엇이고 왜 필요한가
   1.1 정의와 범위 (단어 단위 표준 정의 — 관계는 B-3, 필드 속성은 A-2)
   1.2 Pain Point: 부서·계열사·글로벌 거점별 용어 불일치 → AI 검색/해석 실패
   1.3 기대 효과와 체계 내 위치 (의미 스택 최상위 → A-2·B-3·RAG에 공급)
2. 표준 용어 구성 체계
   2.1 Glossary 표준 템플릿 (표준명·영문·약어·동의어·금지어·정의·예시·관련 필드)
   2.2 용어 분류 체계 (도메인·업무영역)
   2.3 동의어·약어·다국어 매핑 기준 (★ 현장 용어 → 표준 용어 변환의 핵심)
3. 표준화 대상 선정
   3.1 대상 기준 (AI 검색·해석 영향 큰 용어 우선: 결함명·검사항목·공정명·원인)
   3.2 용어 후보 자동 추출 (코드값·문서에서 — ★ 백지에서 만들지 않는다)
   3.3 충돌·중복 진단 / 우선순위
4. 운영: 등록 → 충돌 조정 → 변경 관리
   4.1 승인 구조 (현업 SME·오너·중앙 조직)
   4.2 용어 충돌 조정 프로세스
   4.3 변경·버전 관리 + 영향도 점검 (관련 데이터·문서·Prompt)
5. AI 활용 연계
   5.1 카탈로그·메타·RAG·Prompt·Agent에 표준 용어 변환 공급
   5.2 도구 선택지 (Business Glossary 전용 vs 카탈로그 내장형)
6. 계열사 예시 — 두산밥캣 글로벌 용어 표준화 (환경 → 동의어/다국어 매핑 → 변환 검색 → 효과)
7. KPI & 고도화 (적용률·매핑 커버리지·검색 일치율 / 수기 → 자동 후보추출 → 실시간 의미변환)
```

# B. 변환·지식화 (Transformation & Knowledge)

## B-1 데이터 전처리  *(자동화 우선 — 변환 파이프라인이 주연)*
> 파일명: `B-1 데이터 전처리.md`
> 정의: 문서·표·이미지 등 이미 디지털로 존재하지만 구조화되지 않은 비정형·반정형 데이터를 AI가 읽을 수 있는 구조화 형태로 변환하는 작업.
> 핵심 계열사 예시: 두산에너빌리티 (기술문서·점검 보고서·시험 성적서 전처리)
> 골격 변경: 추진역할 제거(운영에 흡수), "솔루션 선정"→파서·파이프라인 도구 선정, "구조 추출"과 "Chunking/위치보존"을 What의 두 핵심 섹션으로 분리 승격 → 8개 섹션.
> Why·What·How: Why=1.2 / What=1.1·2·3장(대상유형·추출·구조화) / How=4·5·6장(대상선정·파이프라인·운영)
> ★ 핵심 원칙: 표준 양식부터 파서·파이프라인으로 자동화해 수작업 변환을 최소화한다.

```
1. 전처리란 무엇이고 왜 필요한가
   1.1 정의 (이미 디지털인 자료의 "구조화 변환" — AI 가공의 진입점)
   1.2 Pain Point: PDF·PPT·Excel·표·이미지가 방치돼 AI가 못 읽음, 파서가 양식 변경에 깨짐, 원본 위치 소실 → RAG 정확도 저하
   1.3 기대 효과와 체계 내 위치 (가치사슬 "③ AI 가공" 진입점 → B-2 의미 부여·RAG에 공급)
   1.4 경계 (이미 디지털=B-1 / 아날로그 대량전환=F-3 · 사용 판정=C-2 · 의미 라벨=B-2)
2. 무엇을 전처리하는가 — 대상 유형과 추출 기준  [KQ1·KQ2]
   2.1 전처리 대상 데이터 유형 (PDF·PPT·Excel·이미지·표·반정형 보고서)
   2.2 문서 유형별 추출 기준 (PDF=문단·표·이미지·페이지 / PPT=제목·본문·도형텍스트·표 / Excel=시트·셀·수식·병합셀)
   2.3 추출 요소 정의 (텍스트·수치·표 구조·이미지 내 텍스트·섹션 구조)
3. 어떤 구조로 변환하는가 — 구조화·Chunking·위치 보존  [KQ3]
   3.1 구조화 데이터 포맷 (JSON·테이블·청크·인덱스 — AI 활용 가능 형태)
   3.2 Chunking 기준 (절차/항목 단위 청크, 표는 행 단위 보존)
   3.3 원본 위치 정보 보존 (문서명·페이지·셀좌표 — ★ C-3 Lineage·Citation 출발점)
4. 전처리 대상 선정  [KQ1]
   4.1 대상 자료 Inventory (카탈로그(A-1) 비정형 자산 기반 목록화)
   4.2 포함/제외 기준 (이미 디지털=포함, 종이·스캔 대량전환=F-3 이관)
   4.3 우선순위 산정 (AI 과제 연관성·재참조 빈도·문서 규모)
   4.4 보안·민감정보 사전 식별 (F-4 마스킹 대상 표시)
5. 어떻게 변환하는가 — 파이프라인 구성  [KQ4 일부]
   5.1 To-Be 파이프라인 (추출 → 정규화 → Chunking → Embedding → 적재)
   5.2 파서·도구 선정 (Document Parser·Layout 분석·Table Extraction·Chunking/Embedding — 표·수식 정확도, 양식 변동 대응으로 비교)
   5.3 PoC (대표 문서셋 추출 정확도 측정 / 정형 표·기술도면·혼합 보고서별 파서 조합)
   5.4 ★ 자동화 우선 — 표준 양식부터 파서·파이프라인으로 수작업 최소화
6. 운영: 양식 변경 대응과 산출물 제공  [KQ4·KQ5]
   6.1 양식 변경 안정성 (파서 버전 관리·테스트 문서셋·예외 처리 — KQ4 핵심)
   6.2 변환 결과 검증 (구조 보존·추출 누락 점검)
   6.3 AI 환경 제공 (검색 인덱스·Vector DB·데이터레이크·분석 테이블 적재 — KQ5)
   6.4 변경 요청 → 파이프라인 갱신, 권한·보안(마스킹은 F-4)
7. 계열사 예시 — 두산에너빌리티 기술문서 전처리 (환경 → 대상선정·F-3 제외 → 추출·Chunking·위치보존 → 양식변경 운영 → 효과)
8. KPI & 고도화 (전처리 자동화율·구조 추출 정확도·양식 변경 장애율·원본 위치 보존율·RAG 활용 문서 비중 / 수기 → 규칙 파서 → LLM 구조 추출 → 양식 자가복구 파서 → Agent 자율 전처리)
```

## B-2 데이터 해설/주석  *(AI 1차 라벨 + 선별 검수가 주연 — 라벨체계·일관성·HITL 중심)*
> 파일명: `B-2 데이터 해설_주석.md`
> 정의: AI 학습을 위해 데이터에 라벨·분류·해석 기준 등 사람이 부여한 의미 정보를 붙이는 체계 (학습 데이터 라벨 한정).
> 핵심 계열사 예시: 두산전자 (결함 이미지·고객 클레임 라벨링)
> 골격 변경: 추진역할 제거, "솔루션 선정"→라벨링 도구 선정(운영에 통합), 라벨체계·일관성가이드·HITL·버전관리를 What/How의 독립 섹션으로 승격 → 8개 섹션.
> Why·What·How: Why=1.2 / What=1.1·2장(라벨체계·일관성) / How=3·4·5장(대상선정·라벨링 운영·HITL·버전)
> ★ 핵심 원칙: AI 1차 라벨 후 저신뢰·고위험 건만 전문가가 검수한다(사람 노력 최소화).

```
1. 해설/주석이란 무엇이고 왜 필요한가
   1.1 정의 (학습용 의미 정보 부여 — 전처리(B-1) 결과에 라벨을 붙임)
   1.2 Pain Point: 학습 라벨 부재, 작업자마다 다른 기준, 애매 케이스 불일치, 라벨 이력 미관리 → 학습 품질 저하·재작업
   1.3 기대 효과와 체계 내 위치 (③ AI 가공 — 학습 데이터 품질·일관성·추적성 확보)
   1.4 경계 (학습 라벨=B-2 / 평가용 정답셋·Rubric=E-3(최고 혼동쌍) · 개념 관계망=B-3 · 용어 표준=A-3)
2. 라벨 체계를 어떻게 설계하는가  [KQ2]
   2.1 라벨 Taxonomy / 라벨 정의서 (결함 유형·원인 유형·조치 유형·판정 결과)
   2.2 라벨 경계·상호배타성·다중 라벨 허용 기준
   2.3 라벨명-표준 용어 정합 (A-3 Glossary 연계)
3. 일관성을 어떻게 확보하는가 — Annotation Guideline  [KQ3]
   3.1 Annotation Guideline / 사례집 (주석 기준·대표 사례·애매 케이스·금지 판단 — 예: 소재 문제 vs 공정 문제)
   3.2 작업자 캘리브레이션·작업자 간 일치도(IAA) 관리
   3.3 검수·합의(Consensus) 워크플로우
4. 주석 대상 선정  [KQ1]
   4.1 주석 대상 식별 (결함 이미지·클레임 문장·원인 분석 문서·실험 결과·조치 이력)
   4.2 포함/제외 기준 (학습용=포함 / 평가용 정답셋=E-3 제외)
   4.3 중요도·규모 산정 (SME 필요 여부·주관성·건수)
   4.4 보안·개인정보 검토 (클레임 내 고객 식별정보 — F-4 마스킹)
5. 어떻게 주석하는가 — AI 1차 라벨 + 선별 검수  [KQ4] ★ (이 주제의 핵심)
   5.1 AI-assisted Labeling 운영 모델 (AI 1차 라벨 → 저신뢰·고위험 건만 SME 검수, Human-in-the-loop)
   5.2 라벨링 워크플로우 (AI 사전라벨 → 검수 → 합의 → 확정)
   5.3 라벨링 도구 선정 (Annotation Tool·AI 사전라벨·데이터셋 관리 — 이미지/텍스트별 조합, 검수 워크플로·HITL로 비교)
   5.4 ★ 사람 노력 최소화 — 전수 검수가 아니라 위험 기반 선별 검수
6. 운영: 라벨 버전 관리와 학습셋 이력  [KQ5]
   6.1 라벨 버전 관리 (라벨 버전·작업자·검수자·변경 사유·학습 사용 시점 추적 — KQ5 핵심)
   6.2 라벨 변경/추가 운영 (신규 결함 유형·기준 변경 → Taxonomy·가이드라인 갱신)
   6.3 라벨 품질 검증 (IAA·검수 합격률) / 권한·보안
7. 계열사 예시 — 두산전자 결함 이미지·클레임 라벨링 (환경 → 대상선정·E-3 분리 → Taxonomy·가이드라인 → AI 1차 라벨+선별 검수 → 버전 관리 → 효과)
8. KPI & 고도화 (라벨 정확도·IAA·검수 합격률·라벨링 생산성·AI 사전라벨 채택률·학습셋 버전 추적률 / 수기 전수 → AI 사전라벨+전수 검수 → AI 1차+선별 검수 → 능동 학습 → 자율 라벨 보강)
```

## B-3 온톨로지  *(무거운 주제 — 깊이 상단. 모델링·계층·매핑·추론을 핵심 섹션으로)*
> 파일명: `B-3 온톨로지.md`
> 정의: 업무 개념 간 관계·계층·인과 구조를 정의해 AI가 데이터의 맥락과 연결성을 이해하도록 하는 지식 구조 (개념 간 관계망 한정).
> 핵심 계열사 예시: 두산에너빌리티 (품질 지식그래프 — 제품·공정·결함·원인·조치 관계망)
> 골격 변경: 추진역할 제거, 무거운 주제이므로 적용판단·엔티티/관계 모델링·공통/계열사 계층·개념-데이터 매핑·AI 추론을 각각 독립 섹션으로 전개 → 10개 섹션(깊이 상단).
> Why·What·How: Why=1.2 / What=1.1·3·4·5장(엔티티/관계·계층·매핑) / How=2·6·7·8장(적용판단·구축·추론·운영)
> ★ 핵심 원칙: 전사 거대 온톨로지 대신 한 업무를 푸는 최소 가치 그래프(MVG)부터 만든다.

```
1. 온톨로지란 무엇이고 왜 필요한가
   1.1 정의 (개념 간 관계·계층·인과 구조 — 데이터의 맥락·연결성)
   1.2 Pain Point: 단순 검색으로 원인-결과·연관 지식 추론 불가, 흩어진 지식 미연결, 유사 사례·조치 추천 불가
   1.3 기대 효과와 체계 내 위치 (② 의미 기반 — A-3·A-2를 입력받아 관계망 형성, RAG·Agent 추론 토대)
   1.4 경계 (개념 간 관계망=B-3 / 단일 용어 정의=A-3 · 학습 라벨 Taxonomy=B-2 · 자산 소재=A-1)
2. 언제 온톨로지를 구축하는가 — 적용 판단  [KQ1]
   2.1 적용 판단 기준 (관계 추론·인과 분석·Agent 판단 필요 vs 단순 용어 통일=A-3 / 분류 라벨=B-2)
   2.2 적용 대상 업무 식별 (원인 분석·조치 추천·연관 검색)
   2.3 우선 모델링 도메인 선정 (지식 밀도·관계 복잡성·AI 활용 가치)
   2.4 ★ 최소 가치 그래프(MVG) 우선 — 한 업무를 푸는 최소 관계망부터, 전사 거대 온톨로지 금지
3. 무엇을 모델링하는가 — 엔티티와 관계  [KQ2]
   3.1 엔티티(Entity) 목록 (제품·공정·결함·검사항목·원인·조치·고객불만·설비·조건)
   3.2 관계(Relation) 정의서 (발생한다·검출된다·원인이 된다·조치한다 등 관계 타입)
   3.3 계층·인과 구조 표현 (상위/하위 개념, 원인→결과 체인)
4. 공통 지식과 계열사 지식을 어떻게 나누는가  [KQ3]
   4.1 공통/계열사 확장 구조 (지주=공통 상위 개념·표준 관계 / 계열사=현장 특화 확장)
   4.2 공통-로컬 온톨로지 연결 방식 (확장점·매핑 규칙)
5. 무엇과 연결하는가 — 개념-데이터-문서 매핑  [KQ4]
   5.1 개념-데이터-문서 매핑표 (Glossary 표준 용어·메타데이터 필드·카탈로그 자산)
   5.2 문서 연결 (SOP·PFMEA·C/S Report 등 지식 원천 매핑)
6. 어떻게 구축하는가 — 지식 추출·도구·적재  [KQ2·KQ4 실행]
   6.1 지식 원천 취합·정제 (SME 인터뷰·문서 분석, 중복 개념·용어 충돌 정리 — A-3 연계)
   6.2 관계 추출·적재 (수동 모델링 + 문서 기반 관계 추출(LLM))
   6.3 도구 선정 (Graph DB·온톨로지 모델링 도구·지식그래프 플랫폼 / RDF·OWL vs Property Graph — 표현력·추론·SME 친화로 비교)
   6.4 PoC (한 도메인 관계망 → 관계 기반 질의·추천 검증)
7. AI 활용에 어떻게 적용하는가 — 관계 기반 추론  [KQ5] ★
   7.1 RAG 개념 확장 검색 (키워드가 아닌 관계 기반 탐색)
   7.2 Agent 원인 분석·유사 사례 추천·조치 후보 추천
   7.3 D-2/D-3(Tool·Prompt) 연계로 추론 실행
8. 운영: 개념·관계 변경 관리  [KQ6]
   8.1 변경 접수·전문가 검토 (신규 개념·관계·계층 변경이 추론에 미치는 영향 — KQ6 핵심)
   8.2 버전 관리·반영 / 그래프 Pipeline 운영 (매핑 갱신·관계 추출 재실행)
   8.3 권한·보안 (기밀 노드 접근 통제) / E-4 환류 (추론 오류·누락 관계 피드백 → 보강)
9. 계열사 예시 — 두산에너빌리티 품질 지식그래프 (지식 분산 환경 → 핵심 도메인 MVG → 엔티티·관계 모델링 → 공통/계열사 계층 → 개념-문서 매핑 → 관계 기반 원인·조치 추론 → 변경 운영 → 효과)
10. KPI & 고도화 (관계 커버리지·추론 정확도·유사 사례 추천 적중률·원인 분석 정답률·개념-데이터 매핑률·그래프 최신성 / 수동 모델링 → SME 반자동 → 문서 기반 관계 자동 추출 → 현업 검수 → 관계 기반 RAG·Agent 자율 추론)
```

# C. 신뢰·관측 (Trust & Observability)

## C-1 데이터 Observability  *(운영 관제 중심 — 감지·알림이 주연)*
> 파일명: `C-1 Observability.md`
> 정의: 데이터 흐름의 지연·누락·변경·이상 징후를 운영 중(실시간) 감지·알림하는 모니터링 체계. "지금 데이터에 무슨 일이 일어나고 있는가"를 본다.
> 핵심 계열사 예시: 두산밥캣 (다국가 공장 파이프라인·텔레매틱스 관측)
> 골격 변경: 추진역할(4장)·구축(8장)을 운영 섹션에 흡수, 솔루션 선정 축소, "관찰 지표·이상 판정·알림"을 최상위 핵심으로 승격 → 8개 섹션.
> Why·What·How: Why=1.2 / What=1.1·2·3장(관찰 지표·판정·알림·원인분리) / How=4·5·6장(대상선정·구축·운영)
> ★ 핵심 원칙: 오탐·중복 알림을 억제해 알림 피로(Alert Fatigue)를 최소화한다. "사용 가능 판정"은 하지 않는다(C-2).

```
1. Observability란 / 왜 필요한가
   1.1 정의 ("데이터 흐름의 실시간 계기판·관제실 CCTV" — 발생 순간 인지)
   1.2 Pain Point: 이상을 사후에야 인지 → AI 결과 신뢰 붕괴 (야간 배치 누락 → 수요예측 모델 오작동 무인지)
   1.3 경계: 사후 출처·영향 추적=C-3 / 투입 전 사용가능 판정(게이트)=C-2 / 재처리·롤백·복구=F-1
2. 무엇을 관찰하나 — 관찰 지표  [KQ1]
   2.1 핵심 지표 5종 (Freshness 지연 · Volume 건수급변 · Schema 변경 · Distribution 분포변화 · Null/이상값)
   2.2 흐름 유형별 지표 매핑 (배치 / 스트리밍 / API / 파일 — 유형별 핵심 지표)
   2.3 데이터-모델-Prompt 관찰 체계 (데이터·모델·Prompt·전처리 버전, 입력 분포를 함께 기록 — ★ 원인 후보 분리의 토대)  [KQ4]
3. 이상을 어떻게 판정하나 — 판정 룰·알림  [KQ2·KQ3]
   3.1 이상 판정 룰 / 임계값 기준 (고정 임계값 · 과거 추세 · 계절성 · 업무 캘린더) — ★ "사용 가능 판정"은 하지 않는다(C-2)
   3.2 임계값 베이스라인 수집 (정상 구간 학습 · 캘린더 반영)
   3.3 알림 기준 / Severity Level (이상 유형·심각도별 수신자 라우팅: 오너·DataOps·AI 운영자)  [KQ3]
   3.4 ★ 알림 피로(Alert Fatigue) 최소화 — 오탐 억제 · 중복 알림 묶기 · 심각도 차등
4. 무엇을 관찰 대상으로 삼나 — 대상 선정  [KQ1]
   4.1 핵심 데이터 흐름 식별 (AI 서비스·RAG·리포팅 직결 흐름 우선)
   4.2 관찰 우선순위·커버리지 (다운스트림 영향도 기반 Tier 1/2/3)
   4.3 자동 수집 가능 흐름 vs 수동 점검 지표 구분
5. 어떻게 구축하나 — 관측 체계·솔루션
   5.1 To-Be 관측 아키텍처 (지표 자동 수집 → 룰 판정 → 알림·에스컬레이션 워크플로)
   5.2 솔루션 선택지 (전용 Data Observability / 모니터링·APM 연계 / 오케스트레이터 내장 / 자체 구축) + 비교 기준(자동 지표 수집·룰/ML 탐지·알림 연동·C-3 Lineage 연계)
   5.3 PoC·검증 (파이프라인 2~3종 연결, 인위적 이상 주입 탐지 검증)
6. 어떻게 운영하나 — 탐지 → 트리아지 → 핸드오프  [KQ4]
   6.1 이상 탐지·알림 발생 운영 (룰·임계값 버전 스탬프 후 배포)
   6.2 이상 대응 트리아지 — 원인 후보 분리(데이터/모델/Prompt), ★ 복구 실행은 F-1으로 핸드오프
   6.3 게이트 연계 — C-2 Quality Gate에 관측 근거 제공
   6.4 커버리지 유지 (신규 흐름 자동 등록 · 룰 튜닝 · 오탐 회수)
7. 계열사 예시 — 두산밥캣 다국가 파이프라인 관측 (환경: MES/ERP·텔레매틱스·시차 운영 → 지표·임계값 설계 → Severity 라우팅 → 모델 오작동 원인 분리 → 효과)
8. 운영 수준 측정 & 고도화  [KQ5]
   8.1 KPI (이상 탐지 시간 · 미탐지(무인지 장애) 건수 · 오탐률 · 알림 후 확인 시간 · 핵심 파이프라인 커버리지)
   8.2 고도화 (수기 점검 → 룰 기반 탐지·알림 → 학습 기반 이상 탐지·자동 트리아지)
```

## C-2 데이터 품질 관리  *(무거운 주제 — 품질·권한 두 관문을 본문에 두텁게. 깊이 상단)*
> 파일명: `C-2 데이터 품질 관리.md`
> 정의: 데이터가 AI 활용 기준을 충족하는지(품질) 그리고 누가 쓸 수 있는지(권한)를 판정해 AI 투입을 통제하는 게이트(Quality Gate) 체계. "이 데이터를 써도 되는가"를 결정한다.
> 핵심 계열사 예시: 두산퓨얼셀 (스택 시험·운전·필드 데이터 + 고객/계약 민감정보 게이트)
> 골격 변경: 무거운 주제이므로 "품질 기준·사용 제한·권한·게이트·예외"를 각각 본문 섹션으로 분산 유지, 추진역할·구축을 운영에 흡수 → 9개 섹션(상단 깊이).
> Why·What·How: Why=1.2 / What=1.1·2·3·4장(품질 기준·제한·권한·게이트·예외) / How=5·6·7장(대상선정·구축·운영)
> ★ 핵심 원칙: 룰 기반 자동 판정 우선, 사람 검토는 예외·고위험만. 접근 차단은 C-2 역할, 마스킹은 F-4.

```
1. 데이터 품질 관리란 / 왜 필요한가
   1.1 정의 ("AI 투입 전 검문소(Gate)" — 통과/차단/예외 판정, 두 관문: 품질 + 권한)
   1.2 Pain Point: 품질·권한 미검증 데이터가 AI에 투입 → 오류 증폭·정보 유출 (미승인 시험데이터로 학습 → 잘못된 스택 수명 예측)
   1.3 경계: 운영 중 이상 감지=C-1 / 마스킹·가명화·비식별=F-4 (접근 "차단"은 C-2, 마스킹은 F-4) / 출처 근거 제공=C-3
2. 무엇이 "쓸 수 있는 데이터"인가 — 품질 기준  [KQ1·KQ2]
   2.1 데이터 품질 기준 (완전성·정확성·일관성·최신성·유효성·중복성) + 업무별 허용 임계(Critical/Major/Minor)  [KQ1]
   2.2 AI 사용 제한 데이터 기준 (출처 불명 · 미승인 · 품질 미달 · 최신성 미보장 · 목적 외 사용 제한 — 차단 대상)  [KQ2]
3. 누가 쓸 수 있나 — 접근 권한  [KQ3]
   3.1 접근 권한 정책 (RBAC/ABAC — 역할·조직·목적·민감도 기반)
   3.2 활용 단계별 권한 구분 (조회 / 분석 / AI 학습 / 외부 반출) — ★ 접근 "차단"은 C-2 역할, 마스킹은 F-4
   3.3 데이터 민감도·권한 등급 분류
4. 어떻게 판정하나 — Quality Gate & 예외  [KQ4·KQ5]
   4.1 Quality Gate 구조 (품질 + 권한 동시 통과해야 AI 학습·RAG·분석·자동화 투입 허용)  [KQ4]
   4.2 게이트 판정 분기 (통과 / 차단 / 예외)
   4.3 예외 사용 승인 (승인자·사유·사용 기간·범위·책임 기록 — ★ 예외에도 추적성)  [KQ5]
5. 무엇을 게이트 대상으로 삼나 — 대상 선정  [KQ1·KQ2·KQ3]
   5.1 품질 측정·게이트 대상 자산 선정 (AI 투입 예정 자산 우선)
   5.2 업무별 품질 허용 기준 정의 (차원별 임계)
   5.3 AI 사용 제한·민감도 식별 (출처·승인·목적 제한 + 권한 등급 매핑)
   5.4 게이트 우선순위 (다운스트림 AI 영향도 기반)
   5.5 ★ 판정 자동화 원칙 — 룰 기반 자동 판정 우선, 사람 검토는 예외·고위험만
6. 어떻게 구축하나 — 게이트·솔루션
   6.1 To-Be 게이트 아키텍처 (품질 판정 + 권한 판정 결합, 파이프라인 인입점에 배치)
   6.2 솔루션 선택지 (Data Quality 엔진 / 정책 엔진(Policy Engine) / 권한관리(IAM·ABAC) / 통합 거버넌스) + 비교 기준(자동 프로파일링·RBAC/ABAC·게이트 연동·예외 워크플로·C-1/C-3 연계)
   6.3 PoC·검증 (정상/불량/미승인 데이터로 통과·차단 검증)
7. 어떻게 운영하나 — 판정 → 차단/예외 → 공급  [KQ4·KQ5]
   7.1 Quality Gate 판정 운영 (투입 전 통과/차단)
   7.2 예외 승인 프로세스 운영
   7.3 품질/권한 위반 처리 (차단 · 재검토 · 개선 요청)
   7.4 AI 서비스 연계 (게이트 통과 데이터만 활용 단계로 공급) + 룰·정책 버전 관리
8. 계열사 예시 — 두산퓨얼셀 Quality Gate (환경: 시험·운전·필드 데이터 + 민감정보 혼재 → 품질 임계·권한 정책 설계 → 게이트 룰 → 미승인 데이터 차단 + 예외 승인 흐름 → 효과)
9. 측정 & 고도화  [KQ6]
   9.1 KPI (품질 기준 통과율 · 권한 위반 건수 · 미승인 데이터 사용 건수 · AI 사용 차단 건수 · 품질 개선 리드타임 · 예외 승인 적정성)  [KQ6]
   9.2 고도화 (수동 점검 → 자동 Quality Gate·정책 기반 차단 → 자가 교정·동적 권한)
```

## C-3 데이터 계통 Lineage  *(추적성 중심 — 정형 Lineage + RAG Citation 두 축)*
> 파일명: `C-3 데이터 계통 Lineage.md`
> 정의: 데이터의 출처→이동→변환→활용→AI 답변 근거까지의 이력을 기록해 AI 결과의 근거를 추적하고 변경 영향을 분석하는 체계. "이 결과는 어디서 왔는가"를 되짚는다.
> 핵심 계열사 예시: 두산로보틱스 (매뉴얼·정비 절차 RAG 답변 근거 추적)
> 골격 변경: 추진역할·구축을 운영에 흡수, 솔루션 축소, "범위·변환이력·Citation·영향도"를 본문 핵심으로, 감사 증빙을 독립 섹션으로 승격 → 8개 섹션.
> Why·What·How: Why=1.2 / What=1.1·2장(범위·변환이력·Citation·영향도) / How=3·4·5장(대상선정·구축·운영)
> ★ 핵심 원칙: 메타데이터·로그 기반 자동 수집을 우선하고, 수동 매핑은 미연동 구간만.

```
1. 데이터 계통(Lineage)이란 / 왜 필요한가
   1.1 정의 ("데이터 족보·이력서" — 출처에서 AI 답변까지의 계보)
   1.2 Pain Point: AI 답변·보고서 근거 역추적 불가, 원천 변경 영향 파악 불가 (RAG 답변이 어느 매뉴얼 버전을 참조했는지 알 수 없어 책임 규명 불가)
   1.3 경계: 실시간 이상 탐지=C-1 / 보존·아카이빙·폐기=F-2 / 출처 근거를 받아 판정=C-2
2. 무엇을 추적하나 — 범위·변환이력·근거·영향  [KQ1·KQ2·KQ3·KQ4]
   2.1 AI-ready Lineage 범위 (원천 → 전처리 → 분석셋 → 보고서 → RAG 인덱스 → Prompt 입력 → AI 답변)  [KQ1]
   2.2 변환 이력 기록 기준 (ETL/ELT 로직 · 전처리 버전 · 병합 · 필터 · 파생 변수 · 문서 참조)  [KQ2]
   2.3 AI 답변 근거 로그 / Citation 구조 (참조 문서·청크·테이블·Tool 결과 — 답변→근거 역추적)  [KQ3]
   2.4 영향도 분석 기준 (원천·로직 변경 → 리포트·모델·Prompt·서비스 영향 추적)  [KQ4]
3. 무엇을 추적 대상으로 삼나 — 대상 선정  [KQ1·KQ3·KQ5]
   3.1 Lineage 대상 흐름 선정 (AI 답변 직결 흐름 우선)
   3.2 기록 수준(Granularity) 정의 (컬럼/테이블/문서/청크 — 흐름별 차등)
   3.3 Citation 필요 과제 식별 (RAG·Agent·자동 보고서 — 근거 필수)
   3.4 감사·규제 증빙 대상 식별 (규제·고객 협약 흐름)
   3.5 ★ 자동 수집 우선 원칙 — 메타데이터·로그 기반 자동 수집, 수동 매핑은 미연동 구간만
4. 어떻게 구축하나 — Lineage 체계·솔루션
   4.1 To-Be Lineage 아키텍처 (정형 Lineage + RAG Citation 두 축 결합)
   4.2 메타데이터·카탈로그 정합성 (A-1 자산 식별자 정렬)
   4.3 솔루션 선택지 (전용 Data Lineage(OpenLineage 류) / 카탈로그·메타 내장 / 오케스트레이터 연계 / LLM Observability·tracing) + 비교 기준(자동 수집 범위·컬럼 레벨·문서/RAG Citation·영향도 UI·감사 리포트)
   4.4 PoC·검증 (RAG 파이프라인 연결, 답변 → 근거 역추적 검증)
5. 어떻게 운영하나 — 수집 → Citation → 영향도  [KQ3·KQ4]
   5.1 Lineage 수집·Citation 로깅 운영 (룰·스키마 버전 관리)
   5.2 영향도 분석 운영 (원천·로직 변경 시 다운스트림 영향 — ★ 이상 "탐지"는 C-1으로)
   5.3 AI 서비스 연계 (RAG 답변에 근거 Citation 자동 첨부)
   5.4 커버리지 유지 (끊긴 Lineage 보수)
6. 감사·규제 대응  [KQ5]
   6.1 감사 대응 Lineage Report (출처 · 변환 이력 · 접근 이력 · 사용 목적 · AI 답변 근거 증빙)
   6.2 권한·보안 (Lineage 그래프 접근 · 민감 경로 노출 통제)
7. 계열사 예시 — 두산로보틱스 RAG 근거 추적 (환경: 매뉴얼·정비 절차 RAG, 매뉴얼 버전 빈번 갱신 → 청크 단위 Citation 설계 → RAG 오답 근거 역추적 + 매뉴얼 개정 영향도 분석 → 효과)
8. 측정 & 고도화
   8.1 KPI (Lineage 커버리지 · AI 답변 근거 추적 가능률 · 영향도 분석 소요시간 · 감사 대응 리드타임 · 끊긴 Lineage 비중)
   8.2 고도화 (수동 매핑 → 자동 Lineage 수집·Citation 로깅 → 근거 자동 제시·영향도 자동 분석)
```

## D-3 Prompt/Harness 자산화  *(업무 지식의 자산화 관점. Registry·의존성·재사용이 주인공)*

```
1. Prompt/Harness 자산화란 / 왜 필요한가
   1.1 정의 (Prompt = 업무 지시·판단의 구조화 자산, Harness = Prompt+입력+Tool 순서+출력을 묶은 반복 실행 단위)
   1.2 Pain Point ★ (Prompt가 개인 PC에 산재·버전 미관리·의존성 불명 → 재사용·개선·감사 불가, 데이터/Tool 변경 시 깨짐)
   1.3 체계 내 위치 (활용 단계 — 사람의 업무 지식을 AI 실행 자산으로 옮기는 계층)
2. 업무 지식을 어떻게 Prompt로 옮기나 — 자산 정의 & 변환  [KQ1·KQ2]
   2.1 Prompt/Harness 자산 목록 (업무 Prompt·시스템 지시문·역할 정의·판단 기준·출력 형식·예외 처리)  [KQ1]
   2.2 업무 절차 → Prompt 변환 템플릿 (입력·판단 단계·참조 기준·금지 행동·출력 형식)  [KQ2]
   2.3 SME 업무 절차 구조화 (인터뷰 → 단계·판단 기준 추출 — 암묵지를 명시 절차로)
3. Prompt Registry & 버전 관리  [KQ3]

5. 무엇을 자산화하나 — 대상 선정  [KQ1]
   5.1 자산화 대상 기준 (★ 반복 사용성·업무 영향도·판단 복잡도·재사용 가치 / 1회성 실험 제외)
   5.2 의존성·민감도 검토 (참조 데이터/Tool 보안, 출력 민감정보 — 마스킹은 F-4)

   6.1 구축 흐름 (Legacy 자산 수집 → 변환·등록 → 의존성·재현성 검증)
   6.2 Prompt 성능 저하 감지 및 개선 (평가 결과·사용자 수정률·실패 사례 근거 / 평가 기준 자체는 E-3)  [KQ5]
```

> Why·What·How: Why=1.2 / What=1.1·2장 / How=3·4·5·6장

```
1. 데이터 Product화란 / 왜 필요한가

2. 데이터 Product의 구성요소 — 명세서  [KQ2]
   2.1 데이터 Product 명세서 표준 (대상 사용자·사용 목적·제공 데이터·품질 기준·SLA·갱신 주기·Owner·사용 조건·예시 쿼리)

   3.2 원천-Product 매핑 (계열사마다 다른 테이블/스키마를 동일 Product 형태로 정규화)

4. 소비자는 어떻게 쓰나 — 소비 방식  [KQ4]

   4.2 소비자별 제공 기준 (분석가·AI Agent·BI 사용자가 각각 무엇을 원하는가)
   4.3 셀프서비스 소비 경험 (요청 → 승인 없이 바로 쓰는 데까지)
5. 누가 책임지나 — Data Product Owner  [KQ5]
   5.1 Data Product Owner 역할·책임 (품질 유지·변경 공지·문의 대응·개선 Backlog·버전/폐기 판단)
   5.2 소비자 요청·변경 운영 (신규 요청 → Owner 검토 → 명세·채널 반영 → 변경 공지)
   5.3 사용 조건·권한 (소비 단위 접근 통제 — 상세 정책은 C-2/F-4)
6. Product화 대상 선정  [KQ1]
   6.1 Product 후보 식별 기준 (반복 요청 빈도·다(多)팀 사용·AI 과제 활용 빈도·책임 명확화 필요성)
   6.2 후보 진단 (요청 로그·중복 추출 패턴 분석 — ★ 백지에서 만들지 않는다)

7. 계열사 예시 — 두산테스나 수율 데이터 Product화 (반복 요청 진단 → "Lot별 수율 Product" 정의 → 명세·추상화 → API/대시보드/RAG 소비 → Owner 운영 → 중복 추출 제거 효과)
8. KPI & 고도화  [KQ6] (재사용률·소비자 수·중복 데이터 감소·요청 처리시간 단축·AI 과제 활용 건수 / 수동 제공 → 셀프서비스 카탈로그 → AI 추천 소비·자동 Product 합성)
```

## E-2 합성데이터 (Synthetic)  *(방식 중심 — 솔루션 도입이 아니라 생성→검증→위험관리)*
> 파일명: `E-2 합성데이터.md`
> 정의: 실데이터가 부족·제한될 때 AI 학습·검증을 위해 인공 생성한 데이터와 그 생성·검증·위험 관리 체계.
> 핵심 계열사 예시: 두산전자 (희소 불량 사례 합성)
> 골격 변경: "솔루션 선정"→방식·도구 선정, "구축"→생성 파이프라인, 추진역할 제거, 검증·위험을 독립 섹션으로 승격 → 8개 섹션.
> Why·What·How: Why=1.2(왜·언제 합성이 필요한가) / What=1.1·2·3장 / How=4·5·6장
> ★ 핵심 원칙: 합성은 보충재 — 부족/제한을 메우되 실데이터와 절대 혼동되지 않게 태깅·검증한다. 평가셋은 E-3, 비식별은 F-4.

```
1. 합성데이터란 / 언제 쓰나
   1.1 정의 (실데이터 부족·보안 제한 시 인공 생성 — 원본 대체 아님)
   1.2 적용 판단 기준 ★ (희소 이벤트·클래스 불균형·보안 차단·테스트 부족 / "실데이터 우선" 원칙)
   1.3 경계: 평가셋은 E-3, 비식별은 F-4, 사용 판정은 C-2
2. 무엇을 합성할 것인가
   2.1 합성 대상 정의 (불량·장애·예외·테스트 케이스)
   2.2 데이터 부족 영역 진단 / reference(실데이터 기준) 확보
3. 어떻게 생성할 것인가 — 방식·도구 선정
   3.1 생성 방식 비교 (통계·규칙·시뮬레이션·생성모델/LLM) — 데이터 특성별 매칭
   3.2 도구·라이브러리 후보 / PoC (1종을 2방식으로 생성해 비교)
4. 생성 파이프라인 구성 (설계 → 실행·튜닝 → Synthetic Tag 부착)
5. 검증 — 실데이터를 닮았는가
   5.1 검증 항목 (분포 유사성·상관 보존·희소 패턴 재현·도메인 타당성·모델 성능 개선)
   5.2 도메인 전문가 검토 게이트
6. 위험 관리 & 운영
   6.1 편향·과적합·현실 왜곡·재식별 점검
   6.2 Synthetic Tag·사용 정책 (★ 실데이터와 절대 혼동 금지)
   6.3 버전·재현성, 학습 파이프라인 연계
7. 계열사 예시 — 두산전자 희소 불량 합성 (요약)
8. KPI & 고도화 (분포 유사도·재현율·성능 개선폭·태깅 누락0 / 규칙 → 생성모델 → 검증 자동화)
```

## E-3 AI 평가 데이터  *(검증 게이트 — E-2와 동결: 6장=평가 방식·Eval 도구 선정, 8장=Gold Dataset·Harness 구성)*
> 파일명: `E-3 AI 평가 데이터.md`
> 정의: AI 모델·Prompt·Agent의 성능을 객관적으로 판단하기 위한 정답셋(Gold Dataset)과 평가 기준(Rubric) 데이터 및 그 평가 체계.
> 핵심 계열사 예시: 두산로보틱스 (협동로봇 기술문의 RAG·작업 Agent 응답 평가셋)
> 골격 변경: "추진 역할" 독립 섹션 제거, "솔루션 선정"→평가 방식·Eval 도구 선정으로 변형, "구축"→Gold Dataset·Harness 구성으로 변형, 계열사 13단계→핵심 흐름 1섹션 → 8개 섹션.
> Why·What·How: Why=1.2 / What=1.1·2·3장 / How=4·5·6·8장 (대상선정·방식선정·구성·운영)
> ★ 핵심 원칙: 평가 데이터는 "맞았다/틀렸다"가 아니라 "왜 맞고 왜 틀렸는지"를 축적한다.

```
1. AI 평가 데이터란 / 왜 필요한가
   1.1 정의 (성능 판단용 정답셋(Gold Dataset) + 평가 기준(Rubric) — ★ "왜 맞고 왜 틀렸는지"를 축적)
   1.2 Pain Point: 객관적 정답셋 부재로 성능 판단 불가, Prompt 수정 후 회귀(Regression) 미검출, "좋아진 것 같다"는 주관 판단, 전문가 판단 휘발
   1.3 경계: 학습 라벨은 B-2, Prompt 자산은 D-3, 운영 중 모니터링은 C-1 (★ E-3=사전 검증 게이트)
2. 무엇을 평가하나 / 어떤 과제에 평가 데이터가 필요한가  [KQ1]
   2.1 평가 데이터 필요 AI 과제 식별 (RAG·Agent·분류·예측·보고서 생성 등 — 성능을 객관 검증해야 하는 과제)
   2.2 과제별 평가 항목 도출 (정확성·근거성·완전성·안전성·실행 가능성 중 무엇이 핵심인가)
   2.3 평가 제외/간이 평가 대상 (저위험·실험적 과제)
   2.4 ★ "대표성·실패 케이스 우선" — 쉬운 문제만 모으지 않고 경계·실패 케이스를 포함
3. 평가 데이터의 구성요소 — Gold Dataset · Rubric · 전문가 평가  [KQ2·KQ3·KQ4]
   3.1 Gold Dataset 표준 (기준 질문·입력 데이터·기대 답변·정답 근거·허용 답변 범위·전문가 코멘트)  [KQ2]
   3.2 평가 Rubric(채점 기준표) (정확성·근거성·완전성·일관성·안전성·실행 가능성 — 항목별 채점 기준)  [KQ3]
   3.3 전문가(SME) 평가 프로세스 (점수 + 판단 근거 + 오류 유형 구조화, 평가자 간 불일치 조정 기준)  [KQ4]
4. 변경을 어떻게 통과시키나 — Evaluation Harness · 회귀  [KQ5]
   4.1 Evaluation Harness(자동 평가 실행기) 구조 (변경 트리거 → 자동 실행 → 점수 집계)
   4.2 회귀 테스트셋(Regression Test Set) (기존에 맞던 문제가 다시 틀리지 않는지 — baseline 대비)
   4.3 변경 검증 게이트 (모델·Prompt 변경 전후 비교로 통과/반려 — D-3·E-4 연계)
5. 평가 데이터 대상 선정 · 운영  [KQ6]
   5.1 우선 구축 대상·규모 산정 (대표 케이스 수·난이도 분포)
   5.2 운영: 신규 실패 사례·평가 케이스 추가 → SME 검토 → Gold·Rubric 반영 → 버전 등록
   5.3 평가 데이터 갱신 기준 (신규 실패 사례·업무/정책 변경·새 데이터 유형 반영)  [KQ6]
   5.4 권한·보안 (★ 평가셋 유출 시 성능 부풀림 — 학습/외부 노출 차단)
6. 평가 방식·Eval 도구 선정  ※ 솔루션 도입이 아니라 "방식·도구 선정"으로 변형
   6.1 평가 방식 유형 비교 (정답 매칭·룰 기반·LLM-as-judge·전문가 정성 평가) — 과제 특성별 매칭
   6.2 방식별 적용 조건 (정형 답 vs 자유 생성 vs Agent 행동 평가)
   6.3 Eval 도구·프레임워크 후보 / PoC (동일 과제를 LLM-as-judge와 전문가 평가로 교차 검증 → 신뢰도 확인)
7. 계열사 예시 — 두산로보틱스 RAG/Agent 평가셋 (평가 대상 과제 식별 → 기술문의 Gold Dataset·근거 문서 작성 → Rubric(정확성·근거성·안전성) → 전문가 평가 축적 → Harness·회귀 구성 → Prompt 변경 검증)
8. Gold Dataset 작성 · Harness 구성 & KPI/고도화  ※ "구축"을 평가 자산 구성으로 변형
   8.1 평가 케이스 수집 → Gold Dataset 작성(기대 답변·근거·허용 범위) → Rubric 채점 기준 표준화 → Harness/회귀셋 구성 → baseline 기준선 확정
   8.2 KPI (평가 커버리지·회귀 검출률·전문가 간 일치도(IAA)·LLM-judge–전문가 일치도·평가셋 갱신 주기)
   8.3 고도화 (수동 평가 → LLM-as-judge 자동 회귀 → 실패 사례 자동 보강·평가셋 자율 확장)
```

## F-3 데이터 디지털화  *(원천 확보 주제 — 일회성 전환을 넘어 Digital-first 체질 전환이 핵심, KQ 6개)*

```
1. 데이터 디지털화란 / 왜 필요한가
   1.1 정의 (아날로그→디지털 전환 + 처음부터 디지털 생성(Digital-first) — "원천 확보": AI가 쓸 데이터를 새로 만들어 공급)

   1.3 경계 ★: 이미 디지털 문서의 구조화 전처리 = B-1 / 전환 데이터의 업무 품질 판정 = C-2 / F-3 = 아날로그→디지털 대량·일회성 전환 + 신규 생성 방식 전환
2. 무엇이 누락되어 있나 — 아날로그 자산 Inventory  [KQ1]
   2.1 누락 자산 전수 조사 (종이 문서·수기 검사표·출력물·현장 메모·사진·스캔본)
   2.2 암묵지 식별 (SOP/FMEA에 안 적힌 작업자 판단 기준 — 인터뷰 구조화 대상)
3. 무엇을 먼저 디지털화하나  [KQ2]
   3.1 우선순위 기준 (업무 중요도·AI 활용 가능성·소실 위험·반복 활용성·현장 의존도)
   3.2 자산 유형별 전환 난이도·소실 위험 평가
   3.3 ★ 소실 위험 높고 AI 활용 가치 큰 자산부터, Wave로 단계 전환
4. 어떤 방식으로 전환하나  [KQ3]
   4.1 방식별 적용 기준 (인쇄물=OCR · 수기=ICR · 신규 입력=모바일/태블릿 폼 · 음성=STT · 사진=메타정보 태깅)
   4.2 암묵지 전환 — 현장 인터뷰 구조화·템플릿화
   4.3 대상 특성 ↔ 전환 방식 매핑
5. 흩어진 Excel·개인 PC 데이터를 어떻게 공식 자산으로 승급하나  [KQ4]
   5.1 산재 데이터 식별·승급 대상 선정 (비공식 데이터의 공식 자산화)
   5.2 지속 수집 체계로 전환 (일회성 수집이 아닌 상시화)
6. 전환 오류·누락을 어떻게 검증하나  [KQ5]
   6.1 검증 기준 (원본 대비 검수·샘플링 검수·필수항목 누락 확인·OCR/ICR 인식률)
   6.2 ★ 여기서는 전환 "정확도"만 — 전환 데이터의 업무 품질 판정은 C-2
7. 앞으로 처음부터 디지털로 — Digital-first 전환 ★ (이 주제의 지향점)  [KQ6]
   7.1 Digital-first 양식 설계 (표준 코드·필수 입력값·드롭다운·자동 시간 기록)
   7.2 종이 양식 단계적 폐지 → 신규 데이터의 디지털 원천 생성
   7.3 ★ 핵심 원칙 — 일회성 전환에 그치지 않고 아날로그 유입 자체를 차단
8. 솔루션·계열사 예시
   8.1 솔루션 메모 (OCR/ICR 엔진·모바일/태블릿 입력 앱·STT — 수기·현장 환경 인식률·한글/도면/표 인식이 관건)
   8.2 계열사 예시 — 두산에너빌리티 현장 디지털화 (자산 Inventory → 우선순위 → 전환 방식 매핑(ICR/OCR/모바일/인터뷰) → 산재 Excel 승급 → 검증 → Digital-first 양식 → 암묵지 보존)
9. KPI & 고도화
   9.1 KPI (디지털화율(대상 대비)↑·전환 정확도/인식률↑·산재 데이터 공식 승급 건수↑·Digital-first 양식 적용률↑·암묵지 구조화 건수↑)
   9.2 고도화 (일회성 수동 전환 → 자동 OCR/ICR·암묵지 추출 보조·산재 데이터 자동 탐지(AI보조) → 상시 수집·Digital-first 정착(자율))
```

## F-4 AI 데이터 보안  *(횡단 보안 주제 — 무게 상단, KQ 7개, "차단"이 아닌 "쓸 수 있게 변환")*

> 핵심 계열사 예시: 두산밥캣 (북미/유럽 딜러·고객 개인정보 등 글로벌 사업의 비식별, GDPR 등 규제 대응)

```
   1.3 경계 ★: 접근 차단·권한 통제 = C-2 / 원본 대체 합성 = E-2 / F-4 = 탐지·마스킹·가명화·비식별
```
