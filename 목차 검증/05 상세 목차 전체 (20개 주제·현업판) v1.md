# AI-Ready Data 매뉴얼 — 상세 목차 전체 (20개 주제·현업판) v1

> **이 파일이 20개 주제 상세 목차의 최종본**입니다. (이전 01·02 = 골격 실험본, 04 = 형식 데모본 — 본 파일로 대체)
> 작성 기준은 `CLAUDE.md`의 「목차 구조 표준」을 따릅니다:
> - **12개 강제 X** — 주제에 필요한 것만(보통 7~8개).
> - **순서**: 개요 → 왜(Why) → 무엇(What) → 언제/어디에(When) → **예시 시나리오** → 어떻게 준비·운영(How) → 다른 주제와 관계 → 지표·고도화.
> - **예시 시나리오는 How 앞**에 두어 "적용하면 이렇게 좋아진다 + 흐름"을 먼저 그려줌(맨 끝 계열사 예시 통합).
> - **세부 H3 목차는 두되 내용은 현업 눈높이로 쉽게**, 어려운 약어는 풀이.
> - **When 규칙**: 골라서 하는 주제(온톨로지·합성·평가·Product화·Tool·Prompt·디지털화)는 "언제/무엇을 하나(적용 판단)", 기본 다 필요한 주제(카탈로그·메타·Glossary·전처리·품질·관측·Lineage·DataOps·생애주기·보안)는 "어디부터(우선순위)".

# A. Findable — 발견·의미 (데이터를 찾을 수 있게)

## A-1 데이터 카탈로그  *(기본적으로 다 필요한 주제 → "어디부터")*
> 한 줄: 데이터의 "주소록" — 어떤 데이터가 어디에 있고, 누가 주인이며, 어떻게 접근하는지 적어둔 자산 목록.
> 핵심 계열사: 두산전자

```
1. 개요
   1.1 데이터 카탈로그란 (데이터 자산의 주소록 — 소재·오너·접근경로)
   1.2 적용 범위와 체계 내 위치 (소재는 카탈로그, 항목 뜻은 메타데이터(A-2)·용어 뜻은 Glossary(A-3)와 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (데이터가 어디 있는지 몰라 사람마다 찾아 헤매고, AI도 못 찾음)
   2.2 기대 효과 (찾는 시간↓·중복 수집↓·AI가 필요한 데이터에 바로 도달)
3. 무엇을 갖추나 (필수 등록 항목)
   3.1 자산 기본 정보 (데이터명·보유 시스템·저장 위치)
   3.2 오너·접근 정보 (데이터 오너·보유 부서·접근 경로·갱신 주기)
   3.3 탐색용 분류 (업무 도메인·데이터 유형·보유 조직 — 찾기 위한 분류, 뜻 해석은 아님)
4. 어디부터 하나 (등록 우선순위)   ← 카탈로그는 다 필요하므로 "순서"
   4.1 등록 대상 범위 (시스템 데이터·문서·보고서·품질·실험·설비 데이터)
   4.2 우선순위 기준 (AI 활용 가능성·업무 중요도·재사용 많은 핵심 데이터부터)
   4.3 자동으로 모으고 사람은 검수만 (★ 사람이 다 찾아 등록하지 않는다)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (검사 데이터가 어느 시스템·폴더에 있는지 몰라 담당자 수소문 → 카탈로그에서 검색 한 번으로 위치·오너·접근법 확인)
   5.2 흐름 미리보기 (원천에서 자산 정보 자동 수집 → 오너·접근경로 채우기 → 분류 붙여 검색 가능 → AI·현업이 바로 탐색)
6. 어떻게 준비·운영하나 (상세)
   6.1 수집·관리 도구 검토 (자동 수집 가능 자산 vs 수동 등록 자산 구분)
   6.2 구축 절차 (여러 원천 연결: ERP·MES·QMS·문서저장소·데이터레이크 → 하나의 목록으로 통합 → 검색 화면 구성)
   6.3 운영 (신규·폐기·위치 변경·오너 변경 시 갱신: 자동 감지 + 오너 확인 + 정기 점검) / 역할·책임
7. 다른 주제와의 관계
   7.1 메타데이터(A-2 항목 속성)·Glossary(A-3 용어 뜻)·Lineage(C-3 데이터 흐름)와의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (핵심 자산 등록률·자동 수집률·검색 성공률·미등록 데이터 비율)
   8.2 고도화 (등록 자동 감지 범위↑ → 사람은 검수만)
```

## A-2 메타데이터  *(기본적으로 다 필요한 주제)*
> 한 줄: 데이터의 "설명서" — 이 항목이 무슨 뜻이고, 단위·갱신주기·담당자가 누구인지 적어두는 것.
> 핵심 계열사: 두산에너빌리티

```
1. 개요
   1.1 메타데이터란 (데이터의 설명서)
   1.2 적용 범위와 체계 내 위치 (소재는 카탈로그, 용어 뜻은 Glossary와 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (항목 이름·단위만으론 뜻을 몰라 AI가 잘못 해석)
   2.2 기대 효과 (재사용이 쉬워지고 오해석이 줄어듦)
3. 무엇을 갖추나
   3.1 표준 항목 (타입·단위·생성 시스템·갱신 주기·담당자)
   3.2 항목 설명서 (약어·코드 풀이 — 예: DEF_CD = 결함 유형 코드)
   3.3 단위·기준일·집계 기준 (숫자를 잘못 비교하지 않도록)
4. 어디부터 하나 (우선순위)   ← 메타데이터는 다 필요하므로 "순서"
   4.1 정비 우선 대상 (설명이 비었거나 단위가 뒤죽박죽인 데이터부터)
   4.2 자동으로 모으고 사람은 검수만 (★)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (AI가 'DEF_CD'를 못 알아봄 → 설명서를 달면 '결함 유형 코드'로 바로 해석)
   5.2 흐름 미리보기 (자동 수집 → 빈 설명·단위 채우기 → 담당자 검수 → AI가 바로 활용)
6. 어떻게 준비·운영하나 (상세)
   6.1 수집·관리 도구 검토
   6.2 구축 절차 (자동 수집 연결 → 설명·단위 보완 → 검증)
   6.3 운영 (항목이 바뀌면 갱신·승인) / 역할·책임
7. 다른 주제와의 관계
   7.1 카탈로그(소재)·Glossary(용어)·품질과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (작성 완성률·자동 수집률·설명 충실도)
   8.2 고도화 (AI가 설명 초안 작성 → 사람 검수)
```

## A-3 비즈니스 Glossary  *(기본적으로 다 필요한 주제 → "어디부터")*
> 한 줄: 회사 용어 "표준 사전" — 같은 결함·공정을 부서·계열사마다 다르게 부르는 말을 하나로 통일.
> 핵심 계열사: 두산밥캣

```
1. 개요
   1.1 비즈니스 Glossary란 (업무 용어·약어·동의어를 표준 정의로 통일한 사전)
   1.2 적용 범위와 체계 내 위치 (단어 "뜻"은 Glossary, 필드 설명은 메타데이터(A-2), 개념 "관계"는 온톨로지(B-3)와 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (같은 결함을 현장마다 다르게 불러 AI 검색·집계가 따로 놂 — 특히 글로벌 현장)
   2.2 기대 효과 (현장 용어로 물어도 표준 용어로 변환돼 관련 데이터를 한 번에 검색·해석 일관)
3. 무엇을 갖추나 (표준 용어 템플릿)
   3.1 용어 기본 항목 (표준명·영문명·약어·정의·사용 예시)
   3.2 동의어·금지어 (현장식 표현을 표준 용어로 이어주는 연결고리)
   3.3 연계 정보 (관련 데이터 필드·책임 부서)
4. 어디부터 하나 (표준화 우선순위)   ← Glossary는 다 필요하므로 "순서"
   4.1 표준화 대상 용어 (결함명·검사항목·공정명·제품군·원인 유형·조치 유형·고객 불만 유형)
   4.2 우선순위 기준 (AI 검색·해석에 영향 크고, 부서·계열사마다 다르게 쓰는 용어부터)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (현장식 결함명으로 검색하면 결과가 안 나옴 → 동의어를 표준명에 묶으면 어느 표현으로 물어도 같은 데이터가 검색)
   5.2 흐름 미리보기 (대상 용어 추리기 → 표준명·동의어 정의 → 충돌 조정·승인 → AI 검색에 연결)
6. 어떻게 준비·운영하나 (상세)
   6.1 용어 수집·정의 (현장·문서에서 실제 쓰는 표현 모으기 → 표준명·동의어·금지어 정리)
   6.2 용어 충돌 조정 (같은 말 다른 뜻·다른 말 같은 뜻을 현업 전문가·오너·중앙 조직이 조정·승인)
   6.3 AI 활용 연계 (카탈로그 검색·메타데이터·AI 검색에 연결 — 현장 용어→표준 용어 자동 변환)
   6.4 운영 (신규 등록·변경·폐기 시 버전 관리·승인, 변경이 데이터·문서에 주는 영향 점검) / 역할·책임
7. 다른 주제와의 관계
   7.1 메타데이터(A-2 필드 설명)·온톨로지(B-3 개념 관계)·카탈로그(A-1 검색)와의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (표준화 완료 용어 수·동의어 매핑률·현장 용어 변환 성공률·용어 충돌 해소 건수)
   8.2 고도화 (문서에서 후보 용어·동의어를 AI가 초안 추출 → 전문가 검수)
```

# B. Understandable — 변환·지식화 (AI가 이해할 수 있게)

## B-1 데이터 전처리  *(기본적으로 다 필요한 주제 → "어디부터")*
> 한 줄: 이미 디지털인 문서·표·이미지를 AI가 바로 읽을 수 있는 구조로 바꾸는 작업.
> 핵심 계열사: 두산에너빌리티

```
1. 개요
   1.1 데이터 전처리란 (PDF·PPT·Excel·이미지를 AI가 읽을 구조로 변환 — 형식 변환에 집중)
   1.2 적용 범위와 체계 내 위치 (이미 디지털인 자료가 대상 — 종이·수기 전환은 디지털화(F-3), 쓸 수 있나 판정은 품질(C-2), 의미 라벨은 주석(B-2)과 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (보고서·검사표는 사람 눈엔 읽혀도 AI는 표·문단 구조를 못 읽어 그대로 못 씀)
   2.2 기대 효과 (문서 속 표·수치·내용을 AI가 검색·분석에 바로 활용 가능)
3. 무엇을 갖추나
   3.1 문서 유형별 추출 기준 (PDF=문단·표·페이지 / PPT=제목·본문·표 / Excel=시트·셀·병합 구조)
   3.2 구조화 결과물 (텍스트·표·수치를 검색·분석 가능한 형태로 — 원본 위치 정보도 함께 보존해 되짚기 가능)
   3.3 변환 안정장치 (양식이 바뀌어도 안 깨지게: 테스트용 문서셋·예외 처리 기준)
4. 어디부터 하나 (전처리 우선순위)   ← 전처리는 다 필요하므로 "순서"
   4.1 전처리 대상 (PDF·PPT·Excel·이미지·표·반정형 보고서 등 AI가 바로 읽기 어려운 디지털 자료)
   4.2 우선순위 기준 (AI 과제에 자주 쓰고·반복 활용 많은 문서부터)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (점검 보고서 표를 사람이 일일이 옮겨 적음 → 전처리하면 표·수치가 구조화돼 AI가 바로 검색·집계)
   5.2 흐름 미리보기 (대상 문서 고르기 → 유형별로 표·텍스트 추출 → 검색 가능한 구조로 변환 → AI 활용 환경에 제공)
6. 어떻게 준비·운영하나 (상세)
   6.1 변환 도구 검토 (문서 유형별 추출 방식 비교·선정)
   6.2 변환 절차 (추출 → 구조화 → 원본 위치 정보 보존 → 검색 인덱스·Vector DB·분석 테이블 등 AI 활용 환경에 적재)
   6.3 운영 (원천 양식이 바뀌면 변환 로직 점검·갱신: 파서 버전·테스트셋·예외 처리) / 역할·책임
7. 다른 주제와의 관계
   7.1 디지털화(F-3 아날로그 전환)·품질(C-2 사용 판정)·주석(B-2 의미 라벨)과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (전처리 완료 문서 수·표·수치 추출 정확도·양식 변경 시 무중단율·AI 활용 적재율)
   8.2 고도화 (추출·구조화 자동화↑ → 사람은 어려운 양식만 검수)
```

## B-2 데이터 해설/주석  *(라벨/주석 — 학습용 의미 부여)*
> 한 줄: AI가 학습할 수 있게 데이터에 라벨·해석을 붙이는 것 (예: 결함 이미지에 결함 유형 표시).
> 핵심 계열사: 두산전자

```
1. 개요
   1.1 데이터 해설/주석이란 (학습용으로 라벨·해석을 붙이는 것)
   1.2 적용 범위와 체계 내 위치 (평가용 정답은 평가데이터(E-3), 개념 관계는 온톨로지와 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (라벨이 없거나 기준이 제각각이면 AI 학습 품질이 떨어짐)
   2.2 기대 효과 (분류 자동화·판단 기준 일관·재작업 감소)
3. 무엇을 갖추나
   3.1 분류 체계(taxonomy)·라벨 정의 (결함·원인·조치 유형)
   3.2 주석 기준·사례집 (헷갈리는 경우의 판단 기준)
   3.3 라벨 이력(버전) 관리 (누가·언제·왜 바꿨나)
4. 어디부터 하나 (대상 선정)
   4.1 주석 대상 (결함 이미지·고객 클레임·원인 분석 등)
   4.2 우선순위·규모 (자주 쓰고 영향 큰 것부터)
5. 예시 시나리오 (한눈에)   ← How 앞 ★
   5.1 적용 전/후 (결함 이미지를 사람이 일일이 분류 → 라벨을 달면 AI가 자동 분류, 검사 시간↓·기준 일관↑)
   5.2 흐름 미리보기 (분류 체계 정하기 → AI가 1차 라벨 → 사람은 헷갈리는 것만 검수 → 학습 데이터 완성)
6. 어떻게 준비·운영하나 (상세)
   6.1 분류 체계(taxonomy) 만드는 기준 (유형 간 경계·중복 없이)
   6.2 라벨링 시키는 법 (AI가 1차 라벨 + 가이드라인 + 사람 검수 워크플로)
   6.3 운영 (라벨 변경·버전 관리) / 역할·책임
7. 다른 주제와의 관계
   7.1 평가 정답(E-3)·개념 관계(B-3)·용어 뜻(A-3)과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (라벨 정확도·검수 합격률·작업 생산성)
   8.2 고도화 (AI 1차 라벨 비중↑ → 사람은 어려운 것만 검수)
```

7. 다른 주제와의 관계
   7.1 Glossary(단어 뜻)·주석(분류 라벨)·AI 검색/에이전트와의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (관계 채움 정도·추천 적중률)

# C. Trustworthy — 신뢰·관측 (믿고 쓸 수 있게)

## C-1 데이터 Observability  *(기본적으로 다 필요한 주제 — "어디부터 관측하나"가 핵심)*
> 한 줄: 데이터가 흐르는 동안 지연·누락·이상 징후를 실시간으로 감지하고 바로 알려주는 "데이터용 관제실".
> 핵심 계열사: 두산밥캣

```
1. 개요
   1.1 데이터 Observability란 (운영 중 데이터 상태를 지켜보고 알려주는 관제실)
   1.2 적용 범위와 체계 내 위치 (실시간 감지가 C-1, 사후 추적은 계통(C-3), 사용 판정은 품질관리(C-2)와 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (수집이 멈췄거나 값이 튀어도 모른 채 AI가 잘못된 데이터를 그대로 씀)
   2.2 기대 효과 (이상을 빨리 잡아 AI 오작동·잘못된 보고를 사전에 막음)
3. 무엇을 갖추나
   3.1 관측 지표 (수집 실패·지연, 건수 급변, 빈칸(결측) 증가, 값 분포 변화)
   3.2 이상 판단 기준 (고정 한계값 vs 평소 추세 대비 변화)
   3.3 알림 체계 (심각도 등급별로 누구에게 어떻게 알릴지)
   3.4 원인 구분용 기록 (데이터·모델·Prompt 버전을 함께 남겨 "데이터 탓인지"를 가림)
4. 관측 대상 우선순위   ← 다 필요하므로 "어디부터·얼마나 촘촘히"
   4.1 기본 필수 (AI 서비스·핵심 리포트로 바로 흘러가는 데이터 흐름부터)
   4.2 적용 판단 (변동이 잦거나 멈추면 타격이 큰 흐름을 더 촘촘히)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (센서 수집이 새벽에 끊긴 걸 아침에야 발견 → 관측을 걸면 멈춘 즉시 알림이 떠 당일 대응)
   5.2 흐름 미리보기 (관측 대상 고르기 → 지표·기준 설정 → 이상 시 등급별 알림 → 담당자 확인·조치)
6. 어떻게 준비·운영하나 (상세)
   6.1 모니터링 도구 검토 (대시보드·알림 연동)
   6.2 구축 절차 (대상 흐름 연결 → 정상 기준선 잡기 → 알림 규칙 설정 → 시범 운영)
   6.3 운영 ★ 알림 피로 최소화 (사소한 알림 줄이기·등급 조정·중복 묶기 — 진짜 중요한 알림만 살아남게)
7. 다른 주제와의 관계
   7.1 사후 추적(C-3)·사용 판정(C-2)·복구 조치(F-1 운영관리)와의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (이상 감지까지 걸린 시간·놓친 건수·오탐률·핵심 흐름 관측 범위)
   8.2 고도화 (평소 패턴을 학습해 임계값 자동 조정 → 오탐·알림 피로↓)
```

## C-2 데이터 품질 관리  *(기본적으로 다 필요한 주제 — "어디부터 게이트를 거나"가 핵심)*
> 한 줄: 데이터를 AI에 넣기 전 "이거 써도 되는가"를 판정하고, 쓸 자격(접근 권한)까지 통제하는 마지막 관문.
> 핵심 계열사: 두산퓨얼셀

```
1. 개요
   1.1 데이터 품질 관리란 (AI 투입 전 "쓸 수 있는가"를 판정하는 통제 관문)
   1.2 적용 범위와 체계 내 위치 (사용 판정·접근 차단이 C-2, 마스킹은 보안(F-4), 이상 감지는 Observability(C-1)와 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (빈칸·오류·오래된 데이터가 그대로 들어가 AI 답이 틀리고, 봐선 안 될 데이터까지 학습됨)
   2.2 기대 효과 (믿을 수 있는 데이터만 AI에 들어가 답 신뢰도↑·사고 위험↓)
3. 무엇을 갖추나
   3.1 품질 기준 (빠짐없음·정확·일관·최신·유효 — 업무별 합격선)
   3.2 사용 금지 기준 (출처 불명·미승인·기준 미달·목적 외 데이터는 AI 사용 제한)
   3.3 접근 권한 정책 (역할·목적·민감도에 따라 조회/분석/AI 학습/반출 가능 여부 구분)
   3.4 예외 승인 항목 (기준 미달인데 꼭 써야 할 때: 승인자·사유·기간·범위 기록)
4. 게이트 대상 우선순위   ← 다 필요하므로 "어디부터 게이트를 거나"
   4.1 기본 필수 (AI 학습·RAG·자동화로 바로 투입되는 데이터부터 게이트)
   4.2 적용 판단 (틀리면 영향이 큰 데이터일수록 기준을 엄격히)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (오래된 검사 데이터가 섞여 AI가 엉뚱한 판정 → 게이트를 걸면 기준 미달 데이터는 투입 전 자동 차단)
   5.2 흐름 미리보기 (품질 기준·권한 정해두기 → 투입 직전 자동 점검(게이트) → 통과만 AI로, 미달은 차단·예외 승인)
6. 어떻게 준비·운영하나 (상세)
   6.1 품질 점검 도구 검토 (자동 검사·게이트 연동)
   6.2 구축 절차 ★ 룰 기반 자동 판정 우선 (사람이 일일이 보지 않게, 규칙으로 자동 합/불 판정 → 애매한 것만 사람)
   6.3 운영 (기준·권한 갱신, 예외 승인 관리) / 역할·책임
7. 다른 주제와의 관계
   7.1 접근 차단(C-2)·마스킹(F-4 보안)·이상 감지(C-1)와의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (품질 기준 통과율·권한 위반 건수·미승인 사용 차단 건수·개선 리드타임)
   8.2 고도화 (자동 판정 규칙 비중↑ → 사람은 예외만 검토)
```

## C-3 데이터 계통 Lineage  *(기본 필수 — 단, 추적 범위는 골라서 정함)*
> 한 줄: 데이터가 어디서 와서(출처) → 어떻게 바뀌어 → AI 답의 근거가 됐는지, 그 "이동 경로 지도"를 남기는 것.
> 핵심 계열사: 두산로보틱스

```
1. 개요
   1.1 데이터 계통(Lineage)이란 (출처→변환→AI 답변 근거까지 이어지는 경로 기록)
   1.2 적용 범위와 체계 내 위치 (사후 경로 추적이 C-3, 실시간 이상 감지는 Observability(C-1), 보존·폐기는 생애주기(F-2)와 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (AI가 내놓은 답·보고서가 "어떤 데이터에서 나왔는지" 못 짚어 신뢰·검증이 안 됨)
   2.2 기대 효과 (답의 근거를 즉시 역추적, 데이터 바뀔 때 영향 범위 파악, 감사 대응 가능)
3. 무엇을 갖추나
   3.1 추적 대상 흐름 (원천 데이터→전처리→데이터셋→리포트→RAG 인덱스→AI 답변)
   3.2 변환 이력 기록 (병합·필터·파생 생성 등 주요 가공 단계)
   3.3 AI 답변 근거 연결 (그 답이 참조한 문서·청크·데이터셋 기록)
   3.4 영향도·감사 증빙 (변경 시 영향받는 곳 / 출처·이력 증빙 리포트)
4. 추적 대상 우선순위 (AI 답변 직결 흐름부터)   ← 기본 필수, 범위는 선택
   4.1 기본 필수 (AI 답변·핵심 보고서로 바로 이어지는 흐름부터 추적)
   4.2 적용 판단 (감사·규제 대상이거나 틀리면 영향 큰 흐름을 더 깊게)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (AI 점검 리포트의 근거를 못 찾아 신뢰 불가 → 계통을 남기면 어느 센서·문서에서 나왔는지 클릭 한 번에 추적)
   5.2 흐름 미리보기 (추적할 흐름 고르기 → 출처·변환·근거 연결 기록 → AI 답에 근거 링크 → 역추적·영향분석)
6. 어떻게 준비·운영하나 (상세)
   6.1 계통 수집 도구 검토 (변환 로그·근거 자동 수집)
   6.2 구축 절차 (대상 흐름 정의 → 출처·변환 단계 연결 → AI 답변 근거 로깅 → 검증)
   6.3 운영 (흐름·로직 바뀌면 계통 갱신, 근거 보존) / 역할·책임
7. 다른 주제와의 관계
   7.1 이상 탐지(C-1)·보존 폐기(F-2 생애주기)와의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (계통 연결 범위·근거 추적 성공률·영향분석 소요시간·감사 대응 시간)
   8.2 고도화 (변환·근거 연결을 자동 수집 → 사람은 누락만 보완)
```

# D. Actionable — 활용·실행 (실행에 쓸 수 있게)

## D-1 Physical 데이터  *(기본 필수 — 설비 데이터 범위가 핵심)*
> 한 줄: 센서·설비가 쏟아내는 현장 신호를 AI가 바로 쓸 수 있게 시간·단위·설비ID를 맞춰 연결하는 것.
> 핵심 계열사: 두산에너빌리티

```
1. 개요
   1.1 Physical 데이터란 (센서·설비·현장 시스템이 만드는 신호를 AI 입력으로 연결)
   1.2 적용 범위와 체계 내 위치 (사용해도 되는지 최종 판정은 사용성(C-2), 문서 전처리는 비정형(B-1)과 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (시간·단위·설비 이름이 제각각이라 데이터를 합치거나 비교할 수 없음)
   2.2 기대 효과 (이상 감지·예지보전·품질 분석에 현장 데이터를 바로 활용)
3. 무엇을 갖추나
   3.1 수집 대상 목록 (센서값·설비 상태·검사 장비값·알람 로그·작업 조건·환경 데이터)
   3.2 표준 3종 맞추기 (시간 동기화·단위·설비ID/공정ID·샘플링 주기 — "언제·무슨 단위·어느 설비"를 통일)
   3.3 기초 정제 기준 (센서 잡음·결측·이상값·신호 끊김을 AI에 넣기 전 1차 정리)
4. 수집 대상 우선순위 (핵심 설비부터)   ← 기본 필수이므로 "전 태그 일괄 X, 순서"
   4.1 우선 대상 (고장·품질 영향 큰 핵심 설비·핵심 태그부터)
   4.2 처리 방식 구분 (즉시 대응용 실시간 vs 분석·리포팅용 배치 — 급한 신호와 모아서 보는 신호를 나눔)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (설비마다 시간·단위가 달라 합산 불가 → 표준을 맞추면 여러 설비를 한 화면에서 비교·이상 감지)
   5.2 흐름 미리보기 (핵심 설비 태그 선정 → 수집 연결 → 시간·단위·설비ID 표준화 → 잡음·결측 1차 정제 → AI 입력)
6. 어떻게 준비·운영하나 (상세)
   6.1 수집·연결 방식 (설비·센서·Edge·IoT 플랫폼·MES/QMS 연결, 실시간/파일/수동 입력 구분)
   6.2 구축 절차 (대상 태그 정의 → 수집 연결 → 시간·단위·설비ID 표준화 → 기초 정제 → 검증)
   6.3 운영 (태그 추가·설비 교체 시 표준 갱신, 정제 기준 점검) / 역할·책임
7. 다른 주제와의 관계
   7.1 사용 가능 최종 판정(C-2)·문서 전처리(B-1)·복구/백업(F-1)과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (표준화율·결측/이상값 비율·실시간 수집 지연)
   8.2 고도화 (이상값·결측 자동 보정, 신규 태그 자동 표준 매핑)
```

## D-2 API/Tool 연계 데이터  *(선택 적용 — Agent를 쓸 때만)*
> 한 줄: AI 에이전트가 외부 시스템을 안전하게 부르도록 "Tool 사용 설명서(기능·입력·출력·제약)"를 정리하는 것.
> 핵심 계열사: 두산로보틱스

```
1. 개요
   1.1 API/Tool 연계 데이터란 (AI 에이전트가 호출할 기능의 설명서·규격)
   1.2 적용 범위와 체계 내 위치 (호출 절차/실행 흐름은 Prompt·Harness(D-3), 로그 활용·개선은 운영개선(E-4)과 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (설명이 부실하면 AI가 엉뚱한 Tool을 부르거나 위험한 동작을 실행)
   2.2 기대 효과 (에이전트가 맞는 도구를 정확·안전하게 호출, 오작동·사고 예방)
3. 무엇을 갖추나
   3.1 Tool 후보 목록 (조회·계산·보고서 생성·시스템 검색·업무 실행)
   3.2 Tool 설명서 (이름·기능·목적·입력값·출력값·제약·호출 예시)
   3.3 입출력 규격 (필수값·데이터 타입·허용값·응답 구조·오류 응답 — 무엇을 넣고 무엇이 나오는지 약속)
4. 연계 대상 선정 (조회형부터, 위험도 분류)   ← 선택 적용이므로 "무엇을 연결하나"
   4.1 적용 판단 (에이전트가 실제로 외부 시스템을 부를 때만, 단순 답변형이면 불필요)
   4.2 위험도 분류·우선순위 (조회형 → 추천형 → 실행형 → 외부 전송형, 위험할수록 사람 승인)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (에이전트가 어떤 시스템을 어떻게 부를지 몰라 실패 → 설명서를 주면 정확히 조회하고 위험 동작은 승인 거침)
   5.2 흐름 미리보기 (조회형 Tool 골라 설명서 작성 → 입출력 규격 정의 → 위험도 분류·승인 규칙 → 등록 후 에이전트가 호출)
6. 어떻게 준비·운영하나 (상세)
   6.1 설명서·규격 작성 도구·표준 검토 (Tool 등록소 Registry 형태)
   6.2 구축 절차 (Tool 식별 → 설명서·입출력 규격 작성 → 위험도/승인 규칙 → 호출 테스트 → 등록)
   6.3 운영 — 버전·오류 관리 (기능·규격·응답이 바뀌면 버전·변경 이력 관리해 오호출 방지)
   6.4 호출 기록 (호출 시점·입력·출력·주체·성공/실패·오류를 로그로 남김 — 활용·개선은 E-4)
7. 다른 주제와의 관계
   7.1 호출 절차·실행 흐름(D-3)·로그 활용 개선(E-4)·거버넌스 승인과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (호출 성공률·오호출률·위험 Tool 승인 준수율)
   8.2 고도화 (시스템 명세에서 설명서 초안 자동 생성 → 사람 검수)
```

## D-3 Prompt/Harness 자산화  *(선택 적용 — 반복·고영향 업무만)*
> 한 줄: 사람이 하던 업무 절차·판단 기준을 AI가 그대로 실행할 Prompt로 정리하고 버전으로 관리하는 것.
> 핵심 계열사: 두산밥캣

```
1. 개요
   1.1 Prompt/Harness 자산화란 (업무 지시·판단을 AI 실행용 Prompt로 구조화, Harness = 묶음 패키지)
   1.2 적용 범위와 체계 내 위치 (잘했는지 평가 기준은 평가데이터(E-3), Tool 명세 자체는 API/Tool(D-2)과 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (사람마다 Prompt를 따로 만들어 결과가 들쭉날쭉하고 노하우가 흩어짐)
   2.2 기대 효과 (검증된 Prompt를 공용 자산으로 재사용 → 결과 일관·재작업 감소·노하우 축적)
3. 무엇을 갖추나
   3.1 자산 목록 (업무별 Prompt·시스템 지시문·역할 정의·판단 기준·출력 형식·예외 처리)
   3.2 업무→Prompt 변환 틀 (입력·판단 단계·참조 기준·출력 형식·금지 행동으로 나눠 적기)
   3.3 Harness (Prompt + 입력 데이터 + Tool 호출 순서 + 출력 형식을 묶은 반복 실행 단위)
4. ★ 무엇을 자산화하나 (반복·고영향 업무만)   ← 선택 적용의 핵심 판단
   4.1 적용 판단 (자주 쓰거나 업무 판단에 영향이 큰 것만, 일회성·실험용은 제외)
   4.2 우선순위 (현업 노하우가 담긴 반복 업무부터 작게 시작)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (담당자마다 Prompt가 달라 결과가 제각각 → 검증된 공용 Prompt로 누가 써도 같은 품질)
   5.2 흐름 미리보기 (반복·고영향 업무 선정 → 업무 절차를 Prompt로 변환 → 의존성 연결 → 버전 등록 → Harness로 재사용)
6. 어떻게 준비·운영하나 (상세)
   6.1 관리 도구·표준 검토 (Prompt 등록소 Registry·버전 관리 도구)
   6.2 구축 절차 (대상 업무 선정 → 절차→Prompt 변환 → 의존성 연결 → 테스트 → Harness 패키징)
   6.3 버전·의존성 관리 (변경 이력·사유·승인자 기록, 어떤 데이터·용어(Glossary)·Tool에 의존하는지 연결)
   6.4 운영 — 성능 저하 감지·개선 (평가 결과·사용자 수정·실패 사례·정책 변경으로 개선 판단, 영향 큰 것은 승인 후 배포)
7. 다른 주제와의 관계
   7.1 평가 기준(E-3)·Tool 명세(D-2)·용어(Glossary, A-3)와의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (재사용률·사용자 수정률·업무 결과 일관성)
   8.2 고도화 (성능 저하 자동 감지 → 개선안 초안 추천, 사람 검수 후 배포)
```

# E. Sustainable — 가치화·순환 (지속·순환되게)

## E-1 데이터 Product화  *(골라서 하는 주제 — 무엇을 Product화하나가 핵심)*
> 한 줄: 자주 쓰는 데이터를 책임자·품질·제공방식을 갖춘 "상품"으로 한 번 잘 만들어 여러 팀이 갖다 쓰게 하는 것.
> 핵심 계열사: 두산테스나

```
1. 개요
   1.1 데이터 Product화란 (매번 새로 뽑지 말고, 한 번 잘 만들어 공유하는 데이터 "상품")
   1.2 적용 범위와 체계 내 위치 (검색·등록은 카탈로그(A-1), 쓸 수 있나 판정은 품질(C-2)과 구분 — E-1은 "잘 만들어 책임지고 제공")
2. 왜 필요한가
   2.1 현업 Pain Point (같은 데이터를 팀마다 따로 뽑아 버전·기준이 제각각, 담당자도 불분명)
   2.2 기대 효과 (한 번 만든 데이터를 여러 팀·AI 과제가 재사용, 중복 작업·오류 감소)
3. 무엇을 갖추나
   3.1 Product 명세서 (누가 쓰나·무슨 데이터·품질 기준·갱신 주기·사용 조건)
   3.2 책임자(Owner)와 제공 약속 (언제까지·어떤 수준으로 줄지)
   3.3 소비 방식 (테이블·파일·API·대시보드·AI 검색 소스 중 업무에 맞게)
4. ★ 무엇을 Product화하나 (대상 선정 — 과잉 금지)
   4.1 적용 판단 (반복 요청 많고·여러 팀이 쓰고·AI 과제에 자주 쓰는 데이터만 — 한두 번 쓰는 데이터는 굳이 안 함)
   4.2 우선순위 (요청 빈도·사용 팀 수·중복 제거 효과가 큰 것부터)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (월 보고용 수율 데이터를 팀마다 따로 집계해 숫자가 어긋남 → "수율 데이터 Product" 하나로 통일하면 누구나 같은 기준으로 바로 사용)
   5.2 흐름 미리보기 (대상 고르기 → 명세서·Owner 정하기 → 소비 방식으로 제공 → 사용 현황 보고 개선)
6. 어떻게 준비·운영하나 (상세)
   6.1 만드는 절차 (대상 선정 → 명세서 작성 → 품질·갱신 기준 합의 → 소비 방식으로 공개)
   6.2 계열사마다 다른 시스템에서 같은 Product 제공하기 (원천이 달라도 사용자에겐 동일한 모습으로)
   6.3 운영 (Owner의 책임: 품질 유지·변경 공지·문의 대응·개선 요청 관리·버전/폐기 판단)
7. 다른 주제와의 관계
   7.1 카탈로그(검색·등록, A-1)·품질(사용 판정, C-2)·생애주기(보존·폐기, F-2)와의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (재사용률·사용 팀 수·중복 데이터 감소·AI 과제 활용 건수·요청 처리시간 단축)
   8.2 고도화 (자주 쓰는 Product를 표준 패키지로, 사용 로그로 개선 우선순위 자동 파악)
```

## E-2 합성데이터  *(골라서 하는 주제 + 만드는 방식이 핵심)*
> 한 줄: 실제 데이터가 부족할 때 인공으로 만들어 채우는 데이터.
> 핵심 계열사: 두산전자

```
1. 개요
   1.1 합성데이터란 (인공으로 만든 데이터 — 원본 대체가 아님)
   1.2 적용 범위와 체계 내 위치
2. 왜 필요한가
   2.1 현업 Pain Point (불량 사례가 너무 적거나, 보안 때문에 원본을 못 씀)
   2.2 기대 효과 (부족한 사례를 채워 학습·검증 가능)
3. 무엇을 갖추나
   3.1 무엇을 합성할지 (불량·장애·예외 사례)
   3.2 만드는 방식 (통계·시뮬레이션·생성모델)
   3.3 검증 항목 + "합성" 표시
4. ★ 언제 합성하나   ← 실데이터로 충분하면 안 함
   4.1 적용 판단 (데이터가 부족·희소하거나 보안으로 원본을 못 쓸 때만)
   4.2 비교 기준이 될 실데이터 먼저 확보
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (특정 불량 샘플이 10건뿐이라 학습 불가 → 합성으로 보강하면 검사 AI 정확도↑)
   5.2 흐름 미리보기 (만들 대상 정하기 → 방식 골라 생성 → 검증 → "합성" 표시 후 학습)
6. 어떻게 만들고 운영하나 (상세)
   6.1 만드는 방식·도구 비교·선정 (시범 적용)
   6.2 만드는 절차 (생성 → 검증 → "합성" 표시·등록)
   6.3 운영 (재생성, 위험 점검: 편향·현실 왜곡·재식별)
7. 다른 주제와의 관계
   7.1 평가셋(E-3)·비식별(F-4)·사용 판정(C-2)과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (실데이터와 닮은 정도·모델 성능 개선)
   8.2 고도화 (검증 자동화)
```

## E-3 AI 평가 데이터  *(골라서 하는 주제 — 어떤 과제에 평가가 필요한가가 핵심)*
> 한 줄: AI가 잘하는지 못하는지를 객관적으로 판단하기 위한 "정답셋"과 채점 기준.
> 핵심 계열사: 두산로보틱스

```
1. 개요
   1.1 AI 평가 데이터란 (AI 답을 채점할 정답셋 + 무엇을 보고 맞다 할지의 기준)
   1.2 적용 범위와 체계 내 위치 (학습용 라벨은 주석(B-2), AI 지시문은 Prompt(D-3)와 구분 — E-3은 "성능 채점용")
2. 왜 필요한가
   2.1 현업 Pain Point ("AI가 잘 되는 것 같다"는 감(感)뿐, 좋아졌는지·나빠졌는지 숫자로 말 못 함)
   2.2 기대 효과 (성능을 숫자로 비교, 바꾼 뒤 더 나빠졌는지 바로 확인 가능)
3. 무엇을 갖추나
   3.1 정답셋(Gold Set) (기준 질문·입력 데이터·기대 답변·정답 근거)
   3.2 채점 기준 (정확성·근거 제시·완전성·안전성 등 업무에 맞는 항목)
   3.3 전문가 코멘트 (왜 맞고 틀렸는지 — 단순 O/X를 넘어 이유 기록)
4. ★ 어떤 AI 과제에 평가가 필요한가 (대상 선정)
   4.1 적용 판단 (틀리면 손해가 큰 과제·검증이 꼭 필요한 과제부터 — 가벼운 보조 기능은 약식으로)
   4.2 우선순위 (AI 검색·원인 분석·자동 보고서처럼 답의 옳고 그름이 중요한 과제부터)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (Prompt를 바꿨는데 좋아졌는지 알 수 없어 불안 → 정답셋 100문항으로 채점하면 "85→78점, 더 나빠짐"을 바로 확인)
   5.2 흐름 미리보기 (평가할 과제 정하기 → 정답셋·채점 기준 만들기 → 자동 채점 → 점수 비교로 채택 결정)
6. 어떻게 준비·운영하나 (상세)
   6.1 평가 방식·도구 선정 (정답 매칭 / 사람 평가 / AI가 채점(LLM 평가) 중 과제에 맞게)
   6.2 만드는 절차 (실제 업무 사례로 정답셋 만들기 → 채점 기준 합의 → 자동 채점 연결)
   6.3 운영 (바꿀 때마다 다시 채점·이전에 맞던 게 또 틀리지 않는지 점검 / 새 실패 사례를 정답셋에 추가)
7. 다른 주제와의 관계
   7.1 학습 라벨(B-2)·Prompt(D-3)·피드백 루프(E-4)와의 경계 (실패 사례는 E-4에서 받아 정답셋 보강)
8. 성과 지표·고도화
   8.1 성과 지표 (정답셋 규모·평가 자동화율·변경 시 성능 변화 포착 여부)
   8.2 고도화 (사람 평가 → AI 1차 채점 + 사람은 애매한 것만, 정답셋 자동 보강)
```

## E-4 데이터 Feedback Loop  *(기본 필수 주제 — 어디서 피드백을 모으나가 핵심)*
> 한 줄: AI를 쓰다 나온 오류·사용자 피드백을 모아 "진짜 원인" 주제로 보내 데이터·라벨·Prompt를 고치는 되돌림 고리.
> 핵심 계열사: 두산밥캣

```
1. 개요
   1.1 데이터 Feedback Loop란 (AI 운영 중 나온 문제를 모아 개선으로 되돌리는 고리)
   1.2 적용 범위와 체계 내 위치 (★ E-4는 "증상을 모아 원인 주제로 라우팅" — 고치는 일 자체는 각 주제에서)
2. 왜 필요한가
   2.1 현업 Pain Point (같은 오류가 반복되는데 어디를 고쳐야 할지 몰라 방치됨)
   2.2 기대 효과 (오류가 개선으로 이어지고, 반복 실수가 점점 줄어듦)
3. 무엇을 갖추나
   3.1 피드백 수집 항목 (틀린 답·사용자 수정·근거 누락·Tool 호출 실패·사람 승인/거절)
   3.2 오류 유형 분류 (데이터 문제 / 라벨 문제 / Prompt 문제 / Tool 문제 / 권한 문제)
   3.3 개선 과제 연결판 (각 유형을 어느 주제로 보낼지의 지도)
4. ★ 어디서 피드백을 모으나 (수집 지점 — 우선순위)
   4.1 기본 수집 지점 (AI가 답을 주는 모든 화면·사용자 수정·실패 로그에서 빠짐없이)
   4.2 우선순위 (사용량 많고·틀리면 손해 큰 과제의 피드백부터 먼저 처리)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (AI가 자꾸 엉뚱한 답을 내는데 원인을 못 찾아 그대로 둠 → 피드백을 모아 분류하니 "근거 데이터 누락"이 원인 → A영역 정비로 해결)
   5.2 흐름 미리보기 (피드백 모으기 → 원인 유형 분류 → 해당 주제로 라우팅 → 고친 뒤 효과 확인)
6. 어떻게 준비·운영하나 (상세)
   6.1 수집·분류 절차 (피드백 자동 수집 → 원인 유형 태깅 → 개선 Backlog로 정리)
   6.2 ★ 원인 주제로 라우팅 (데이터 누락→A / 라벨 부족→B-2 / 평가 실패→E-3 / Prompt 문제→D-3 / Tool 문제→D-2 / 권한 문제→C-2)
   6.3 운영 (자동/수동 반영 기준: 저위험은 자동 반영 후보, 고위험·고객 영향·보안은 전문가 검토 후 반영)
7. 다른 주제와의 관계
   7.1 각 원인 주제로의 연결 (A·B-2·C-2·D-2·D-3·E-3) — E-4는 "허브", 실제 수정은 각 주제
8. 성과 지표·고도화
   8.1 성과 지표 (반복 오류 감소·피드백 반영 리드타임·사용자 수정률 감소·AI 성공률 개선)
   8.2 고도화 (오류 자동 분류·라우팅 정확도↑, 자주 나오는 실패는 정답셋(E-3)에 자동 반영)
```

# F. Governed — 운영·기반 (통제·기반이 갖춰지게)

## F-1 데이터 운영관리 (DataOps)  *(기본적으로 다 필요한 주제 — "어느 파이프라인부터/얼마나" 차등)*
> 한 줄: 데이터가 흐르는 길(수집→정제→변환→제공)이 멈추지 않게 실행·복구·변경을 관리하는 것 — 장애 나면 되살리고, 바꿀 때 사고 안 나게.
> 핵심 계열사: 두산에너빌리티

```
1. 개요
   1.1 데이터 운영관리(DataOps)란 (데이터 흐르는 길을 안 멈추게 운영)
   1.2 적용 범위와 체계 내 위치 (이상 감지는 Observability(C-1), 추적은 Lineage(C-3), 품질 판정은 품질관리(C-2)와 구분 — 여기는 '실행·복구·변경')
2. 왜 필요한가
   2.1 현업 Pain Point (야간 수집이 조용히 실패 → 다음 날 AI가 옛 데이터로 잘못 답함)
   2.2 기대 효과 (멈춰도 빨리 되살리고, 바꿔도 사고가 안 남 → AI 결과가 끊기지 않음)
3. 무엇을 갖추나
   3.1 운영 표준 (수집·정제·변환·적재·제공의 실행 주기와 배포 절차)
   3.2 장애 대응 절차(Runbook) (실패하면 다시 돌리기·되돌리기(롤백)·담당자 알림·복구 확인)
   3.3 변경 관리 (양식·연결방식·제공방식이 바뀔 때 미리 알리고 영향 점검)
4. 어디부터/얼마나 하나 (운영 대상 파이프라인 우선순위)   ← 다 필요하므로 "차등"
   4.1 운영 대상 고르기 (AI 서비스·RAG·리포팅에 쓰는 핵심 흐름)
   4.2 중요도 차등 (멈추면 타격 큰 흐름은 촘촘히, 그 외는 가볍게)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (야간 수집 실패를 아침에야 발견 → 운영 체계가 있으면 즉시 재처리·알림으로 그날 안에 정상화)
   5.2 흐름 미리보기 (핵심 흐름 정하기 → 실행·배포 표준화 → 실패 시 자동 재처리·알림 → 복구 확인 → 변경은 사전 공지)
6. 어떻게 준비·운영하나 (상세)
   6.1 운영·자동화 도구 검토 (스케줄·배포·복구 자동화)
   6.2 구축 절차 (핵심 파이프라인 등록 → 실행/배포 표준 → 장애 대응 절차 마련 → 변경 절차 연결)
   6.3 운영 (정기 점검·재처리·변경 승인) / 역할·책임
7. 다른 주제와의 관계
   7.1 이상 감지(C-1)·추적(C-3)·품질 판정(C-2)과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (파이프라인 성공률·평균 복구 시간·약속 시간(SLA) 준수율·재처리 건수·제공 지연)
   8.2 고도화 (이상 감지(C-1)와 연동해 자동 복구 범위 확대)
```

## F-2 데이터 생애주기 관리  *(기본적으로 다 필요한 주제 — "무엇을 언제까지/언제 버릴지")*
> 한 줄: 데이터를 언제까지 보관하고 언제 버릴지 정하는 것 — 단, 재사용 가치 있는 과정 데이터(실패한 실험 등)는 일부러 남긴다.
> 핵심 계열사: 두산퓨얼셀

```
1. 개요
   1.1 데이터 생애주기 관리란 (생성→사용→보관→정리(아카이빙)→폐기의 시간축 관리)
   1.2 적용 범위와 체계 내 위치 (무엇이 어떻게 바뀌었나=변환 이력(C-3), 민감정보 보호=보안(F-4)과 구분 — 여기는 '얼마나 두고 언제 버리나')
2. 왜 필요한가
   2.1 현업 Pain Point (다 쌓아두면 비용·위험↑, 막 버리면 나중에 AI가 학습할 데이터가 사라짐)
   2.2 기대 효과 (보관 비용·보안 위험은 줄이고, 쓸모 있는 데이터는 남겨 AI 재사용)
3. 무엇을 갖추나
   3.1 생애주기 단계 정의 (생성·사용·보관·아카이빙·폐기·재사용 + 단계별 책임자)
   3.2 보존 기간 기준 (법적 의무·업무 가치·AI 재사용 가치·비용으로 유형별 기간 결정)
   3.3 ★ 과정 데이터 보존 기준 (실패한 실험·중간 결과·검토 의견처럼 재사용 가치 있는 것은 일부러 보존)
4. 어디부터 하나 (보존/폐기 대상 우선순위)   ← 다 필요하므로 "대상 선정"
   4.1 우선 정리 대상 (보존 기간 끝난 것·안 쓰는 것부터 폐기/아카이빙)
   4.2 우선 보존 대상 (AI 학습 가치·감사 필요성 큰 것은 먼저 챙겨 보존)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (실패 실험 기록을 그때그때 삭제 → 보존 기준이 있으면 그 데이터로 불량 원인 AI를 학습 가능)
   5.2 흐름 미리보기 (단계·기간 정하기 → 폐기/아카이빙 자동화 → 과정 데이터는 따로 보존 → 저장소 계층으로 비용 관리)
6. 어떻게 준비·운영하나 (상세)
   6.1 보존·폐기 자동화 도구 검토 (저장소 계층화(Hot/Warm/Cold) 포함)
   6.2 구축 절차 (단계 정의 → 유형별 보존 기간 → 폐기/아카이빙 절차 → 과정 데이터 보존 규칙)
   6.3 운영 (폐기 승인·복원 절차·보존 예외 관리) / 역할·책임
7. 다른 주제와의 관계
   7.1 변환 이력(C-3)·보안(F-4)·품질(C-2)과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (보존 정책 적용률·미사용 데이터 비중·저장 비용 절감·과정 데이터 재사용 건수)
   8.2 고도화 (사용 빈도를 보고 보관 등급을 자동 추천 → 사람 승인)
```

## F-3 데이터 디지털화  *(골라서 하는 주제 — "무엇부터 디지털로 바꾸나"가 핵심)*
> 한 줄: 종이·수기·사진·머릿속 노하우(암묵지)를 디지털로 바꾸고, 앞으로는 처음부터 디지털로 입력(Digital-first)하게 만드는 것.
> 핵심 계열사: 두산에너빌리티

```
1. 개요
   1.1 데이터 디지털화란 (아날로그 자산을 AI가 쓸 수 있는 디지털로 전환)
   1.2 적용 범위와 체계 내 위치 (이미 디지털인 문서를 AI용으로 다듬는 건 전처리(B-1), 쓸 수 있나 판정은 품질(C-2)과 구분 — 여기는 '아날로그→디지털 전환')
2. 왜 필요한가
   2.1 현업 Pain Point (수기 검사표·현장 노하우가 종이·머릿속에만 있어 AI가 아예 못 봄)
   2.2 기대 효과 (사라질 뻔한 자산을 데이터로 살려 AI 학습·검색에 활용)
3. 무엇을 갖추나
   3.1 아날로그 자산 목록(Inventory) (종이·수기 검사표·사진·현장 메모·작업자 노하우)
   3.2 전환 방식 (스캔·글자 인식(OCR/ICR)·음성→텍스트·모바일/태블릿 입력·인터뷰 구조화)
   3.3 전환 품질 검증 + Digital-first 기준 (원본 대비 검수 / 앞으로는 처음부터 디지털 입력)
4. 무엇부터 하나 (디지털화 우선순위)   ← 다 못 하므로 "고르기"
   4.1 적용 판단 (소실 위험 크거나 활용 가치 큰 것부터, 안 쓸 자료는 굳이 안 함)
   4.2 작게 시작 (한 업무·한 양식부터 → 효과 확인 후 확대)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (수기 검사표가 캐비닛에만 있어 분석 불가 → 디지털화하면 AI가 불량 패턴을 분석)
   5.2 흐름 미리보기 (자산 조사 → 우선순위 정하기 → 방식 골라 전환·검수 → 자산 등록 → 신규는 Digital-first로 입력)
6. 어떻게 준비·운영하나 (상세)
   6.1 전환 방식·도구 비교·선정 (대상 특성에 맞춰 시범 적용)
   6.2 전환 절차 (조사 → 우선순위 → 전환 → 검수 → 자산 등록)
   6.3 운영 (개인 PC·엑셀 산재 데이터를 공식 자산으로 승급·지속 수집 / Digital-first 정착) / 역할·책임
7. 다른 주제와의 관계
   7.1 전처리(B-1)·품질 판정(C-2)·카탈로그 등록(A-1)과의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (우선 대상 전환율·글자 인식 정확도·검수 합격률·Digital-first 적용률)
   8.2 고도화 (AI가 글자·표를 1차 추출 → 사람은 헷갈리는 것만 검수)
```

## F-4 AI 데이터 보안  *(기본적으로 다 필요한 주제 — "막는" 게 아니라 "쓸 수 있게 변환")*
> 한 줄: 개인·민감정보를 찾아 가리고(마스킹)·가명으로 바꿔(가명화) AI에 안전하게 쓰게 하는 것 — 차단이 목적이 아니라 활용이 목적.
> 핵심 계열사: 두산밥캣

```
1. 개요
   1.1 AI 데이터 보안이란 (민감정보를 탐지·마스킹·가명화해 안전하게 활용)
   1.2 적용 범위와 체계 내 위치 (누가 볼 수 있나 접근 차단은 품질/권한(C-2), 부족분 인공 생성은 합성(E-2)과 구분 — 여기는 '데이터를 안전하게 변환')
2. 왜 필요한가
   2.1 현업 Pain Point (보고서·로그에 고객명·계약가가 섞여 있어 AI에 그냥 못 씀 → 그렇다고 다 막으면 활용도 안 됨)
   2.2 기대 효과 (식별정보는 가리되 업무 정보는 살려 AI를 안전하게 활용)
3. 무엇을 갖추나
   3.1 보호 대상 분류 (개인정보뿐 아니라 고객사명·가격·설계정보·기술기밀·영업비밀)
   3.2 탐지·처리 방식 (정형·비정형 모두에서 탐지 / 삭제·마스킹·가명화·익명화)
   3.3 ★ 활용성 보존 기준(Utility-Preservation) (식별자는 가리되 제품군·공정·결함유형은 남겨 AI 성능 유지)
4. 어디부터 하나 (보안 처리 대상 우선순위)   ← 다 필요하므로 "대상 선정"
   4.1 우선 처리 대상 (AI 학습·RAG·외부 모델에 들어가는, 민감정보 많은 데이터부터)
   4.2 활용 목적별 차등 (학습·RAG·로그·외부 모델 여부에 따라 처리 강도를 다르게)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (고객명이 들어 있어 클레임 데이터를 AI에 못 씀 → 고객명만 가명화하면 원인 분석 AI에 안전하게 활용)
   5.2 흐름 미리보기 (보호 대상 정하기 → 탐지 → 목적별 처리(마스킹·가명화) → 학습 가능 등급 부여 → 재식별 위험 점검)
6. 어떻게 준비·운영하나 (상세)
   6.1 탐지·마스킹 도구 검토 (정규식·사전·이름찾기(NER)·LLM 문맥 탐지 조합)
   6.2 구축 절차 (보호 대상 정의 → 탐지 → 목적별 처리 정책 → 학습 가능 등급 부여)
   6.3 운영 (AI 입력(Prompt)·실행 로그도 저장 전 마스킹 / 재식별 위험 정기 점검) / 역할·책임
7. 다른 주제와의 관계
   7.1 접근 차단·권한(C-2)·합성(E-2)·평가데이터(E-3)와의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (민감정보 탐지율·마스킹 적용률·학습 가능 등급 분류율·재식별 위험 점검 건수)
   8.2 고도화 (LLM 문맥 탐지로 탐지 자동화↑ → 사람은 고위험만 검수)
```
