## 1. Why — 데이터 카탈로그의 필요성
1.1 Key Objectives — ① 데이터 존재 유무 파악 ② 데이터 중복처리 방지 ③ 데이터 신뢰도 확보
1.2 AI Data Pain Points — 존재 유무 파악 불가 / 전처리 실행 여부 확인 불가 / 관리·최신화 어려워 신뢰 낮음
   - 현업 인터뷰 인용(두산 에너빌리티) — Parent-child 수동 추적, 데이터 소유자 감각적 파악

## 2. What — 데이터 카탈로그의 정의·구성
2.1 정의 (1/2) — 조직 내 데이터 Source → 카탈로그 등록 (존재 유무·위치·오너십 식별)
   - 주요 구성: 일반 식별 정보 / AI 데이터(전처리 여부·활용 목표·원천 표기) / 공통 데이터
2.2 정의 (2/2) — 카탈로그 활용 3단계: ① 전체 데이터 확인 ② 데이터 탐색(검색·필터·쿼리) ③ 소유자 확인
2.3 [Backup] OvalEdge Data Catalog — 글로벌 탐색(Horizontal) / 카탈로그 내 탐색(Vertical)
2.4 카탈로그 항목 구성 5축 — Business / Technical / Compliance / Operational / AI Data
2.5 기본 구성 案 (1/2) — Technical·Business 항목 표(고유번호·이름·유형·Source·주소·스키마·소유자·조직·설명·업무분야·태그)
2.6 기본 구성 案 (2/2) — AI(전처리·원천·활용도)·Operational(등록/업데이트일·현황)·Compliance(접근권한·보안·규제) 항목 표
2.7 [Backup] 데이터 Type 상세 — Enum / List / Date / String / List<Enum> 작성 기준

## 3. Who — 주요 담당자 정의
3.1 역할 5종 — 데이터 오너 / 현업 / IT / 보안 / AI 조직 (전담 부재 시 대체 지정)

## 4. How — 데이터 카탈로그 구축 주요 업무
4.1 전체 업무 개요 — 기준 정의 → 데이터 취합 → 보안 검토 → 구축 기획 → 대상 선정 → (구축) → (운영) + 변경 관리 루프
4.2 데이터 현황 조사·대상 선정 (1/6) — 항목별 작성 기준 정의, 업무 분야·단일 row 기준
4.3 데이터 현황 조사·대상 선정 (2/6) — 정형/비정형/AI 데이터 등록·제외 기준
4.4 [Backup] 정형 데이터 사용량 기반 중요 테이블 선별 — 로그 소스(Query/ETL/BI/API/Batch/Access), Tier 1~3·Unknown 분류
4.5 데이터 현황 조사·대상 선정 (3/6) — 비정형 데이터 등록·제외 기준(OneDrive·Local·격리 시스템)
4.6 데이터 현황 조사·대상 선정 (4/6) — 취합 양식 작성, Method 1 수동 등록 / Method 2 자동화(Crawling)
4.7 [Backup] Databricks Unity Catalog 카탈로그 생성 자동화 — 7대 어려움 vs 자동화 기능
4.8 데이터 현황 조사·대상 선정 (5/6) — 기술·솔루션 검토, 보안 검토(내부 학습/외부 환경 운영)
4.9 [Backup] 데이터 카탈로그 서비스 비교 — Databricks Unity Catalog / AWS(Glue·DataZone·Lake Formation)
4.10 데이터 현황 조사·대상 선정 (6/6) — 선행 작업 검토, 등록 대상·우선순위 정의(AI 활용성·중요도·재사용성)
4.11 구축 프로세스 — 취합 데이터 검토(무결성), 카탈로그 구축(To-Be 아키텍처·Legacy 연동·Pipeline 개발)
4.12 운영 (1/4) — 변경 관리: 변경 요청 → 검토·승인 → 반영 (신규 등록/삭제/수정 영향 검토)
4.13 운영 (2/4) — 카탈로그 서비스 흐름도(메타데이터 생성·등록 → 조회 → 서비스 연계)
4.14 운영 (3/4) — Pipeline 운영: 자동화 Pipeline / 수동 등록, 갱신 주기·접근 권한(RBAC·ABAC)·관리 책임
4.15 [Backup] Uber Databook 최신화 방식 — Scheduled Collection vs Event-based Collection
4.16 운영 (4/4) — 사용자 Front-end / 데이터 담당자 Back-end / Admin 기능 구분
