## C-1 데이터 Observability  *(기본적으로 다 필요한 주제 — "어디부터 관측하나"가 핵심)*
> 한 줄: 데이터가 흐르는 동안 지연·누락·이상 징후를 실시간으로 감지하고 바로 알려주는 "데이터용 관제실".
> 핵심 계열사: 두산밥캣

```
1. 개요
   1.1 데이터 Observability란 (운영 중 데이터 상태를 지켜보고 알려주는 관제실)
   1.2 적용 범위와 체계 내 위치 (실시간 감지가 C-1, 사후 추적은 계통(C-3), 사용 판정은 품질관리(C-2)와 구분)
2. 왜 필요한가
   2.1 현업 Pain Point (수집이 멈췄거나 값이 튀어도 모른 채 AI가 잘못된 데이터를 그대로 씀)
   2.2 기대 효과 (이상을 빨리 잡아 AI 오작동·잘못된 보고를 사전에 막음)
3. 무엇을 갖추나
   3.1 관측 지표 (수집 실패·지연, 건수 급변, 빈칸(결측) 증가, 값 분포 변화)
   3.2 이상 판단 기준 (고정 한계값 vs 평소 추세 대비 변화 — KQ2)
   3.3 알림 체계 (심각도 등급별로 누구에게 어떻게 알릴지 — KQ3)
   3.4 원인 구분용 기록 (데이터·모델·Prompt 버전을 함께 남겨 "데이터 탓인지"를 가림 — KQ4)
4. 관측 대상 우선순위   ← 다 필요하므로 "어디부터·얼마나 촘촘히"
   4.1 기본 필수 (AI 서비스·핵심 리포트로 바로 흘러가는 데이터 흐름부터)
   4.2 적용 판단 (변동이 잦거나 멈추면 타격이 큰 흐름을 더 촘촘히)
5. 예시 시나리오 (한눈에)   ← How 앞
   5.1 적용 전/후 (센서 수집이 새벽에 끊긴 걸 아침에야 발견 → 관측을 걸면 멈춘 즉시 알림이 떠 당일 대응)
   5.2 흐름 미리보기 (관측 대상 고르기 → 지표·기준 설정 → 이상 시 등급별 알림 → 담당자 확인·조치)
6. 어떻게 준비·운영하나 (상세)
   6.1 모니터링 도구 검토 (대시보드·알림 연동)
   6.2 구축 절차 (대상 흐름 연결 → 정상 기준선 잡기 → 알림 규칙 설정 → 시범 운영)
   6.3 운영 ★ 알림 피로 최소화 (사소한 알림 줄이기·등급 조정·중복 묶기 — 진짜 중요한 알림만 살아남게)
7. 다른 주제와의 관계
   7.1 사후 추적(C-3)·사용 판정(C-2)·복구 조치(F-1 운영관리)와의 경계
8. 성과 지표·고도화
   8.1 성과 지표 (이상 감지까지 걸린 시간·놓친 건수·오탐률·핵심 흐름 관측 범위)
   8.2 고도화 (평소 패턴을 학습해 임계값 자동 조정 → 오탐·알림 피로↓)
```
