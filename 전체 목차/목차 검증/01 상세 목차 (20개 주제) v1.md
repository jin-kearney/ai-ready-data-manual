```
11. KPI 및 성과 관리 (라벨 정확도·작업자 간 일치도(IAA)·검수 합격률·라벨링 생산성(건/시간)·재작업률·AI 사전라벨 채택률·학습셋 버전 추적률·모델 성능 기여도)
12. 고도화 Roadmap (수기 전수 라벨링 → AI 사전라벨+전수 검수 → AI 1차 라벨+저신뢰·고위험 선별 검수 → 능동 학습(Active Learning) 기반 우선 라벨링 → 자율 라벨 보강)
```

# C. 신뢰·관측 (Trust & Observability)

## C-1 데이터 Observability
> 파일명: `C-1 Observability.md`
> 정의: 데이터 흐름의 지연·누락·변경·이상 징후를 운영 중(실시간) 감지하고 알림하는 모니터링 체계. "지금 데이터에 무슨 일이 일어나고 있는가"를 본다.
> 핵심 계열사 예시: 두산밥캣 (다국가 공장 파이프라인 관측)
> ★ 핵심 원칙: 오탐·중복 알림을 억제해 알림 피로(Alert Fatigue)를 최소화한다. "사용 가능 판정"은 하지 않는다(그것은 C-2).

```
1. 개요
   1.1 데이터 Observability의 정의 ("관제실 CCTV" — 데이터 흐름 실시간 계기판)
   1.2 목적 (이상을 사후가 아니라 발생 순간에 인지)
   1.3 적용 범위 (IN: 실시간 감지·알림 / OUT: 사후 추적 C-3·사용 판정 C-2·복구 F-1)
   1.4 주요 대상 조직 (DataOps·데이터 오너·AI 서비스 운영자·플랫폼팀)
   1.5 AI-ready 데이터 체계 내 역할 (④ 신뢰 게이트의 상시 감시등 — C-2 게이트에 근거 제공)
2. 필요성 및 기대 효과
   2.1 AI 과제 수행 시 주요 Pain Point (이상을 사후에야 인지, AI 결과 신뢰 붕괴) — 야간 배치 누락→수요예측 모델 오작동 무인지
   2.2 구축의 필요성 (파이프라인 급증·다국가 시차, 사람 눈 감시 한계)
   2.3 기대 효과 (탐지 시간 단축·무인지 장애 제거·AI 신뢰 회복·원인 분리 가속)
   2.4 주요 기능 (① 관찰 지표 · ② 이상 판정 룰 · ③ 알림·에스컬레이션 · ④ 원인 후보 분리)
3. 구성 체계
   3.1 관찰 지표 (Freshness·Volume·Schema·Distribution·Null/이상값 — 수집실패·지연·건수급변·스키마변경·결측·분포변화)
   3.2 이상 판정 룰 / 임계값 기준 (고정 임계값·과거 추세·계절성·업무 캘린더, ★ "사용 가능 판정"은 하지 않음)
   3.3 알림 기준 / Severity Level (수신자별·심각도별 라우팅)
   3.4 데이터-모델-Prompt 관찰 체계 (버전·입력분포 함께 기록해 원인 후보 분리)
4. 추진 역할 및 책임
   4.1 주요 담당자 정의 (Observability 관리자·DataOps 엔지니어·데이터 오너·AI 서비스 운영자)
   4.2 역할별 책임 구분 (RACI) — 지표 정의/룰 설정/알림 대응/트리아지
   4.3 계열사 상황에 따른 담당 조직 조정 기준 (다국가·시차 운영 시 Follow-the-sun 당직)
5. 데이터 흐름 현황 조사 및 관찰 대상 선정
   5.1 관찰 대상 기준 정의 — 핵심 데이터 흐름 식별 (AI 서비스·RAG·리포팅 직결 흐름 우선)
   5.2 흐름 유형별 관찰 지표 매핑 (배치/스트리밍/API/파일 — 유형별 핵심 지표)
   5.3 관찰 우선순위·커버리지 정의 (다운스트림 영향도 기반 Tier 1/2/3)
   5.4 임계값 베이스라인 수집 방식 (정상 구간 학습·캘린더 반영)
   5.5 ★ 알림 피로 최소화 기준 — 오탐 억제·중복 알림 묶기·심각도 차등
6. 데이터 Observability 솔루션 선정 검토
   6.1 솔루션 유형 (전용 Data Observability / 모니터링·APM 연계 / 오케스트레이터 내장 / 자체 구축)
   6.2 적용 범위 (관찰 지표·연결 원천·알림 채널)
   6.3 주요 솔루션 후보군 (전용 Observability 도구 / 오픈소스 / 클라우드 내장)
   6.4 기능 비교 기준 (자동 지표 수집·이상 탐지(룰/ML)·알림 연동·Lineage 연계(C-3)·커버리지·확장성)
   6.5 평가 및 PoC 기준 (실제 파이프라인 2~3종 연결, 인위적 이상 주입 탐지 검증)
   6.6 세부 영역 제안
7. 계열사 적용 예시: 두산밥캣 파이프라인 관측 시나리오
   7.1 데이터 환경 가정 (다국가 공장 MES/ERP·텔레매틱스·데이터레이크, 시차 운영)
   7.2 관측 목표 (핵심 파이프라인 무인지 장애 0)
   7.3 관찰 대상 흐름 식별 예시
   7.4 관찰 지표·임계값 설계 예시 (텔레매틱스 수집 건수·MES 야간 배치 Freshness)
   7.5 Severity·알림 라우팅 설계 예시
   7.6 핵심 산출물 (관찰 지표 카탈로그·알림 룰북·트리아지 Runbook)
   7.7 솔루션 선정 검토 예시
   7.8 To-Be 관측 아키텍처 예시
   7.9 구축 단계 예시 (Wave별 일정)
   7.10 운영 시나리오 예시 — 모델 오작동 원인 분리(데이터/모델/Prompt)
   7.11 기대 효과 예시 (탐지 시간·무인지 장애·오탐률)
8. 데이터 Observability 구축
   8.1 관찰 대상 파이프라인 취합·검토
   8.2 지표·메타 정합성 검토 (Lineage·카탈로그 연동 점검)
   8.3 To-Be 관측 아키텍처 설계
   8.4 솔루션 기반 목표 아키텍처 반영
   8.5 Legacy/미지원 원천 관측 연동 방안
   8.6 미연동 흐름 관측 파이프라인(에이전트·로그 수집) 개발
   8.7 수동 점검 지표 정의 (자동 수집 불가 흐름)
   8.8 이상 탐지 룰·임계값 기능 설정
   8.9 알림·에스컬레이션 워크플로 구성
   8.10 솔루션-계열사 시스템 연동 테스트 (이상 주입 시뮬레이션)
   8.11 초기 베이스라인 적재 및 탐지 검증
9. 데이터 Observability 운영
   9.1 변경 요구사항 요청 (신규 지표·임계값 조정·알림 룰 변경)
   9.2 검토·승인 (룰 변경 영향·오탐 가능성 확인)
   9.3 반영 (임계값·룰 버전 스탬프 후 배포)
   9.4 이상 탐지·알림 발생 운영
   9.5 이상 대응 트리아지 — 원인 후보 분리(데이터/모델/Prompt), ★ 복구 실행은 F-1으로 핸드오프
   9.6 AI 서비스·게이트 연계 (C-2 Quality Gate에 관측 근거 제공)
   9.7 관측 파이프라인 운영 (커버리지 유지·신규 흐름 자동 등록)
   9.8 권한·보안 (관측 대시보드 접근·민감 지표 노출 통제)
   9.9 역할별 기능 (오너=자기 자산 알림 / DataOps=룰 관리 / 운영자=대시보드)
   9.10 솔루션 운영 (룰 튜닝·오탐 회수·임계값 재학습)
10. AI-ready 데이터 체계 내 연계 범위
   10.1 과제 범위 재확인 (★ 경계: 실시간 감지·알림까지만)
   10.2 C-2 데이터 품질 관리 (C-1=운영 중 감지 / C-2=투입 전 사용가능 판정 게이트) — 최고 혼동쌍, 양방향
   10.3 C-3 데이터 계통 Lineage (C-1=실시간 이상 탐지 / C-3=사후 출처·영향 추적)
   10.4 F-1 데이터 운영관리 (C-1=탐지 / F-1=재처리·롤백·복구)
   10.5 A-1 카탈로그 (관찰 대상 자산 소재 참조)
11. KPI 및 성과 관리 (이상 탐지 시간·미탐지(무인지 장애) 건수·오탐률·알림 후 확인 시간·핵심 파이프라인 커버리지)
12. 고도화 Roadmap (수기 점검 → 룰 기반 탐지·알림(AI보조) → 학습 기반 이상 탐지·자동 트리아지(자율))
```

## C-2 데이터 품질 관리
> 파일명: `C-2 데이터 품질 관리.md`
> 정의: 데이터가 AI 활용 기준을 충족하는지(품질) 그리고 누가 쓸 수 있는지(권한)를 판정해 AI 투입을 통제하는 게이트 체계. "이 데이터를 써도 되는가"를 결정한다.
> 핵심 계열사 예시: 두산퓨얼셀 (품질·권한 게이트)
> ★ 핵심 원칙: 룰 기반 자동 판정을 우선하고, 사람 검토는 예외·고위험만. 접근 차단은 C-2 역할, 마스킹은 F-4.

```
1. 개요
   1.1 데이터 품질 관리의 정의 ("AI 투입 전 검문소(Gate)" — 통과/차단/예외 판정)
   1.2 목적 (불량·미승인 데이터의 AI 유입 차단, 두 관문: 품질+권한)
   1.3 적용 범위 (IN: 사용가능 판정·접근 차단 통제 / OUT: 이상 감지 C-1·마스킹·비식별 F-4)
   1.4 주요 대상 조직 (품질관리 책임·데이터 오너·보안/권한 관리자·AI 과제 승인자)
   1.5 AI-ready 데이터 체계 내 역할 (④ 신뢰 게이트의 최종 판정자 — 통과 데이터만 ⑤ 활용으로)
2. 필요성 및 기대 효과
   2.1 AI 과제 수행 시 주요 Pain Point (품질·권한 미검증 데이터가 AI에 투입돼 오류·정보유출) — 미승인 시험데이터로 학습→잘못된 스택 수명 예측
   2.2 구축의 필요성 (AI는 "쓰레기 입력→쓰레기 출력" 증폭, 권한 미통제 시 규제 위반)
   2.3 기대 효과 (품질 통과율 향상·권한 위반 차단·AI 신뢰·예외의 책임 추적)
   2.4 주요 기능 (① 품질 기준·판정 · ② 접근 권한(RBAC/ABAC) · ③ Quality Gate · ④ 예외 승인)
3. 구성 체계
   3.1 데이터 품질 기준 (완전성·정확성·일관성·최신성·유효성·중복성, 업무별 허용 기준)
   3.2 AI 사용 제한 데이터 기준 (출처 불명·미승인·품질 미달·목적 외)
   3.3 접근 권한 정책 (RBAC/ABAC — 조회/분석/AI학습/외부반출 구분, ★ 접근 차단은 C-2 역할)
   3.4 Quality Gate 구조 (품질+권한 동시 통과해야 AI 투입 허용)
4. 추진 역할 및 책임
   4.1 주요 담당자 정의 (품질관리자·데이터 오너·권한 관리자·예외 승인권자·AI 과제 책임자)
   4.2 역할별 책임 구분 (RACI) — 기준 정의/룰 설정/게이트 운영/예외 승인
   4.3 계열사 상황에 따른 담당 조직 조정 기준 (품질·보안 조직 분리/통합)
5. 데이터 현황 조사 및 품질·권한 적용 대상 선정
   5.1 적용 대상 기준 정의 — 품질 측정·게이트 대상 데이터 선정 (AI 투입 예정 자산 우선)
   5.2 업무별 품질 허용 기준 정의 (Critical/Major/Minor 차원별 임계)
   5.3 AI 사용 제한 데이터 식별 (출처·승인·목적 제한 — 차단 대상)
   5.4 데이터 민감도·권한 등급 분류 (역할·조직·목적·민감도 매핑)
   5.5 품질 프로파일링·룰 수집 방식 (자동 프로파일링 우선)
   5.6 게이트 우선순위 (다운스트림 AI 영향도 기반)
   5.7 ★ 판정 자동화/효율 원칙 — 룰 기반 자동 판정 우선, 사람 검토는 예외·고위험만
6. 데이터 품질·권한 관리 솔루션 선정 검토
   6.1 솔루션 유형 (Data Quality / 정책엔진(Policy Engine) / 권한관리(IAM·ABAC) / 통합 거버넌스)
   6.2 적용 범위 (품질 룰·권한 정책·게이트 연동)
   6.3 주요 솔루션 후보군 (품질 룰 엔진 / 권한·정책 엔진 / 클라우드 내장)
   6.4 기능 비교 기준 (품질 룰 엔진·자동 프로파일링·RBAC/ABAC·Gate/파이프라인 연동·예외 워크플로·C-1/C-3 연계)
   6.5 평가 및 PoC 기준 (실제 자산에 품질 룰+권한 정책 적용, 통과/차단 검증)
   6.6 세부 영역 제안 (품질 솔루션 + 권한 솔루션 조합)
7. 계열사 적용 예시: 두산퓨얼셀 Quality Gate 적용 시나리오
   7.1 데이터 환경 가정 (스택 시험·운전·필드 데이터, 고객/계약 민감정보 혼재)
   7.2 구축 목표 (AI 투입 데이터 100% 게이트 통과, 미승인 사용 0)
   7.3 품질·권한 대상 자산 식별 예시
   7.4 품질 기준·허용 임계 정의 예시 (시험 결측률·운전 로그 정확성)
   7.5 권한 등급·정책(RBAC/ABAC) 설계 예시
   7.6 Quality Gate 룰 설계 예시
   7.7 핵심 산출물 (품질 룰북·권한 정책표·게이트 판정 로그)
   7.8 솔루션 선정 검토 예시
   7.9 To-Be 게이트 아키텍처 예시
   7.10 구축 단계 예시 (Wave별 일정)
   7.11 운영 시나리오 예시 — 미승인 데이터 차단 + 예외 승인 흐름
   7.12 기대 효과 예시 (품질 통과율·권한 위반·차단 건수)
8. 데이터 품질·권한 게이트 구축
   8.1 대상 자산·품질 룰 취합·검토
   8.2 품질 룰·권한 정책 정합성 검토
   8.3 To-Be 게이트 아키텍처 설계 (품질 판정 + 권한 판정 결합)
   8.4 솔루션 기반 목표 아키텍처 반영 (품질 엔진·정책 엔진 매핑)
   8.5 Legacy 시스템 권한·품질 연동 방안
   8.6 미연동 파이프라인 품질 검증 개발
   8.7 수동 품질 점검 기준 (자동 룰 불가 항목)
   8.8 Quality Gate 기능 개발 (통과/차단/예외 분기)
   8.9 솔루션 설정 (품질 룰·RBAC/ABAC 정책·게이트 임계)
   8.10 솔루션-계열사 시스템 연동 테스트 (정상/불량/미승인 데이터 게이트 검증)
   8.11 초기 품질 측정·권한 적용 및 검증
9. 데이터 품질·권한 운영
   9.1 변경 요구사항 요청 (품질 기준·권한 정책 변경)
   9.2 검토·승인 (변경 영향·게이트 통과율 영향 검토)
   9.3 반영 (룰·정책 버전 스탬프 후 배포)
   9.4 Quality Gate 판정 운영 (AI 학습·RAG·분석 투입 전 통과/차단)
   9.5 예외 사용 승인 프로세스 (승인자·사유·기간·범위·책임 기록)
   9.6 AI 서비스 연계 (게이트 통과 데이터만 ⑤ 활용 단계로 공급)
   9.7 품질/권한 위반 처리 파이프라인 (차단·재검토·개선 요청)
   9.8 권한·보안 운영 (RBAC/ABAC 정책 갱신, ★ 접근 차단 통제 — 마스킹은 F-4)
   9.9 역할별 기능 (오너=품질 / 권한관리자=정책 / 승인자=예외 / 이용자=게이트 결과)
   9.10 솔루션 운영 (룰 튜닝·정책 동기화·게이트 임계 재조정)
10. AI-ready 데이터 체계 내 연계 범위
   10.1 과제 범위 재확인 (★ 경계: 사용가능 "판정" + 접근 "차단"까지)

11. KPI 및 성과 관리 (품질 기준 통과율·권한 위반 건수·미승인 데이터 사용 건수·AI 사용 차단 건수·품질 개선 리드타임·예외 승인 적정성)
```

## C-3 데이터 계통 Lineage
> 파일명: `C-3 데이터 계통 Lineage.md`
> 정의: 데이터의 출처→이동→변환→활용→AI 답변 근거까지의 이력을 기록해 AI 결과의 근거를 추적하고 변경 영향을 분석하는 체계. "이 결과는 어디서 왔는가"를 되짚는다.
> 핵심 계열사 예시: 두산로보틱스 (RAG 답변 근거 추적)
> ★ 핵심 원칙: 메타데이터·로그 기반 자동 수집을 우선하고, 수동 매핑은 미연동 구간만. 이상 "탐지"는 C-1.

```
1. 개요
   1.1 데이터 계통(Data Lineage)의 정의 ("데이터 족보·이력서" — 출처에서 답변까지의 계보)
   1.2 목적 (AI 답변 근거 역추적·변경 영향도 분석·감사 증빙)
   1.3 적용 범위 (IN: 연결성·추적성 / OUT: 이상 탐지 C-1·보존·폐기 F-2)
   1.4 주요 대상 조직 (데이터 오너·DataOps·AI 서비스 운영자·감사/규제 담당)
   1.5 AI-ready 데이터 체계 내 역할 (④ 신뢰 게이트의 근거 제공자 — C-2 판정에 출처 증빙)
2. 필요성 및 기대 효과
   2.1 AI 과제 수행 시 주요 Pain Point (AI 답변·보고서 근거 역추적 불가, 변경 영향 파악 불가) — RAG 답변이 어느 매뉴얼 버전을 참조했는지 알 수 없어 책임 규명 불가
   2.2 구축의 필요성 (RAG/Agent는 근거 없으면 신뢰 불가, 원천 변경 시 영향 범위 미상)
   2.3 기대 효과 (근거 추적 가능·영향도 분석 가속·감사 대응·오류 원인 격리)
   2.4 주요 기능 (① Lineage 범위 · ② 변환 이력 기록 · ③ 답변 근거(Citation) · ④ 영향도 분석)
3. 구성 체계
   3.1 AI-ready Lineage 범위 (원천→전처리→분석셋→보고서→RAG 인덱스→Prompt 입력→AI 답변)
   3.2 변환 이력 기록 기준 (ETL/ELT·전처리 버전·병합·필터·파생·문서 참조)
   3.3 AI 답변 근거 로그 / Citation 구조 (참조 문서·청크·테이블·Tool 결과)
   3.4 영향도 분석 기준 (원천·로직 변경 → 리포트·모델·Prompt·서비스 영향 추적)
4. 추진 역할 및 책임
   4.1 주요 담당자 정의 (Lineage 관리자·데이터 오너·AI 서비스 운영자·감사 담당)
   4.2 역할별 책임 구분 (RACI) — 범위 정의/연동/Citation 로깅/영향도 분석/감사 대응
   4.3 계열사 상황에 따른 담당 조직 조정 기준 (자동 수집 역량·문서 기반 Lineage 비중)
5. 데이터 흐름 현황 조사 및 추적 대상 선정
   5.1 추적 대상 기준 정의 — Lineage 대상 흐름 선정 (AI 답변 직결 흐름 우선)
   5.2 기록 수준(Granularity) 정의 (컬럼/테이블/문서/청크 — 흐름별 차등)
   5.3 Citation 필요 AI 과제 식별 (RAG·Agent·자동 보고서 — 근거 필수)
   5.4 감사·규제 증빙 대상 식별 (규제·고객 협약 흐름)
   5.5 수집 방식 구분 (자동 Lineage 수집 vs 수동 매핑)
   5.6 추적 우선순위 (영향도·감사 중요도 기반)
   5.7 ★ 자동 수집 최소화/효율 원칙 — 메타데이터·로그 기반 자동 수집 우선, 수동 매핑은 미연동 구간만
6. 데이터 Lineage 솔루션 선정 검토
   6.1 솔루션 유형 (전용 Data Lineage / 카탈로그·메타 내장 / 오케스트레이터 연계 / LLM Observability(근거 추적))
   6.2 적용 범위 (정형 Lineage·문서·RAG Citation)
   6.3 주요 솔루션 후보군 (OpenLineage 류 / 카탈로그 내장 / RAG Citation·LLM tracing)
   6.4 기능 비교 기준 (자동 수집 범위·컬럼 레벨 Lineage·문서·RAG Citation 추적·영향도 분석 UI·카탈로그(A-1) 연계·감사 리포트)
   6.5 평가 및 PoC 기준 (RAG 파이프라인 연결, 답변→근거 역추적 검증)
   6.6 세부 영역 제안 (정형 Lineage + RAG Citation 두 축 결합)
7. 계열사 적용 예시: 두산로보틱스 RAG 근거 추적 시나리오
   7.1 데이터 환경 가정 (매뉴얼·정비 절차·고객 문의 RAG, 매뉴얼 버전 빈번 갱신)
   7.2 구축 목표 (모든 AI 답변에 근거 Citation 100%, 영향도 분석 즉시화)
   7.3 추적 대상 흐름 식별 예시
   7.4 Lineage 범위·기록 수준 설계 예시 (문서 청크 단위 Citation)
   7.5 변환·Citation 로깅 설계 예시
   7.6 핵심 산출물 (Lineage 맵·Citation 로그 스키마·영향도 분석 리포트)
   7.7 솔루션 선정 검토 예시
   7.8 To-Be Lineage 아키텍처 예시
   7.9 구축 단계 예시 (Wave별 일정)
   7.10 운영 시나리오 예시 — RAG 오답 근거 역추적 + 매뉴얼 개정 영향도 분석
   7.11 기대 효과 예시 (근거 추적 가능률·영향도 분석 소요시간)
8. 데이터 Lineage 구축
   8.1 추적 대상 흐름·메타 취합·검토
   8.2 메타데이터·카탈로그 정합성 검토 (A-1 자산 식별자 정렬)
   8.3 To-Be Lineage 아키텍처 설계
   8.4 솔루션 기반 목표 아키텍처 반영
   8.5 Legacy/미지원 변환 Lineage 연동 방안
   8.6 미연동 흐름 Lineage 수집 파이프라인(로그·OpenLineage 이벤트) 개발
   8.7 수동 Lineage 매핑 기준 (자동 수집 불가 문서·수기 변환)
   8.8 Citation 로깅·영향도 분석 기능 개발
   8.9 솔루션 설정 (Lineage 그래프·기록 수준·Citation 스키마)
   8.10 솔루션-계열사 시스템 연동 테스트 (변경 주입→영향도 추적 검증)
   8.11 초기 Lineage 적재 및 근거 추적 검증

   9.2 검토·승인 (변경 영향·수집 부하 검토)

   9.4 Lineage 수집·Citation 로깅 운영
   9.5 영향도 분석 운영 (원천·로직 변경 시 다운스트림 영향, ★ 이상 탐지는 C-1으로)
   9.6 AI 서비스 연계 (RAG 답변에 근거 Citation 자동 첨부)
   9.7 감사 대응 Lineage Report 운영 (출처·변환·근거 증빙)
   9.8 권한·보안 (Lineage 그래프 접근·민감 경로 노출 통제)
   9.9 역할별 기능 (오너=자기 자산 계보 / 운영자=영향도 분석 / 감사=증빙 리포트)
   9.10 솔루션 운영 (수집 커버리지 유지·끊긴 Lineage 보수)
10. AI-ready 데이터 체계 내 연계 범위
   10.1 과제 범위 재확인 (★ 경계: 연결성·추적성만)
   10.2 C-1 데이터 Observability (C-1=실시간 이상 탐지 / C-3=사후 출처·영향 추적) — 양방향
   10.3 F-2 데이터 생애주기 관리 (C-3=이력 추적 / F-2=보존·아카이빙·폐기)
   10.4 A-1 데이터 카탈로그 (A-1=자산 소재 / C-3=자산 간 흐름·이력)
   10.5 C-2 데이터 품질 관리 (C-3 출처 근거를 C-2 게이트 판정에 제공)
11. KPI 및 성과 관리 (Lineage 커버리지·AI 답변 근거 추적 가능률·영향도 분석 소요시간·감사 대응 리드타임·끊긴 Lineage 비중)
```

## D-2 API/Tool 연계 데이터
> 파일명: `D-2 API_Tool 연계 데이터.md`
> 정의: AI Agent가 외부 시스템과 Tool을 안전하게 호출하도록 기능·입력·출력·제약·위험도를 정의한 명세 체계. Tool을 "Agent가 읽고 고를 수 있는 데이터 자산"으로 정비.
> 핵심 계열사 예시: 두산로보틱스 (운영·주문·서비스 Agent Tool)
> ★ 핵심 원칙: 모든 API를 Tool로 열지 않는다 — 명세가 명확하고 위험이 통제되는 조회형부터, 실행형은 승인 게이트와 함께 개방.

```
1. 개요
   1.1 API/Tool 연계 데이터의 정의 (Tool = Agent가 호출하는 "기능 데이터 자산", 명세가 곧 데이터)
   1.2 목적 (Agent가 외부 시스템을 안전·정확하게 호출하도록 명세 표준화)
   1.3 적용 범위 (IN: Tool 명세·스키마·위험분류·Registry·Call Log / OUT: Prompt/Harness D-3·로그 기반 개선 환류 E-4·모델/Agent 오케스트레이션 구현)
   1.4 주요 대상 조직 (AI 플랫폼팀·시스템 API 오너·보안·업무 현업·DataOps)
   1.5 AI-ready 체계 내 역할 (⑤ 활용 단계, Agent의 손발(Tool) 명세 입구)
2. 필요성 및 기대 효과
   2.1 AI 과제 수행 시 주요 Pain Point (Tool 명세 부재·스키마 불일치·위험 Tool 무분별 호출로 오작동·실패)
   2.2 필요성 (Agent는 자연어 설명만으로 Tool을 고른다 → 명세 품질이 곧 호출 정확도, 실행형 Tool의 안전성)
   2.3 기대 효과 (호출 성공률↑·오작동↓, 위험 Tool 통제, Tool 재사용·횡전개, 변경에 따른 Agent 깨짐 방지)
   2.4 주요 기능 (① Tool 명세(Description) · ② Tool Schema · ③ Risk 분류 · ④ Registry·버전·Call Log)
3. 구성 체계
   3.1 Tool 후보 목록 (조회형·계산형·보고서 생성형·시스템 검색형·업무 실행형)
   3.2 Tool Description 표준 (이름·목적·기능·입력·출력·제약·호출 예시)
   3.3 Tool Schema (input/output schema·데이터 타입·필수값·허용값·오류 응답)
   3.4 Tool Risk Classification (조회/추천/실행/외부전송형 + 위험도별 승인 기준) ★
   3.5 Tool Registry & 버전 관리 / Call Log Schema (호출 시점·입출력·주체·성공실패·오류)
4. 추진 역할 및 책임
   4.1 주요 담당자 정의 (Tool 오너=시스템 API 소유 부서, AI 플랫폼/Registry 관리자, 보안 검토자, Agent 개발자)
   4.2 역할별 책임 구분 (RACI: 명세 작성·스키마 검증·위험 등급 승인·Registry 등록·Call Log 운영)
   4.3 계열사 상황별 담당 조직 조정 기준 (API 거버넌스 성숙도·MCP 도입 단계)
5. API/Tool 현황 조사 및 연계 대상 선정
   5.1 연계 대상 기준 정의 (Agent 활용 빈도·업무 가치·자동화 효과)
   5.2 연계 대상 시스템·API·I/F 식별 (사내 시스템·외부 SaaS·내부 함수)
   5.3 Tool 유형별 등록/제외 기준 (조회형 우선, 외부 전송형 신중)

   5.5 인증·인가·소재 보안 검토 (자격증명 관리·최소 권한·호출 범위 제한)

   6.4 주요 솔루션 후보군 검토 (MCP·OpenAPI·API Gateway 등)

   6.6 검토 항목 (기존 API 자산·인증 체계 정합)
   6.7 평가 및 선정 기준
   6.8 PoC 기준 (실제 Tool 3종 등록→Agent 호출→로그 추적 재현)

7. 계열사 적용 예시: 두산로보틱스 서비스 Agent Tool 연계 시나리오
   7.1 시스템·API 환경 가정 (주문/재고·로봇 상태·서비스 티켓·문서 검색 API)
   7.2 구축 목표
   7.3 주요 Tool 후보 식별 예시 (조회·계산·실행·외부 전송 혼합)
   7.4 연계 대상 Tool 선정 예시 (Wave 1 핵심 Tool)
   7.5 연계 제외 Tool 판단 예시

   7.9 솔루션 선정 검토 예시
```

## D-3 Prompt/Harness 자산화
> 파일명: `D-3 Prompt_Harness 자산화.md`
> 정의: 사람의 업무 절차와 판단 기준을 AI가 실행 가능한 Prompt·Harness 형태로 구조화·버전 관리·재사용하는 체계. Prompt·Harness를 "산재한 텍스트"가 아니라 "등록·버전·의존성이 관리되는 데이터 자산"으로.
> 핵심 계열사 예시: 두산밥캣 (딜러 견적·서비스 진단·품질 분석 업무 Harness)
> ★ 핵심 원칙: 모든 Prompt를 자산화하지 않는다 — 반복되고 업무 판단에 영향을 주며 재사용 가치가 있는 것부터.

```
1. 개요
   1.1 Prompt/Harness 자산화의 정의 (Prompt = 업무 지시·판단의 구조화 자산, Harness = 반복 실행 업무 묶음)
   1.2 목적 (현업의 절차·판단 기준을 재사용·감사 가능한 자산으로 구조화)
   1.3 적용 범위 (IN: 자산 목록·변환 템플릿·Registry·Dependency Map·재사용 Harness / OUT: 평가 기준 자체 E-3·Tool 명세 D-2·모델/Agent 구현 코드)
   1.4 주요 대상 조직 (현업 SME·AI 플랫폼/Prompt 관리자·업무 오너·품질/감사)
   1.5 AI-ready 체계 내 역할 (⑤ 활용, 사람의 업무 지식을 AI 실행 자산으로 옮기는 계층)
2. 필요성 및 기대 효과
   2.1 AI 과제 수행 시 주요 Pain Point (Prompt 개인 PC 산재·버전 미관리·의존성 불명으로 재사용·개선·감사 불가)
   2.2 필요성 (업무 판단이 1회용 Prompt에 갇힘, 누가 왜 바꿨는지 추적 불가, 데이터/Tool 변경 시 깨짐)
   2.3 기대 효과 (재사용·표준화, 변경 이력·승인으로 감사 가능, Dependency 가시화, Harness 패키징 횡전개)
   2.4 주요 기능 (① 업무절차→Prompt 변환 · ② Prompt Registry·버전 · ③ Dependency Map · ④ 재사용 Harness)
3. 구성 체계
   3.1 Prompt/Harness 자산 목록 (업무 Prompt·시스템 지시문·역할 정의·판단 기준·출력 형식·예외 처리)
   3.2 업무 절차 → Prompt 변환 템플릿 (입력·판단 단계·참조 기준·금지 행동·출력 형식)
   3.3 Prompt Registry & 버전 관리 기준 (변경 이력·사유·승인자·적용 업무)
   3.4 Prompt Dependency Map (데이터·Glossary·Tool·평가 데이터 의존 관계) ★
   3.5 재사용 Harness 구조 (Prompt+입력+Tool 호출 순서+출력 포맷 패키징)
4. 추진 역할 및 책임
   4.1 주요 담당자 정의 (Prompt 오너=업무 SME, Prompt/Harness 관리자, 승인자, AI 과제자)
```

> ★ 핵심 원칙: 데이터 Product화는 "데이터를 만들어 주는 것"이 아니라 "상품처럼 책임지고 제공하는 것" — 등록·검색은 A-1, E-1은 소비자 경험.


## E-2 합성데이터 (Synthetic)
> 파일명: `E-2 합성데이터.md`
> 정의: 실데이터가 부족하거나 보안상 사용이 제한될 때, AI 학습·검증을 위해 인공적으로 생성한 데이터와 그 생성·검증·위험 관리 체계.
> 핵심 계열사 예시: 두산전자 (CCL 불량 — 희소 불량 사례 합성으로 검사 AI 학습 보강)
> ★ 핵심 원칙: 합성은 보충재 — 부족/제한을 메우되 실데이터와 절대 혼동되지 않게 태깅·검증한다. 평가셋은 E-3, 비식별은 F-4. (6장=방식·도구 선정, 8장=생성 파이프라인 구성으로 변형)

```
1. 개요
   1.1 합성데이터의 정의 (실데이터 부족·제한 → 인공 생성, 원본 대체 아님)
   1.2 목적 (희소 이벤트 보강·보안 제약 우회·테스트 케이스 확보)
   1.3 적용 범위 (IN: 적용 판단·생성·검증·위험관리 / OUT: 평가셋=E-3·비식별=F-4·품질판정=C-2)
   1.4 주요 대상 조직 (AI 과제팀·데이터 과학자·도메인 전문가·보안 담당)
   1.5 AI-ready 체계 내 역할 (③ 가공 단계 "보충" 경로 — 부족분만 채우는 보조 공급선)
2. 필요성 및 기대 효과
   2.1 AI 과제 수행 시 주요 Pain Point (희소 불량 샘플 부족·보안으로 원본 학습 불가·예외 케이스 부재)
   2.2 적용의 필요성 (실데이터 확보 한계·클래스 불균형·테스트 부족)
   2.3 기대 효과 (희소 패턴 재현·보안 안전 학습·검증 케이스 확보)
   2.4 주요 기능 (① 적용 판단 · ② 생성 · ③ 검증 · ④ 위험 태깅)
3. 합성데이터 구성 체계
   3.1 합성데이터 적용 판단 기준 (데이터 부족·희소 이벤트·보안 제한·테스트 부족 — 적용/비적용 결정 트리)
   3.2 합성 대상 데이터 정의 (불량 사례·설비 장애·클레임 유형·예외/테스트 케이스 — 대상·목적 명세)
   3.3 합성 방식 분류 (통계 기반·규칙 기반·시뮬레이션·생성모델/LLM 기반 — 방식별 적용 조건)
   3.4 합성데이터 검증 항목 (분포 유사성·상관관계 보존·희소 패턴 재현·도메인 타당성·모델 성능 개선)
   3.5 Synthetic Tag·사용 정책·위험 관리 (편향·과적합·현실 왜곡·재식별 점검, 원본 혼동 방지 태깅)
4. 추진 역할 및 책임
   4.1 주요 담당자 정의 (생성 담당·검증 담당·도메인 전문가·보안 검토자)
   4.2 역할별 책임 구분 (RACI: 적용 판단·생성·검증 서명·태깅·위험 승인)
   4.3 계열사 상황에 따른 담당 조직 조정 기준 (전문 인력 부재 시 지주 지원·외부 협력)
5. 합성데이터 현황 조사 및 합성 대상 선정
   5.1 데이터 부족 영역 진단 (클래스 불균형·희소 이벤트·보안 차단 데이터 식별)
   5.2 합성 대상·목적 정의 (무엇을·왜 합성하는가)
   5.3 합성 비적용 기준 (실데이터 충분·합성 위험 > 이득)
   5.4 검증 기준선 데이터 확보 (합성 품질을 비교할 실데이터 reference)
   5.5 최종 합성 대상 및 우선순위
   5.6 ★ "실데이터 우선" 원칙 (확보 가능한 실데이터를 먼저 쓴다)
6. 합성 방식·생성 도구 선정 검토  ※ 솔루션 도입이 아닌 "방식·도구 선정"으로 변형
   6.1 목적·적용 범위 (데이터 특성·목적별 생성 방식 매칭)

   6.4 도구·라이브러리 후보군 (Synthetic Data Generation 도구·시뮬레이터·생성모델 프레임워크)
   6.5 비교·평가 기준 (재현 충실도·희소 패턴 표현력·재식별 위험·운영 난이도)

7. 계열사 적용 예시: 두산전자 불량 사례 합성 시나리오
   7.1 환경 (CCL 검사 AI, 특정 불량 클래스 샘플 극소)

   8.2 생성 파이프라인 설계 (방식별)
```

## E-3 AI 평가 데이터
> 파일명: `E-3 AI 평가 데이터.md`

> 핵심 계열사 예시: 두산로보틱스 (RAG 기반 기술문의 응답/Agent 동작 평가셋)
> ★ 핵심 원칙: 평가 데이터는 "맞았다/틀렸다"가 아니라 "왜 맞고 왜 틀렸는지"를 축적한다. 학습 라벨은 B-2, Prompt 자산은 D-3. (6장=방식·Eval 도구 선정, 8장=Gold Dataset·Harness 구성으로 변형)

```
1. 개요

   1.2 목적 (객관적 성능 판단·회귀 방지·개선 근거 제공)

6. 평가 방식·Eval 도구 선정 검토  ※ 솔루션 도입이 아닌 "평가 방식·도구 선정"으로 변형

   6.4 Eval 도구·프레임워크 후보군 (Eval 플랫폼·LLM-as-judge·회귀 테스트 러너)

   7.5 Rubric 정의 (정확성·근거성·안전성)
```

## E-4 데이터 Feedback Loop
> 파일명: `E-4 데이터 Feedback Loop.md`
> 정의: AI 운영 결과·오류·사용자 피드백을 다시 데이터·모델·Prompt·Tool 개선으로 환류하는 체계.
> 핵심 계열사 예시: 두산밥캣 (현장 AI 응답·Agent 실행 오류를 수집해 전 주제로 개선 환류)
> ★ 핵심 원칙: 오류를 모으는 것이 아니라 올바른 주제로 라우팅해 개선으로 닫는다 — 데이터 누락→A-1/A-2, 용어→A-3, 라벨→B-2, 평가→E-3, Prompt→D-3, Tool→D-2.

```
1. 개요
   1.1 데이터 Feedback Loop의 정의 (운영 결과 → 개선 환류, 일회성 보고 아님)
   1.2 목적 (반복 오류 차단·지속 개선·전 주제 정비 우선순위 제공)
   1.3 적용 범위 (IN: 수집·분류·라우팅·반영 / OUT: 실제 데이터/Prompt/Tool 수정은 각 주제, E-4는 환류 흐름)
   1.4 주요 대상 조직 (AI 서비스 운영자·DataOps·각 주제 담당·개선 Backlog 관리자)
   1.5 AI-ready 체계 내 역할 (가치사슬의 "되먹임" — AI 서비스 → 전 주제로 환류하는 순환 고리)
2. 필요성 및 기대 효과
   2.1 AI 과제 수행 시 주요 Pain Point (운영 중 오류가 기록만 되고 개선으로 안 이어짐·같은 오류 반복·원인 주제 불명)
   2.2 구축의 필요성 (개선 단절·반복 오류·암묵적 수정의 휘발)
   2.3 기대 효과 (오류 재발률 감소·개선 리드타임 단축·사용자 수정 감소)
   2.4 주요 기능 (① 수집 · ② 분류(Taxonomy) · ③ 라우팅 · ④ 반영)
3. 데이터 Feedback Loop 구성 체계
   3.1 피드백 수집 항목 (Agent 실행 결과·사용자 수정·실패 답변·누락 근거·Tool 호출 실패·Human 승인/거절)
   3.2 오류 유형 Taxonomy (데이터·Prompt·모델·Tool·권한·업무 정책 문제로 분류)
   3.3 개선 Backlog 라우팅 기준 ★ (데이터 누락→A-1/A-2 · 용어→A-3 · 라벨→B-2 · 평가→E-3 · Prompt→D-3 · Tool→D-2)
   3.4 자동/수동 반영 기준 (위험도 기반 — 저위험 자동 후보, 고위험/고객영향/보안은 전문가 검토)
4. 추진 역할 및 책임
   4.1 주요 담당자 정의 (피드백 운영자·분류 담당·각 주제 담당·Backlog 관리자)
   4.2 역할별 책임 구분 (RACI: 수집·분류·라우팅·반영 승인·효과 측정)

5. Feedback 수집 현황 조사 및 대상 선정
   5.1 피드백 수집 가능 지점 식별 (AI 서비스 로그·사용자 인터페이스·Tool Call Log(D-2)·Human 승인 로그)

6. Feedback Loop 솔루션·연계 방식 선정 검토
   6.1 목적·적용 범위 (피드백 수집·Backlog·실험/배포 연계)
   6.2 솔루션·방식 유형 (피드백 수집 도구·이슈/Backlog 시스템·실험 추적·라우팅 룰 엔진)
```

## F-2 데이터 생애주기 관리
> 파일명: `F-2 데이터 생애주기 관리.md`
> 정의: 데이터의 생성·사용·보관·아카이빙·폐기까지 시간축을 정책 기반으로 통제하는 체계. 무엇을 얼마나 오래 어디에 두고 언제 버릴지를 정한다.
> 핵심 계열사 예시: 두산퓨얼셀 (시험·운전 데이터의 장기 보존과 과정 데이터 관리)
> ★ 핵심 원칙: "최종 결과"만이 아니라 AI 재사용 가치가 있는 과정 데이터(실패·중간 산출물)를 의도적으로 보존한다. 변환 이력은 C-3, F-2는 데이터 실체의 보존·폐기.

```
1. 개요
   1.1 데이터 생애주기 관리의 정의
   1.2 목적 (가치 있는 데이터의 적시 보존 + 불필요 데이터의 적기 폐기로 비용·리스크 통제)
   1.3 적용 범위 (IN: 시간·보존·아카이빙·폐기 / OUT: 변환 이력=C-3·비식별/보안=F-4·품질 판정=C-2)
   1.4 주요 대상 조직 (전사 데이터 거버넌스·계열사 데이터 담당·데이터 오너·법무/컴플라이언스·인프라 운영)
   1.5 AI-ready 체계 내 역할 (F-1과 함께 가치사슬을 떠받치는 "기반" 횡단 레이어 — 시간축 담당)
2. 필요성 및 기대 효과
   2.1 AI 과제 수행 시 주요 Pain Point (무기한 보관으로 비용·리스크 누적, AI 재사용 가치 있는 과정 데이터 소실, 법적 보존 의무 누락)
   2.2 구축의 필요성 (데이터는 시간이 지나며 가치·위험·비용이 변함, AI는 과거 데이터로 학습하므로 "언제 버렸는가"가 학습 자산을 좌우)
   2.3 기대 효과 (저장 비용 절감·보안 리스크 축소·AI 학습 자산 보존·감사 대응력)
   2.4 주요 기능 (① 단계 통제 · ② 보존 정책 · ③ 과정 데이터 보존 · ④ 저장소 계층화)
3. 데이터 생애주기 관리 구성 체계
   3.1 데이터 생애주기 단계 정의 (생성→수집→사용→보관→아카이빙→폐기·재사용, 단계별 책임자)
   3.2 데이터 보존 기간 기준 (법적 의무·업무 재사용 가치·AI 학습 가치·감사 필요성·비용 매트릭스)
   3.3 과정 데이터 보존 기준 (실패 실험·중간 분석·검토 의견 등 AI 재사용 가치 있는 과정 산출물 보존 판단)
   3.4 저장소 계층화 기준 (Hot/Warm/Cold — 활용 빈도·신선도 요구) (★ 과정 데이터 의도적 보존, 이력 추적은 C-3)
4. 추진 역할 및 책임
   4.1 주요 담당자 정의 (생애주기 정책 관리자·데이터 오너·법무/컴플라이언스·인프라/스토리지 운영)
   4.2 역할별 책임 구분 (RACI: 보존 기간 결정·폐기 승인·아카이빙·복원)
   4.3 계열사 상황에 따른 담당 조직 조정 기준 (지주 공통 보존 정책 + 계열사 데이터 유형별 적용)
5. 생애주기 적용 데이터 현황 조사 및 대상 선정
   5.1 적용 대상 기준 정의 — 모든 데이터에 동일 보존 정책을 적용하지 않는 이유
   5.2 데이터 유형별 분류 (정형/문서/시계열/실험·과정 데이터)
   5.3 보존 대상 식별 (법적 보존·업무 재사용·AI 학습 가치)
   5.4 폐기·아카이빙 대상 식별 (보존 기간 만료·사용 중단)
   5.5 과정 데이터 보존 여부 판단 (AI 재사용 가치 평가)
   5.6 보존 예외·법적 보존 대상 식별 (소송·규제 Hold)
   5.7 데이터 유형별 보존 기간·저장 계층 우선순위 확정
   5.8 ★ 보존은 "최대한 오래"가 아니라 가치·의무·비용의 균형점에서 유형별로
6. 데이터 생애주기 관리 솔루션 선정 검토
   6.1 솔루션 도입 목적·적용 범위
   6.2 솔루션 유형 (Lifecycle Policy 엔진·아카이빙/콜드 스토리지·스토리지 계층화·보존 정책 자동화 ILM)
   6.3 후보군 (스토리지 라이프사이클 정책·객체 스토리지 티어링·아카이브 솔루션)
   6.4 비교 기준 (정책 자동 적용·법적 Hold 지원·복원 속도·비용 모델)
   6.5 검토 항목 (보존 만료 자동 폐기·예외 잠금·계열사 이기종 스토리지 수용)
   6.6 평가·선정
   6.7 PoC (티어링·아카이빙·복원 검증)
7. 계열사 적용 예시: 두산퓨얼셀 데이터 생애주기 정책 시나리오
   7.1 환경 (시험·운전·품질 데이터, 장기 운전 시계열)
   7.2 목표
   7.3 대상 데이터 식별
   7.4 생애주기 단계 정의
   7.5 보존 기간 매트릭스
   7.6 과정 데이터(실패 시험) 보존 결정
   7.7 Hot/Warm/Cold 배치
   7.8 폐기·아카이빙 절차
   7.9 저장소 아키텍처
   7.10 운영
   7.11 기대 효과 (비용 절감·학습 자산 확보)
8. 데이터 생애주기 관리 구축
   8.1 데이터 자산 인벤토리·보존 현황 취합
   8.2 보존 기간·유형 정합성 검토 (법적·업무 기준 대조)
   8.3 To-Be 생애주기 정책·저장소 계층 설계
   8.4 솔루션에 보존 정책·티어링 규칙 반영
   8.5 Legacy 장기 보관 데이터 분류·이관
   8.6 과정 데이터 수집·보존 파이프라인 구성
   8.7 수동 보존/폐기 대상 처리 절차
   8.8 폐기·아카이빙·복원 기능 개발
   8.9 솔루션 설정 (티어 전환 임계·만료 트리거·법적 Hold)
   8.10 테스트 (폐기·복원·예외 잠금 검증)
   8.11 초기 분류·적재·검증
9. 데이터 생애주기 관리 운영
   9.1 생애주기 변경 요청 접수 (보존 기간 변경·예외 신청)
   9.2 검토·승인 (법무·오너 협의)
   9.3 정책 반영·티어 전환 실행
   9.4 폐기 실행·폐기 증적 기록
   9.5 아카이빙·복원 운영
   9.6 AI 학습/분석과의 연계 (보존 데이터의 재사용 제공)
   9.7 비용·미사용 데이터 정기 점검 운영
   9.8 권한·보안 (폐기 권한·아카이브 접근 통제)
   9.9 역할별 기능 (오너/법무/운영 화면)
   9.10 솔루션 운영 (정책 엔진 유지·버전 관리)
10. AI-ready 데이터 체계 내 연계 범위
   10.1 과제 범위 (시간축 보존·폐기만)
   10.2 C-3 Lineage (C-3=변환·흐름 이력 추적 / F-2=데이터 실체의 보존·폐기 시간 통제)

   10.4 A-1 카탈로그 (자산 소재 등록) · C-2 품질 (사용 가능 판정)
11. KPI 및 성과 관리 (보존 정책 준수율↑·미사용/장기 보관 데이터 비중↓·저장 비용 절감액↑·법적 보존 누락 건수↓·AI 재사용된 과정 데이터 건수↑)
12. 고도화 Roadmap (수기 보존 정책·수동 폐기 → 데이터 가치·민감도 자동 분류·보존 기간 추천(AI보조) → 가치 기반 자동 보존·폐기 최적화(자율))
```

## F-3 데이터 디지털화
> 파일명: `F-3 데이터 디지털화.md`
> 정의: 종이·수기·사진·암묵지 등 아날로그 자산을 AI가 활용 가능한 디지털 자산으로 전환하고, 처음부터 디지털로 생성되도록(Digital-first) 업무를 바꾸는 체계.
> 핵심 계열사 예시: 두산에너빌리티 (현장 수기 점검표·작업일지·암묵지)
> ★ 핵심 원칙: 일회성 전환에 그치지 않고 데이터가 처음부터 디지털로 생성되도록(Digital-first) 업무 양식을 바꾼다. 기존 디지털 문서 전처리는 B-1.

```
1. 개요
   1.1 데이터 디지털화의 정의
   1.2 목적 (AI 활용에서 누락된 아날로그·암묵지 자산을 디지털 자산으로)
   1.3 적용 범위 (IN: 아날로그→디지털 전환·Excel/개인PC 산재 데이터 승급·Digital-first / OUT: 이미 디지털 문서의 전처리=B-1·전환 데이터의 업무 품질 판정=C-2)
   1.4 주요 대상 조직 (현장 운영 부서·데이터 담당·IT·현업 SME(암묵지 보유자))
   1.5 AI-ready 체계 내 역할 (① 원천 확보 — AI가 쓸 데이터를 새로 만들어 공급)
2. 필요성 및 기대 효과
   2.1 AI 과제 수행 시 주요 Pain Point (핵심 판단 근거가 종이·수기·작업자 머릿속에 있어 학습 대상에서 누락, 개인 Excel/PC 산재 데이터 비공식·휘발성·소실 위험)
   2.2 구축의 필요성 (제조 현장 노하우의 상당 부분이 비디지털, AI는 디지털 자산만 학습, 숙련자 퇴직 시 암묵지 소실)
   2.3 기대 효과 (누락 자산 확보·암묵지 보존·산재 데이터 공식화·신규 데이터의 처음부터 디지털화)
   2.4 주요 기능 (① Inventory · ② 우선순위 · ③ 전환 방식 · ④ Digital-first 전환)
3. 데이터 디지털화 구성 체계
   3.1 아날로그 자산 Inventory (종이 문서·수기 검사표·출력물·현장 메모·사진·스캔본·암묵지)
   3.2 디지털화 우선순위 기준 (업무 중요도·AI 활용 가능성·소실 위험·반복 활용성·현장 의존도)
   3.3 디지털화 방식별 적용 기준 (스캔·OCR/ICR·음성-텍스트·모바일/태블릿 입력·사진 메타정보·현장 인터뷰 구조화)
   3.4 Excel·개인 PC 산재 데이터 승급·지속 수집 기준 (비공식 데이터의 공식 자산화)
   3.5 Digital-first 업무 전환 기준 (표준 코드·필수 입력값·드롭다운·자동 시간 기록) (★ Digital-first 전환, 기존 디지털 전처리는 B-1)
4. 추진 역할 및 책임
   4.1 주요 담당자 정의 (디지털화 추진 담당·현장 부서장·SME(암묵지)·전환 작업/검수자)
   4.2 역할별 책임 구분 (RACI: 대상 선정·전환 실행·검수·Digital-first 양식 승인)
   4.3 계열사 상황에 따른 담당 조직 조정 기준 (현장 인력 활용 전환 vs 외주 스캔/OCR)
5. 아날로그 자산 현황 조사 및 디지털화 대상 선정
   5.1 디지털화 대상 기준 정의 — 모든 아날로그 자산을 전환하지 않는 이유
   5.2 누락 아날로그 자산 조사 (종이·수기·사진·암묵지 전수 식별)
   5.3 자산 유형별 전환 난이도·소실 위험 평가
   5.4 디지털화 우선순위 결정 (중요도·AI 활용성·소실 위험·반복성)
   5.5 Excel·개인 PC 산재 데이터 식별·승급 대상 선정
   5.6 암묵지(작업자 판단 기준) 추출 대상 선정 (인터뷰 구조화 대상)
   5.7 전환 방식 매핑 (대상 특성 ↔ 스캔/OCR/ICR/모바일/인터뷰)
   5.8 최종 디지털화 대상 및 Wave 우선순위 (★ 소실 위험 높고 AI 활용 가치 큰 자산부터)
6. 데이터 디지털화 솔루션 선정 검토
   6.1 솔루션 도입 목적·적용 범위
   6.2 솔루션 유형 (스캐너·OCR/ICR 엔진·모바일/태블릿 입력 앱·음성 STT·사진 캡처/메타태깅)
   6.3 후보군 (인쇄체 OCR vs 수기 ICR·현장 모바일 폼·STT 도구)
   6.4 비교 기준 (수기·현장 환경 인식률·한글/도면/표 인식·오프라인 입력·검수 워크플로)
   6.5 검토 항목 (인식 정확도·필수항목 강제·표준 코드 연동)
   6.6 평가·선정
   6.7 PoC (실제 현장 수기 검사표·작업일지 샘플 인식률 검증)
   6.8 세부영역 제안 (수기=ICR·인쇄물=OCR·신규 입력=모바일 폼·암묵지=인터뷰 구조화)
7. 계열사 적용 예시: 두산에너빌리티 현장 디지털화 시나리오
   7.1 환경 (현장의 수기 점검표·작업일지·설비 사진·숙련자 노하우)
   7.2 목표
   7.3 아날로그 자산 Inventory
   7.4 우선순위 선정
   7.5 전환 방식 매핑 (ICR/OCR/모바일/인터뷰)
   7.6 산재 Excel 승급
   7.7 검증 기준
   7.8 전환 파이프라인 아키텍처
   7.9 Digital-first 양식 전환
   7.10 운영 (상시 수집)
   7.11 기대 효과 (암묵지 보존·현장 데이터 확보)
8. 데이터 디지털화 구축
   8.1 대상 아날로그 자산 취합·검토
   8.2 전환 항목·표준 코드 정합성 정의
   8.3 To-Be 전환 파이프라인 설계 (스캔→OCR/ICR→구조화→적재)
   8.4 솔루션 반영 (OCR/ICR·모바일 폼·STT 설정)
   8.5 Legacy 산재 데이터(개인 Excel/PC) 수집·승급 연동
   8.6 암묵지 인터뷰 구조화·템플릿화
   8.7 수동 검수·교정 절차 구성
   8.8 Digital-first 입력 양식 개발 (필수값·드롭다운·자동 기록)
   8.9 솔루션 설정 (인식 임계·검수 큐)
   8.10 디지털화 검증 (원본 대비 검수·샘플링·필수항목 누락·OCR 인식률) ※ 업무 품질 판정은 C-2
   8.11 초기 전환·적재·검증
9. 데이터 디지털화 운영
   9.1 신규 디지털화 요청 접수 (현장 추가 자산·신규 양식)
   9.2 검토·승인 (대상 적합성·방식 선택)
   9.3 전환 실행·검수 반영
   9.4 디지털 자산 생성·등록 (카탈로그 A-1 연계)
   9.5 전환 품질 검증 운영 (인식률·누락 모니터링)
   9.6 산재 데이터 지속 수집 운영 (Excel 승급 상시화)
   9.7 Digital-first 정착 운영 (종이 양식 단계적 폐지·신규 데이터 디지털 생성)
   9.8 권한·보안 (현장 입력 권한·민감 사진/도면 취급)
   9.9 역할별 기능 (현장 입력/검수자/관리자 화면)
   9.10 솔루션 운영 (OCR/ICR 모델·모바일 폼 유지·버전 관리)
10. AI-ready 데이터 체계 내 연계 범위
   10.1 과제 범위 (오프라인·일회성/대량 전환 + 신규 생성 방식 전환만)
   10.2 B-1 전처리 (B-1=이미 디지털 문서 구조화 / F-3=아날로그→디지털 전환·산재 승급·Digital-first) ★
   10.3 C-2 품질 (전환된 데이터의 업무 품질 판정)
   10.4 A-1 카탈로그 (전환 자산 등록) · A-2 메타데이터 (전환 자산 속성)
11. KPI 및 성과 관리 (디지털화율(대상 대비)↑·전환 정확도/인식률↑·산재 데이터 공식 승급 건수↑·Digital-first 양식 적용률↑·암묵지 구조화 건수↑)
12. 고도화 Roadmap (일회성 수동 전환 → 자동 OCR/ICR·암묵지 추출 보조·산재 데이터 자동 탐지(AI보조) → 상시 수집·Digital-first 정착으로 아날로그 유입 원천 차단(자율))
```

## F-4 AI 데이터 보안
> 파일명: `F-4 AI 데이터 보안.md`
> 정의: AI 학습·추론·RAG·Agent 실행 과정에서 개인정보·민감정보를 탐지·마스킹·가명화해 안전하게 활용하는 비식별 체계. 데이터를 "쓸 수 있게 변환"하는 보안.
> 핵심 계열사 예시: 두산밥캣 (글로벌 사업의 개인정보·고객/딜러 정보 비식별)
> ★ 핵심 원칙: 접근을 막는 것이 아니라 민감정보를 탐지해 비식별 변환해 안전하게 쓰게 한다. 접근 차단/권한은 C-2, 원본 대체 합성은 E-2. "목적이 무엇이냐"가 처리 수준을 결정한다.

```
1. 개요
   1.1 AI 데이터 보안의 정의
   1.2 목적 (민감정보를 보호하면서도 AI 활용 가치(Utility)는 유지)
   1.3 적용 범위 (IN: 탐지·마스킹·가명화·비식별 / OUT: 접근 차단·권한 통제=C-2·원본 대체 합성데이터=E-2)
   1.4 주요 대상 조직 (정보보호/Privacy·데이터 거버넌스·계열사 보안 담당·AI 서비스 운영자·법무)
   1.5 AI-ready 체계 내 역할 (가공·활용 단계를 가로지르며 데이터에 마스킹을 입히는 횡단 보안 레이어)
2. 필요성 및 기대 효과
   2.1 AI 과제 수행 시 주요 Pain Point (학습·RAG·로그에 개인정보/고객사명/계약/설계정보 유입, 외부 모델 전송 시 유출 위험, 속성 결합 재식별, 과도한 마스킹 시 AI 성능 붕괴)
   2.2 구축의 필요성 (AI는 데이터를 대량·교차 활용하므로 일반 보안과 다른 위험, 글로벌 규제(GDPR 등) 대응)
   2.3 기대 효과 (규제 준수·유출 위험 차단·외부 모델 안전 활용·유용성 보존)
   2.4 주요 기능 (① 분류 · ② 탐지 · ③ 목적별 마스킹 · ④ 등급/로그 통제)
3. AI 데이터 보안 구성 체계
   3.1 민감정보 분류 체계 (개인정보·임직원·고객사명·연락처·계약/가격·기술기밀·설계정보·영업비밀)
   3.2 민감정보 탐지 기준 (정규식·사전·NER·LLM 문맥 탐지 조합 / 정형·비정형·로그·Prompt·출력)
   3.3 목적별 마스킹 정책 (학습/RAG/평가/로그/외부모델별 삭제·부분 마스킹·가명화·익명화·토큰화·범주화)
   3.4 Utility-Preservation 기준 (식별자 제거 + 업무 판단 정보(제품군·공정·결함) 보존의 균형)
   3.5 AI 학습 가능 등급 (원본 가능·가명화 후 가능·익명화 후 가능·학습 금지 / 외부 모델 별도 제한) (★ 차단은 C-2, 합성은 E-2)
4. 추진 역할 및 책임
   4.1 주요 담당자 정의 (Privacy/정보보호 담당·데이터 오너·AI 서비스 운영자·법무/DPO)
   4.2 역할별 책임 구분 (RACI: 분류 기준 수립·탐지 룰·마스킹 정책 승인·재식별 위험 판정)
   4.3 계열사 상황에 따른 담당 조직 조정 기준 (글로벌 계열사=현지 규제 / 지주 공통 분류 체계 + 계열사 적용)
5. 민감정보 현황 조사 및 보안 처리 대상 선정
   5.1 보안 처리 대상 기준 정의 — 모든 데이터를 같은 수준으로 비식별하지 않는 이유
   5.2 민감정보 포함 데이터·경로 식별 (정형 DB·문서·로그·Prompt 입력·AI 출력)
   5.3 민감정보 유형별 분류·태깅
   5.4 활용 목적별 처리 수준 매핑 (학습/RAG/평가/로그/외부모델)
   5.5 Utility-Preservation 우선순위 (보존할 업무 정보 vs 제거할 식별자)
   5.6 AI 학습 가능 등급 부여
   5.7 재식별 위험 평가 대상 선정 (속성 결합 고위험 데이터)
   5.8 최종 보안 처리 대상 및 우선순위 (★ "목적이 무엇이냐"가 처리 수준 결정)
6. AI 데이터 보안 솔루션 선정 검토
   6.1 솔루션 도입 목적·적용 범위
   6.2 솔루션 유형 (민감정보 탐지(DLP/스캐너)·마스킹/가명화 엔진·토큰화·NER/LLM 기반 문맥 탐지·재식별 위험 분석)
   6.3 후보군 (정형 DB 마스킹·비정형/문서 비식별·LLM 입출력 가드)
   6.4 비교 기준 (정형+비정형+로그 커버리지·다국어/글로벌 PII 인식·유용성 보존·외부 모델 연계 가드)
   6.5 검토 항목 (탐지 정밀도/재현율·가명화 일관성(같은 값→같은 가명)·가역/비가역)
   6.6 평가·선정
   6.7 PoC (실제 클레임·계약 문서·AI 로그 비식별 검증)
   6.8 세부영역 제안 (정형=마스킹·문서/로그=NER+LLM·외부모델=입출력 가드)
7. 계열사 적용 예시: 두산밥캣 글로벌 AI 데이터 비식별 시나리오
   7.1 환경 (북미/유럽 딜러·고객 개인정보, 글로벌 규제)
   7.2 목표
   7.3 민감정보 분류
   7.4 탐지 대상·경로 식별
   7.5 목적별 마스킹 정책
   7.6 Utility-Preservation 설계
   7.7 학습 가능 등급
   7.8 비식별 파이프라인 아키텍처
   7.9 AI Log/Prompt 마스킹
   7.10 재식별 위험 점검
   7.11 기대 효과 (규제 준수·유출 위험 차단)
8. AI 데이터 보안 구축
   8.1 대상 데이터·민감정보 현황 취합·검토
   8.2 분류 체계·탐지 룰 정합성 정의 (정규식·사전·NER·LLM)
   8.3 To-Be 비식별 아키텍처 설계 (탐지→처리 수준 결정→마스킹/가명화)
   8.4 솔루션에 분류·마스킹 정책 반영
   8.5 Legacy 데이터/문서 일괄 비식별 연동
   8.6 정형·비정형·로그별 탐지 파이프라인 개발
   8.7 목적별 마스킹 처리 로직 개발 (가명화 일관성 유지)
   8.8 AI Log/Prompt/출력 마스킹 가드 개발 (저장 전 자동 마스킹)
   8.9 솔루션 설정 (탐지 임계·등급별 처리 규칙·외부 모델 제한)
   8.10 테스트 (탐지율·오탐·유용성·재식별 검증)

   9.1 보안 처리 변경 요청 접수 (신규 데이터 유형·새 규제·탐지 룰 변경)

   9.6 외부 모델 사용 시 비식별 게이트 운영
```
