# CLAUDE.md — AI-Ready Data 매뉴얼 프로젝트

이 레포는 `최종 주제.md`에 정의된 **AI-Ready Data 20개 주제(A-1~F-4)** 각각에 대해
장표/매뉴얼용 가이드 `.md`를 작성하는 프로젝트다. 작업은 "가이드 작성 → 고객 확인 →
피드백 반영"을 주제별로 반복한다.

## 폴더 구조

```
CLAUDE.md                  ← 이 파일(루트 고정). 프로젝트 전역 규칙의 진입점
공통 규칙/                  ← Global Rule(전역). 모든 주제에 공통으로 적용
  ├ 01 가이드 작성 표준.md
  ├ 02 다이어그램 표준.md
  ├ 03 주제 가이드 템플릿.md
  └ 최종 주제.md            ← 20개 주제 정의·Key Question(합격 기준)
가이드 작성/                ← 실제 가이드 작업 공간. 주제별 폴더로 분리
  └ A-1 데이터 카탈로그/
      ├ A-1 데이터 카탈로그.md   ← 가이드 본문(피드백 반영하며 버전 갱신)
      └ _작업 메모.md            ← 이 주제 전용 규칙·피드백 메모(주제 로컬)
전체 목차/                  ← 주제별 목차를 잡아가는 공간(기존 매뉴얼 기반)
  ├ 00 전체 목차 (20개 주제).md
  └ 목차 검증/              ← 상세목차·갭분석 등 목차 작업 이력(아카이브)
기존 매뉴얼 작성본/          ← Kearney 기존 PPTX + 목차 추출본(입력 자료)
파트너님 예시/              ← 파트너 작성 예시(A-1예시 등, 형식 레퍼런스)
```

- **전역 규칙(Global Rule)** = `CLAUDE.md` + `공통 규칙/`. 모든 주제에 적용. 여기만 고치면 전 주제에 반영.
- **주제 로컬 규칙** = 각 `가이드 작성/<주제>/_작업 메모.md`. 그 주제의 피드백·결정·주의점만. 가이드를 작성/수정하기 전 **해당 폴더의 `_작업 메모.md`를 먼저 확인**한다.
- 새 주제를 시작하면 `가이드 작성/<주제코드> <주제명>/` 폴더를 만들고 그 안에 본문 `.md`와 `_작업 메모.md`를 둔다.

> 위 표준 파일을 읽지 않고 가이드를 쓰지 않는다.
> **작성 원칙**(아래 절대 원칙·01 표준의 0장·2~6장)은 주제와 무관하게 그대로 따른다 — 원칙과 실제 가이드가 어긋나면 원칙을 따른다.
> 다만 **섹션 목차 구성(12개 섹션)은 "권장 기본 골격"일 뿐, 주제에 따라 섹션을 더하고·빼고·순서를 바꿀 수 있다.** 주제에 더 필요한 목차가 있으면 추가하고, 그 주제에 의미 없는 섹션은 생략한다. 합격 기준은 "12개를 다 채웠는가"가 아니라 **"Key Question에 전부 답했고 원칙을 지켰는가"**다 (상세는 01 표준 §1).

## 절대 원칙 (요약 — 상세는 01번 표준)

- **목적은 "이 주제가 무엇이고(What) 어떻게 구성하는가(How)"를 가르치는 것.** 리서치 덤프가 아니다.
- **"AI 구축"이 아니라 "AI를 위한 데이터 준비".** 모델/RAG/Agent 만드는 법이 아니라 데이터를 준비·정비하는 법.
- **이 주제가 주연.** 인접 주제(카탈로그·품질·리니지 등)는 짧은 포인터로. 어느 주제에나 들어갈 일반론(보일러플레이트)은 잘라낸다.
- **`최종 주제.md`의 Key Question에는 전부 답해야 한다 (합격 기준).** 단, **"KQ1·KQ3" 같은 내부 코드를 제목·본문에 노출하지 않는다** — 독자는 그 코드가 뭔지 모른다. 대신 ① 문서 앞에 **"이 가이드가 답하는 핵심 질문" 박스**(질문 평문·한 줄 답·다루는 위치)를 두고, ② 답하는 절에는 **"❓ 이 절이 답하는 질문: …" 안내**를 단다 (상세는 아래 목차 구조 표준).
- **섹션 목차는 주제에 맞춘다.** 12개 섹션은 권장 기본 골격이다 — 주제에 더 필요한 목차는 더하고, 의미 없는 섹션은 빼거나 합친다. "N개 섹션 채우기"가 목적이 아니다. 단, 원칙·이해장치·Key Question 커버리지는 어느 구성에서나 유지한다.
- **정본 모델 하나로 일관되게.** 같은 개념을 두 번 설명하지 않고, 용어·개수가 문서 끝까지 일치.
- **형식은 내용을 따른다.** "N가지" 숫자 채우기 금지.
- **이해 장치 의무:** 섹션/슬라이드마다 한 줄 요약 + 어려운 용어 즉시 풀이 + 제조 현업(두산) 예시.
- **언어:** 한국어. 기술용어·솔루션명·표준명은 `한글(English)` 병기 (예: 데이터 계보(Data Lineage)).
- **다이어그램:** 02번 표준의 파랑6/초록3 컬러 스키마·테마 헤더를 그대로 쓴다. **그림은 칸 채우기가 아니다** — 구조·분기·루프·관계·시간축·좌표처럼 그림이라야 빨리 이해되는 곳에만 만들고, 단순 나열·본문 재진술은 표·불릿으로 둔다(판단 테스트: "표/한 문장으로 바꿔도 정보 손실이 없으면 그림을 쓰지 않는다" — 02 표준 §4-1).
- **링크:** ① 문서 상단에 **목차(앵커 링크)**를 둔다. ② 인접 주제 언급은 해당 **가이드 파일로 링크**한다(예: `[A-2 메타데이터](가이드%20작성/A-2%20메타데이터/A-2%20메타데이터.md)`). ③ **솔루션·도구·표준을 소개할 때는 반드시 출처 URL을 링크**로 단다(본문 인라인 + 하단 `## 참고자료(References)` 절). 상세는 01번 표준 §4.
- **솔루션 2층 구조(필수):** 솔루션은 한 제품이 여러 주제에 걸치므로 2층으로 관리한다. **1층** = 각 주제 가이드 '솔루션' 섹션(그 주제 관점 **기능 비교**), **2층** = [`전체 목차/01 Tech Stack 비교 (솔루션×주제).md`](전체%20목차/01%20Tech%20Stack%20비교%20(솔루션×주제).md)(솔루션 **자체를 묶어 평가·선정** — 플랫폼×주제 매트릭스). 가이드 솔루션 섹션은 정본으로 링크하되, **정본 Part A 반영(reflect-back)은 매번 자동이 아니라 — 사용자가 가이드를 최종 승인하며 "정본에 반영" 을 명시 요청할 때만** 한다(별도 커밋). 상세는 01번 표준 §4-5.

## 목차 구조 표준 (고객 피드백 반영 — 가이드/목차 작성 시 반드시 따른다)

목차는 아래 **논리 흐름**을 기본으로 한다. **12개 섹션을 다 채우지 않으며**(A-1 12-섹션은 참고용 풀일 뿐), 주제에 맞게 빼거나 합친다. **섹션 수는 주제 무게에 따라 다르게 한다(고정 금지) — 가벼운 주제는 6~7개로 합치고, 무거운 주제(예: 카탈로그·품질·보안·Physical)는 솔루션·구축·운영을 나눠 9~11개로 깊게.** H3 소목차 개수도 섹션마다 다르게(2~6개) — 다룰 게 많으면 많이, 적으면 적게.

**기본 순서 (왜 → 무엇 → 언제 → 예시 → 어떻게 → 관계 → 지표):**
1. **개요** — 이게 뭐고 체계 내 어디 (한 줄 설명 포함)
2. **왜 필요한가 (Why)** — 현업 Pain Point·기대효과
3. **무엇을 갖추나 (What)** — 구성 요소
4. **언제/어디에 하나 (When)** — 대상 선정·적용 판단
5. **예시 시나리오** — "적용하면 이렇게 좋아진다(효과) + 흐름 미리보기"를 계열사 맥락으로 (★ **How 바로 앞에** 둔다)
6. **어떻게 준비·운영하나 (How)** — 방식·도구·**구축 절차**·운영·역할
7. **다른 주제와의 관계** (경계)
8. **성과 지표·고도화**

세부 규칙:
- **순서**: A-1 12-섹션 순서를 그대로 베끼지 말고 위 논리 흐름으로 배열한다.
- **When 처리**: 온톨로지·합성데이터·평가데이터처럼 **골라서 하는 주제**는 "언제 해야 하나(적용 판단)"를 명시한다(전 데이터에 다 하는 게 아님). 데이터 카탈로그·메타데이터처럼 **기본적으로 다 필요한 주제**는 "어디부터(정비/등록 우선순위)"로 쓴다.
- **예시 시나리오(5번)**: How 들어가기 전에 효과+흐름을 먼저 그려준다(예: "라벨을 달면 검사 시간↓·기준 일관↑ → 분류 체계 정하고 → AI 1차 라벨 → 사람 검수"). 이것이 worked example 역할을 하므로 **맨 끝에 별도 계열사 예시를 또 두지 않는다**(중복 방지). **구축 절차는 예시와 별개로 How(6번)에** 둔다.
- **세부 H3 목차는 반드시 둔다**(없애지 말 것): 각 섹션 아래 번호 매긴 소목차(3.1·3.2…)는 있어야 한다. 단 **H3 내용은 현업이 바로 이해할 쉽고 얕은 수준** — 어려운 전문 내용·약어(RDF/OWL·ABAC·ICR·MVG 등)는 빼거나 한 줄로 풀이한다.
- **★ Key Question 가시화 (제목에 내부 코드 금지)**: KQ는 합격 기준이지 독자용 표기가 아니다. 따라서:
  - ① **제목·본문에 `(KQ3)`·`— KQ1` 같은 코드를 쓰지 않는다.** 제목은 내용으로만 짓는다.
  - ② 문서 앞(목차 아래)에 **"이 가이드가 답하는 N가지 핵심 질문" 박스**를 둔다 — `질문(평문) | 한 줄 답 | 다루는 곳(링크)` 표로. 독자가 "무슨 질문에 어떻게 답하는지"를 한눈에 본다.
  - ③ 그 질문에 답하는 절 첫머리에 **`> ❓ 핵심 질문 N — "…질문 평문…"에 답하는 절` 안내 박스**를 단다(여러 절에 걸치면 이어지는 절을 링크).
  - ④ **링크는 안정적 앵커로**: 제목에 `①②③`·`—`(em dash)·괄호가 있으면 자동 생성 앵커가 깨질 수 있으므로, 해당 절 바로 위에 `<a id="kq1"></a>`·`<a id="sec63"></a>` 같은 **명시적 HTML id**를 심고 그 id로 링크한다.
  - (작성용 KQ→섹션 매핑 주석은 최종본에서 위 ② 박스로 대체한다. 적용 예: `가이드 작성/B-2 데이터 해설·주석/B-2 데이터 해설·주석.md` 상단.)
- **"다른 주제와의 관계"는 항상 독립 섹션**: 어느 주제든 이 섹션을 **끝에서 두 번째 독립 섹션**으로 두고(경계만), 마지막은 **"성과 지표·(로드맵·)고도화"**로 둔다. 지표·고도화와 한 섹션으로 합치지 않는다(20개 위치·형태 일관 — 독자가 항상 같은 자리에서 찾게).
- **현업 눈높이·분량 최소**: 제조 현업이 읽고 바로 이해할 표현. 섹션당 핵심 1~3개. "많이 적기"가 아니라 "필요한 것만".
- **4축(Why·What·When·How)**은 어떤 구성에서도 빠지지 않게 유지한다.
- **★ 현업 실행 키트(따라 만들 수 있게)**: 각 주제는 "현업이 실제 만들어 내는 산출물을 어떻게 채우는가"를 보여주는 5개 장치를 기존 섹션에 끼운다 — ㉠ **항목 사전**(항목·쉬운 의미·예시값·**필수/선택·작성 주체** 열, §3 What), ㉡ **Before→After 작성 규칙**(나쁜 예→좋은 예, §6 How), ㉢ **표준값 목록**(고르는 허용값, 해당 시), ㉣ **빈 템플릿+1건 완성 예시**(§5 또는 별첨), ㉤ **플랫폼·도구 구체 매핑**("어디서 하나"). 본문엔 대표 항목만, 전체 사전은 별첨/외부 엑셀로. 상세는 [`01 가이드 작성 표준.md`](공통%20규칙/01%20가이드%20작성%20표준.md) §2-A.
- **★ 데이터 준비 관점 고정**: 모든 주제·모든 섹션은 "AI·에이전트·보안 시스템을 만드는 법"이 아니라 "그 AI가 쓸 **데이터를 준비·정비하는 법**"으로 쓴다(절대 원칙과 동일, `/ai-ready-data-guide-en`). 솔루션/구축/운영 섹션도 AI 서비스 구축이 아니라 데이터 준비 도구·절차다. 특히 **D-2(에이전트가 읽을 Tool 명세 데이터), D-3(재사용할 Prompt 지식 자산), F-4(데이터 비식별 준비)**가 "에이전트 구축/보안 구축"으로 흐르지 않게 주의.

## 파일·작업 규칙

- **가이드 위치·파일명:** 주제마다 `가이드 작성/<주제코드> <주제명>/` 폴더를 만들고 그 안에 본문 `<주제코드> <주제명>.md`를 둔다 (예: `가이드 작성/A-1 데이터 카탈로그/A-1 데이터 카탈로그.md`). 같은 폴더의 `_작업 메모.md`에 그 주제의 피드백·결정 이력을 누적한다.
- **목차 작업:** 주제별 목차는 `전체 목차/`에서 잡는다(기존 매뉴얼 작성본·파트너님 예시를 입력으로). 전체 목차를 사용자와 피드백하며 갱신.
- **커밋:** 주제 단위로 의미 있게 커밋. 메시지는 무엇을/왜. 고객 피드백 반영은 별도 커밋으로 남겨 이력을 추적.
- **새 파일을 만들면 요청을 기다리지 말고 바로 커밋·푸시한다**(원격까지 반영). 기존 파일 수정도 의미 단위로 마무리되면 같이 올린다. 단, 세션 시작 시 이미 있던 무관한 변경은 함께 올리지 않는다.
- **피드백:** 각 가이드 하단 `## 변경 이력 / 피드백 반영` 표에 고객 피드백·반영 내용을 누적 기록 (상세는 01번 표준).
- 로컬 전용 설정(`.claude/settings.local.json`), `.DS_Store`는 커밋하지 않는다(.gitignore 처리됨).
